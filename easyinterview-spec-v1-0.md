# 修订说明

\begin{bluebox}
\textbf{为什么要重写这份 spec}\\
原版 v0.9 的愿景完整度很高，但首发范围过大：同时覆盖用户深画像、全球搜岗、公司情报、简历工坊、模拟面试、成长系统、语音视频分析与未来 Team 版。新版的核心任务不是否定这些方向，而是把它们重新排序，让产品先在一个真实、强需求、高复购的场景中站稳。
\end{bluebox}

## 本版与原版的关键差异

| 维度 | 原版 v0.9 | 本版 v1.0 |
|---|---|---|
| 核心切口 | 全流程 AI 求职陪跑 | 以具体 JD 为中心的面试训练系统 |
| 首发入口 | 深画像 -> 岗位匹配 -> 全链路 | JD 导入/面后复盘直达核心价值 |
| MVP 范围 | 覆盖较广 | 收敛为 5 个核心能力 |
| 评分方式 | 倾向精细打分与通过率判断 | 证据优先，评分配置信心区间，弱化伪精确 |
| 视频分析 | 较早引入 | 后置、可选、仅做训练反馈 |
| 岗位数据策略 | 尽早做多平台聚合 | 先做用户导入 + 合规数据源的轻量情报 |
| 商业定位 | 全能求职工具 | 先赢在“这场面试准备得更好” |

## 修订后的三条主线

1. **先把“目标岗位训练”做透。**  
   首发必须让用户在最短路径内完成一次高质量练习：导入 JD、定制问题、拿到可执行反馈、知道下一轮该怎么补。

2. **所有反馈都要可追溯、可复练。**  
   报告不能只给一个好看的分数，而要明确：具体哪一题、哪一句、缺了什么、如何改、下次怎么测。

3. **从 Day 1 开始按生产系统而不是 Demo 设计。**  
   包括模型适配层、Prompt/Rubric 版本化、离线评估集、异步任务、数据留存策略、来源可追溯与灰度发布。

# 产品定义与范围

## 一句话定义

**easyinterview 是一款围绕具体目标岗位和真实面试流程设计的 AI 面试训练产品。**  
它帮助用户把“看到一个岗位 -> 开始准备 -> 模拟练习 -> 拿到反馈 -> 修正表达 -> 复练 -> 面后复盘”的过程标准化、重复化和可视化。

## 核心问题

当前主流求职工具在以下几处同时失效：

- 用户往往知道“我要面什么岗位”，但不知道“这份 JD 真正在考什么”。
- 面试练习常停留在泛泛而谈，缺少对具体岗位、公司语境、轮次风格的适配。
- 传统复盘只告诉你“答得一般”，却不告诉你“为什么一般、怎么改、改完怎么验证”。
- 求职者在重要面试前的时间窗口很短，无法承受冗长 onboarding 和高学习成本。

## 目标用户

### 主目标用户

1. **高意图求职者**  
   已经拿到或即将投递某个具体岗位，希望在 24 到 72 小时内完成高质量准备。

2. **面后复盘用户**  
   刚完成一轮真实面试，希望快速复盘问题、补写更好的答案，并准备下一轮。

3. **跨语言 / 跨国家求职者**  
   对面试语言、文化表达、岗位期望不够熟悉，需要获得结构化训练与本地化表达建议。

### 次目标用户

- 正处于换工作周期、需要逐步建立面试能力的中长期用户。
- 求职训练营、学校职业中心、求职教练等 B2B2C 组织型用户。

## 用户任务（JTBD）

- 当我手里有一个 JD 时，我想快速知道它在考什么，以及我当前最大的短板是什么。
- 当我准备一轮面试时，我想针对具体轮次练，而不是做泛用练习。
- 当我答不好一道题时，我想知道怎么把自己的真实经历改写成更强的回答。
- 当我已经面过一轮时，我想把真实题目沉淀为下一轮的准备清单。
- 当我练了很多次时，我想看到自己是否真的在变好，而不是只积累报告。

## 产品原则

### 1. 极速进入价值
首屏优先服务“今天就有用”的需求。用户不必先完成深画像，才有资格开始练习。

### 2. JD 与轮次优先于大而全
任何练习、反馈、简历建议，都应优先绑定目标岗位、面试角色和当前轮次。

### 3. 证据优先于伪精确
报告优先呈现证据、观察和建议，再呈现评分。所有推断性判断都应带上依据与置信度。

### 4. 复练优先于一次性报告
产品核心不是“生成一份漂亮报告”，而是“帮助用户在下一次更好地回答同类问题”。

### 5. 隐私默认保守
音视频、情绪、行为信号属于高敏数据；所有采集都必须可解释、可关闭、可删除。

### 6. 工程与评估前置
模型质量、延迟、成本、可观测性、离线评估和回归检测，都属于功能的一部分。

## 明确的非目标

\begin{greenbox}
\textbf{本产品不做的事}\\
\begin{itemize}
\item 不做真实面试中的隐形实时辅助或作弊型能力。
\item 不在 MVP 阶段成为完整的招聘网站替代品。
\item 不把视频/音频分析结果用于企业端的候选人淘汰决策。
\item 不在没有校准和证据的前提下，对用户输出过强的录用概率承诺。
\end{itemize}
\end{greenbox}

# 市场定位与商业假设

## 赛道切口

本产品不从“求职一切都管”切入，而从一个更锋利的子问题切入：

**“我有一个具体岗位，我需要把这场面试准备得更好。”**

这意味着产品要先打透三类高频场景：

- **面试前 48 小时冲刺**
- **真实面试后的立即复盘**
- **同一岗位的多轮持续训练**

## 差异化定位

### 与泛用 AI 聊天工具的差异
不是“陪你聊面试”，而是“围绕目标岗位组织训练流程、记录错误、推动下一次更好”。

### 与泛题库产品的差异
不是“给你更多题”，而是“给你更贴合这个岗位、这个轮次、这个短板的题”。

### 与求职平台的差异
不是“帮你看更多岗位”，而是“帮你把已经看中的岗位赢下来”。

## 商业模式假设

| 层级 | 目标用户 | 核心权益 |
|---|---|---|
| 免费版 | 新用户与低频用户 | 有限次数的 JD 解析、快速练习与基础报告 |
| Pro | 高频求职用户 | 无限练习、错题本、复练追踪、简历定制、真实面试复盘 |
| Sprint Pack | 临近重要面试用户 | 7-14 天高强度训练包、专项计划与高频复练 |
| Team / EDU | 学校、训练营、教练团队 | 组织看板、班级进度、模板与质量管理 |

## 渠道假设

- B2C 早期通过内容和搜索获取：JD 解析、行为面试、反问问题、真实面试复盘。
- 用户增长以复购和复练驱动，而不是靠一次性注册。
- B2B2C 在产品稳定后再推进，不与 MVP 抢资源。

# 完整产品结构

## 模块地图

easyinterview 的完整产品仍保留原版的五大能力，但重新定义职责与优先级：

1. **M1 · Progressive Profile（渐进式画像）**  
   不是深画像前置，而是随简历、JD、练习、复盘逐步长成的候选人记忆层。

2. **M2 · Target Job Workspace（目标岗位工作台）**  
   管理 JD、岗位要点、公司轻情报、轮次假设、简历定制与练习入口。

3. **M3 · Mock Interview Orchestrator（模拟面试编排器）**  
   负责定制题目、轮次角色、追问逻辑、练习模式与会话状态。

4. **M4 · Evidence-based Review（证据化复盘）**  
   负责逐题观察、报告、错题本、复练机制、真实面试复盘。

5. **M5 · Growth System（成长系统）**  
   负责跨时间、跨岗位、跨轮次的能力趋势、准备度和行动计划。

## 完整用户闭环

1. 导入一个目标岗位  
2. 系统抽取岗位重点与面试假设  
3. 生成一轮练习或多轮计划  
4. 产出带证据的报告与错题  
5. 用户复练、改写材料、继续推进  
6. 真实面试后回填真实问题与结果  
7. 系统更新画像、错题本与下一轮建议

# 路线图：从 MVP 到完整产品

## 分阶段总览

| 阶段 | 目标 | 核心结果 | 典型时长 |
|---|---|---|---|
| P0 · MVP | 验证核心闭环 | 用户能针对一个 JD 完成练习、反馈、复练 | 8-10 周 |
| P1 · Beta | 做深单人价值 | 加入真实面试复盘、多轮计划、成长看板、多语言 | 再 8-10 周 |
| P2 · Growth | 扩大适用场景 | 语音模式、跨岗位对比、轻量岗位情报、Offer 支持 | 再 10-12 周 |
| P3 · Full Product | 形成平台化能力 | 视频训练、合规岗位发现、组织版、评估基础设施完备 | 再 12-16 周 |

## 每个阶段的准入标准

### P0 -> P1
- 至少一条核心链路可稳定完成：JD 导入 -> 模拟面试 -> 生成报告 -> 写入错题本 -> 复练。
- 激活用户中，至少 40\% 在 7 天内发生第二次练习。
- 报告延迟、失败率、内容质量达到内部阈值。
- 删除请求、数据导出、日志追踪链路可运行。

### P1 -> P2
- 多轮计划和真实面试复盘被证明确实驱动复练。
- 多语言输出质量达到可上线标准。
- 评分一致性与人工评审的相关性达到内部基线。

### P2 -> P3
- 语音模式稳定，媒体处理成本可控。
- 轻量岗位情报和跨岗位成长模块证明有留存价值。
- 组织版对齐单人版边界，不损害消费者体验与伦理底线。

# P0：MVP 详细规格

\begin{bluebox}
\textbf{MVP 只回答一个问题：}\\
当一个用户带着一份具体 JD 来到产品里，我们能不能在最短时间内让他完成一轮有效练习，并明确知道下一次该怎么练？
\end{bluebox}

## MVP 目标

- 让第一次使用在 10 分钟内产生明确价值。
- 让每次练习都沉淀为可复用的错题与改写建议。
- 验证用户是否愿意为了“更贴合岗位的训练”反复回来。

## MVP 包含的五个能力

### M1-lite：渐进式画像（Lite）

#### 目标
用尽量少的前置成本，获得足够支撑定制练习的上下文。

#### 输入来源
- 用户上传简历或粘贴履历摘要
- 用户补充目标角色、语言、地区、工作年限
- 用户在练习中暴露出的经历、优势和短板
- 用户在真实面试复盘中补充的信息

##### 输出
- 画像摘要
- 可用于练习的核心经历库
- 语言与岗位偏好
- 待补充信息提示

#### MVP 不做
- 冗长的深度职业咨询流程
- 复杂雷达图和大段性格分析
- 过度依赖单次 onboarding 完成所有画像

#### 验收标准
- 用户在 3 到 5 分钟内完成最小画像输入
- 没有简历也能开始练习
- 每次练习结束后，系统能将新的经历要点写回画像

### M2：目标岗位工作台

#### 目标
把一份 JD 变成可练、可复盘、可关联材料的工作台。

#### 用户入口
- 粘贴 JD 文本
- 上传 JD 文件
- 粘贴岗位链接
- 手动新建目标岗位

#### 系统处理
- 解析岗位标题、职级、地点、语言、显性要求
- 提炼岗位关注点、可能的面试主题与轮次假设
- 标注用户当前的直接命中项与明显缺口
- 生成练习入口与简历定制入口

#### 工作台字段
- 基础信息：岗位名、公司名、地区、语言、来源链接
- 要求拆解：必需项、加分项、隐性关注点
- 风险提示：缺少案例、技能不够强、语言短板、经验跨度
- 练习上下文：推荐练习模式、推荐轮次、历史会话、错题关联
- 材料上下文：关联简历版本、求职信版本、反问问题草稿

#### MVP 不做
- 大规模自动搜岗
- 全面的外部公司情报拼图
- 复杂的职位收藏生态

#### 验收标准
- 任意一份 JD 能在 60 秒内完成结构化摘要
- 结构化摘要能被练习引擎直接消费
- 工作台是所有后续动作的统一入口

### M3：模拟面试编排器（Core）

#### 目标
围绕目标岗位组织一次像样的练习，而不是只抛出随机问题。

#### MVP 练习模式
1. **快速热身**：5 到 8 分钟，适合首次试用与临近面试快速启动  
2. **岗位定制核心面**：15 到 25 分钟，围绕 JD 的混合式练习  
3. **单题深钻**：针对一道错题或高频题连续追问  
4. **反问环节专练**：只练“你有什么问题要问我”  
5. **真实面试复现**：由用户输入真实问过的问题，再做补答

#### MVP 面试角色
- 综合面试官（默认）
- HR / Hiring Manager 两种语气模板作为轻度变体

#### MVP 交互模式
- 先做文字模式
- 允许逐题追问
- 支持暂停、继续、结束
- 支持“给我一点提示”开关

#### 关键逻辑
- 每题结束时记录观察，而不是等整轮结束才回忆
- 追问必须与用户刚刚的回答直接相关
- 问题优先覆盖：自我介绍、动机、核心经历、岗位相关难题、反问环节

#### MVP 不做
- 语音和视频
- 超复杂多轮编排器
- 群面和 Assessment Center

#### 验收标准
- 模拟过程无明显跑题或角色漂移
- 至少 70\% 的追问与上一轮回答直接相关
- 用户能清楚感知到“这轮问题就是围绕我的岗位来的”

### M4：证据化复盘与错题本（Core）

#### 目标
给出足够具体、可操作、可复练的反馈。

#### 报告结构
- **总览**：本次练习目的、完成时长、总体准备度档位
- **三类亮点**：内容、表达、岗位契合
- **三类问题**：缺证据、结构弱、答非所问、追问失守等
- **逐题复盘**：每题摘要、做得好的点、缺的点、推荐框架
- **错题写入**：进入错题本并生成复练任务
- **下次建议**：下一轮最值得练的 1 到 3 个主题

#### 错题本最小结构
- 原始问题
- 用户当时的回答摘要
- 失分原因
- 推荐回答框架
- 与用户经历的可映射案例
- 复练状态：未解决 / 改善中 / 已攻克

#### 错题攻克规则
- 对同一能力点的相似问题，连续两次达到“达标”或以上，才标记为已攻克。
- 若两次达标后再次明显退步，重新回到“改善中”。

#### MVP 用户可见评分
\begin{greenbox}
\textbf{MVP 的前台展示不主打精细分数，而主打“准备度档位 + 维度状态”。}\\
推荐展示：\\
- 准备度：未就绪 / 基本可面 / 建议再练 / 较为充分\\
- 维度状态：强项 / 达标 / 待加强\\
- 每条判断附简短证据说明
\end{greenbox}

#### MVP 不做
- 显示极精确的通过率
- 显示“预测击败多少候选人”
- 在没有校准的前提下输出强烈录用结论

#### 验收标准
- 每一条主要反馈都能回溯到具体题目或回答片段
- 用户看完报告后，能立刻知道下一轮该练什么
- 错题本能被后续练习直接调用

### M5-lite：简历定制（Lite）

#### 目标
基于具体岗位，给出现实、直接、低门槛的简历改写建议。

#### MVP 范围
- 给出现有简历与目标岗位的匹配观察
- 推荐补充哪些关键词、哪类成果、哪段经历
- 按 bullet 级别给出改写建议
- 提示哪些内容不必强行加入

#### MVP 不做
- 复杂排版编辑器
- 多模板设计系统
- 深度求职信生成工作台

#### 验收标准
- 用户能为一个目标岗位生成一版“更贴近 JD 的表达建议”
- 改写建议与面试错题能互相联动

## MVP 的用户主流程

### 流程 A：带着 JD 来的用户
1. 导入 JD  
2. 生成岗位工作台  
3. 选择“岗位定制核心面”  
4. 完成一轮 15 到 25 分钟练习  
5. 查看报告与错题本  
6. 直接开始“单题深钻”或“反问环节专练”  
7. 导出简历改写建议

### 流程 B：刚面完一轮的用户
1. 新建目标岗位或进入已有工作台  
2. 选择“真实面试复现”  
3. 输入真实问题与自己当时的回答  
4. 获取更优答案结构与下一轮准备建议  
5. 将高频问题写入错题本  
6. 开始针对性复练

## MVP 明确排除项

| 排除项 | 原因 |
|---|---|
| 全球多平台搜岗 | 数据接入、合规与维护成本过高 |
| 视频情绪识别 | 风险高、解释难、前期价值不稳定 |
| 社区与 Team 版 | 会稀释首发资源 |
| 复杂多轮全流程模拟 | 先验证单轮与轻计划更稳妥 |
| 企业端候选人评估 | 与训练产品定位冲突，且伦理负担高 |

# P1：Beta 规格

P1 的目标不是把产品做“大”，而是把单人价值做“深”。

## 新增能力

### 深化画像：从 Lite 到 Progressive Profile
- 通过历史练习、真实复盘、简历改写自动补全经历库
- 支持“经历卡片”沉淀：每张卡片包含背景、行动、结果、适用题型
- 为常见行为题自动推荐最相关经历

### 多轮计划
- 提供 HR 面、经理面、文化面等预设轮次
- 支持用户按顺序完成，并共享上下文
- 每轮结束后自动把需要补强的点传递给下一轮

### 真实面试复盘（正式版）
- 结构化记录真实问题、面试官反应、自己当时的回答
- 生成感谢信草稿、后续跟进建议
- 对比“模拟表现”与“真实表现”的差距

### 成长看板
- 近 7 / 30 / 90 天练习次数
- 各维度趋势
- 错题攻克率
- 与具体岗位关联的准备度变化

### 多语言支持
- 中文、英文优先
- 支持目标岗位语言与界面语言分离
- 输出本地化表达建议，而不是简单翻译

### 简历工坊增强
- 多版本简历管理
- 基于目标岗位生成求职信草稿
- 将错题中暴露的“无案例维度”反推到简历修改建议

## P1 退出标准

- 用户愿意围绕同一岗位进行 2 次以上复练
- 真实面试复盘功能能产生明显回访
- 成长看板不是“好看但无用”，而能驱动下一步练习
- 多语言输出的可用性通过人工抽检与用户反馈验证

# P2：Growth 规格

P2 开始扩大适用范围，但仍围绕训练闭环。

## 新增能力

### 语音练习模式
- 实时语音转写
- 记录语速、停顿、口头禅等可观察信号
- 面后报告增加“表达层”反馈

### 轻量岗位情报
- 在合规数据源范围内，补充公司官方信息、公开新闻和职位上下文
- 生成更真实的反问建议与面试风格提示
- 强制附上来源与抓取时间

### 跨岗位对比
- 如果用户同时准备多个岗位，显示不同岗位的准备度差异
- 识别“可迁移能力”与“岗位专属短板”
- 帮助用户决定先冲哪一个岗位

### Offer 与谈判支持
- Offer 对比模板
- 谈判话术与准备清单
- 与前期练习历史联动，帮助用户提炼自己的卖点

### 题型扩展
- 压力面试
- 案例面试
- 技术深钻
- Presentation 准备

## P2 退出标准

- 语音练习成本与延迟可控
- 公开情报模块不会过度增加维护成本
- 跨岗位对比被证明确实帮助用户做决策，而不是信息噪音

# P3：完整产品规格

完整产品阶段，easyinterview 从“高强度个人训练器”升级为“结构化求职辅导平台”。

## 能力边界

### 完整能力集合
- 渐进式候选人画像
- 目标岗位工作台
- 合规岗位发现与岗位清单管理
- 多轮、多模式模拟面试
- 证据化报告与错题本
- 简历 / 求职信 / 反问问题材料工坊
- 真实面试复盘、感谢信与跟进策略
- 成长仪表盘与跨岗位分析
- 语音与可选视频训练
- Team / EDU 版组织功能

### 仍然坚持的边界
- 不做真实面试作弊工具
- 不做企业筛选式情绪识别产品
- 不把训练反馈直接转化为用人淘汰依据

## 可选视频训练的约束

视频仅在以下条件下提供：

- 用户明确开启
- 只分析与训练直接相关的可观察信号，例如注视镜头、语速、停顿、口头禅、明显分心
- 不输出“你看起来不够自信所以不适合这家公司”这类不可解释、容易伤害用户的推断
- 所有媒体留存策略、用途与删除能力都清晰可见

## Team / EDU 方向

面向学校、训练营和教练团队时，组织版只提供：

- 学员进度看板
- 模板管理
- 练习完成率与错题类型聚合
- 不直接输出“候选人淘汰建议”

# 关键功能规格（跨阶段）

## 目标岗位工作台

### 核心对象

一个目标岗位工作台是所有动作的主容器。它至少包含：

- 岗位基础信息
- 用户与岗位的匹配观察
- 练习会话列表
- 错题本关联
- 简历与材料关联
- 真实面试记录
- 来源记录与更新时间

### 信息结构

| 分区 | 内容 |
|---|---|
| 概览 | 岗位名、公司、地区、语言、状态、来源 |
| 要求拆解 | 必需项、加分项、隐性关注点、轮次假设 |
| 我的准备 | 当前准备度、强项、短板、最近一次练习 |
| 练习历史 | 会话记录、报告、复练入口 |
| 材料工坊 | 简历版本、求职信、反问问题 |
| 真实进展 | 已投递、面试中、Offer、拒绝、备注 |

### 状态机

- 草稿
- 准备中
- 已投递
- 面试中
- Offer
- Rejected
- Archived

## 模拟面试编排器

### 练习计划对象

每个练习计划包含：

- 目标岗位
- 当前目标（摸底 / 冲刺 / 修错 / 复盘）
- 练习模式
- 角色设定
- 难度等级
- 问题覆盖范围
- 会话预算时长
- 输出报告模板版本

### 问题生成逻辑

问题生成分三层：

1. **岗位层**：从 JD 中抽取必须覆盖的主题  
2. **用户层**：从画像与错题本中挑选最需要检验的能力点  
3. **轮次层**：根据 HR、经理、技术、文化等角色切换问法和追问深度

### 追问原则

- 追问必须基于回答中的缺口，而不是模板化追问
- 追问优先验证三类问题：是否真实、是否具体、是否贴合岗位
- 单题追问不宜无限延长，应在信息增量不足时及时切换

### 会话事件

- 问题开始
- 用户回答
- AI 追问
- 用户请求提示
- 题目结束
- 观察写入
- 会话完成

## 证据化报告与评分体系

### 展示原则

### 用户前台优先展示
- 观察
- 证据
- 建议
- 复练任务

### 分数使用原则
- 分数是导航，不是结论
- 对外展示尽量使用档位或区间
- 对内可以保留更细粒度分值，用于趋势分析和模型校准

### 基础评估维度

| 维度 | 定义 | 典型证据 |
|---|---|---|
| 内容具体性 | 是否提供了具体场景、行动和结果 | 是否有案例、数字、关键动作 |
| 结构清晰度 | 是否有清晰的回答框架 | 是否有起承转合，是否易跟随 |
| 岗位相关性 | 是否对准 JD 真正关心的能力 | 是否主动映射到岗位要求 |
| 追问应对 | 面对深挖是否还能稳定补充 | 是否出现明显慌乱、矛盾或空转 |
| 自我认知 | 是否能客观地谈优势、短板和成长 | 是否过度包装，是否能反思 |
| 表达质量（语音后引入） | 语速、停顿、口头禅、冗长度 | 语速波动、长停顿、重复词 |

#### 输出结构

每题至少要产出：

- 题目意图
- 用户回答摘要
- 做得好的点
- 缺失的点
- 建议框架
- 是否写入错题本
- 与用户哪段经历最相关

### 置信度

所有推断性反馈都带置信度标签：

- **高**：证据充分，多个回答都支持该判断
- **中**：本次会话中有明显迹象，但样本不多
- **低**：仅有弱迹象，只作为提醒，不作为结论

### 评分校准

评分系统必须具备以下治理机制：

1. Rubric 版本化
2. Prompt 版本化
3. 基准题集与标准答案样本
4. 人工抽检
5. 离线回归评估
6. 分模型、分语言、分题型的表现监控

### 不建议在前台直接展示的指标

- 精确通过率
- “你超过了多少百分比的候选人”
- 强烈录用 / 不录用标签
- 文化契合度绝对分

这些指标可以作为内部研究对象，但不应在早期产品中直接作为用户结论。

## 错题本与复练系统

### 错题进入条件

满足任一条件即可入本：

- 回答明显空泛
- 与岗位严重不相关
- 被追问后仍无法补充
- 逻辑混乱影响理解
- 在真实面试中确实答砸
- 用户主动标记“这题我想重练”

### 错题粒度

错题不是“这个问题答得不好”而已，而是要归类到能力点，例如：

- 自我介绍
- 离职原因
- 冲突处理
- 失败经历
- 领导力
- 系统设计
- 反问问题质量
- 薪资预期

### 复练策略

| 情况 | 系统动作 |
|---|---|
| 首次失守 | 进入错题本，安排轻量复练 |
| 多次失守 | 提升优先级，安排单题深钻 |
| 已改善但不稳定 | 随机插入相似题检验 |
| 连续达标 | 标记为已攻克并降低频率 |

### 错题本与其他模块的联动

- 与目标岗位工作台联动：显示该岗位最危险的未攻克题
- 与简历工坊联动：提示哪些经历需要补素材
- 与真实面试复盘联动：真实问题自动提升权重
- 与成长看板联动：显示攻克率和顽固薄弱点

## 简历工坊

### 功能边界

简历工坊服务于面试准备，不追求成为完整文档设计器。

### 核心能力

1. **岗位贴合度诊断**  
   判断当前简历与目标岗位的表达差距。

2. **Bullet 改写建议**  
   输出更有结果感、更贴近岗位语言的表达方式。

3. **案例缺口提示**  
   如果用户在行为题上总缺案例，简历工坊要同步提醒并建议补充素材。

4. **多版本管理（P1 开始）**  
   面向不同岗位维护不同版本。

### 不做的能力

- 复杂模板编辑
- 花哨排版
- 大量无依据的“关键词堆砌”

## 真实面试复盘

### 为什么是高优先级功能

真实面试复盘往往比模拟更接近用户愿意付费的时刻。它把“已经发生的高成本事件”转化成下一轮更强的准备素材。

### 复盘流程

1. 选择目标岗位  
2. 记录真实轮次和面试角色  
3. 输入真实问题与面试官反馈印象  
4. 回忆自己当时的回答  
5. 系统生成更优结构、补答建议与下一轮计划  
6. 自动写入错题本与经历库

### 输出

- 复盘摘要
- 高风险问题列表
- 下一轮准备 checklist
- 感谢信草稿
- 跟进建议

# 技术架构

## 架构原则

- 单人主流程必须稳定、低延迟、易扩展
- 所有模型调用都可替换，不绑定单一供应商
- 同步链路尽量短，重分析放到异步任务
- 任何生成结果都要携带版本信息与来源记录

## 系统分层

### 客户端
- Web 优先
- 负责会话交互、工作台、报告展示、错题本、材料工坊
- 后续扩展移动端容器

### 应用服务层
- 身份与权限
- 目标岗位服务
- 模拟面试服务
- 报告与错题服务
- 简历工坊服务
- 真实面试复盘服务

### AI 编排层
- Prompt / Rubric Registry
- 会话状态机
- 问题生成器
- 报告生成器
- 证据抽取器
- 评估与校准服务

### 数据与索引层
- PostgreSQL：用户、岗位、会话、报告、错题、材料
- Redis：会话状态、缓存、限流
- Object Storage：简历、音频、关键帧等媒体文件
- Vector Index：JD、经历卡片、错题、问题模板的语义检索

### 异步任务层
- JD 解析
- 公司信息刷新
- 报告后处理
- 语音转写
- 媒体清理
- 评估回归任务

## 服务边界建议

| 服务 | 核心职责 |
|---|---|
| Profile Service | 渐进式画像、经历卡片、偏好与画像版本 |
| Target Job Service | JD 导入、工作台、状态管理、来源记录 |
| Practice Service | 计划、会话、题目、追问、结束逻辑 |
| Review Service | 报告、评分、错题本、成长趋势 |
| Resume Service | 简历诊断、bullet 改写、版本管理 |
| Intelligence Service | 公司轻情报、来源抓取与缓存 |
| Media Service | 音频、视频、关键帧与保留策略 |
| Eval Service | Rubric、离线题集、回归监控、人工抽检 |

## 模型与供应商抽象

必须有统一适配层：

- LLM Adapter
- Embedding Adapter
- STT Adapter
- TTS Adapter
- Vision Adapter

这样做的原因：
- 降低成本波动风险
- 便于按语种和任务切换模型
- 避免单点故障
- 支持离线评估和 A/B 对比

## Prompt 与 Rubric 治理

每次结果都必须记录：

- prompt version
- rubric version
- model id
- language
- feature flag
- data source version

否则成长趋势无法解释，分数也无法比较。

## 可观测性

### 必须埋点
- JD 导入成功率
- 练习开始率 / 完成率
- 报告生成成功率与耗时
- 用户在报告页的停留与复练触发率
- 错题写入率
- 真实面试复盘使用率
- 删除与导出请求完成率

### 质量告警
- 角色漂移
- 追问重复
- 报告空泛
- 不合理高分
- 语言混乱
- 来源过期

# 数据模型（核心实体）

## 核心实体清单

| 实体 | 说明 |
|---|---|
| User | 账号与基础设置 |
| CandidateProfile | 渐进式画像与经历卡片 |
| TargetJob | 目标岗位工作台 |
| PracticePlan | 一次练习计划 |
| PracticeSession | 一次具体练习会话 |
| QuestionAssessment | 逐题观察与评分 |
| FeedbackReport | 会话报告 |
| MistakeEntry | 错题本条目 |
| ResumeAsset | 简历与材料资产 |
| Debrief | 真实面试复盘记录 |
| SourceRecord | 外部来源及 freshness 信息 |
| EvaluationRun | 离线评估与模型质量记录 |

## 关键关系

- 一个用户可有多个目标岗位
- 一个目标岗位可绑定多个练习计划与复盘记录
- 一个练习会话可产生多个逐题观察和一个报告
- 多个逐题观察可汇总为一个或多个错题
- 错题可跨会话复用，但必须标注来源岗位与能力点

# 质量评估体系

## 线上质量指标

### 体验类
- 首次价值达成率
- 会话完成率
- 报告阅读完成率
- 7 日复练率
- 真实面试复盘回流率

### 内容类
- 追问相关率
- 报告可执行性评分
- 用户对“贴合岗位”的主观评分
- 错题命中率：后续练习是否确实围绕薄弱点展开

### 工程类
- P95 会话响应时延
- P95 报告生成耗时
- 任务失败率
- 每次会话成本
- 关键链路错误率

## 离线评估集

系统需要维护一个持续增长的评估集，覆盖：

- 行为题
- 动机题
- 反问题
- 技术深钻题
- 多语言样本
- 不同资历层级
- 不同岗位类别

每个样本包含：
- 标准任务描述
- 参考优质回答
- 参考低质量回答
- Rubric 标签
- 人工审阅结论

## 人工质量环

- 每周抽检高风险语言与题型
- 每次模型或 rubric 更新前后跑回归
- 对低评分与高评分都抽样，防止通胀
- 对真实用户申诉的报告进行复审

# 信任、安全与合规

## 核心原则

1. 用户知道系统收集了什么  
2. 用户知道系统为什么收集  
3. 用户可以选择关闭  
4. 用户可以随时删除  
5. 用户获得的反馈可解释

## 数据留存建议

| 数据类型 | 默认策略 |
|---|---|
| 画像与工作台 | 长期保留，用户可删除 |
| 文字会话 | 默认保留，支持一键删除 |
| 音频 | P2 起默认短期保留，超期自动删除 |
| 视频关键帧 | 仅用户开启时保存，并提供独立删除入口 |
| 外部来源快照 | 保留摘要与来源链接，不长期缓存无必要原文 |

## 区域合规

- 欧洲用户优先按 GDPR 设计删除、导出与最小必要原则
- 对音视频与简历等高敏数据使用单独告知与同意
- 所有外部来源显示抓取时间与用途
- 组织版默认展示聚合统计，不展示不必要的个人敏感内容

## 伦理边界

\begin{greenbox}
\textbf{红线}\\
\begin{itemize}
\item 不承诺“帮你在真实面试中隐身拿答案”。
\item 不把“你看起来不自信”等高主观判断包装成科学结论。
\item 不将训练数据在未获同意的情况下用于外部候选人评估或销售。
\end{itemize}
\end{greenbox}

# 设计与前端体验约束

## 首屏原则

首屏只做三件事：

- 导入一个岗位
- 选择一个练习目标
- 开始一轮练习

不要在首页堆叠太多模块入口。

## 报告页原则

报告页优先级从高到低：

1. 我这次最该改什么
2. 哪几题最危险
3. 我哪些地方做得不错
4. 我下一轮该怎么练
5. 历史趋势

## 错题本原则

- 错题应像学习工具，不像惩罚工具
- 每条错题都要能直接复练
- 标签体系要稳定，不能一版一个叫法

# 发布计划与资源估算

## 建议团队配置

### P0
- 1 产品 / Founder
- 1 设计
- 2 全栈工程
- 1 AI / 数据工程（可兼任）
- 0.5 QA / Ops

### P1 以后
- 增加 1 后端或平台工程
- 增加 1 内容 / 质量运营
- 语音与媒体引入后再增加媒体处理能力

## 推荐交付节奏

### P0（8-10 周）
- 第 1-2 周：信息架构、JD 导入、工作台骨架
- 第 3-4 周：模拟面试主链路
- 第 5-6 周：报告、错题本、简历改写
- 第 7-8 周：埋点、评估、删除链路、灰度
- 第 9-10 周：Beta 打磨与首批用户测试

### P1（8-10 周）
- 多轮计划
- 真实面试复盘
- 成长看板
- 多语言
- 简历工坊增强

### P2 / P3
- 语音、轻情报、跨岗位对比、Offer 支持
- 视频训练、岗位发现、组织版

## 发布门槛

任意阶段上线前必须满足：

- 功能能闭环，不靠人工补锅
- 质量有离线评估支撑
- 核心埋点齐全
- 法务与隐私说明到位
- 有明确灰度开关与回滚策略

# 风险与应对

| 风险 | 描述 | 应对 |
|---|---|---|
| 范围膨胀 | 很容易回到“全能求职平台” | 严格按阶段 gate 控制，需求进未来池 |
| 评分失真 | 分数看似精细但不可信 | 前台弱化精确分，强化证据和校准 |
| 数据源复杂 | 搜岗和情报接入维护成本高 | 先做用户导入与轻量来源，逐步扩展 |
| 成本失控 | 会话、报告、语音分析叠加成本高 | 模型适配层、缓存、异步后处理、套餐限制 |
| 用户惧怕暴露短板 | 面试报告容易造成挫败 | 反馈文案以“如何改进”为主，不做羞辱式表达 |
| 合规压力 | 音视频与外部来源存在风险 | 明确授权、最小化留存、来源记录、保守上线 |

# 附录 B：一句话产品判断

\begin{bluebox}
\textbf{easyinterview 不应该从“AI 求职操作系统”开始做，而应该从“最懂具体 JD、最会帮你复练的 AI 面试训练器”开始做。}\\

\end{bluebox}

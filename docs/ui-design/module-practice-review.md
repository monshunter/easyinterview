# Interview Session 与 Report Dashboard 目标模块

> **版本**: 1.11
> **状态**: active
> **更新日期**: 2026-05-02

## 1 文档目的

本文档定义当前静态 UI 中完整模拟面试与仪表盘报告的闭环。目标是保证用户从一场明确的面试规划进入完整面试，结束后看到一份隶属于该会话的报告，并能明确选择复练当前轮或进入下一轮。

## 2 模块职责

Interview Session 负责：

- 运行一场完整模拟面试。
- 允许用户选择 `文本面试` 或 `语音面试` 作为整场面试形式。
- 在文本面试中提供 `语音转文字`，把语音识别结果填入文本回答框。
- 记录问题、回答、追问、提示、跳过和会话状态。
- 允许在面试页切换面试官角色：综合面试官、HR、用人经理。
- 结束后进入报告生成。

Report Dashboard 负责：

- 以仪表盘形式展示本场面试结果。
- 明确展示报告 `sessionId`、目标岗位、绑定简历、面试轮次、沟通形式、练习方式和提示使用记录。
- 展示准备度、能力维度、题目回顾、证据详情和复练计划。
- 区分 `复练当前轮` 与 `进入下一轮`。

本模块不负责：

- 选择热身、反问、单题深钻等练习模式。
- 独立追问树。
- 独立错题本或单题复练队列。
- 报告时间线。
- 独立刊物式报告页。
- 复盘。

## 3 面试 session 生命周期

```text
MockInterviewPlan
  -> CreateInterviewSession
  -> ChooseInterviewForm
     ├─ TextInterview
     │  ├─ TypeAnswer
     │  └─ SpeechToText -> FillTextAnswer
     └─ VoiceInterview
        └─ LiveSpokenConversation
  -> InProgress
     ├─ Question
     ├─ Answer
     ├─ FollowUp
     ├─ Hint / Skip
     └─ NextQuestion
  -> Finished
  -> GeneratingReport
  -> ReportDashboard(sessionId)
```

文本面试和语音面试共享同一个面试生命周期、同一套上下文、同一个 `PracticeScreen` 外层骨架和同一种报告形态。旧 `voice` hash / route 只作为兼容入口折回 `practice?mode=voice&modality=voice`，不再渲染独立的语音页面框架。

当前 `screen-report.jsx` 已删除 `ReportEditorial`、`ReportTimeline` 等历史组件定义，`ReportScreen` 固定返回 `ReportDashboard`。设计画板里的报告变体标签和 `reportLayout` hash 参数也已从静态 UI 清理，不改变当前目标。

## 5 语音相关边界

```text
面试形式:
├─ 文本面试
│  └─ 用户主要通过文字提交回答
└─ 语音面试
   ├─ 用户与 AI 进行实时语音对话
   ├─ 保留波形、停顿、口头禅、语速标注和实时转写
   └─ 复用文本面试的顶部控制、题目地图和会话上下文

文本输入辅助:
└─ 语音转文字
   ├─ 只在文本面试输入框内出现
   ├─ 识别结果填入文本回答
   └─ 不改变当前面试形式
```

因此，按钮文案不能再写成 `切到语音`。如果在文本输入框中使用麦克风，应使用 `语音转文字`、`插入转写` 等表达。

## 7 复练与下一轮

```text
Report Next Actions
├─ Path A: 复练当前轮
│  ├─ 直接进入同一轮次的面试 session
│  ├─ 不推进到下一轮
│  ├─ 重复同一面试轮次
│  ├─ 带入题目回顾里的错题
│  ├─ 带入证据缺口
│  └─ 带入追问风险
└─ Path B: 进入下一轮
   ├─ 直接进入下一轮面试 session
   ├─ 沿用同一目标岗位 / JD
   ├─ 沿用同一绑定简历
   ├─ 切换到下一轮面试官视角
   └─ 改变题目结构和考察重点
```

这两条路径不能混用，也都不是返回面试前规划页的动作。`返回面试前确认` 可以回到准备页，但报告页上的 `复练当前轮` 与 `进入下一轮` 都应立即发起对应面试 session。`确认下一场模拟面试` 这类模糊文案应避免，因为用户无法判断是重复当前轮还是进入下一轮。

## 8 题目回顾边界

```text
Question Review
├─ 题目列表
├─ 回答分析
├─ 回答有效点
├─ 缺口
├─ 建议框架
├─ 证据片段
├─ 下次面试追问
└─ 加入本轮复练
```

题目回顾承载“错题本”的用户价值，但仍属于报告内部二级详情。它不产生独立错题本模块，不自动进入复盘。

## 9 复盘边界

复盘属于一级 `Debrief` 模块，不属于模拟面试报告。

```text
Debrief
├─ 选择目标岗位 / JD
├─ 选择关联模拟面试
├─ 选择绑定简历
├─ 复盘记录
├─ 复盘分析
└─ 复盘面试
```

模拟报告里的题目回顾可以作为用户理解本场表现的依据，但不自动写入复盘记录。

## 10 后续实现输入

1. `practice` 是 `InterviewSession` 的目标运行 route；`mode/modality=text` 表示文本面试，`mode/modality=voice` 表示语音面试。
2. 旧 `voice` route 必须折回 `practice` 并补齐 `mode=voice` 与 `modality=voice`，不得重新进入独立语音页面骨架。
3. 文本面试输入框的麦克风必须是 `语音转文字`。
4. 面试官角色是同一场面试的上下文参数，不是练习模式入口。
5. 报告生成后只进入 `ReportDashboard(sessionId)`。
6. 报告不得作为顶部一级导航。
7. 报告必须显示完整会话上下文；无 `sessionId` 时必须回到当前面试规划或历史列表，不展示假报告。
8. 任何刊物式长内容都必须挂到仪表盘模块详情，不新增报告形态。
9. 报告不得提供时间线视图。
10. 工作台和报告页的“再练”必须说明是复练当前轮还是进入下一轮；报告页 CTA 直接进入对应面试 session，不先回工作台二次确认。
11. 报告生成页不得再出现“错题条目 / 入本”等旧文案；当前文案应指向报告内部题目回顾和本轮复练线索，不代表恢复独立错题本。

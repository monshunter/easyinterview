# Resume 一级模块目标结构

> **版本**: 2.0
> **状态**: active
> **更新日期**: 2026-06-12

## 1 文档目的

本文档定义当前静态 UI 中简历作为一级模块的目标结构。简历和 JD 一样是一等公民：它们不是模拟面试规划页的附属信息，而是模拟面试和报告分析的重要输入。

当前运行时入口仍是 `resume_versions`，目标实现以 `ui-design/src/screen-resume-workshop.jsx` 为准。按 product-scope D-20，简历资产已扁平化：不存在原始简历树、结构化主版本、岗位定制版本、分叉流程或版本管理；旧树形实现（含 `screens-p1-depth.jsx` 中的 `_LegacyResumeVersionsScreen` dead code）已全部删除。

## 2 核心决策

1. 简历是一级模块，对应顶部导航 `简历`，当前运行时入口是 `resume_versions`。
2. 简历是平铺列表中的独立资产：每份简历自带原始来源（文件或粘贴文本，只读）、解析文本快照和结构化内容，不区分原始简历 / 主版本 / 岗位定制版本（D-20）。
3. 简历首页是 `Resume Workshop` 平铺列表，无树形分组、无视图切换、无“选为底稿”。
4. 创建新简历必须经历 `上传 / 粘贴 -> Agent 解析 -> 预览确认 -> 保存`，不能在未确认前写入正式简历；轻量问答建档已删除。
5. 简历详情页包含 `预览`、`改写建议`、`手动编辑` 三个标签，对列表中任意一份简历一致可用。
6. 改写建议每条仅提供 `采纳` 一个动作；默认不动作即忽略，不提供逐条 `拒绝 / 编辑`。
7. 采纳一条或多条改写后，用户通过 `预览并保存` 进入确认前预览，查看整份改写结果，并选择 `覆盖原简历` 或 `保存为新简历`；未确认前不写入任何简历。
8. 详情页必须能回看原始来源，弹层展示来源关系、原始文件视图和解析文本快照。
9. `导出 PDF`、`复制纯文本`、`保存改动` 和保存简历都必须有可见反馈；`保存为新简历` 应在当前列表态体现新行。
10. 用户上传或粘贴生成的简历都必须有可识别名称。
11. 系统必须保留原始文件或原始文本；解析、编辑和改写采纳不能覆盖原件快照。
12. 模拟面试规划页和解析确认页只选择当前面试使用哪份简历，不承担完整简历管理职责。
13. 更换绑定简历应在对应页面打开简历选择弹窗，不跳转到简历模块。

## 3 模块结构

```text
Resume / resume_versions
├─ Resume Workshop List（平铺）
│  ├─ Header
│  │  ├─ 简历工坊
│  │  └─ 新建简历
│  └─ 列表行
│     ├─ 简历名称 / 语言标签
│     ├─ 来源（上传文件 / 粘贴文本）
│     ├─ 最近编辑
│     └─ 打开
├─ Resume Create Flow
│  ├─ 上传文件
│  ├─ 粘贴内容
│  ├─ Agent 解析中
│  └─ 预览确认并保存 -> 加入当前列表
└─ Resume Detail(resumeId)
   ├─ Preview Tab
   │  ├─ 只读简历预览
   │  ├─ 导出 PDF
   │  ├─ 复制纯文本
   │  └─ 查看原件弹层: 来源关系 / 原始文件 / 解析文本
   ├─ Rewrites Tab
   │  ├─ 建议改写列表
   │  ├─ 原句 / AI 改写 diff
   │  ├─ 为什么这么改
   │  ├─ 采纳（每条唯一动作，默认不动作）
   │  └─ 预览并保存 -> 确认前预览
   │     ├─ 覆盖原简历
   │     └─ 保存为新简历
   └─ Edit Tab
      ├─ 一句话标题
      ├─ 简介
      ├─ 工作经历
      ├─ 技能
      └─ 保存改动
```

## 4 数据对象

```text
Resume
├─ resumeId
├─ displayName
├─ sourceType: upload / paste
├─ originalFileRef
├─ originalText
├─ parsedTextSnapshot
├─ languageTag
├─ summary
├─ structuredContent
├─ rewriteSuggestions
├─ acceptedChanges
├─ manualEdits
├─ createdAt
├─ updatedAt
├─ model / prompt / parserVersion
└─ userConfirmedFields
```

原始内容只读保存。采纳改写时由用户决定覆盖本简历的结构化内容，或以改写结果另存一份新简历；两种结果都不修改原始来源快照。

## 5 与 JD / 模拟面试规划的关系

```text
MockInterviewPlan
├─ targetJobId
├─ jdId
├─ resumeId
└─ roundId

Resume
├─ 管理全部平铺简历资产
└─ 提供 Resume Picker 数据
```

模拟面试输入至少包括：

- 当前目标岗位 / JD。
- 当前绑定简历。
- 当前面试轮次。
- 当前岗位历史模拟面试结果。

## 6 首次无简历入口

当用户没有任何简历时，首页和简历模块都可以触发同一套 `Resume Intake`。

```text
No Resume
├─ 还没有简历？1 分钟创建
├─ resume_versions(flow=create)
├─ 上传 / 粘贴
├─ Agent 解析原始内容
├─ 预览结构化草稿
└─ 用户确认后保存为一份简历
```

## 7 一级导航入口

```text
[Top Navigation]
└─ 简历
   ├─ 查看简历工坊平铺列表
   ├─ 新建简历（上传 / 粘贴）
   ├─ 打开简历详情
   ├─ 预览 / 改写建议 / 手动编辑
   ├─ 采纳改写 -> 预览 -> 覆盖或另存
   ├─ 查看原始文件
   ├─ 导出 PDF
   └─ 复制纯文本
```

`Settings` 只保留账号、隐私、导出和删除等账户级操作，不承担简历资产管理职责。

旧 `resume` / `experiences` / `star` / `onboarding` route 通过 `ROUTE_ALIASES` 折回 `resume_versions`；历史 `ResumeScreen`、旧 `ResumeVersionsScreen`（含 `_LegacyResumeVersionsScreen` dead code）、`ResumeTreeView` 与 `ResumeBranchFlow` 均已删除，当前目标入口以 `screen-resume-workshop.jsx` 为准。

## 8 后续实现输入

1. 简历列表和简历选择弹窗都必须展示简历名称、来源和最近编辑时间。
2. 简历列表是单层平铺，不得出现树形分组、主版本 / 定制版本标签或“选为底稿”动作。
3. 简历预览必须能看到来源关系、原始文件和解析文本快照。
4. 改写建议必须可追溯到原句，并解释为什么这么改。
5. 改写建议每条只有 `采纳`；不得恢复逐条 `拒绝 / 编辑` 按钮。
6. 采纳改写后必须经过确认前预览，由用户显式选择覆盖原简历或保存为新简历；取消确认不写入任何简历。
7. 创建新简历时，必须先展示解析进度和结构化预览，用户确认后才保存。
8. 导出、复制、保存改动和保存简历等按钮不得是无反馈空动作；保存为新简历应在列表中可见。
9. 解析确认页与模拟面试规划页更换简历时不应直接跳转到简历模块。

# Interview Session 与 Report Dashboard 目标模块

> **版本**: 1.6
> **状态**: active
> **更新日期**: 2026-05-02

## 1 文档目的

本文档定义当前静态 UI 中完整模拟面试与仪表盘报告的闭环。目标是保证用户从一场明确的面试规划进入完整面试，结束后看到一份隶属于该会话的报告，并能明确选择复练当前轮或进入下一轮。

## 2 模块职责

Interview Session 负责：

- 运行一场完整模拟面试。
- 允许用户选择 `文本面试` 或 `语音面试` 作为整场面试形式。
- 在文本面试中提供 `语音转文字`，把语音识别结果填入文本回答框。
- 记录问题、回答、追问、提示、跳过和会话状态。
- 允许在面试页切换面试官角色：综合面试官、HR、用人经理。
- 结束后进入报告生成。

Report Dashboard 负责：

- 以仪表盘形式展示本场面试结果。
- 明确展示报告所属会话、目标岗位、绑定简历、面试轮次、完成时间和沟通形式。
- 展示准备度、能力维度、题目回顾、证据详情和复练计划。
- 区分 `复练当前轮` 与 `进入下一轮`。

本模块不负责：

- 选择热身、反问、单题深钻等练习模式。
- 独立追问树。
- 独立错题本或单题复练队列。
- 报告时间线。
- 独立刊物式报告页。
- 真实面试复盘。

## 3 面试 session 生命周期

```text
MockInterviewPlan
  -> CreateInterviewSession
  -> ChooseInterviewForm
     ├─ TextInterview
     │  ├─ TypeAnswer
     │  └─ SpeechToText -> FillTextAnswer
     └─ VoiceInterview
        └─ LiveSpokenConversation
  -> InProgress
     ├─ Question
     ├─ Answer
     ├─ FollowUp
     ├─ Hint / Skip
     └─ NextQuestion
  -> Finished
  -> GeneratingReport
  -> ReportDashboard(sessionId)
```

文本面试和语音面试共享同一个面试生命周期、同一套上下文和同一种报告形态。

当前 `screen-report.jsx` 中仍保留 `ReportEditorial`、`ReportTimeline` 等历史组件定义，但 `ReportScreen` 运行时固定返回 `ReportDashboard`。设计画板里的报告变体标签和 `reportLayout` hash 参数不改变当前目标。

## 5 语音相关边界

```text
面试形式:
├─ 文本面试
│  └─ 用户主要通过文字提交回答
└─ 语音面试
   └─ 用户与 AI 进行实时语音对话

文本输入辅助:
└─ 语音转文字
   ├─ 只在文本面试输入框内出现
   ├─ 识别结果填入文本回答
   └─ 不改变当前面试形式
```

因此，按钮文案不能再写成 `切到语音`。如果在文本输入框中使用麦克风，应使用 `语音转文字`、`插入转写` 等表达。

## 6 报告仪表盘框架

```text
[Report Dashboard(sessionId)]
├─ Back
│  └─ 返回面试前确认
├─ Header
│  ├─ 面包屑: 模拟面试 / 会话 # / 面试报告
│  ├─ 标题: 岗位 · 当前轮次模拟报告
│  ├─ 说明: 报告隶属于一次已完成模拟面试
│  ├─ 复练当前轮: 当前轮次
│  └─ 进入下一轮: 下一轮次
├─ Context Strip
│  ├─ 所属会话
│  ├─ 目标岗位
│  ├─ 面试轮次
│  ├─ 绑定简历
│  ├─ 完成时间
│  └─ 沟通形式
├─ Summary Cards
│  ├─ 准备度
│  ├─ 维度详情
│  ├─ 题目回顾
│  └─ 下一动作
└─ Detail Surface
   ├─ 准备度详情
   ├─ 维度详情
   ├─ 题目回顾页
   ├─ 证据详情
   └─ 复练计划
```

## 7 复练与下一轮

```text
Report Next Actions
├─ Path A: 复练当前轮
│  ├─ 不推进到下一轮
│  ├─ 重复同一面试轮次
│  ├─ 带入题目回顾里的错题
│  ├─ 带入证据缺口
│  └─ 带入追问风险
└─ Path B: 进入下一轮
   ├─ 沿用同一目标岗位 / JD
   ├─ 沿用同一绑定简历
   ├─ 切换到下一轮面试官视角
   └─ 改变题目结构和考察重点
```

这两条路径不能混用。`确认下一场模拟面试` 这类模糊文案应避免，因为用户无法判断是重复当前轮还是进入下一轮。

## 8 题目回顾边界

```text
Question Review
├─ 题目列表
├─ 回答分析
├─ 回答有效点
├─ 缺口
├─ 建议框架
├─ 证据片段
├─ 下次面试追问
└─ 加入本轮复练
```

题目回顾承载“错题本”的用户价值，但仍属于报告内部二级详情。它不产生独立错题本模块，不自动进入真实复盘。

## 9 真实面试复盘边界

真实面试复盘属于一级 `Debrief` 模块，不属于模拟面试报告。

```text
Debrief
├─ 选择目标岗位 / JD
├─ 选择关联模拟面试
├─ 选择绑定简历
├─ 复盘记录
├─ 复盘分析
└─ 复盘面试
```

模拟报告里的题目回顾可以作为用户理解本场表现的依据，但不自动写入真实复盘记录。

## 10 后续实现输入

1. `practice` 与 `voice` 都归属于 `InterviewSession`，分别表示文本面试和语音面试。
2. 文本面试输入框的麦克风必须是 `语音转文字`。
3. 面试官角色是同一场面试的上下文参数，不是练习模式入口。
4. 报告生成后只进入 `ReportDashboard(sessionId)`。
5. 报告不得作为顶部一级导航。
6. 报告必须显示完整会话上下文，避免用户不知道报告针对哪次面试。
7. 任何刊物式长内容都必须挂到仪表盘模块详情，不新增报告形态。
8. 报告不得提供时间线视图。
9. 工作台和报告页的“再练”必须说明是复练当前轮还是进入下一轮。
10. 报告生成中出现的“错题条目 / 入本”旧文案应解释为报告内部题目回顾和本轮复练线索，不代表恢复独立错题本。

# BUG-0149: hidden signals stayed blank when JD parse omitted hidden_signal classification

> 模块: workspace | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-09 | 关联 Commit: `fix(frontend-home): derive hidden signals from JD parse (BUG-0149)`
> 关联日志: `-`

## 1 现象

- 用户在真实本地页面打开 Parse 只读详情时，`隐性关注点 · 推断` 区块只显示标题、说明和置信度 badge，没有实际 hidden signal 条目。
- 同一页面的 `轮次假设` 在旧后端进程下也为空；重启到当前结构化轮次代码并重新解析后，轮次恢复，但 hidden signal 仍为空。
- 影响范围是 TargetJob JD parse 生成的只读详情。前端没有本地固定兜底，空白直接反映后端没有返回 `target_job_requirements.kind='hidden_signal'`。

## 2 诊断过程

### 假设 1：前端丢弃了后端 hidden signal 数据

- 验证方法：用 Playwright 打开用户提供的 URL，并统计 `[data-testid^="parse-hidden-signal-"]`。
- 结果：排除。页面渲染逻辑只消费 API 返回的 `requirements[].kind === "hidden_signal"`；数据库查询也显示该 TargetJob 当时没有 hidden signal requirement。

### 假设 2：后端解析结果完全没有 LLM 推断信号

- 验证方法：检查本地 `target_import` 解析日志、`fit_summary.riskSignals` 和 requirements 落库结果。
- 结果：排除。LLM 输出包含 `riskSignals`，但没有把这些风险/隐含筛选点归类成 `requirements[].kind="hidden_signal"`。

### 假设 3：后端 parser 接受了缺少 hidden_signal 的“有效”输出

- 验证方法：反查 `backend/internal/targetjob/parse_executor.go` 的 `buildRequirements` 与 focused tests。
- 结果：确认。旧 parser 只校验 requirements 非空、kind 合法和 label 非空；只要有 `must_have`，就会成功落库，导致 hidden section 可以为空。

## 3 根因分析

- 根本原因：`target.import.parse` prompt/schema 没有把“至少一个隐性关注点”写成强约束，backend parser 也没有把缺少 `hidden_signal` 视为输出漂移。
- 涉及路径：
  - `config/prompts/target.import.parse/v0.1.0.md`
  - `config/prompts/target.import.parse/v0.1.0.schema.json`
  - `config/prompts/target.import.parse/v0.1.0.yaml`
  - `migrations/000002_seed_baseline_prompt_rubric_versions.up.sql`
  - `backend/internal/targetjob/parse_executor.go`
  - `backend/internal/targetjob/pipeline_test.go`
  - `backend/internal/targetjob/parse_executor_cross_layer_test.go`
- 为什么之前没有发现：历史 fixture 覆盖了存在 hidden signal 的前端渲染路径，但没有覆盖真实 provider 把隐含信息放在 `riskSignals[]`、却漏掉 `hidden_signal` requirement 的情况。

## 4 修复方案

- Prompt/schema 明确要求 `requirements[]` 至少包含一条 `hidden_signal`，且 `evidenceLevel` 为 `inferred`。
- 后端新增 `ensureHiddenSignalRequirements`：
  - 优先使用 LLM 明确输出的 `hidden_signal`。
  - 如果缺失，则从同一份 LLM 输出的 `riskSignals[]` 派生 inferred hidden signal requirements。
  - 如果 `hidden_signal` 和 `riskSignals` 都缺失，则返回 `AI_OUTPUT_INVALID`，避免页面继续空白。
- 补充 focused tests 覆盖：
  - `riskSignals[]` 可以补偿 hidden signal。
  - hidden signal 和 risk signal 同时缺失时，解析失败。
  - prompt contract 继续包含 `hidden_signal` / `inferred`。

## 6 经验教训

- 对 LLM 输出的 UI 必填区域，仅靠前端 fixture 覆盖“有数据时能渲染”不够；后端 parser 也要拒绝或补偿会造成空白的半结构化输出。
- 如果 prompt 同时输出 summary/risk 与 requirements，后端可以用同一份 LLM 输出做结构化归一，但不能用前端固定文案兜底。
- 验收截图应同时断言 DOM 数据量，避免截图只证明区块存在而没有证明条目存在。

# BUG-0154: resume create prototype discarded the created detail handoff

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-10 | 关联 Commit: `fix(ui-design): preserve created resume detail handoff (BUG-0154)`
> 关联日志: -

## 1 现象

- 在静态原型的 Resume Workshop 中，从 `flow=create` 或列表内 “New resume” 进入 Paste 创建并点击 “Save and open”。
- 首次复现会返回仅含三条 fixture 的列表；保留 screen instance 后，从列表内再次创建仍会停留在 create 页面。
- 正式前端 P0.081 与 focused tests 均通过，影响范围限定在 `ui-design/` 静态真理源的创建后详情 handoff。

## 2 诊断过程

### 假设 1：删除未读 `ResumeCreateFlow.nav` 改变了提交回调

- 验证方法：检查 `ResumeCreateFlow` 调用点、`onCreateResume` 绑定及 Babel 参数消费 inventory。
- 结果：排除。提交仍调用 `addCreatedResume`，被删除的 `nav` 从未在 child 内读取。

### 假设 2：同一路由参数切换卸载了创建资产 owner

- 验证方法：浏览器复现后检查 `App` route key 和 `ResumeWorkshopScreen` 本地 state 生命周期。
- 结果：确认。`route.params.flow` 参与 wrapper key，`flow=create` 切换到 `resumeId` 时 screen 被重新挂载，`createdResumes` 随之丢失。

### 假设 3：移除 flow key 已覆盖列表内创建路径

- 验证方法：从详情返回列表，通过本地 “New resume” 再次执行 Paste 创建。
- 结果：排除。该路径前后 `params.flow` 都为空，依赖参数变化的 effect 不会退出 create mode。

## 3 根因分析

- `ui-design/src/app.jsx` 把瞬时 `flow` 参数放进 screen wrapper key，错误地将同一 Resume Workshop 内的 create-to-detail handoff建模为组件重建。
- `ui-design/src/screen-resume-workshop.jsx` 只在 `params.flow === "create"` 时同步本地 flow，且提交 handler 没有主动退出 create；因此移除 remount 后，列表内创建仍无法进入详情分支。
- 原契约只断言 `nav("resume_versions", { resumeId })` 调用存在，没有验证 route key、本地资产存活或真实 DOM 状态，所以未发现导航调用后的错误页面。

## 4 修复方案

- 从 App screen wrapper key 删除 `route.params.flow`，在同一 Resume Workshop route 内保留 `createdResumes`。
- 将 flow effect 改为同时同步 create/list，并在 `addCreatedResume` 中先切回 list，再导航到新建 `resumeId`。
- 强化 UI contract，锁定非 remount key、双向 flow 同步和提交时的显式 list handoff；没有增加兼容参数、持久化层或平行状态 owner。
- 修改文件：`ui-design/src/app.jsx`、`ui-design/src/screen-resume-workshop.jsx`、`ui-design/ui-design-contract.test.mjs`，并原地更新 `frontend-resume-workshop/002-create-flow` owner checklist。

## 5 验证

- UI contract 两轮 RED 均仅命中 direct-detail 契约，最终 44/44 通过。
- 静态浏览器从 hash create 入口返回列表，再从本地 “New resume” 完成 Paste 创建；50ms 内显示解析中详情，1.2 秒后显示 Markdown ready 详情，page errors 为空。
- P0.081 real-mode generated-client gate 1/1、focused scenario tests 28/28、create-flow focused tests 32/32 通过。
- 全量 frontend 137 files / 841 tests、typecheck、build 通过；owner contexts、`git diff --check` 与 core-loop pruning surface gate 通过。

## 6 经验教训

- 导航函数被调用不等于目标页面正确，包含本地瞬时数据的 handoff 必须验证 owner instance 生命周期和目标 DOM。
- 同一 route 内的 mode 参数不应无条件参与组件 identity；若本地模式也可进入 create，提交 handler 必须显式完成模式转换。

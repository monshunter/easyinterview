# BUG-0087: async runner L2 review exposed outbox startup and live gate drift

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(backend-async-runner): wire outbox startup and live gates (BUG-0085)`
> 关联日志: [2026-05-22](../work-journal/2026-05-22.md)

## 1 现象

- `backend-async-runner/001-internal-job-outbox-runner` 已标记 completed，但 L2 code review 反查 production startup path 时发现 `cmd/api/main.go` 只创建 `runner.Runtime`，没有把 `runner.OutboxDispatcher` 挂到 kernel；`outbox_events` 在真实启动后仍不会被扫描。
- 相关 BDD rerun 中，`E2E.P0.033` 起初因缺少 live DB / MinIO env 未能提供有效证据；补齐 live env 后又暴露 MinIO signed PUT 被客户端强制 SSE header 拒绝、privacy completion SQL 参数无类型、integration tombstone test 重复运行撞固定 audit id。
- 影响范围覆盖 async runtime bootstrap、outbox delivery、upload privacy delete live BDD 和重复运行的场景 gate。

## 2 诊断过程

### 假设 1：outbox dispatcher 已由 main runtime 自动接管

- 验证方法：`rg "NewOutboxDispatcher|SetOutboxDispatcher" backend/cmd backend/internal/runner`，并读取 `backend/cmd/api/main.go` startup path。
- 结果：确认 `NewOutboxDispatcher` 仅存在于 runner tests；production `cmd/api` 没有调用 `SetOutboxDispatcher`，completed checklist 的 D-6 / Phase 3 证据没有覆盖真实启动接线。

### 假设 2：p0-033 失败只是本机环境缺失

- 验证方法：按 `deploy/dev-stack` contract 运行 `make dev-doctor`，补齐 live DB 与 `OBJECT_STORAGE_*` 后重跑 `p0-033` setup/trigger/verify/cleanup。
- 结果：dev-stack 依赖健康后仍失败；MinIO 返回 `NotImplemented`，原因是 signed PUT 带 `x-amz-server-side-encryption: AES256`，而本地 MinIO 未配置 KMS。

### 假设 3：privacy runner 本身未处理 job

- 验证方法：补充 `TestUploadPresignRegisterPrivacyDeleteLiveRoundtrip` 失败诊断，查询 `async_jobs.error_code/error_message`。
- 结果：runner 已处理 `privacy_delete`，但 `MarkDeleteRequestCompleted` 的 `jsonb_build_object('deletedFileCount', $3)` 在 live Postgres 中报 `could not determine data type of parameter $3`，导致 retry。

### 假设 4：p0-033 修复后可重复运行

- 验证方法：连续执行 `p0-033` live gate。
- 结果：第二轮触发 `TestInsertAuditTombstoneIntegrationDoesNotPersistObjectKey` 固定 audit id 唯一键冲突，说明 integration test 只在结束后清理，没有在开始前清理历史行。

## 3 根因分析

- Production bootstrap 未覆盖 D-6：`cmd/api/main.go` 使用单一 `runner.Runtime` 后，没有把 outbox dispatcher 纳入同一 lifecycle；测试只覆盖 runner package 内部 dispatcher 行为，未覆盖 `cmd/api` startup aggregator。
- p0-033 live gate 曾经依赖历史 PASS 和离线 focused tests，缺少对 live DB / MinIO / no-skip / no-tests 的重复运行约束。
- `MinIOStore.Presign` 把服务端加密作为客户端 signed header 强制下发，本地 MinIO 无 KMS 时无法执行 PUT；对象存储加密策略应由 storage-side policy 或 future config 承担。
- `privacyrunner.SQLStore.MarkDeleteRequestCompleted` 的 Postgres JSON 参数缺少显式类型 cast，只有 live DB 才能暴露。
- `file_objects` integration tombstone test 使用固定 audit id，但没有先清理历史行，导致场景重复运行不可幂等。

## 4 修复方案

- 在 `backend/cmd/api/main.go` 新增 `buildRunnerKernel`，从 typed async config 创建 `runner.Runtime`，并创建 `runner.NewSQLOutboxStore` / `runner.NewOutboxDispatcher` 后通过 `SetOutboxDispatcher` 挂入 kernel；新增 `TestMainRunnerKernelDrivesOutboxDispatcher` 覆盖 production-style startup。
- 调整 MinIO presign header，仅返回 `Content-Type`，不再强制 `x-amz-server-side-encryption`；新增 focused test 固定该约束。
- 修复 `MarkDeleteRequestCompleted` SQL，把 `$3` cast 为 `int`。
- 增强 p0-033 live roundtrip 失败诊断，失败时输出 privacy job status / error；修复 audit tombstone integration test 的前置清理，保证重复运行不撞固定 id。
- 原地修订 `backend-async-runner` spec/history/plan/checklist/test-plan/test-checklist 至 1.2，补齐 BDD 场景清单和当前 evidence。

## 6 经验教训

- Completed plan 审查必须反查 production caller，而不是只看 package tests；已知模式对应 `PATTERNS.md` 模式 4 / 模式 5。
- Live BDD gate 必须显式拒绝缺 env、skip、no tests 和不可重复运行；否则一次历史 PASS 会掩盖真实场景不可验收。
- 对象存储 provider-specific header 不应无配置地进入通用 presign contract；本地 MinIO live gate 是发现这类 drift 的必要证据。

# BUG-0108: P0.050 task-run gate lagged answer summary observation

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-26 | 关联 Commit: `fix(backend-practice): align p0050 task-run gate (BUG-0108)`
> 关联日志: [2026-05-26](../work-journal/2026-05-26.md)

## 1 现象

- `go test ./cmd/api -run '^TestE2EP0050PracticeAssistantActionProvenanceAndTaskRuns$' -count=1 -v` 执行了目标测试并失败。
- 失败点是 `ai_task_runs` 数量与固定下标断言：实际行中包含 start-session 的 `question_generate`、`answer_submitted` 的 `hint_generate` answer summary observation、`followup_generate`、`hint_requested` 的 `hint_generate`，以及测试 fake 输出缺少 `answerSummary` 造成的 `AI_OUTPUT_INVALID` failed row。
- 影响范围是 `E2E.P0.050` 与其 shell wrapper；旧 BDD 文档仍写着 fake writer 只捕获 2 行 `followup_generate` / `hint_generate`。

## 2 诊断过程

### 假设 1：生产实现重复写入 hint task-run

- 验证方法：运行 focused test，读取失败输出中的每一行 `ai_task_runs`，反查 `backend/internal/practice/append_session_event_service.go` 与 `backend/internal/practice/hint_ai.go`。
- 结果：排除。start-session 写 `question_generate` 是既有 A3 observed path；`answer_submitted` 在 BUG-0105 后会调用 `applyAnswerObservationAI` 生成并持久化 `answer_summary`，因此新增 `hint_generate` observation row 是当前产品语义。

### 假设 2：测试和 BDD 文档落后于 BUG-0105

- 验证方法：读取 `docs/bugs/BUG-0105.md`、`docs/work-journal/2026-05-26.md` 与 `docs/spec/backend-practice/spec.md` 中 answer_summary 语义，再对照 `backend-practice/003` 的 `bdd-plan.md` / `bdd-checklist.md`。
- 结果：确认。BUG-0105 明确要求 `appendSessionEvent(answer_submitted)` 写入 `practice_turns.answer_summary`，但 `E2E.P0.050` 仍用旧的全局 2 行 task-run 断言。

## 3 根因分析

- `TestE2EP0050PracticeAssistantActionProvenanceAndTaskRuns` 把全局 `ai_task_runs` 行数和数组下标当作 provenance 证据，无法承受 BUG-0105 新增的 answer summary observation path。
- `scenarioPracticeAIClient` 对 `practice.turn.lightweight_observe` 成功响应只返回 `hint`，而当前 parser 在 answer observation 场景需要 `answerSummary`；这让测试自身制造了 parse-after-success `AI_OUTPUT_INVALID` failed row。
- 之前没有发现，是因为 BUG-0105 的真实 provider UAT 修复验证了 report 语义与 answer_summary 持久化，但没有同步扫旧的 focused BDD gate 对 `ai_task_runs` 数量和下标的假设。

## 4 修复方案

- 更新 `scenarioPracticeAIClient`，让 `practice.turn.lightweight_observe` 成功响应同时包含 `hint` 和 `answerSummary`。
- 更新 `TestE2EP0050PracticeAssistantActionProvenanceAndTaskRuns`，按每个 HTTP trigger 的 task-run 增量断言：
  - start-session baseline 写 1 行 `question_generate`
  - 第一次 `answer_submitted` 写 1 行 `hint_generate` answer summary observation + 1 行 `followup_generate`
  - `hint_requested` 写 1 行 `hint_generate`
  - `turn_skipped` / `session_paused` 写 0 行
  - 最终 `answer_submitted` 写 1 行 `hint_generate` answer summary observation，且不写 completion-specific task run
- 同步修订 `backend-practice/003-mode-policies-and-provenance` 的 `bdd-plan.md` 与 `bdd-checklist.md`，移除旧的固定 2 行 / 固定下标口径。

## 6 经验教训

- 修改真实 provider / full funnel runtime 语义后，必须反向扫旧 BDD gates 中对行数、顺序、固定下标、wrapper evidence 的假设。
- 对 provenance / task-run 类场景，优先按 action 前后增量断言，而不是用全局行数证明行为。

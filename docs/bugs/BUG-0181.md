# BUG-0181: Full staticcheck and UI pruning baselines blocked AI transport closeout

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-16 | 关联 Commit: `refactor(ai): adopt openai-go and restore quality gates (BUG-0181)`
> 关联日志: [2026-07-16](../work-journal/2026-07-16.md)

## 1 现象

- A3 `openai-go` transport migration 的 focused tests、`go vet`、根 `make test` 与 `make build` 已通过，但 Phase 15.4 的全仓 `staticcheck ./...` 仍报告 4 条既有 finding。
- `make lint-ui-demo-pruning` 同时报告 1 条 active residual，导致 owner plan 无法恢复 `completed`。
- 失败分布在 `cmd/api`、`cmd/evalkit`、AI registry、review store 与 Home plan 文档，不能归因于单个 adapter 实现。

## 2 诊断过程

### 假设 1：SDK migration 引入了静态检查回归

- 验证方法：逐个比较 5 个失败文件相对 `main` 的 diff，并独立运行 full `staticcheck` 与 UI pruning gate。
- 结果：排除。失败文件在 SDK migration 前均无当前分支 diff，属于既有基线问题。

### 假设 2：所有 U1000 helper 都可直接删除

- 验证方法：对三个 helper 做 repo-wide symbol inventory，并检查 build tags。
- 结果：部分排除。`liveReportActionTypeJSON` 与 review `nullableString` 确认零引用；`newTestKernel` 仅由 `integration` build-tag 测试使用，不能删除。

### 假设 3：UI pruning finding 是 lint 误报

- 验证方法：读取 lint forbidden patterns 与 Home plan 上下文，核对 `docs/ui-design/` 的当前 owner 语义。
- 结果：排除。`Product and UI truth sources` 重新引入了已删除的并行 UI truth-source 口径，当前合同应表述为 product/design contracts。

## 3 根因分析

- integration-only `newTestKernel` 放在无 build tag 的共享测试文件中，默认 build 的 `staticcheck` 看不到它的唯一消费者并报告 U1000。
- evalkit 与 review store 在后续清理后各遗留一个零引用 helper；judge payload 使用可等价转换的冗长 struct literal，触发 S1016。
- UI Demo pruning 完成后，Home plan 的后续文档修订重新使用了 `UI truth sources` 正向措辞；零残留 lint 正确阻止该口径进入 completed baseline。
- 根 `make test` 不执行 full staticcheck 或 `lint-ui-demo-pruning`；这些问题直到 A3 Phase 15.4 的显式全仓收口 gate 才同时暴露。

## 4 修复方案

- 将 `newTestKernel` 移入其唯一 `integration` build-tag 测试文件，保留真实集成测试能力并消除默认 build 假死代码。
- 删除两个零引用 helper，把 judge score-level mapping 收敛为等价类型转换。
- 将 Home plan 标题改为 `Product and design contracts`，保留 `docs/ui-design/` 的设计 owner 语义，不恢复并行 truth source。
- 不增加 lint suppression，不放宽 staticcheck 或 pruning gate，也不修改业务 API、持久化或用户行为。

## 5 验证

- RED：full `staticcheck ./...` 复现 4 条 finding；`make lint-ui-demo-pruning` 复现 `active_residuals (1)`。
- GREEN：full `staticcheck ./...` 零输出；UI pruning 为 `active_residuals (0)`。
- 受影响 4 个 Go package 测试通过；`go test -tags=integration ./cmd/api -run '^$' -count=1` 证明 integration 包仍可编译。
- 根 `make test` 通过：Python 567 tests / 4481 subtests、Go 全包、frontend 126 files / 1004 tests；`make build`、`go vet ./...`、AI lint/config、context/docs/index/diff 与 pruning gates 全部通过。

## 6 经验教训

- 只由 build-tag consumer 使用的 helper 应与 consumer 保持相同 build constraint，避免默认静态分析把合法测试支撑代码判为死代码。
- 根 `make test` 与 lint/static-analysis 是不同证据层；owner closeout 应显式执行计划声明的全仓 lint gate，不能用单测/build PASS 替代。
- pruning lint 的 active residual 是设计 owner 漂移证据；应修正文档口径，而不是增加例外或 suppression。

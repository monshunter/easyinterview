# BUG-0168: P0.006 verifier treated business failed counters as Playwright failures

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-14 | 关联 Commit: `fix(interview): close turn UX and report navigation (BUG-0167, BUG-0168, BUG-0169)`
> 关联日志: [2026-07-14](../work-journal/2026-07-14.md)

## 1 现象

- P0.006 实际运行 160 个 Playwright tests 并全部通过，但 `verify.sh` 仍返回非零。
- `trigger.log` 包含 ReportsScreen 业务状态证据 `failed=1`；旧 verifier 用宽泛的 `[0-9]+ failed` 搜索失败摘要，把这段业务 marker 误判为测试失败。
- 误报阻断 UI 像素一致性最终验收，但没有产品行为失败，因此严重度为 `medium`。

## 2 诊断过程

### 假设 1：Playwright 存在被汇总输出隐藏的失败

- 验证方法：检查 runner exit code、最终 summary、13 个 spec markers 与 desktop/mobile project 输出。
- 结果：排除。runner 明确记录 `160 passed`，所有当前 spec 和两个 viewport project 均执行完成。

### 假设 2：verifier 将业务日志当成 runner-owned summary

- 验证方法：对 `trigger.log` 的所有 `failed` 命中逐行分类，并单独执行旧 regex 与行首 summary regex。
- 结果：确认。唯一相关命中来自合法业务计数 `failed=1`；旧 regex 命中，锚定行首的 Playwright summary regex 不命中。

## 3 根因分析

- verifier 没有区分测试 runner 拥有的汇总行与测试用例打印的业务 marker，只按任意位置的“数字 + failed”判失败。
- ReportsScreen 新增失败状态覆盖后，合法业务输出首次包含该文本形态，触发已有 gate 的 false positive。
- 原 scaffold 只校验 spec 数量和 runner 存在，没有用业务 `failed=1` 作为 verifier 负例，因而无法在新增报告状态前发现边界过宽。

## 4 修复方案

- 将失败摘要检查收紧为 `^[[:space:]]*[0-9]+ failed([[:space:]]|$)`，只接受 Playwright 行首 summary 形态。
- 在 `pixelParityScaffold.test.ts` 增加 source contract，要求 verifier 保留锚定 regex，并禁止恢复旧宽泛 pattern。
- 同步当前 parity spec inventory 为 13 个并显式加入 `reports.spec.ts`，确保独立报告列表进入完整 desktop/mobile runner。

## 5 验证

- RED：旧 verifier 对包含 `failed=1` 且 `160 passed` 的当前 trigger log 返回非零；新增 source contract 在旧 pattern 下失败。
- GREEN：focused pixel parity scaffold 10/10 通过。
- P0.006 最终完整 wrapper 退出 0，13/13 specs、desktop/mobile、160 Playwright tests 全部通过；持久日志 SHA-256 为 `b85c098efe73c33c0f0b51c24605ce3213af30940fbd538dedf93ba39f18266e`。

## 6 经验教训

- Scenario verifier 的失败检测必须锚定 runner-owned status/summary 行，不能对整份混合业务日志使用宽泛文本搜索。
- 对日志 parser 修复应添加与真实冲突形态相同的负例，防止未来业务 marker 再次触发误报。
- 本问题归入 PATTERNS 模式 4 的 runner evidence 精确性范畴；无需新增模式。

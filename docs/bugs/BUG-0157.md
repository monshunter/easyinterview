# BUG-0157: P0.098 shadowed TargetJob fixture left a stale persistence assertion

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-10 | 关联 Commit: `fix(test): align full-funnel target fixture (BUG-0157)`
> 关联日志: -

## 1 现象

- P0.098 完整 setup/trigger/verify/cleanup 在真实 PostgreSQL 上执行时，`resume_parse` 与 `target_import` runner 均成功，但 `assertTargetImportPersisted` 报告 requirement 数量为 2，场景仍期望 1。
- `TestE2EP0098CreatePracticePlanAcceptsEmptyFocusCodes` 和 out-of-scope negative 用例通过，失败集中在 TargetJob 导入后的持久化断言。
- 影响 Ready 场景门禁可靠性，不影响生产 TargetJob 解析或数据持久化。

## 2 诊断过程

### 假设 1：TargetJob store 重复写入 requirement

- 验证方法：读取 parse executor、deterministic fixture、store merge 路径和场景 harness 运行时构造。
- 结果：排除。test-mode deterministic fixture 明确包含一条 `must_have` 和一条 `hidden_signal`，store 正确地各写入一次。

### 假设 2：环境残留或 cleanup 不完整

- 验证方法：运行 `env-status.sh` / `env-verify.sh`，检查四个共享依赖，并确认场景 setup 与 cleanup 退出码和场景用户隔离边界。
- 结果：排除。PostgreSQL、Redis、MinIO、Mailpit 均健康；cleanup 成功删除场景自有数据，没有全局清库或环境重启。

### 假设 3：场景自定义 TargetJob AI fixture 是当前数据来源

- 验证方法：从 `newFullFunnelJourneyHarnessWithTimeout` 反查 `buildTargetJobRuntime`，再检查 `APP_ENV=test` 下的 AI client 包装顺序。
- 结果：排除。`buildTargetJobRuntime` 用 `targetjob.NewDeterministicParseAIClient` 拦截 `target.import.parse`；`fullFunnelScenarioAIClient` 中同 feature 分支及 JSON helper 从未进入该路径。

### 假设 4：场景断言与真实 fixture 漂移

- 验证方法：对比 `deterministicParseFixtureContent` 的 requirement kinds 与 `assertTargetImportPersisted` 的历史总数断言。
- 结果：确认。当前 fixture 的语义是 `must_have=1`、`hidden_signal=1`，场景仍只断言总数为 1。

## 3 根因分析

- P0.098 为 TargetJob 解析保留了两套看似可用的 test fixture：运行时真正使用 `buildTargetJobRuntime` 内的 deterministic client，场景文件同时保留了一个被完全遮蔽的 `fullFunnelScenarioAIClient` 分支。
- TargetJob deterministic fixture 扩展为两个 requirement kinds 后，场景的单行总数断言没有同步，且死分支让代码阅读时容易误判实际数据来源。
- 常规无 `DATABASE_URL` 的 `go test ./...` 会跳过 live PostgreSQL journey；历史 focused/static gates没有把 requirement 断言与运行时 fixture 的语义绑定。
- 涉及路径：`backend/cmd/api/main.go` 的 `buildTargetJobRuntime`、`backend/internal/targetjob/parse_fixture_ai.go`、`backend/cmd/api/full_funnel_journey_scenario_test.go`。

## 4 修复方案

- 删除 `fullFunnelScenarioAIClient` 中不可达的 `target.import.parse` 分支及 `fullFunnelTargetImportJSON` helper，只保留 practice/report 等真实消费者。
- 将 `assertTargetImportPersisted` 改为分别断言 `must_have=1`、`hidden_signal=1`，并断言其他 requirement kind 为 0。
- 在原 `e2e-scenarios-p0/001-full-funnel-happy-journey` plan/checklist 中新增 Phase 5，记录 test-mode fixture owner 和回归 gate，不创建 sibling plan。
- 未采用把历史总数从 1 改成 2 的方案；总数断言无法说明两条记录各自的业务语义，也可能掩盖错误 kind。

## 5 验证

- P0.098 setup、trigger、verify、cleanup 全部通过，三个目标测试均实际执行；日志包含 `resume_parse`、`target_import`、`report_generate` succeeded marker。
- `go test ./cmd/api -count=1` 与 `go test ./... -count=1` 通过。
- `go vet ./...`、`staticcheck ./...` 通过；scoped `dupl -t 100` 为 0 clone group。
- 死 fixture 标识零命中；e2e/product contexts、文档 Header/INDEX/link、`git diff --check` 与 pruning surface 均通过，`real_residuals=0`。

## 6 经验教训

- 集成场景中的 deterministic fixture 应只有一个明确 owner；runtime builder 已接管某 feature 时，应删除场景级平行 fixture，而不是保留看似可注入的备用分支。
- 对 enum/kind 集合的持久化验证应断言语义分布，不应只检查总数。
- Ready 场景的 live DB gate 不能由缺少 `DATABASE_URL` 时会 skip 的常规包测试替代；运行时 fixture 变化后应显式重跑四段 lifecycle。

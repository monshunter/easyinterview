# BUG-0166: unpaired report prompt example leaked unsupported facts into live output

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-07-13 | 关联 Commit: `fix(report): ground report semantics and reliability (BUG-0164, BUG-0165, BUG-0166)`
> 关联日志: [2026-07-13](../work-journal/2026-07-13.md)

## 1 现象

- 真实 provider 生成的一份英文报告通过结构、枚举、长度与跨字段校验，但摘要写入了候选人从未陈述的 “production incident”。
- 反查发现完整 JSON 示例只有输出、没有相邻的合成候选人输入；模型可能把示例中的业务事实当成当前报告可复用内容。
- 直接删除完整示例会损失复杂 JSON 结构与跨字段一致性的生成引导，因此不能用“无示例”规避复制风险。

## 2 诊断过程

### 假设 1：问题来自 schema 或长度限制

- 验证方法：运行 prompt schema、200 code-point fuse、英文 24 words、中文 64 code points、focus/action/readiness 跨字段校验。
- 结果：排除。该输出机械格式合法，问题是事实没有候选人消息锚点。

### 假设 2：完整示例本身不应保留

- 验证方法：对照既有完整 example contract 与真实输出稳定性需求，并按用户反馈评估删除示例的影响。
- 结果：排除。完整示例是复杂结构的重要生成约束，正确修复应让示例自足且明确不可复制。

### 假设 3：无配对输入的示例诱导了语义复用

- 验证方法：比较 prompt 示例事实、真实候选人消息和错误摘要，再加入相邻 synthetic input 与 anti-copy 规则做静态/真实模型复验。
- 结果：确认。旧示例的事实来源不在同一示例上下文中；配对后每个示例字段都能追溯到 synthetic seq 2，且规则明确要求从当前冻结上下文重新生成。

## 3 根因分析

- prompt 同时承担结构示范和语义约束，但旧 complete JSON example 只展示输出，没有展示支撑它的候选人输入。
- 模型能学到合法 JSON 形状，却无法从示例局部判断哪些文本只是示例事实，形成语义复制风险。
- 既有 lint 证明 example schema-valid，却没有证明 example 的事实自洽、与 synthetic input 配对，也没有禁止删除示例或复用示例语义。

## 4 修复方案

- 保留完整 JSON example，在其前加入 synthetic candidate message seq 2：候选人按 user impact / delivery effort 排序，并明确没有解释 tie-breaking rule。
- 将示例输出改为同一 prioritization/tie-breaking 事实链，使 highlight、issue、readiness、retry action 与 source seq 2 完整闭环。
- 增加 anti-copy 指令：示例只用于 JSON shape 与 cross-field coherence，不得复用示例事实、维度、准备度、措辞或行动，所有字段必须从当前冻结上下文与候选人消息重新生成。
- lint 锁定 paired input、完整 output 与 anti-copy 三者同时存在；migration、YAML hash、resolved prompt 与 dev DB 使用同一正文/hash。

## 6 经验教训

- 复杂结构的完整示例应保留，但示例输入与输出必须相邻配对，且显式隔离“结构学习”和“当前事实生成”。
- schema-valid 只能证明格式；事实真实性必须由当前候选人消息锚点、语义 judge 与人工抽查共同评价。
- 本问题不新增 PATTERNS 条目；paired-example lint、真实模型样本与当前报告验收口径已形成可执行防线。

# BUG-0086: home targetjob plan stayed fixture-only after real backend landed

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(frontend-home): close targetjob real backend gate drift (BUG-0086)`
> 关联日志: [2026-05-22](../work-journal/2026-05-22.md)

## 1 现象

- 触发条件 / 发现方式：对 `frontend-home-job-picks-and-parse/001-home-jd-import-and-parse` 执行 L2 code review 时，用户要求按同一 frontend-first handoff 模式做 real-backend drift sweep，确认 TargetJobs/import/parse 的 fixture-first 口径是否仍与当前 backend owner 事实一致。
- 可观察到的错误表现：`plan.md` §3.6 operation matrix 仍把 `listTargetJobs`、`createUploadPresign`、`importTargetJob`、`getTargetJob`、`updateTargetJob` 的 backend handler 写为 `not-yet-implemented`；P0.014-P0.016 scenario scripts 只证明 fixture-backed UI 子用例，没有证明 `VITE_EI_API_MODE=real` 下 production generated client 会打到真实 backend base URL。
- 影响范围：Home JD import / parse plan 可能被误判为只处于 frontend-first fixture handoff 状态，而当前 backend-targetjob 与 backend-upload 已有真实 route/handler；缺少 paired evidence 会让后续 review 无法区分 UI fixture variants 与真实 API 联调闭环。

## 2 诊断过程

### 假设 1：plan001 的 fixture-first wording 仍是当前事实

- 验证方法：读取 `docs/spec/frontend-home-job-picks-and-parse/plans/001-home-jd-import-and-parse/plan.md` §3.6、`spec.md` 模块边界、P0.014-P0.016 README/trigger/verify。
- 结果：排除。文档仍是 2026-05-08 的 frontend-first handoff 口径，但这只反映当时 backend 尚未完成，不是 2026-05-22 的当前 backend owner 事实。

### 假设 2：TargetJobs/import/parse backend 仍未落地

- 验证方法：读取 `backend/cmd/api/main.go` route mounting、`backend/internal/targetjob` handler/service、`backend-targetjob/001` BDD docs 与 P0.010-P0.013 scenario scripts。
- 结果：排除。`GET /api/v1/targets`、`POST /api/v1/targets/import`、`GET /api/v1/targets/{targetJobId}`、`PATCH /api/v1/targets/{targetJobId}` 已挂载到 `targetjob.Handler`，backend E2E.P0.010-P0.013 覆盖 live HTTP TargetJob import/list/detail/update/parse failure/manual form paths。

### 假设 3：upload presign 支撑 operation 也还只是 future owner

- 验证方法：读取 `backend/cmd/api/main.go` upload route、`backend/internal/upload/handler/presign.go`、`backend-upload/001` docs，并运行 focused route/handler tests。
- 结果：排除。`POST /api/v1/uploads/presign` 已挂载到 `upload.Handler.CreateUploadPresign`，focused tests 通过。

## 3 根因分析

- 根本原因：plan 001 完成时 backend-targetjob/backend-upload 尚未完成，后续 backend owner 完成后没有回到原 completed frontend plan 做 real-mode reconciliation，导致 fixture-first handoff wording 没有转为“fixture UI variants + real backend gate + backend live scenarios”。
- 涉及路径：
  - `docs/spec/frontend-home-job-picks-and-parse/plans/001-home-jd-import-and-parse/plan.md`
  - `docs/spec/frontend-home-job-picks-and-parse/spec.md`
  - `test/scenarios/e2e/p0-014-home-default-render/`
  - `test/scenarios/e2e/p0-015-jd-import-and-parse/`
  - `test/scenarios/e2e/p0-016-parse-confirm-to-workspace/`
  - `frontend/src/api/targetJob.realApiMode.test.ts`
- 为什么之前没有发现：历史 P0.014-P0.016 gate 只要求 fixture-backed UI 子用例 PASS；backend P0.010-P0.013 虽证明 live backend，但没有被 frontend owner plan 作为 paired evidence 反向接入。

## 4 修复方案

- 选择的方案及理由：原地修订 completed plan 001，不新建 sibling；新增 `frontend/src/api/targetJob.realApiMode.test.ts`，在 `VITE_EI_API_MODE=real` 下覆盖 5 个相关 generated-client operation 的真实 backend base URL、cookie credentials、side-effect `Idempotency-Key`、无 fixture `Prefer` header 与 TargetJob provenance roundtrip。
- 同步 P0.014-P0.016 trigger/verify/README，让每个 frontend scenario 先跑 real-mode generated-client gate，再跑 fixture-backed UI variants；P0.017 保持 jd_match UI-only smoke，不纳入 TargetJobs overlay。
- 更新 plan/spec/BDD/scenario docs，将 backend handler 状态从 `not-yet-implemented` 改为当前真实 handler 路径，并把 backend E2E.P0.010-P0.013 与 backend-upload focused tests 作为 paired evidence。

## 5 验证

- `VITE_EI_API_MODE=real VITE_EI_API_BASE_URL=http://localhost:8080/api/v1 pnpm --filter @easyinterview/frontend exec vitest run src/api/targetJob.realApiMode.test.ts` PASS。
- E2E.P0.014 / P0.015 / P0.016 全部 setup→trigger→verify→cleanup PASS；每个 trigger log 均包含 `VITE_EI_API_MODE=real`、`VITE_EI_API_BASE_URL=http://localhost:8080/api/v1` 与 `targetJob.realApiMode.test.ts`。
- Backend E2E.P0.010 / P0.011 / P0.012 / P0.013 全部 setup→trigger→verify→cleanup PASS。
- Backend upload focused tests PASS：`go test ./cmd/api -run TestBuildUploadRoutesAlignsIdempotencyTTLWithPresignTTL -count=1`；`go test ./internal/upload/handler -run 'TestCreateUploadPresignReturnsCreatedResponse|TestCreateUploadPresignIdempotencyReplayAndTTL' -count=1`。
- `python3 .agent-skills/sync-doc-index/scripts/sync-doc-index.py --fix-index` zero drift。

## 6 经验教训

- Frontend-first fixture plans 在 backend owner 完成后必须回到原 completed plan 做 real-mode reconciliation；fixture UI PASS 不能替代真实 API mode 的 production generated-client proof。
- 同一 subspec 内出现一次 handoff drift 后，应对相同模式的 sibling/completed plans 做 sweep，而不是只修用户最先指出的 plan。

# BUG-0044: resume additive generated client and cleanup drift escaped contract gates

> 模块: schema | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-12 | 关联 Commit: `fix(openapi): harden resume additive client contracts`
> 关联日志: [2026-05-12](../work-journal/2026-05-12.md)

## 1 现象

- L2 review 发现 `branchResumeVersion` 的 `seedStrategy=ai_select` 合法 `202` fixture 返回 `{resumeVersionId, version, job}`，但 generated TS client 仍承诺 `Promise<ResumeVersion>`。
- `exportResumeVersion` 与既有 `requestPrivacyExport` 在 OpenAPI 中声明 P0 `501 + ApiErrorResponse`，但 generated client 的 generic request path 对所有 non-OK status 先 throw，调用方无法走 typed fallback body。
- `frontend/src/api/devMockClient.ts` 未导入 Resume Workshop 9 个新 operation fixtures，导致 generated operationId coverage test 失败。
- live migration tests 的 cleanup 只尝试删除 `users` 且忽略错误；当 `resume_versions` 等 child rows 存在时，固定 UUID seed 可能残留并污染下一次 live run。

## 2 诊断过程

### 假设 1：OpenAPI response schema 与 generated client return type 不一致
- 验证方法：读取 `openapi/openapi.yaml` 的 `branchResumeVersion` responses、`openapi/fixtures/Resumes/branchResumeVersion.json` 的 `ai-select-202-with-job` scenario，以及 `backend/cmd/codegen/openapi/render_ts.go` 的 `tsResponseType`。
- 结果：确认 generator 按 `200/201/202/204/501` 优先级返回第一个 schema，`201 ResumeVersion` 遮蔽了 `202` response；`202` body 还是匿名 object，无法生成命名 TS type。

### 假设 2：P0 501 fallback 应作为 typed response 被解析
- 验证方法：读取 generated `request<T>`，运行 `pnpm --filter @easyinterview/frontend test src/api/devMockClient.test.ts`。
- 结果：确认 `request()` 在解析 JSON 前对所有 `!response.ok` throw；新增 dev mock fixture coverage 还暴露了 9 个 Resumes operation 未导入。

### 假设 3：migration cleanup 不按 FK 顺序删除
- 验证方法：读取 `backend/internal/migrations/resume_versions_integration_test.go` 的 `cleanupUser` 与 `resume_versions` FK 设计。
- 结果：确认 helper 删除 `users` 时会被 child rows 阻止，且错误被忽略；当前环境无 `DATABASE_URL`，live rows 未实际产生，但缺陷在已迁移 live DB 上可复现。

## 3 根因分析

- TS codegen 的 response model 假设“一个 operation 只有一个可解析 response body”，没有覆盖同一 operation 的多个显式 success status。
- P0 `501` export exception 被当作普通 error path，而不是 OpenAPI 明确声明的 typed fallback path。
- Resume Workshop fixtures 已落地，但 frontend dev fixture registry 这个 consumer 未纳入同一同步检查。
- migration integration cleanup 没有按 B4 privacy deletion matrix 的 child-first 顺序实现，也没有把 cleanup failure 暴露为测试失败。

## 4 修复方案

- 新增 OpenAPI schema `BranchResumeVersionAccepted`，把 `branchResumeVersion` 的 `202` response 引用到命名 schema。
- 修改 TS generator：显式 typed response status 生成 union return type；声明的 P0 `501` status 传入 request allowlist，默认 / 未声明的 4xx/5xx 继续 throw。
- 更新 generated Go/TS artifacts，并为 generator 补充 union response 与 501 allowlist 单测。
- 将 9 个 Resumes fixtures 接入 dev mock client，新增 Vitest 覆盖 operationId 完整性、`exportResumeVersion` typed 501 fallback 与 `ai_select` branch response shape。
- 修改 migration cleanup helper：按 suggestions → versions → tailor runs → target jobs → assets → users 的顺序清理，并在 cleanup error 时 fail test。

## 6 经验教训

- OpenAPI generator gate 需要覆盖 “operation 多显式 response status” 与 “显式 non-OK typed response” 两类 response matrix，而不仅检查第一个 success schema。
- 新增 operation 后必须同步 fixture-backed frontend consumer；operationId coverage test 应作为 OpenAPI additive plan 的直接回归 gate。
- live DB integration tests 使用固定 UUID 时，cleanup helper 必须 child-first 且 report failure；忽略 cleanup error 会把一次失败拖成后续重复运行冲突。

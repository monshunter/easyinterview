# BUG-0163: completed practice rounds were not persisted as TargetJob progress

> 模块: schema | 严重度: high | 状态: resolved
> 发现日期: 2026-07-12 | 关联 Commit: `fix(interview): ground resumes and persist round progress (BUG-0162, BUG-0163)`
> 关联日志: [2026-07-12](../work-journal/2026-07-12.md)

## 1 现象

- 第一轮面试完成后，Workspace 与首页面试卡片仍显示第一轮；刷新页面或点击“立即面试”仍创建/复用第一轮 plan。
- `practice_sessions` 已持久化完成时间，`practice_session_events` 也有 `session_completed` 事实，但 `practice_plans` 没有 `roundId` / `roundSequence`，无法审计该完成事实属于哪一轮。
- TargetJob read model 只选择目标岗位下全局最新的 ready plan；前端再从 TargetJob lifecycle status 和固定/推断轮次补齐当前轮，页面刷新后无法恢复真实进度。
- 影响已完成轮次、下一轮、复练和末轮判断，用户会重复进入错误面试轮次，因此严重度为 `high`。

## 2 诊断过程

### 假设 1：前端完成后没有重新请求 TargetJob

- 验证方法：检查 Home/Workspace 完成后的 query invalidation、页面 reload 请求和真实浏览器网络记录。
- 结果：排除。两个页面都会重新请求 TargetJob；刷新后仍错误，说明后端响应没有可恢复的轮次事实。

### 假设 2：更新 TargetJob 上的可变 `currentRound` 字段即可修复

- 验证方法：对比 session completion 台账、重复完成/重试语义和另一个 TargetJob 可变进度字段的并发更新方式。
- 结果：排除。额外可变字段会与 session/event 完成事实形成两份状态，重复请求、重试或部分失败时容易漂移，且无法解释每一轮由哪个 session 完成。

### 假设 3：只在前端 localStorage 保存下一轮即可修复

- 验证方法：对比跨设备、刷新、服务端列表/详情一致性、删除/换绑简历和审计要求。
- 结果：排除。轮次进度是业务状态，不属于主题/外观偏好；浏览器持久化无法成为服务端 plan/session 的权威依据。

## 3 根因分析

- 原 `practice_plans` 没有规范化轮次身份，只有时间预算和目标岗位；相同时长轮次、非连续 sequence、复练与下一轮无法可靠区分。
- session completion 虽有持久化台账，但 TargetJob Get/List 没有按当前绑定简历、规范 round、完成事件与 session 终态投影 `completedRounds + currentRound`。
- plan create/reuse 没有验证客户端 round 意图是否等于服务端投影出的当前/后继轮次，source report/session 也没有携带可审计的 round provenance。
- 前端 `roundAssumptions` 用 lifecycle status 和本地推断补齐业务进度，掩盖后端 read model 缺失；刷新只能重复得到第一轮。
- 之前的 gate 没有覆盖“完成 → Home/Workspace/Parse reload → 创建下一轮 plan → GET 回读”的真实数据库与浏览器链路。

## 4 修复方案

- 采用方案 A：migration 000017 为 `practice_plans` 增加成对的 nullable `round_id` / `round_sequence`、正数约束和查询索引；新 plan 由后端根据 TargetJob 结构化轮次与完成台账推导并持久化。
- legacy backfill 只处理当前用户、TargetJob 当前绑定简历、未删除目标岗位且时间预算唯一匹配的合法轮次；歧义、非法 type、int32 overflow 或无法绑定的记录保持 `NULL`，不猜测历史事实。backfill 支持 dry-run、apply 和幂等重跑。
- TargetJob Get/List 在分页结果上使用 lateral facts：仅接受当前绑定简历、合法 plan round 和 `session_completed` 台账，投影规范的 completed prefix、首个未完成 current round 与当前 ready plan；生命周期/报告状态不参与轮次判断。
- OpenAPI 增加 `PracticeRoundRef`、`PracticeProgress` 和 PracticePlan round pair；CreatePlan 只接收可选 `roundId` 意图，`roundSequence` 由服务端推导，非法/过期/末轮请求 fail closed。
- frontend 严格消费后端 `practiceProgress`，不再从 lifecycle status 或 localStorage 推导业务进度；plan 复用必须精确匹配 round identity，支持 1/2/4 等非连续 sequence 和相邻同 duration 轮次。
- completion replay 保持事件、报告任务和 outbox 单次落地；重复完成不会追加第二份轮次事实。
- 未采用 TargetJob 可变进度字段方案，也未采用浏览器持久化方案，因为两者都会引入与 session 台账并列的第二真理源。

## 6 经验教训

- 进度不是一个需要额外写入的计数器，而是持久化事实台账的 read projection；只有这样才能同时满足完成、重试、重复请求和审计。
- 轮次身份必须在 plan 创建时规范化落地；只保存 duration、名称或前端 index 都不足以处理同预算、重名和非连续 sequence。
- 业务状态必须由后端持久化。除主题、外观、语言等展示偏好外，前端 storage 不得成为进度、选择、审批或流程状态的事实源。
- Get/List、页面刷新和 direct-start 必须共享同一 read model，并用真实数据库 + 浏览器 reload 证明跨请求恢复，而不是只测组件内状态更新。

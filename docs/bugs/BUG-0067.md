# BUG-0067: practice voice turn contract was not mounted as HTTP route

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-17 | 关联 Commit: `fix(practice-voice): wire voice turn route and E2E gate (BUG-0067)`
> 关联日志: [2026-05-17](../work-journal/2026-05-17.md)

## 1 现象

- Phase 5 创建 `E2E.P0.007` 前反查当前实现时发现，OpenAPI 已声明 `POST /practice/sessions/{sessionId}/voice-turns`，generated types/client/server contract 已包含 `createPracticeVoiceTurn`，domain service 与 SQL store 也已实现 voice turn orchestration。
- 真实 HTTP runtime 没有挂载 `/api/v1/practice/sessions/{sessionId}/voice-turns`，`practice.Handler` 也没有 `CreatePracticeVoiceTurn` adapter 方法。
- 影响范围是完整语音 turn 主路径：前端提交 voice audio 时只能得到 404，无法触发后端 STT -> chat -> TTS 级联，也无法验证 request-level idempotency replay。

## 2 诊断过程

### 假设 1：OpenAPI / generated contract 尚未产出 voice turn operation

- 验证方法：读取 `openapi/openapi.yaml`、`openapi/fixtures/PracticeSessions/createPracticeVoiceTurn.json` 与 `backend/internal/api/generated/types.gen.go`。
- 结果：排除。operation、fixture、request/response schema 均存在。

### 假设 2：domain service 或 persistence 尚未实现

- 验证方法：读取 `backend/internal/practice/voice_turn_service.go`、`backend/internal/store/practice/voice_turn.go` 和对应测试。
- 结果：排除。service 已实现独立 STT / chat / TTS profiles、failure fallback、privacy persistence 与 store 写入。

### 假设 3：HTTP adapter / mux 缺失导致 runtime 404

- 验证方法：读取 `backend/internal/api/practice/handler.go`、`backend/internal/api/practice/session_event_handlers.go` 与 `backend/cmd/api/main.go`，并新增 `TestE2EP0007PracticeVoiceTurnHTTPRoute`。
- 结果：确认。修复前测试失败为 `POST /api/v1/practice/sessions/{sessionId}/voice-turns status=404 want=200`。

## 3 根因分析

- 根本原因：Phase 2 只完成了 voice turn domain/store contract，没有同步落地 generated operation 对应的 HTTP handler adapter 与 `cmd/api` mux route。
- 涉及的代码路径：
  - `backend/internal/api/practice/handler.go`
  - `backend/internal/api/practice/voice_turn_handler.go`
  - `backend/cmd/api/main.go`
- 之前没有发现的原因：已有 tests 覆盖了 domain/service/store 和前端 mock transport，但缺少从 `cmd/api` mux 入口验证 `createPracticeVoiceTurn` 已真实挂载的 E2E / HTTP route gate。

## 4 修复方案

- 新增 `Handler.CreatePracticeVoiceTurn`，解码 `audio.contentBase64`，映射 generated request 到 domain request，并把 domain result 映射为 `PracticeVoiceTurnResult`。
- 在 `cmd/api` 挂载 `POST /api/v1/practice/sessions/{sessionId}/voice-turns`，并用 shared idempotency middleware 包裹 `operation=createPracticeVoiceTurn`。
- handler 成功后写入 idempotency resource handoff `practice_voice_turn/{voiceTurnId}`，确保 replay 只返回同一 voice turn payload，不重复写 session event。
- 新增 handler tests 与 `cmd/api` HTTP scenario test，并创建 `E2E.P0.007` 场景 wrapper，verify 中固定 route mount 与 idempotency handoff 断言。

## 6 经验教训

- OpenAPI operation、generated types、domain service 和 store tests 绿色，仍不足以证明 runtime 可用；side-effect endpoint 必须有 `cmd/api` route-level test。
- BDD wrapper 的 `verify.sh` 需要检查真实 runner marker、route mount 和 idempotency resource handoff，避免 contract-only 或 mock-only 绿色掩盖 mux wiring drift。

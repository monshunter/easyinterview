# BUG-0144: workspace plan cards dropped bound resume context

> 模块: workspace | 严重度: high | 状态: resolved
> 发现日期: 2026-07-08 | 关联 Commit: `fix(workspace): close plan list UX and context regressions (BUG-0143, BUG-0144, BUG-0145)`
> 关联日志: [2026-07-09](../work-journal/2026-07-09.md)

## 1 现象

- 用户在面试规划列表点击任意卡片的 `进入规划`。
- 规划创建时已经绑定简历，但详情页总是进入缺简历绑定提示。
- 影响面试规划主入口，用户无法从列表直接回到已绑定简历的 JD 解析结果 / 规划详情。

## 2 诊断过程

### 假设 1：前端列表卡片只能依赖瞬时 route state

- 验证方法：读取 `WorkspacePlanList.openPlan` 与 `WorkspaceEmptyState.test.tsx`。
- 结果：确认。修复前 `openPlan` 只传 `targetJobId/jobId/jdId`，并合成 `plan-${targetJobId}`；Home 导入时选中的 `resumeId` 只存在于前端 route params，刷新或从列表重新进入时无法恢复。

### 假设 2：后端没有持久化 TargetJob 级简历绑定

- 验证方法：检查本地 DB、`openapi/openapi.yaml`、TargetJob store/service/API mapper。
- 结果：确认。截图同款 TargetJob 有 `analysis_status=ready`，但修复前 `target_jobs` 没有 `resume_id` 列，且该行没有 `practice_plans` 记录；因此 `listTargetJobs` / `getTargetJob` 无法返回绑定简历。

### 假设 3：只从 ready practice plan 投影绑定不够

- 验证方法：对比“导入 JD 后尚未创建 practice plan”的用户路径。
- 结果：确认。用户创建规划时的第一步是导入 JD + 绑定简历；如果还没有 ready `practice_plans` 行，绑定必须存在于 `target_jobs.resume_id`，不能只依赖下游 plan 投影。

## 3 根因分析

- Home 导入 JD 时已经要求用户选择简历，但这个 `resumeId` 没有写入 TargetJob 持久化模型，只作为前端路由参数传到 Parse / Workspace。
- Workspace plan list 是从后端 `listTargetJobs` 重建入口；后端没有 TargetJob 级绑定字段时，列表重新进入只能得到 JD 身份，得不到简历身份。
- 仅从 latest ready `practice_plans` 投影绑定会漏掉“刚导入 JD、尚未创建 practice plan”的正常路径。
- 之前的测试只断言“不要合成 report/resume 数据”，没有断言 `进入规划` 必须携带真实 bound resume，且没有覆盖无 practice plan 行的 re-entry。

## 4 修复方案

- OpenAPI `ImportTargetJobRequest.resumeId` 设为 required；Home paste/upload/URL import 都发送选中的 `resumeId`。
- 新增迁移 `target_jobs.resume_id`，后端 import 校验该简历属于当前用户且未删除，然后把绑定写入 TargetJob。
- `listTargetJobs` / `getTargetJob` 的 `TargetJob.resumeId` 来自 `target_jobs.resume_id`；`currentPracticePlanId` 仍只来自 latest ready practice plan。
- 前端 `WorkspacePlanList.openPlan` 使用声明字段导航；没有 practice plan 时仍携带 target job-level `resumeId`，不得合成 `plan-${targetJobId}` 或 `resume-unbound`。
- 补充 `WorkspaceEmptyState.test.tsx` / `InterviewContext` 回归断言，覆盖无 practice plan 行的列表重进与详情恢复。

## 6 经验教训

- 列表入口不能通过 deterministic route id 猜测 plan identity；必须由 API 显式提供当前绑定。
- 用户在创建 JD 时选择的简历属于 TargetJob 级业务事实；不能只放在 route state 或等到 practice plan 创建后再间接推断。
- 既有历史 TargetJob 如果是在修复前导入且没有 practice plan 行，无法从数据库自动恢复当时选中的简历，需要重新导入或显式重新绑定。

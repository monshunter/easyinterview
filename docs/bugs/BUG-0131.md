# BUG-0131: home immediate interview skipped required resume pre-bind

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-06 | 关联 Commit: `fix(frontend-home): prebind resume for immediate interview (BUG-0131)`
> 关联日志: [2026-07-06](../work-journal/2026-07-06.md)

## 1 现象

- Home JD 输入区仍展示旧说明“粘贴 JD，或继续最近一次模拟面试。每一次练习都绑定具体岗位，而不是泛用题库。”，该文案与当前“首页是新建模拟面试规划快捷入口”的定位重复。
- Home 主按钮仍是“解析并确认面试”，没有表达用户点击后是在启动一场绑定岗位和简历的模拟面试规划。
- Home 主路径允许用户先提交 JD import，再到 `parse` 选择简历；但下游“立即面试”链路必须携带真实 `resumeId`，首页缺少“选择已有简历”导致流程语义前后不一致。

## 2 诊断过程

### 假设 1：Home 只负责解析 JD，简历选择放在 Parse 即可

- 验证方法：反查 `frontend-home-job-picks-and-parse` owner spec / plan、`docs/ui-design/` 和 `ui-design/src/screen-home.jsx`。
- 结果：排除。当前产品语义是 Home 作为“模拟面试”的新建规划快捷方式；由于成功 handoff 必须有真实 ready `resumeId`，Home import 前就应让用户选择已有简历。

### 假设 2：正式前端已经可以在 Home 选择已有简历

- 验证方法：新增 `frontend/src/app/screens/home/HomeResumeSelection.test.tsx`，断言 Home 调用 `listResumes`、渲染 `home-resume-select` / `home-resume-option-*`、未选择时禁用 `home-jd-submit`，并负向断言旧 hero sub 与旧按钮文案。
- 结果：排除。红灯显示旧实现仍渲染 `home-hero-sub`，没有 Home 简历选择控件，也不会调用 `listResumes`。

### 假设 3：只改 Home 实现，不改 Parse 和 BDD gate 也能闭环

- 验证方法：检查 Parse route params、P0.014/P0.015/P0.016 场景脚本、expected outcome 与 verify 断言。
- 结果：排除。Home 成功 import 必须把 `resumeId` 传给 `parse`；Parse 必须继承有效 route `resumeId`，场景 gate 也必须拒绝旧文案、缺少 Home pre-bind 和 `resume-unbound` 成功 marker。

## 3 根因分析

- `frontend/src/app/screens/home/HomeScreen.tsx` 仍沿用“提交 JD → parse 再确认”的旧导入模型，按钮文案、禁用条件和 navigate params 都只围绕 JD source，没有把 ready resume 作为 Home import 的前置条件。
- `ui-design/src/screen-home.jsx` 与 `docs/ui-design/` 没有把 Home 明确建模为新建模拟面试规划快捷入口，导致正式前端缺少可追溯的“选择已有简历” UI 真理源。
- P0.014 / P0.015 / P0.016 场景 gate 之前没有同时覆盖 Home pre-bind、Parse route resume 继承和旧 Home 文案负向搜索。

## 4 修复方案

- 更新 UI 真理源和文档：Home 删除旧 hero sub，主按钮改为“立即面试” / `Start interview now`，在 JD 输入卡下新增“选择已有简历”列表，并保留“还没有简历？1 分钟创建 →”入口。
- `HomeScreen` 在已登录状态调用 generated `listResumes`，过滤 `parseStatus=ready`、未 archived、未 deleted 的简历；用户显式选择 ready 简历前禁用 paste / upload / URL import。
- Home import 成功后 `navigate("parse")` 携带真实 selected `resumeId`；未选择简历时不调用 `importTargetJob`，也不产生 pending import。
- `ParseScreen` 继承有效 route `resumeId`；缺失或无效时仍保持 Phase 7 的阻断逻辑，不默认选中最近简历。
- 同步 owner spec / plan / checklist / BDD、`docs/ui-design/`、static prototype contract test、Home pixel parity 和 P0.014/P0.015/P0.016 场景脚本。

## 6 经验教训

- 入口按钮文案改变通常不是单纯 copy change；当 CTA 从“解析”变为“立即面试”时，必须同步检查其前置业务实体是否已经在入口处绑定。
- Home、Parse、Workspace 这种跨屏漏斗不能只验证单页跳转，还要在 route params 和 BDD verify 中反向拒绝语义不完整的成功路径。
- UI 真理源、正式前端和场景脚本必须同步迁移；否则旧文案或旧 gate 会继续把过期流程当作可接受状态。

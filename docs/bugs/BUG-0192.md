# BUG-0192: core workflow pages drifted from the approved UI references

> 模块: frontend | 严重度: low | 状态: resolved
> 发现日期: 2026-07-19 | 关联 Commit: `fix(frontend): align resume and report record views (BUG-0192)`
> 关联日志: [2026-07-19](../work-journal/2026-07-19.md)

## 1 现象

- 面试进行、上传简历和面试报告三页的主体宽度、导航下方起点、卡片层级、按钮、图标、字体和响应式布局与批准的 UI 参考图存在明显偏差。
- 面试消息角色使用圆形标识，输入区上方提示只显示裸文本；参考图要求方形角色标识和带语义图标的浅色提示胶囊。
- 提示最初仍作为 Transcript 的最后一项渲染，会随消息数量和聊天滚动上下移动，无法持续贴住输入框；输入框本身也缺少显式回归合同来保证始终固定在会话卡底部。
- 报告页桌面 Header 的固定 `flex-basis` 泄漏到 390px 移动端，造成 Header 高度异常并把 Context 区域推到首屏下方。
- 第一轮报告页优化只改变了外层宽度、背景、圆角和局部图标，仍保留四张独立 Context card、无统一左侧语义图标轨和松散正文层级；用户标框区域与目标稿仍是不同的页面构图。
- 后续对照参考图时，简历预览仍保留窄阅读框；报告列表缺少插画、真实岗位摘要与编号时间线；面试记录页的 AI 消息直接落在主体背景中，而“我”使用另一套头像轮廓和独立卡片，角色之间缺少一致的消息容器。

## 2 诊断过程

### 假设 1：仅调整页面外层最大宽度即可恢复一致

- 验证方法：对照三张参考图，逐层检查 Header、内容网格、卡片、CTA、图标和移动端 computed style / bounding box。
- 结果：排除。偏差同时存在于组件 DOM 层级、内联样式、共享 CSS、图标形态和响应式规则，单一容器改动无法闭环。

### 假设 2：既有组件测试已覆盖参考图视觉合同

- 验证方法：先增加 reference visual source/component tests，再运行原实现观察 RED；之后用 Chrome 在 1916×821 和 390×844 两档检查布局与 document overflow。
- 结果：排除。新测试在旧布局上按预期失败；Chrome 进一步发现报告 Header 的移动端 `flex-basis` 回归，以及面试角色和提示细节仍未对齐。

### 假设 3：视觉改动会改变消息、简历注册或报告业务事实

- 验证方法：保留并运行三个 owner 的行为测试、根 `make test`、typecheck/build，并反查相关 spec/plan/BDD 合同。
- 结果：排除。业务请求、路由、幂等、错误恢复、报告投影和完成语义均由原 owner 保持不变。

### 假设 4：通过 max-width、rounded surface 和无溢出即可证明报告页完成目标稿改造

- 验证方法：把目标图标框拆成 Header CTA、单体四列 Context、两列 Metrics、四张带左侧图标轨的 Detail card 和全宽 Overall，并在真实 ready report 上逐块测量 bbox。
- 结果：排除。Phase 16 的 gate 允许旧 DOM 仅套用新外壳而转绿，缺少对共享 Context surface、内部 divider、四种 Detail icon 和首屏 Overall 的结构性拒绝条件。

### 假设 5：面试记录用“AI 裸背景、用户独立卡片”的非对称层级更易区分角色

- 验证方法：把两种消息逐项与用户标框参考图核对，并在 Chrome 读取 assistant/user 卡片与头像的 computed style / bbox。
- 结果：排除。目标是通过角色颜色和文案区分身份，而不是改变消息容器；两种角色必须共用边框、圆角、留白、阴影和头像轮廓。

## 3 根因分析

- 三个页面在历史迭代中分别积累了窄内容面、内联视觉拼装和局部组件样式，缺少以当前参考图为基准的共享 viewport / grid / responsive 合同。
- 既有测试重点验证业务行为和历史布局，没有固定参考图所要求的桌面内容宽度、视觉层级、语义图标、角色形态和移动端 containment。
- 报告 Header 的桌面 flex shorthand 没有在移动端重置 `flex-basis`；普通最终态截图不容易暴露该继承值，必须检查移动端 bbox 与 computed style。
- 报告页首次修订把“视觉对齐”误缩小为宽度和卡片外观，没有把目标稿中标框区域的组件所有权和网格关系写成 RED gate，导致结构未变却被错误收口。
- 面试记录的首轮视觉合同只约束用户消息 surface，没有把 assistant/user 的共同卡片基类与头像几何写成对称断言，允许角色样式再次分叉。

## 4 修复方案

- 上传简历页改为 full-canvas + 1470px 内容面，补齐 Header 插画、分段标签、上传卡、主 CTA 和能力标签，并保留 upload/paste、runtime limit、隐私和 direct-detail 行为。
- 面试进行页建立约 1706px 对齐内容面，重构 Session Header、消息面、Composer 和操作区；将角色标识改为 50px 方形，并把提示改为带 sparkle icon 的 320×42 浅色胶囊。
- 将提示的 DOM ownership 从可滚动 `Transcript` 移交到固定 `InputBar` / Composer，作为输入 shell 的前一个兄弟元素并保持 8px 固定间距；用 source gate 固定 `Conversation` flex-column、`Transcript` 唯一滚动和 Composer `flex: 0 0 auto` 合同。
- 报告页重建为约 1432px 共享网格：复练固定为带刷新 icon 的蓝色主按钮，下一轮固定为带右箭头的描边次按钮；Context 改为一张四列内部竖线分隔的整卡；Metrics、四张 Detail 与 Overall 使用对齐的左侧语义图标轨，并删除优势/风险中重复的 confidence。
- 移动端保持相同阅读顺序单列展开，显式重置 Header copy 的 flex basis；长合法内容完整换行，不用截断换取首屏。
- 为三页新增或强化 reference visual / responsive tests，并把桌面、移动端 Chrome 验收与真实 E2E 边界写回原 owner plan/checklist。
- 简历预览重建为约 `1512/1310/1150px` 的内容面、阅读背景板与白色纸张；报告列表重建为 `1372px` Header 插画、真实事实摘要卡和编号时间线；面试记录采用同宽 Header、三列 Context Strip 与统一消息卡片。
- AI/“我”消息共享浅色 surface、1px 描边、10px 圆角、留白和阴影；desktop 头像统一为 `60×60px` / 9px 圆角，mobile 统一为 `48×48px` / 9px 圆角，仅保留蓝色 AI 与灰色“我”的身份色差。

## 6 经验教训

- 截图驱动的 UI 修订必须把参考图拆成可执行的 grid、bbox、DOM、a11y 和 responsive 约束，不能只依赖肉眼或最终态单测。
- 桌面 flex shorthand、固定宽度和最小高度必须在移动断点逐项审计；仅设置 `flex-direction: column` 不会自动清除继承的 `flex-basis`。
- Fixture-backed 正式前端适合验证完整视觉状态，但必须与真实后端健康检查和真实业务 E2E 证据分开报告。
- 参考图标框必须先翻译成组件结构与几何关系，再写 RED；只测 max-width、圆角、背景和 overflow 会让“旧 DOM 套新皮肤”形成假绿。
- 同一聊天流中的角色差异应由身份 token 表达，共同容器必须由一条基类和对称视觉合同锁定；不能分别拼装 assistant/user 外壳后仅凭肉眼判断一致性。

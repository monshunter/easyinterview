# BUG-0192: core workflow pages drifted from the approved UI references

> 模块: frontend | 严重度: low | 状态: resolved

## 1 现象

- 面试进行、上传简历和面试报告三页的主体宽度、导航下方起点、卡片层级、按钮、图标、字体和响应式布局与批准的 UI 参考图存在明显偏差。
- 面试消息角色使用圆形标识，输入区上方提示只显示裸文本；参考图要求方形角色标识和带语义图标的浅色提示胶囊。
- 提示最初仍作为 Transcript 的最后一项渲染，会随消息数量和聊天滚动上下移动，无法持续贴住输入框；输入框本身也缺少显式回归合同来保证始终固定在会话卡底部。
- 报告页桌面 Header 的固定 `flex-basis` 泄漏到 390px 移动端，造成 Header 高度异常并把 Context 区域推到首屏下方。
- 第一轮报告页优化只改变了外层宽度、背景、圆角和局部图标，仍保留四张独立 Context card、无统一左侧语义图标轨和松散正文层级；用户标框区域与目标稿仍是不同的页面构图。
- 后续对照参考图时，简历预览仍保留窄阅读框；报告列表缺少插画、真实岗位摘要与编号时间线；面试记录页的 AI 消息直接落在主体背景中，而“我”使用另一套头像轮廓和独立卡片，角色之间缺少一致的消息容器。
- 最后一组参考图进一步显示面试规划详情仍是窄 inline 拼装，Header 动作、基本信息、要求、隐性关注点和动态轮次没有形成统一卡片层级；登录、验证码和退出页仍使用 `1160px` 窄壳，设置页仍是 `980px` 纵向区块，五页构图均未达到批准稿。
- 四类真实异步等待态继续沿用不同历史构图：Practice 是旧 panel，Resume 被详情容器收窄，Generating 是隐藏 TopBar 的 inline 页面，JD Parse 使用窄版步骤块；它们均未达到最新四张加载参考图的统一画布、插画、标题、真实进度和导航层级。
- Settings 整体宽幅构图已经完成，但 Header 右侧插画仍只是山形折线、人物轮廓和独立圆形对勾；目标稿要求半透明资料窗口、头像信息、柱状图、前景锁、盾牌对勾与星芒组成的完整安全插画，当前实现与设计稿仍有显著结构差异。
- TopBar 的语言入口仍使用 9px 文本 `▾`，在浅色边框内辨识度不足；圆形设置入口继续显示固定品牌字母 `E`，没有反映当前登录用户名，和用户确认的账号入口语义不一致。
- Home 的 JD textarea 虽然已经横向占满 intake card，但默认可视高度仍只有 `106px`；粘贴多行长 JD 后会保留内部纵向滚动，不能一次完整看到输入内容，删除内容后也没有高度重算合同。
- Settings 的退出登录、注销账号等操作仍沿用 `2px` 尖角；继续反查发现 TopBar/Auth/Home/Workspace/Parse/Practice/Reports/Report/Generating/Resume 中的有框 action 同时存在 `2px`、`6px`、`7px` 和页面私有 small token，和当前页面中已采用的圆角按钮缺少统一语义。
- 报告生成等待态在统一氛围画布内仍叠加一张白色卡片、描边、阴影和局部毛玻璃；修订后的报告生成 UI 要求内容直接落在页面背景上，与其它等待态保持一致。
- 二级/三级页面的返回控件继续使用“返回首页”“返回报告”“返回简历工坊”“返回面试规划”等目标特定文案，同一动作在不同页面形成不一致的可见标签。
- 全面统一为“返回”后，报告生成页的可信 Reports 目标又丢失了用户明确要求的“返回面试报告”语义；Workspace fallback 仍应保持通用“返回”。
- 简历详情 JSX 移除 `ei-resume-detail-preview-card` / `ei-resume-detail-preview-card--pdf` 后，desktop/mobile 背景、边框、阴影和尺寸 CSS 仍成为孤儿；完全删除旧样式后，直接 `article` 又因没有宽度 owner 把短 Markdown 纸张收缩到约 497px。
- Practice Composer 的 send 虽在大外框内，却位于 textarea 焦点内框之外，悬浮在两层边界之间；首轮 overlay 修订在 desktop 看似正确，但 `390px` 下用固定右内边距压窄正文，并叠加 textarea 与 surface 两层 focus ring。

## 2 诊断过程

### 假设 1：仅调整页面外层最大宽度即可恢复一致

- 验证方法：对照三张参考图，逐层检查 Header、内容网格、卡片、CTA、图标和移动端 computed style / bounding box。
- 结果：排除。偏差同时存在于组件 DOM 层级、内联样式、共享 CSS、图标形态和响应式规则，单一容器改动无法闭环。

### 假设 2：既有组件测试已覆盖参考图视觉合同

- 验证方法：先增加 reference visual source/component tests，再运行原实现观察 RED；之后用 Chrome 在 1916×821 和 390×844 两档检查布局与 document overflow。
- 结果：排除。新测试在旧布局上按预期失败；Chrome 进一步发现报告 Header 的移动端 `flex-basis` 回归，以及面试角色和提示细节仍未对齐。

### 假设 3：视觉改动会改变消息、简历注册或报告业务事实

- 验证方法：保留并运行三个 owner 的行为测试、根 `make test`、typecheck/build，并反查相关 spec/plan/BDD 合同。
- 结果：排除。业务请求、路由、幂等、错误恢复、报告投影和完成语义均由原 owner 保持不变。

### 假设 4：通过 max-width、rounded surface 和无溢出即可证明报告页完成目标稿改造

- 验证方法：把目标图标框拆成 Header CTA、单体四列 Context、两列 Metrics、四张带左侧图标轨的 Detail card 和全宽 Overall，并在真实 ready report 上逐块测量 bbox。
- 结果：排除。Phase 16 的 gate 允许旧 DOM 仅套用新外壳而转绿，缺少对共享 Context surface、内部 divider、四种 Detail icon 和首屏 Overall 的结构性拒绝条件。

### 假设 5：面试记录用“AI 裸背景、用户独立卡片”的非对称层级更易区分角色

- 验证方法：把两种消息逐项与用户标框参考图核对，并在 Chrome 读取 assistant/user 卡片与头像的 computed style / bbox。
- 结果：排除。目标是通过角色颜色和文案区分身份，而不是改变消息容器；两种角色必须共用边框、圆角、留白、阴影和头像轮廓。

### 假设 6：沿用通用 Screen Shell 并只调整间距即可对齐规划详情和账号页面

- 验证方法：把五张参考图拆成独立内容宽度、Header/action 关系、卡片数量、装饰 SVG、桌面/移动断点和无溢出约束，并让旧 `1160px` auth、`980px` Settings 与 `1200px` inline 规划详情先形成 RED。
- 结果：排除。三个页面族需要各自 page-scoped composition：规划详情为 `1250px` 内容列，Settings 为 `1372px` Header + 三张功能卡，Auth 三页为 `1450px` 双栏主操作构图；共享业务组件和状态机保持不变。

### 假设 7：分别美化四个现有 loading panel 即可对齐等待态

- 验证方法：反查 Practice/Resume/Generating/Parse 的 DOM owner、TopBar route 策略、轮询/错误状态和四张参考图，再用真实 Chrome 分别触发一次完整异步流程。
- 结果：排除。四页重复实现导致同一等待语义出现不同内容宽度、插画、导航与步骤结构；Generating 还被 route 层显式列为 no-chrome，局部 CSS 无法形成一致体验。

### 假设 8：存在 Header SVG 且使用主题色即可证明 Settings 插画已对齐

- 验证方法：把目标稿拆为窗口、profile、chart、lock、shield 与 sparkle 六类可观察层，并让 component test 查找独立 DOM 锚点、让 CSS test 检查主题色透明层和阴影。
- 结果：排除。旧测试只断言 `.ei-settings-header-art` 存在，旧 SVG 的 7 个基础节点全部属于稀疏人物线稿，没有任何目标图层；因此整体页面验收为绿也无法证明插画内容正确。

### 假设 9：固定品牌字母和文本三角足以表达账号、语言入口

- 验证方法：反查 authenticated runtime 已持有的 `displayName`、TopBar DOM/CSS、中文与拉丁姓名边界，并在真实 Chrome 中读取按钮内容、SVG 尺寸、展开 transform、跳转和控制台日志。
- 结果：排除。固定 `E` 无法区分账号入口与品牌标识，9px 文本三角缺少稳定轮廓和开合状态；两者都需要独立的运行时/视觉合同。

### 假设 10：把 textarea 横向宽度直接翻倍即可增加 JD 可视容量

- 验证方法：读取正式 Home DOM/CSS 与真实 Chrome bbox，对照用户标框检查 textarea、外框和页面宽度，并分别测量空值、36 行长 JD 与删减后的 `clientHeight` / `scrollHeight`。
- 结果：排除。textarea 已经是 `width: 100%`，继续横向放大会突破卡片并产生页面溢出；实际缺口是默认纵向可视高度和随内容重算高度的行为合同。

### 假设 11：使用全局 `button` selector 即可消除尖角

- 验证方法：枚举正式前端所有有框矩形/方形 action，并把 circular initial、pill toggle、无边框 link/back、card、input、status surface 单独分类；先让 token/source contract 在旧实现上 RED，再检查 scoped consumer diff。
- 结果：排除。全局 selector 会破坏圆形账号入口、语言/主题 pill 和无边框导航语义；正确边界是增加 `--ei-radius-control: 8px`，只让显式有框 action consumer 使用该 token。

### 假设 12：报告等待态只需复用共享场景，保留白色卡片也算一致

- 验证方法：在真实报告生成窗口读取内容层 computed style，并对照修订后的无卡片参考稿检查 `card` prop/class、surface、边框、阴影、毛玻璃和 mobile overflow。
- 结果：排除。共享场景仍暴露仅 Generating 消费的 `card` 分支；组件测试虽能证明状态与轮询正确，却不能证明页面已采用同一氛围画布构图。

### 假设 13：目标特定返回文案能帮助用户理解去向

- 验证方法：枚举 Auth、Parse、Resume、Practice、Reports、Generating、Report 和 Conversation 的正式返回消费者、locale key 与现有导航测试，再在真实中英文页面抽样点击。
- 结果：排除。去向已经由页面上下文和 route owner 决定，目标特定标签只制造文案漂移；统一为“返回 / Back”后，各页面原有 target、push/replace、trusted-context 与 fail-closed 行为均可独立保留。

### 假设 14：把 send 绝对定位到 textarea 右下角即可在所有视口完成归属修订

- 验证方法：先以 DOM/CSS RED 要求 textarea/send 同属一个 input surface，再用真实 active session 分别测量 desktop 与 `390×844` 的文本宽度、按钮 containment、textarea/action overlap 和 focus computed style。
- 结果：排除。绝对定位和 `126px` 右内边距在 desktop 可用，但把 mobile 正文压成窄列；全局 textarea focus ring 又与 surface `:focus-within` 形成双框。最终改为 input surface 内的非叠加底部 action area，并由 surface 单独承担焦点反馈。

## 3 根因分析

- 三个页面在历史迭代中分别积累了窄内容面、内联视觉拼装和局部组件样式，缺少以当前参考图为基准的共享 viewport / grid / responsive 合同。
- 既有测试重点验证业务行为和历史布局，没有固定参考图所要求的桌面内容宽度、视觉层级、语义图标、角色形态和移动端 containment。
- 报告 Header 的桌面 flex shorthand 没有在移动端重置 `flex-basis`；普通最终态截图不容易暴露该继承值，必须检查移动端 bbox 与 computed style。
- 报告页首次修订把“视觉对齐”误缩小为宽度和卡片外观，没有把目标稿中标框区域的组件所有权和网格关系写成 RED gate，导致结构未变却被错误收口。
- 面试记录的首轮视觉合同只约束用户消息 surface，没有把 assistant/user 的共同卡片基类与头像几何写成对称断言，允许角色样式再次分叉。
- 规划详情和账号页面长期依赖通用 shell 与大量 inline style，owner test 只验证动作存在和业务状态，没有拒绝旧 max-width、左对齐 Header 动作或纵向 Settings 堆叠，因此旧构图能够在行为测试全绿时持续存在。
- 异步等待态没有 shell-owned 共享组件；业务 owner 各自拼装 loading DOM，route owner 又把 Generating 当作隐藏 chrome 的例外，导致视觉与导航合同同时漂移。旧测试只验证“出现 loading 文案”，没有锁定四种语义插画、真实步骤、无伪百分比、reduced-motion 和全局导航上下文。
- Settings Phase 18 只把 Header 插画记为一个整体存在性锚点，设计文档也只写“低对比度账号/安全插画”；没有把参考图的对象构成、前后景关系和透明度层级转译为 component/CSS RED，导致不相干的稀疏线稿能够长期满足形式合同。
- TopBar 设计文档曾把固定 `E` 写成账号入口合同，并把语言状态交给极小的文本字形；缺少 authenticated `displayName` 到头像首字符的派生规则，以及 SVG chevron 可见底板、旋转状态和空名称 fallback gate。
- Home visual contract 只锁定了 textarea 的 `width: 100%` 与 `min-height: 106px`，没有把空/短/长/删减四种内容状态、`scrollHeight` 重算和禁止内部滚动写入 spec/plan/BDD；因此业务测试全部为绿时，输入容量仍会停留在历史尺寸。
- 共享视觉系统只有 surface/small/medium 等几何 token，没有“有框操作控件”的独立语义 token；各页面因局部视觉迭代直接写入 `2px`、`6px`、`7px` 或复用 surface radius，既有测试只验证单页状态，无法对跨页面 action 一致性做 exact inventory。
- `AsyncTransitionScene` 同时承担共享画布和可选卡片两种构图，但 `card` 只有 Generating 一个消费者；旧视觉 gate 只验证卡片尺寸，没有把修订后的“内容层透明直出”写成负向 source/DOM/CSS 合同。
- 返回文案由各业务页面持有目标特定 locale key，缺少 shell-owned `common.back` 以及正式消费者 inventory；因此 route 行为测试持续为绿时，可见标签仍能跨页面分叉。
- 第一轮返回文案治理又把 `common.back` 误当作无例外合同，没有把 Generating 的 trusted Reports destination 与 Workspace fallback 当作同一 owner 内的显式分支，导致安全目标仍正确但可见文案过度泛化。
- 简历预览的共享背景板同时承担视觉 surface 和宽度 owner；只从 JSX 删除 class、未同步删除 desktop/mobile/后代 CSS 会遗留死样式，只删除全部旧样式又会让 flex item 按短内容 shrink-to-fit。背景与布局职责必须拆开：语义 `article` 保持透明，但仍需占满可用宽度。
- Composer 旧 DOM 只定义 outer shell + textarea + shell-level actions，没有可命名的 input surface owner；设计又只写“按钮位于右下角”，没有区分内边界归属、overlay/flow 关系、窄屏文本宽度和单一 focus ring，因此按钮可以在内外边界之间长期满足行为测试。

## 4 修复方案

- 上传简历页改为 full-canvas + 1470px 内容面，补齐 Header 插画、分段标签、上传卡、主 CTA 和能力标签，并保留 upload/paste、runtime limit、隐私和 direct-detail 行为。
- 面试进行页建立约 1706px 对齐内容面，重构 Session Header、消息面、Composer 和操作区；将角色标识改为 50px 方形，并把提示改为带 sparkle icon 的 320×42 浅色胶囊。
- 将提示的 DOM ownership 从可滚动 `Transcript` 移交到固定 `InputBar` / Composer，作为输入 shell 的前一个兄弟元素并保持 8px 固定间距；用 source gate 固定 `Conversation` flex-column、`Transcript` 唯一滚动和 Composer `flex: 0 0 auto` 合同。
- 报告页重建为约 1432px 共享网格：复练固定为带刷新 icon 的蓝色主按钮，下一轮固定为带右箭头的描边次按钮；Context 改为一张四列内部竖线分隔的整卡；Metrics、四张 Detail 与 Overall 使用对齐的左侧语义图标轨，并删除优势/风险中重复的 confidence。
- 移动端保持相同阅读顺序单列展开，显式重置 Header copy 的 flex basis；长合法内容完整换行，不用截断换取首屏。
- 为三页新增或强化 reference visual / responsive tests，并把桌面、移动端 Chrome 验收与真实 E2E 边界写回原 owner plan/checklist。
- 简历预览使用约 `1512px` 内容面并移除 `1310px` 共享阅读背景板，PDF 页面栈与约 `1150px` Markdown 白色纸张直接位于详情画布；报告列表重建为 `1372px` Header 插画、真实事实摘要卡和编号时间线；面试记录采用同宽 Header、三列 Context Strip 与统一消息卡片。
- AI/“我”消息共享浅色 surface、1px 描边、10px 圆角、留白和阴影；desktop 头像统一为 `60×60px` / 9px 圆角，mobile 统一为 `48×48px` / 9px 圆角，仅保留蓝色 AI 与灰色“我”的身份色差。
- 面试规划详情改为 `1250px` page-scoped shell，把 Start/Reports 放入 Header 右侧，并以基本信息、要求、隐性关注点和动态轮次四层卡面承接后端 TargetJob 投影；saved/missing Resume、无效 progress 与 2~5 个动态轮次仍沿用既有 fail-closed 行为。

## 6 经验教训

- 截图驱动的 UI 修订必须把参考图拆成可执行的 grid、bbox、DOM、a11y 和 responsive 约束，不能只依赖肉眼或最终态单测。
- 桌面 flex shorthand、固定宽度和最小高度必须在移动断点逐项审计；仅设置 `flex-direction: column` 不会自动清除继承的 `flex-basis`。
- Fixture-backed 正式前端适合验证完整视觉状态，但必须与真实后端健康检查和真实业务 E2E 证据分开报告。
- 参考图标框必须先翻译成组件结构与几何关系，再写 RED；只测 max-width、圆角、背景和 overflow 会让“旧 DOM 套新皮肤”形成假绿。
- 同一聊天流中的角色差异应由身份 token 表达，共同容器必须由一条基类和对称视觉合同锁定；不能分别拼装 assistant/user 外壳后仅凭肉眼判断一致性。
- 通用 shell 只能承接跨页面基础节奏；参考图明确要求不同信息架构时，应建立 page-scoped composition 和 source/component RED，同时保留业务 owner 的请求、route 与状态机测试，不能靠 inline style 继续叠加。
- 同一产品中的异步等待态应由 shell 统一视觉语言、a11y 和 reduced-motion，业务 owner 只提供真实状态；Chrome 必须在请求 pending 时截图并读取 bbox/DOM，不能只验最终页或用伪造百分比延长动画。
- “插画存在”不是参考稿一致性的有效 gate；目标插画必须先拆为对象集合、前后景关系、颜色层级和装饰语义，再由 DOM layer、CSS token 与真实 Chrome 三层证据共同验证。
- 账号入口不能把产品品牌字母当作用户身份；共享 shell 应从当前 authenticated runtime 派生最小必要的身份标记，并为中文、拉丁和空值建立显式测试。语言开合状态也应使用 code-native SVG 和可观察 transform，而不是依赖字体渲染的微小文本符号。
- 对 textarea 的“默认尺寸加倍并完整显示内容”不能只写一个静态尺寸值；必须同时锁定横向 containment、纵向下限、长内容 `clientHeight >= scrollHeight`、删减后回缩和内部 overflow，才能避免固定高度或无上限横向扩张形成新的体验回归。
- 跨页面控件一致性不能靠全局元素 selector 或抽样肉眼审查；应先按交互语义建立 token，再维护显式 consumer inventory 与例外清单，并用真实 Chrome 的 computed style 证明代表性页面没有被 surface/pill/circular 规则误伤。
- 共享场景中的可选视觉分支如果只有一个消费者，仍可能让页面族在“复用组件”的同时继续漂移；修订参考稿时必须同时删除无效分支、旧 class/testid 和两档 CSS，并在真实短暂 pending 窗口读取 computed style。
- “按钮在右下角”必须同时定义边界 owner、是否 overlay、文本可用宽度和 focus ring owner；desktop 截图不能替代 exact mobile。若 overlay 需要大块固定内边距，优先改为同一 surface 内的非叠加 action area，并在第一轮 Chrome 后允许依据真实几何修正合同。
- 导航文案与导航目标必须分开拥有：Shell 统一动作标签，业务页面继续拥有 target 与安全语义；source inventory 应同时拒绝旧 locale key 和漏迁正式消费者。
- 共享动作文案可以有极少数 owner 例外，但例外必须绑定既有可信 destination、同时覆盖正常/错误态，并让 fallback 继续复用 shared key；不能因为“统一”而抹平用户已经确认的关键去向语义。
- 删除 presentation class 时要同时做三件事：负向搜索并删除 desktop/mobile/后代死 CSS，保留 renderer 正向测试，再用真实短内容检查直接子项的宽度 owner；透明不等于无布局约束。

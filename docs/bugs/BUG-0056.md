# BUG-0056: backend-practice event replay allowed terminal duplicates and duplicate AI reservations

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-14 | 关联 Commit: `fix(backend-practice): reserve session events before AI`
> 关联日志: `-`

## 1 现象

- Code review 复查 `main` 相较 `checkpoint/008` 的 backend-practice event append 路径时发现，`answer_submitted` 在 session 已 `completed` 或 latest turn 已 `assessed` / `skipped` 后，仍可用新的 `clientEventId` 再次提交同一 `turnId`。
- 终态重复提交会再次进入状态机，可能覆盖 `practice_turns.answer_text` 并重复写 `practice.turn.completed` outbox。
- `ReserveSessionEvent` 只查询 replay，不在 follow-up AI 调用前持久化 reservation；同一 `clientEventId` 并发请求可同时越过 reserve 阶段并各自触发 AI。

## 2 诊断过程

### 假设 1：current-turn 校验已经覆盖终态重复提交

- 验证方法：读取 `AppendSessionEvent` service 的 `turnId` 校验与 `SessionEventService.Route` 状态机分支。
- 结果：排除。service 只校验 payload `turnId` 是否等于 latest turn ID；如果最新 turn 已完成但 ID 相同，仍会继续 route。

### 假设 2：SQL replay reservation 已在 AI 前阻止 same clientEventId 并发

- 验证方法：读取 `backend/internal/store/practice/append_event.go` 的 `ReserveSessionEvent` 与 `AppendSessionEvent` 事务顺序。
- 结果：排除。reserve 事务只执行 context/replay 查询并 commit；真正的 `practice_session_events` unique key 直到 AI 后 append 事务才写入。

## 3 根因分析

- 根本原因：event append 的业务状态 guard 与 idempotency reservation 分散在 route / service / repository 三层，历史测试只覆盖 stale `turnId`、replay hit 与 AI outside transaction，没有断言终态状态不可再写、也没有断言 side effect 前必须有 durable reservation。
- 涉及的代码路径：
  - `backend/internal/practice/session_event.go`
  - `backend/internal/practice/append_session_event_service.go`
  - `backend/internal/store/practice/append_event.go`
- 为什么之前没有发现：existing gate 将 “AI 在事务外” 作为主要安全性证据，但未补充 “AI 前已有 pending reservation” 的并发幂等断言；terminal session/turn 状态也没有作为 route 级负向矩阵覆盖。

## 4 修复方案

- 在 `SessionEventService.Route` 增加 closed session / closed turn guard，阻止 completed/completing/failed/cancelled session 与 answered/assessed/skipped turn 接收新的 current-turn event。
- 在 `AppendSessionEvent` service 中提前校验 `turnId` / `answerText`，生成稳定 `eventID` 并传入 reservation，避免无效 payload 先写 pending event。
- 在 SQL `ReserveSessionEvent` 中 replay 优先；无 replay 时校验 stale/closed 状态，再插入 `pending` payload 的 `practice_session_events` 行。
- 在最终 `AppendSessionEvent` 中要求已有 pending reservation，并将 pending payload 更新为最终 replay payload，避免 AI 前后重复插入窗口。

## 6 经验教训

- AI / external side effect 前的幂等保护必须是 durable reservation，而不是只依赖后置 result insert 的 unique key。
- Event state machine 的 terminal status guard 应与 current cursor guard 同时覆盖；同一个 `turnId` 不代表该 turn 仍可被新事件修改。

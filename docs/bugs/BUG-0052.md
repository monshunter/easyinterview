# BUG-0052: backend resume L2 review exposed validation and retry state drift

> 模块: schema | 严重度: high | 状态: resolved
> 发现日期: 2026-05-13 | 关联 Commit: `fix(backend-resume): remediate asset registration L2 findings`
> 关联日志: [2026-05-13](../work-journal/2026-05-13.md)

## 1 现象

- `$plan-code-review 001-asset-register-parse-and-listing --fix` 发现 backend-resume 已完成计划中仍存在 L2 行为漂移。
- `RegisterResume` 的 upload validation 失败、`ListResumes` invalid cursor 会被 handler 包装成 `500 + VALIDATION_FAILED`，与 spec / BDD 期望的 `422 + VALIDATION_FAILED` 不一致。
- `resume.parse` 遇到 retryable timeout 且未耗尽重试次数时不会写 `resume_assets.parse_status='failed'` 和 `error_code`，后续 retry 也无法从 failed asset 回到 processing。
- 既有 P0.034 / P0.035 gate 主要覆盖 happy path 和 fake service，未直接锁住上述 validation / retry negative path。

## 2 诊断过程

### 假设 1：OpenAPI 或 fixture 定义允许这些失败返回 500

- 验证方法：对照 `backend-resume` plan / bdd-checklist 和 handler fixture parity 范围。
- 结果：排除。计划和 BDD 明确要求 validation 类失败为 422，parse retryable failure 仍要在 `resume_assets` 写 failed/error_code，retry metadata 留在 `async_jobs`。

### 假设 2：service/store 已返回 typed error，但 handler 没有区分

- 验证方法：读取 `backend/internal/resume/service.go`、`backend/internal/resume/store/assets.go`、`backend/internal/resume/handler/register.go`、`backend/internal/resume/handler/list.go`。
- 结果：确认。service/store 已暴露 `resume.ErrValidationFailed` 和 `resumestore.ErrInvalidCursor`，但 handler 将任意 service error 写成 500。

### 假设 3：retryable timeout 没有写 failed 是 async retry 设计要求

- 验证方法：读取 `backend/internal/resume/jobs/parse.go` failure path 与 state transition SQL，再补红灯测试。
- 结果：排除。设计要求 `resume_assets` 始终表达当前用户可见 parse state；retryable 信息只是不应扩展出 `failed_retryable` 状态。当前实现把非耗尽 retry 留在 processing，且 `MarkParsing` 只允许 queued。

## 3 根因分析

- Handler 层缺少 service/store typed error 到 HTTP status 的统一映射，导致业务 validation 错误被 fallback 成 500。
- Parse handler 把 "是否 retryable" 与 "是否写用户可见失败状态" 绑定在一起，只有非 retryable 或重试耗尽才调用 `CompleteParseFailure`。
- Store 层 `MarkParsing` SQL 只允许 `queued -> processing`，没有支持 retry path 的 `failed -> processing`，也没有在 retry 时清理旧 `error_code`。
- E2E 脚本没有 grep 新增 negative 测试名，cmd/api 场景没有覆盖 validation 和 retryable failure，因此历史 PASS 没有暴露漂移。

## 4 修复方案

- 新增 handler error helper，把 `resume.ErrValidationFailed` / `resumestore.ErrInvalidCursor` 映射为 `422 + VALIDATION_FAILED`，其他错误继续走 500 fallback。
- 调整 parse handler：retryable failure 也立即调用 `CompleteParseFailure`，保持 `async_jobs` outcome 的 `Retryable=true`；允许 failed asset 在 retry 时重新 `MarkParsing`。
- 调整 store state machine：`MarkParsing` 支持 `queued` 或 `failed` 到 `processing`，并清理旧 `error_code`。
- 加固测试与场景脚本：新增 handler / job / store / cmd-api negative tests，并让 P0.034 / P0.035 trigger/verify 检查这些测试名且拒绝 skip/no-op。

## 6 经验教训

- L2 review 不能只信 handler fixture parity 或 fake service happy path；validation / retry negative path 必须出现在 cmd/api gate 和 scenario verify 的测试名断言里。
- Parse state machine 需要同时审查用户可见状态列和 async retry metadata，避免用 retry outcome 替代业务状态落库。

# BUG-0076: resume workshop real backend requests drifted from version contracts

> 模块: materials | 严重度: high | 状态: resolved
> 发现日期: 2026-05-18 | 关联 Commit: `fix(resume-workshop): close real backend gaps (BUG-0076, BUG-0077)`
> 关联日志: [2026-05-18](../work-journal/2026-05-18.md)

## 1 现象

- Code review of `main` against `checkpoint/016` found Resume Workshop paths that passed fixture-backed UI tests but failed or misrouted against the real backend.
- Create confirmed a structured master without `structuredProfile.provenance`, so `confirmResumeStructuredMaster` rejected the request.
- Edit/manual rewrite PATCH sent the read-only `structuredProfile.provenance`, so `updateResumeVersion` rejected otherwise valid saves.
- Rerun tailor requests did not include the current `resumeVersionId`, allowing async processing to fall back to the latest matching sibling version.
- Version reads returned empty `suggestions` even after a tailor run had written suggestion rows.

## 2 诊断过程

### 假设 1: Frontend request mappers differed from backend validators

- 验证方法：Compared frontend create/update/rerun request construction with backend `confirm_structured_master.go`, `update_version.go`, and OpenAPI schemas.
- 结果：确认。Create omitted required provenance; update forwarded read-only provenance; rerun omitted an explicit version binding.

### 假设 2: Backend persistence wrote suggestions but read models did not hydrate them

- 验证方法：Traced `CompleteTailorRunSuccess`, `GetVersionByID`, `ListVersionsByAsset`, and `loadVersionSuggestions`.
- 结果：确认。Suggestion rows were written, but get/list paths returned bare `resume_versions` rows without loading `resume_version_suggestions`.

### 假设 3: Branch flow used display text as a server identifier

- 验证方法：Traced `ResumeBranchFlow` submit context into `mapBranchFormToBranchResumeVersionRequest` and backend `BranchResumeVersionRequest.targetJobId`.
- 结果：确认。The UI forwarded free-form target text as `targetJobId`, violating the UUID contract.

## 3 根因分析

- The Resume Workshop implementation used fixture green paths as sufficient evidence and did not assert the real backend wire shape at the frontend boundary.
- `RequestResumeTailorRequest` did not expose the version that should receive suggestions, forcing the async job to infer a target version from asset and target job.
- Backend version read paths had no regression test that checked persisted suggestions are present in `getResumeVersion` / `listResumeVersions`.
- The branch UI treated a human target label and a server `targetJobId` as the same field.

## 4 修复方案

- Added optional `resumeVersionId` to `requestResumeTailor` OpenAPI schema, fixtures, generated Go types, and generated TS types.
- Threaded `resumeVersionId` through frontend rerun requests, backend handler/service/store input, and async job payload creation.
- Validated explicit tailor version ownership against `user_id`, `resume_asset_id`, and `target_job_id` before queueing the async job.
- Loaded `resume_version_suggestions` in `GetVersionByID` and `ListVersionsByAsset`.
- Added frontend guards so branch submission uses a route/server target ID when present, or rejects free-form target text before calling `branchResumeVersion`.
- Added create/update request mapper tests for required provenance and read-only provenance stripping.

## 6 经验教训

- Resume Workshop reviews must assert both frontend generated-client payloads and backend validators for every side-effect operation.
- Fixture-backed UI success is not enough for request mappers; tests need explicit real-backend request-body assertions for required and read-only fields.
- Async jobs that mutate a specific version should carry that version ID in the job payload rather than infer it from broader asset/target context.

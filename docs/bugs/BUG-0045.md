# BUG-0045: resume workshop L2 review exposed original modal and CSS parity drift

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-12 | 关联 Commit: `fix(frontend-resume-workshop): remediate listing detail L2 gaps`
> 关联日志: `-`

## 1 现象

- `/plan-code-review frontend-resume-workshop/001-listing-routing-and-detail-readonly --fix` 复查时发现 `ResumeDetailView` 的 "View original" modal 没有读取原始 resume asset，而是从当前 `ResumeVersion.structuredProfile` 的 headline / summary 拼出所谓原件内容。
- 同一轮复查还发现 `frontend/src/app/screens/resume-workshop/` 组件使用了多组 `ei-resume-workshop-*` / `ei-resume-detail-*` className，但 `frontend/src/app/screens/screens.css` 没有对应 selectors。
- 既有 Playwright pixel parity spec 声称覆盖 computed style，但实际主要断言 DOM anchor、bounding box 与非空截图，未能发现 CSS selectors 缺失。

## 2 诊断过程

### 假设 1：原件弹层只缺少测试断言

- 验证方法：读取 `ResumeDetailView.tsx`、`ResumeDetailView.test.tsx`、generated client 与 `openapi/fixtures/Resumes/getResume.json`。
- 结果：排除。generated client 已有 `getResume(resumeAssetId)`，但 `ResumeDetailView` 未调用，plan context 的 `apiNames` 也未包含 `getResume`。

### 假设 2：CSS selectors 已在其他样式文件中定义

- 验证方法：运行 `rg -n "ei-resume-workshop|ei-resume-detail" frontend/src/app -g '*.css'`。
- 结果：排除。正式前端没有对应 CSS 规则，className 无法追溯到 `ui-design/src/screen-resume-workshop.jsx` 的源级布局值。

### 假设 3：pixel parity gate 足以覆盖该漂移

- 验证方法：读取 `frontend/tests/pixel-parity/resume-workshop.spec.ts`。
- 结果：排除。该 spec 没有读取关键 computed style，也没有源级 CSS selector/value 测试，因此截图 smoke 通过不能证明样式 parity 完整。

## 3 根因分析

- 根本原因一：Phase 3 的数据消费只记录了 `getResumeVersion`，遗漏了 "View original" modal 必须读取 `getResume(version.resumeAssetId)` 的 source asset contract，导致实现用 version profile fallback 替代原件内容。
- 根本原因二：Phase 5 parity gate 只验证可见 DOM 与截图 smoke，没有强制检查 production CSS 是否包含 resume workshop selectors 与原型关键值。
- 涉及代码路径：
  - `frontend/src/app/screens/resume-workshop/components/ResumeDetailView.tsx`
  - `frontend/src/app/screens/resume-workshop/hooks/useResumeAsset.ts`
  - `frontend/src/app/screens/screens.css`
  - `frontend/tests/pixel-parity/resume-workshop.spec.ts`
  - `docs/spec/frontend-resume-workshop/plans/001-listing-routing-and-detail-readonly/context.yaml`
- 为什么之前没有发现：plan/checklist 已完成状态和历史 PASS 被当成线索时，需要 L2 deep reconcile 反查当前真理源、fixtures、generated client 与 CSS runtime truth；旧 gate 没有把 `getResume` 和 computed style 纳入断言。

## 4 修复方案

- 新增 `useResumeAsset(resumeAssetId)` hook，modal 打开后调用 generated client `getResume` 并携带 `Accept-Language`。
- `ResumeDetailView` 优先展示 `mapResumeAssetToUiSource(asset).text`，只有 source asset 没有正文时才回退到 version headline / summary。
- 更新 `getResume.json` default / `master-default` fixture，提供 `parsedTextSnapshot` / `originalText`，让 P0.037 能验证原件文本来自 source asset。
- 补齐 `screens.css` 中 resume workshop list/detail selectors，并新增 `ResumeWorkshopCssParity.test.ts` 锁定关键源级 CSS 值。
- 强化 Playwright pixel parity，读取 runtime computed style 覆盖 shell padding、stats card radius/padding、detail preview grid、resume paper card shadow/min-height/font。
- 同步 plan/checklist/BDD/context，把 `getResume` 纳入 operation matrix、scenario fixture 与 `apiNames`。

## 6 经验教训

- 任何 "查看原件" / source object modal 都必须在 operation matrix 中明确 source fetch operation，不能只依赖 detail object 的投影字段。
- UI parity gate 不能只看 DOM 与截图 smoke；正式 CSS selector/value 与 runtime computed style 都必须有可执行断言。
- Scenario-scoped mock transport 要求跨 operation 使用同名 scenario 时，每个 operation fixture 都提供对应 scenario，不能假设自动 fallback 到 `default`。

# BUG-0186: unanswered terminal prompt leaked into report assessment

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-07-18 | 关联 Commit: `fix(report): exclude unanswered terminal prompt (BUG-0186)`
> 关联日志: -

## 1 现象

- 真实面试结束后，完整会话以一条尚未获得候选人回答的 assistant 追问收尾。
- 报告可以成功生成，但能力维度、问题和练习动作可能围绕这条未回答问题展开，把“没有回答尚未作答的问题”误判为候选人能力缺口。
- 报告状态仍是 ready，schema、evidence user-sequence validator 和页面渲染都可能通过，因此这是会静默损害复盘可信度的核心报告缺陷。

## 2 诊断过程

### 假设 1：报告页面错误拼接了会话末尾文本

- 验证方法：对照 Chrome 报告 DOM、report conversation DOM、持久化 report JSON 与 `practice_messages`。
- 结果：排除。页面忠实显示后端报告；完整会话保留末尾 assistant 消息也符合审计合同。

### 假设 2：模型只在 repair 时错误引用 assistant 序号

- 验证方法：检查 `reportCompletePayload` 的初始 provider 输入和 candidate user allowlist 构造路径，并补充终止 assistant 的 focused test。
- 结果：部分排除。allowlist 已阻止 assistant 序号成为 evidence anchor，但 provider 初始评分输入仍包含末尾追问，模型可以围绕其主题生成锚定到较早 user 消息的负面判断。

### 假设 3：评分输入与完整审计会话缺少显式投影边界

- 验证方法：用三组 transcript 覆盖“末尾未回答 assistant”“assistant 后有 user 回答”“末尾为 user”，同时验证排序和 caller slice 不变性。
- 结果：确认。旧实现把完整 terminal transcript 原样交给 provider，没有区分可审计消息与可评价消息。

## 3 根因分析

- `backend/internal/review/generate_report.go` 的 `reportCompletePayload` 直接序列化完整 `ReportContext.Messages`；完整会话同时承担审计和评分两个不同职责。
- user-sequence allowlist 只限制 evidence anchor 的角色，不能禁止模型从末尾问题主题推导 dimension、readiness、issue 或 corrective action。
- 既有回归覆盖了 assistant anchor 被 validator 拒绝，却没有断言 provider 的 assessment transcript 必须排除未回答的末尾 assistant 消息。

## 4 修复方案

- 先复制并按 `seq_no` 稳定排序消息，建立独立的 provider assessment slice。
- 若且仅若排序后的最后一条消息角色为 `assistant`，从 assessment slice 删除这一条；末尾 user、已有后续回答的 assistant 以及其他消息全部保留。
- candidate user allowlist 从同一 assessment slice 派生；数据库冻结、report conversation API、OpenAPI、prompt/schema、report ID 和 caller-owned slice 均保持不变。
- 没有新增 PATTERNS 条目：BUG-0182 已沉淀 terminal assistant 与 evidence grounding 的同类检查方向，本记录补充的是“合法 user anchor 仍不能替代输入主题边界”的具体变体。

## 5 验证

- TDD RED 证明旧 payload 在两种排序输入下仍包含末尾 assistant；GREEN 后 focused test 和完整 `backend/internal/review` 包通过，并覆盖配对 assistant、末尾 user、排序与源 slice 不变。
- 根 `make test` 通过：Python 584 tests / 4583 subtests、Go 全包、frontend 126 files / 1027 tests；context、docs/index 与 diff gate 通过。
- 后端重新部署后，通过真实 Chrome、真实 frontend/backend/provider 创建新报告；desktop 与 390×844 页面均 ready 且无横向溢出，控制台无 error/warn。
- 同一 report/session 的数据库记录证明完整会话仍有 3 条消息且末条为 assistant；report JSON 不包含末尾追问的规模、分片和锁竞争主题，report conversation 页面仍完整显示该消息。

## 6 经验教训

- “完整可审计上下文”和“可用于评价的证据上下文”必须分别建模；保存更多消息不等于允许模型评价所有消息。
- evidence anchor 的结构校验只能证明引用角色合法，不能证明判断主题来自候选人真正回答过的内容。
- 对以 assistant 问题收尾的会话，应同时验证输入投影、最终自然语言语义和完整 transcript 可读性，不能只验证 report ready 或 JSON schema。

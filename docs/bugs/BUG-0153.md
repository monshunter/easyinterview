# BUG-0153: P0.037 request-count wait raced ready DOM commit

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-10 | 关联 Commit: `fix(test): wait for P0.037 ready DOM (BUG-0153)`
> 关联日志: -

## 1 现象

- 在并行执行 frontend build 与完整 Vitest 时，P0.037 pending PDF upload 用例偶发停留在 `Loading resume...`，完整套件结果为 840 / 841。
- 失败时 `getResume` 已记录第二次调用，但 ready `displayName` 标题尚未出现在 DOM；同一文件 focused 重跑为 6 / 6。
- 影响测试门禁可靠性，不影响生产轮询或用户页面行为。

## 2 诊断过程

### 假设 1：生产 `ResumeDetailView` 轮询回归

- 验证方法：检查本批 diff，并 focused 运行 P0.037 与 `ResumeDetailView.test.tsx`。
- 结果：排除。生产组件无改动，focused 用例均通过，ready 页面最终能正常渲染。

### 假设 2：共享环境或 fixture 污染

- 验证方法：读取 P0.037 场景合同和 runner；确认测试使用 in-tree fixture-backed client，不依赖共享环境，并独立重跑场景。
- 结果：排除。四段 wrapper 通过，场景无外部环境状态。

### 假设 3：请求计数被错误用作 React 提交屏障

- 验证方法：对比场景测试与 owner mirror 的 pending-PDF 用例。
- 结果：确认。两处都先等待 `getResumeSpy` 达到 2 次，随后同步查询 ready heading；高负载时第二次调用记录早于 Promise 结果驱动的 React DOM commit。

## 3 根因分析

- Mock spy 在函数调用时立即增加计数，但 ready response 的 Promise resolution、state update 和 DOM commit 发生在后续调度阶段。
- 原测试把 API 调用次数当成用户可见状态已经完成的同步条件，形成时序竞态。
- 相同断言被复制到场景测试和 owner mirror；此前 Phase 13 只修复 failed-PDF 观察窗口的 `act(...)` warning，没有覆盖 queued-to-ready 的可见状态同步。
- 涉及路径：`frontend/src/app/scenarios/p0-037-resume-detail-preview-readonly.test.tsx:157`、`frontend/src/app/screens/resume-workshop/components/ResumeDetailView.test.tsx:172`。

## 4 修复方案

- 两份用例都先通过 Testing Library `waitFor` 等待 ready `displayName` heading 实际出现在 DOM，再断言 `getResume` 恰好调用两次。
- 保留 PDF page-stack source URL、正文隐藏、native viewer absence 和文件名负向断言。
- 未采用增加固定 sleep 或自动 retry；这些方案会掩盖错误同步条件，仍可能在更慢环境中复发。
- 修改文件：
  - `frontend/src/app/scenarios/p0-037-resume-detail-preview-readonly.test.tsx`
  - `frontend/src/app/screens/resume-workshop/components/ResumeDetailView.test.tsx`

## 5 验证

- 4 个并发 focused 进程均通过，每个进程为 2 files / 14 tests。
- P0.037 `setup.sh` / `trigger.sh` / `verify.sh` / `cleanup.sh` 通过，场景为 6 / 6。
- Resume Workshop owner 回归通过 20 files / 113 tests，frontend typecheck 通过。
- 完整 frontend 回归通过 137 files / 841 tests。
- 旧的 request-count-first source pattern 为零，生产代码无改动。

## 6 经验教训

- 异步 UI 测试应等待用户可观察 DOM 状态；请求次数只能作为行为证据，不能替代渲染完成条件。
- 场景测试与 owner mirror 复制同一异步路径时，修复和 source inventory 必须同时覆盖两处。
- 对 focused 常绿、全量负载下偶发失败的用例，优先检查 wait 条件是否锚定了中间实现信号。

# BUG-0058: backend-practice hint replay leaked hint text and BDD gates under-asserted evidence

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-15 | 关联 Commit: `fix(backend-practice): remediate hint provenance L2 findings`
> 关联日志: `-`

## 1 现象

- `$plan-code-review backend-practice/003-mode-policies-and-provenance --fix` 复查时，003 checklist / BDD checklist 已是 `completed`，但 P0.048-P0.051 Go HTTP scenario 只覆盖了部分声明断言。
- P0.048 BDD 声称 `practice_session_events.payload` 不含 `hint_text` 明文；补强断言后 focused test 首次失败，显示 assisted hint 的 replay payload 会保存生成的 hint 文本。
- P0.050 只验证了 `show_hint` 的 provenance，未覆盖 BDD 中声明的 `ask_follow_up` / `ask_question` / `session_wait` / `session_completed` cross-action parity。
- P0.051 只覆盖 parsed hint empty，未覆盖 BDD 中声明的 F3 unsupported、A3 secret missing、A3 timeout、invalid output 四类 graceful degrade；补强后还发现 observed A3 callErr 行缺少 `error_code` / invalid validation metadata。

## 2 诊断过程

### 假设 1：BDD checklist 的 completed 状态代表 P0.048-P0.051 均已执行完整断言

- 验证方法：直接读取 `docs/spec/backend-practice/plans/003-mode-policies-and-provenance/bdd-checklist.md`，再反查 `backend/cmd/api/practice_http_scenario_test.go::TestE2EP0048...0051`。
- 结果：排除。文档列出了 DB hint_text、no-outbox/no-audit、strict no-pending replay、5 action provenance、4 degrade paths 等断言，但测试只覆盖主干状态和少量字段。

### 假设 2：session event replay payload 已经满足 hint 隐私红线

- 验证方法：在 P0.048 中把 generated hint 文本加入 `practice_session_events.payload` 负向扫描。
- 结果：确认失败。`marshalAppendEventPayload` 会把 `AssistantAction.Hint` 写入 replay payload，导致 `practice_session_events.payload` 保存 hint 明文。

### 假设 3：A3 observability decorator 会为 callErr 自动写入 B1 error_code

- 验证方法：在 P0.051 中分别注入 `AI_PROVIDER_SECRET_MISSING` 和 `AI_PROVIDER_TIMEOUT`，检查 captured `ai_task_runs` row。
- 结果：确认失败。inner AI client 返回 shared `APIError` 但 `AICallMeta.ErrorCode` 为空时，decorator 写出的 row `error_code` 为空且 `validation_status` 未标记 invalid。

## 3 根因分析

- 根本原因：003 的 L2 证据过度信任 scenario 名称和 checklist 勾选，没有逐项对照 BDD checklist 的断言矩阵读取测试实现；同时 replay payload 复用了完整 response shape，没有对 `show_hint` 的敏感 hint 文本做持久化脱敏。
- 涉及的代码路径：
  - `backend/internal/store/practice/append_event.go`
  - `backend/internal/ai/aiclient/observability/decorator.go`
  - `backend/cmd/api/practice_http_scenario_test.go`
  - `test/scenarios/e2e/p0-048-*` 到 `p0-051-*`
- 之前没有发现的原因：focused `go test -run` 能证明命名测试执行，但测试本身没有覆盖 BDD checklist 声称的隐私、cross-action 和多失败路径断言；A3 decorator 的 callErr metadata fallback 没有 dedicated negative assertion。

## 4 修复方案

- `append_event.go` 持久化 replay payload 时对 `show_hint` 的 `AssistantAction.Hint` 置空，避免 `practice_session_events.payload` 保存 hint 明文。
- `selectSessionEventReplay` 在 replay `show_hint` 且 payload 中 hint 为空时，从 `practice_turns.hint_text` 按 `turnId` 取回 response hint，保留 replay 可用性，同时不把 hint 明文放进 session event payload。
- `observability/decorator.go` 在 Complete / Transcribe / Synthesize callErr 场景中，如果错误是 shared `APIError` 且 meta 未填 `ErrorCode`，自动补入 B1 `error_code` 并标记 `validation_status=invalid`。
- 补强 P0.048-P0.051 HTTP scenario：验证 assisted hint 持久化与隐私、strict no-pending replay、5 action provenance parity、4 类 graceful degrade 与 failed `hint_generate` task-run metadata。
- 2026-05-15 follow-up：`marshalAppendEventErrorPayload` 对 strict / unknown 409 finalized replay payload 只持久化 `requestFingerprint + error` sanitized envelope，禁止保存 `requestPayload`、turn id、answer text 或空 `result`。
- 2026-05-15 follow-up：`scripts/lint/backend_practice_legacy.py --phase all` 扩展覆盖 P0.048-P0.051 scenario runtime assets，避免 003 scenario 资产逃逸 legacy-negative gate。
- 2026-05-15 follow-up：P0.048-P0.051 `trigger.sh` 移除 `go test | tee` 假绿风险，保留真实 `go test` exit code；`verify.sh` 明确要求目标 test `--- PASS` 与 `ok ./cmd/api`，并拒绝 `FAIL` / `no tests to run`。

## 6 经验教训

- L2 review 不能只确认 scenario test 名称存在；必须把 BDD checklist 的每条声明断言映射到测试代码中的具体 assertion。
- replay payload 是隐私红线的一部分；当 response 需要返回敏感派生文本时，持久化 replay payload 应保存脱敏形态，并从业务表重建可 replay 字段。
- A3 observability decorator 必须在 inner client 返回 shared `APIError` 但 meta 为空时补齐 typed error metadata，否则运维表会出现 failed row 但无 B1 error_code 的半证据。
- shell wrapper 自身也是 BDD evidence surface；`go test | tee` 在 POSIX `sh` 下不能证明 `go test` 成功，verify 必须要求 `PASS` / package `ok` 并显式拒绝 `FAIL`。

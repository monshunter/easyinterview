# BUG-0146: target import parse rejected valid JD without company name

> 模块: workspace | 严重度: high | 状态: resolved
> 发现日期: 2026-07-09 | 关联 Commit: `fix(targetjob): accept valid JD parses without company names (BUG-0146)`
> 关联日志: [2026-07-09](../work-journal/2026-07-09.md)

## 1 现象

- 用户在 `/parse?...&targetJobId=...` 页面看到「JD 解析失败」。
- DevTools 中同一 `targetJobId` 的详情响应为 `analysisStatus="failed"`，`title=""`、`companyName=""`、`requirements=[]`。
- 原始 JD 文本是有效岗位描述，但没有提供明确公司名；重复解析在真实 provider 下仍可能进入 `AI_OUTPUT_INVALID`。

## 2 诊断过程

### 假设 1：详情接口或前端路由仍有 BUG-0142 的失败态读取问题

- 验证方法：检查本地环境状态、backend 日志、`target_jobs` / `target_job_sources` / `async_jobs` / outbox 记录，并使用同款 target id 请求详情。
- 结果：排除。详情接口能返回 200 和 `analysisStatus="failed"`；失败来自 parse runner，而不是详情读取 500 或前端鉴权。

### 假设 2：真实 provider 输出不可观测，无法定位具体无效字段

- 验证方法：查询 `ai_task_runs`，并检查 `cmd/api` TargetJob runtime wiring。
- 结果：确认。修复前 TargetJob parse path 没有使用 A3 observability decorator，失败 target 没有对应 `ai_task_runs` 记录，无法看到 schema validation 和 provider/model 摘要。

### 假设 3：parser 对公司名的非空要求过严

- 验证方法：给 TargetJob runtime 补 observability 后重新导入同一 JD；检查 `ai_task_runs` 与目标详情。
- 结果：确认。真实 provider 返回了有效 title、requirements 和分析字段，但 `companyName` 为空；当前 parser 把空公司名判为 `AI_OUTPUT_INVALID`，导致有效 JD 被标记 failed。

## 3 根因分析

- BUG-0145 修复后，parse success path 要求 `title` / `companyName` 都非空；这把「JD 本身未披露公司名」误判成 AI 输出无效。
- TargetJob `cmd/api` runtime 没有把 parse AI client 包进 A3 observability decorator，导致真实 provider 失败时没有 `ai_task_runs` 证据，延长了定位路径。
- provider 输出偶尔会把完整 JSON 包在 markdown code fence 中；validator 与 TargetJob decoder 需要接受这种单一 JSON fence，但仍必须拒绝 fence 前后有说明文字或多个 JSON 值，避免回退到 BUG-0095 的宽松解析。

## 4 修复方案

- TargetJob parser 只把空 `title` 视为 identity invalid；空 `companyName` 改为语言相关展示兜底，`zh-*` 使用 `未提供`，其他语言使用 `Unknown company`。
- A3 `outputschema` 增加严格的 `NormalizeJSONContent`：仅接受完整 markdown code fence 包裹的单一 JSON 值，不接受前后 prose 或多个 JSON 值；TargetJob decoder 复用同一 normalize 行为。
- `cmd/api` 的 TargetJob runtime 用 A3 observability decorator 包裹 parse AI client，并写入 `ai_task_runs`，保留 provider/model/status/validation 摘要以支持下一次真实 provider 诊断。
- 补 focused tests 覆盖 fenced JSON、prose 仍失败、缺公司名兜底、TargetJob runtime observability wiring。

## 6 经验教训

- Required prompt fields are not always required business facts. If a JD may legitimately omit a field, the parser should distinguish canonical identity requiredness from display fallback.
- Real-provider parse failures need task-run observability on every runtime path, not only on practice/report paths.
- JSON normalization must be narrow: accepting a full code fence is useful provider tolerance; stripping arbitrary prose would hide prompt/schema violations.

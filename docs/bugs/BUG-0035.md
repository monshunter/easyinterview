# BUG-0035: change-intake mutated main before branch guard

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-10 | 关联 Commit: `fix(change-intake): require branch guard before mutation`
> 关联日志: [2026-05-10](../work-journal/2026-05-10.md)

## 1 现象

- Issue-driven frontend fix started from `/change-intake`, but spec / plan / checklist and frontend files were edited while the worktree was still on `main`.
- `/implement` and `/tdd` have explicit branch resolution, but `/change-intake` was allowed to revise completed owner docs before that handoff.
- `git status` showed uncommitted implementation and documentation changes on `main`, creating a risk that future commits or branch switches would mix work into the default parent branch.

## 2 诊断过程

### 假设 1：只是当前会话忘记手动切分支

- 验证方法：读取 `.agent-skills/change-intake/SKILL.md` and compare it with `.agent-skills/implement/SKILL.md` branch resolution.
- 结果：排除。`/change-intake` only described matching, lifecycle handling, and in-place revision; it did not require a branch guard before mutation.

### 假设 2：`/implement` branch guard would still protect the worktree

- 验证方法：review the actual sequence for completed-plan remediation.
- 结果：排除。For completed plans, `/change-intake` can revise spec / plan / checklist before `/implement` starts, so `/implement` branch resolution happens too late.

### 假设 3：dirty changes can simply remain on the new branch without further rules

- 验证方法：inspect Git behavior for uncommitted changes after `git switch -c`.
- 结果：部分确认。Creating the feature branch preserves current changes and stops further edits on `main`, but the workflow still needs an explicit recovery rule and a contract test so future sessions do not repeat the mistake.

## 3 根因分析

- Branch policy was documented mainly around `/implement`, while `/change-intake` had authority to mutate owner docs before handing off.
- The skill separated "entry skill" from "delivery owner" but did not define which skill owns branch safety before in-place revision.
- Existing change-intake contract tests checked candidate routing and post-pass reconciliation, but not branch-safety wording.

## 4 修复方案

- Move the current dirty worktree to a feature branch after confirming `main...origin/main` was `0 0`.
- Add a `Branch Guard Before Mutation` section to `/change-intake` requiring `git status --short --branch`, fast-forward-only parent update before feature branch creation, and stop/ask behavior for unclear dirty worktrees.
- Add a global `AGENTS.md` Git branch rule covering every file-mutating entry skill, not only `/implement`.
- Add contract test assertions so `/change-intake` cannot silently lose the branch guard wording.

## 5 验证

- `git fetch origin main && git rev-list --left-right --count main...origin/main && git switch -c fix/frontend-dev-mock-runtime`: parent branch matched remote (`0 0`) and dirty changes were moved to the feature branch.
- `python3 -m pytest .agent-skills/change-intake/scripts/test_change_intake_skill_contract.py .agent-skills/change-intake/scripts/test_match_change_context.py`: 7 tests PASS.
- `python3 .agent-skills/skill-creator/scripts/quick_validate.py .agent-skills/change-intake`: PASS.
- `git status --short --branch`: current branch is `fix/frontend-dev-mock-runtime`, not `main`.

## 6 经验教训

- Any skill that can edit files needs its own branch gate; relying on a downstream implementation skill is too late when the entry skill revises docs first.
- Completed-plan in-place revision is a mutation, not just routing, so it must obey the same branch policy as code implementation.
- Branch recovery should preserve current-session changes, but unclear user-owned dirty content must stop and ask before stash, reset, checkout, or commit.

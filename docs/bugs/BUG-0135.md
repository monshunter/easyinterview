# BUG-0135: migration privacy matrix omitted idempotency records

> 模块: schema | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-06 | 关联 Commit: `fix(core-loop): close active doc pruning drift (BUG-0135)`
> 关联日志: `-`

## 1 现象

- Repo pruning review found `docs/spec/db-migrations-baseline/spec.md` still required `idempotency_records` to be hard-deleted during privacy cleanup.
- The executable B4 privacy matrix in `backend/internal/migrations/privacy.go` did not include `idempotency_records`.
- `TestPrivacyMatrixCoversEveryBaselineTableExactly` also omitted `idempotency_records`, so the focused gate was false-green.

## 2 诊断过程

### 假设 1: D-22 table pruning removed the table from current scope

- 验证方法：Read current migrations and migration contract tests for `idempotency_records`.
- 结果：排除。`idempotency_records` remains a current baseline table and is still created by migration SQL.

### 假设 2: Privacy matrix projection drifted from active B4 spec

- 验证方法：Compare `docs/spec/db-migrations-baseline/spec.md` §3.1.2 with `backend/internal/migrations/privacy.go` and the focused privacy matrix test.
- 结果：确认。The spec had the correct hard-delete requirement, while the executable matrix and test allowlist missed the table.

### 假设 3: Active B4 spec also retained obsolete D-22 table rows

- 验证方法：Search B4 active spec for `candidate_profiles`, `experience_cards`, and `debriefs` current matrix/table rows.
- 结果：确认。The active spec still described non-current profile/debrief tables as current baseline rows even though migrations and product-scope D-22 had removed them.

## 3 根因分析

- `backend/internal/migrations/privacy.go` was not derived from the current B4 spec, and its focused test duplicated the same incomplete table allowlist.
- D-22 pruning reconciled migration SQL and broad product-scope gates but left B4 spec count/matrix wording and the executable privacy projection out of sync.
- The previous gate only asserted the matrix's historical count, so missing `idempotency_records` did not fail until the test was changed to expect the current table.

## 4 修复方案

- Add `idempotency_records` to `PrivacyMatrix` with `hard_delete` disposition.
- Update `TestPrivacyMatrixCoversEveryBaselineTableExactly` to require `idempotency_records` and expect 27 current public baseline entries.
- Bump `db-migrations-baseline` spec/history to v1.24 and align current counts: 22 app tables, 25 app/auth support tables, public schema gate >=27.
- Remove non-current `candidate_profiles` / `experience_cards` / `debriefs` from the active B4 privacy matrix wording.
- Add product-scope pruning checklist evidence for the B4 reconciliation.

## 5 验证

- Red: `go test ./backend/internal/migrations -run TestPrivacyMatrixCoversEveryBaselineTableExactly -count=1` failed on missing `idempotency_records`.
- Green: `go test ./backend/internal/migrations -run TestPrivacyMatrixCoversEveryBaselineTableExactly -count=1` passed.
- Regression: `go test ./backend/internal/migrations -count=1` passed.
- Drift gates: `python3 scripts/lint/migrations_lint.py --repo-root .`, `python3 .agent-skills/sync-doc-index/scripts/sync-doc-index.py --check`, `make docs-check`, and `git diff --check` passed.

## 6 经验教训

- Privacy matrix tests must be derived from the current public baseline table set, not from a historical fixed count.
- Pruning work should reconcile executable projections and active contract specs in the same pass; otherwise completed deletion gates can leave a false-green matrix.

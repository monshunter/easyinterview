# BUG-0091: Postgres 18 dev volume layout made local dev stack unhealthy

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(local-dev-stack): repair Postgres 18 dev volume guard (BUG-0087)`
> 关联日志: `-`

> 合并 `main` 后，当前分支原 `BUG-0087` 编号与 `main` 的 async runner 记录冲突；本记录迁移为 `BUG-0091`，关联 commit subject 保留历史文本。

## 1 现象

- `/plan-code-review local-dev-stack/001-bootstrap repo --fix` 复核运行时证据时，`make dev-up` 拉起 Redis / MinIO 成功，但 Postgres 反复 unhealthy，最终 `container easyinterview-postgres-dev is unhealthy`。
- Postgres 容器日志先显示官方 Postgres 18 镜像拒绝旧 `/var/lib/postgresql/data` 挂载布局；compose 修正挂载点后，本机既有卷又因旧根目录数据与半初始化 `/18` 目录表现为 `mkdir: can't create directory '/var/lib/postgresql/18/': Permission denied`。
- 影响范围是本地开发依赖栈：干净 clone / 干净卷需要可一键启动；已有旧卷的开发机不能被自动删卷，但必须得到明确错误与 reset 指引。

## 2 诊断过程

### 假设 1：镜像拉取或 Docker registry 状态导致 Postgres 未启动

- 验证方法：重新执行 `make dev-up`，观察 `postgres:18-alpine` 已成功拉取并创建容器。
- 结果：排除。失败发生在容器启动和健康检查阶段，不是 registry 拉取失败。

### 假设 2：compose 仍使用 Postgres 18 不兼容的旧挂载点

- 验证方法：读取 `deploy/dev-stack/docker-compose.yaml`，并运行 `docker compose ... config` 检查 Postgres volume target。
- 结果：确认。compose 原本将 `easyinterview-pg-data` 挂到 `/var/lib/postgresql/data`，触发 Postgres 18 官方镜像的旧数据库布局保护。

### 假设 3：修正挂载点后，本机旧卷仍会阻止默认 `make dev-up`

- 验证方法：把挂载点改到 `/var/lib/postgresql` 后重跑 `make dev-up`，再只读检查 `easyinterview-pg-data` 内容。
- 结果：确认。既有卷根目录已有旧 `PG_VERSION` 和数据库文件，且存在半初始化 `/18` 目录；不能自动复用，也不能在未经确认时删除。

## 3 根因分析

- 根本原因：Postgres 18 官方 Docker 镜像把默认 `PGDATA` 调整为 major-version-specific 路径 `/var/lib/postgresql/18/docker`，但本地 dev stack 仍沿用旧 compose 挂载点 `/var/lib/postgresql/data`；计划和 README 只记录命名卷名，没有把 Postgres 18 的挂载路径和旧卷 preflight 固化为 gate。
- 涉及路径：
  - `deploy/dev-stack/docker-compose.yaml`
  - `deploy/dev-stack/Makefile`
  - `deploy/dev-stack/scripts/check-postgres-volume-layout.sh`
  - `deploy/dev-stack/README.md`
  - `docs/spec/local-dev-stack/spec.md`
  - `docs/spec/local-dev-stack/plans/001-bootstrap/plan.md`
  - `docs/spec/local-dev-stack/plans/001-bootstrap/checklist.md`
- 为什么之前没有发现：历史验证信任 checklist completed 状态和 compose 结构，没有执行当前镜像版本下的冷启动；旧卷场景也缺少显式只读检测，失败只能从 Postgres 日志里倒推。

## 4 修复方案

- 将 Postgres 命名卷挂载点从 `/var/lib/postgresql/data` 改为 `/var/lib/postgresql`，保留官方镜像 `PGDATA=/var/lib/postgresql/18/docker`。
- 新增 `deploy/dev-stack/scripts/check-postgres-volume-layout.sh`，在 `make dev-up` 启动前只读检查 `easyinterview-pg-data`：若发现卷根 `PG_VERSION`、旧 `/var/lib/postgresql/data/PG_VERSION` 或半初始化 `/var/lib/postgresql/18`，立即退出并提示用户备份后显式执行 `DEV_RESET_FORCE=1 make dev-reset`。
- Review follow-up：guard 容器复用已锁定并由 `make dev-pull` 覆盖的 `postgres:18-alpine`，不再引入额外未声明的 `alpine:3.20` 拉取依赖。
- 同步 `local-dev-stack` spec / plan / checklist / README / history / INDEX，明确 Postgres 18 volume layout 与旧卷处理边界。
- 放弃自动迁移或自动删除旧卷：本地开发数据可能包含用户内容，未获确认时不得清空。

- `check-postgres-volume-layout.sh` 临时目录聚焦断言：空卷通过；卷根 `PG_VERSION`、`data/PG_VERSION`、半初始化 `/18` 均失败并输出明确指引；有效 `/18/docker/PG_VERSION` 通过。

## 6 经验教训

- 依赖镜像 major 升级不能只同步 tag；必须反查官方镜像 runtime contract，尤其是 volume layout、默认 UID、entrypoint 初始化路径。
- 对有持久化命名卷的本地栈，冷启动验证应区分“干净卷通过”和“旧卷被安全阻断”，不能把删卷作为默认修复动作。
- Completed plan 的 L2 code review 必须执行真实 runtime gate；结构性 `docker compose config` 只能证明 YAML 可解析，不能证明当前镜像能启动。

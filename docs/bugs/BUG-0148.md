# BUG-0148: round assumptions ignored structured LLM-derived interview rounds

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-09 | 关联 Commit: `-`
> 关联日志: `-`

## 1 现象

- 三个不同的面试规划详情都显示完全相同的 R1-R4 轮次假设，例如 `HR 初筛 · 20m`、`技术一面 · 45m`、`技术二面 · 60m`、`经理面 · 40m` 及固定 focus 文案。
- Home 最近模拟面试卡片的轮次轨道也使用固定 `R1 Phone S... / R2 Technical / R3 Culture / R4 Final` 风格文案。
- 用户追问后明确产品预期：合理轮次为 2~5 轮；轮数、HR/技术/经理等类型、每轮时长和重点都应由 LLM 根据 JD、行业/公司性质、岗位级别、团队/业务上下文和招聘流程线索推断。
- 用户直观看到的现象像是后端 mock 数据或 LLM 没有真正生成差异化轮次，但实际影响范围包括后端 parse 输出合同、OpenAPI/fixtures、前端 Parse 详情、Home 最近卡片和共享导航上下文。

## 2 诊断过程

### 假设 1：后端返回的是 mock 或固定 LLM 结果

- 验证方法：检查本地环境配置、运行状态、数据库历史 `target_jobs.summary.interviewHypotheses` 和 `ai_task_runs` provenance。
- 结果：排除“纯 mock”。`VITE_EI_API_MODE=real` 指向本地后端；最近 TargetJob 的历史 `summary.interviewHypotheses` 内容不同，`ai_task_runs` 显示 `task_type=jd_parse`、`provider=deepseek`、`model_id=deepseek-v4-flash`、`validation_status=ok`。

### 假设 2：前端展示层没有消费后端 round data

- 验证方法：反查 Parse 详情、Home 最近模拟面试卡片和 `interviewContextFromTargetJob` 的 round 数据来源。
- 结果：确认。Parse 详情曾从 locale 静态 `parse.round*Focus` 构造轮次；Home card 曾维护独立 `DEFAULT_ROUNDS`；导航上下文曾生成静态 round id/name，导致真实 LLM 输出落库后没有进入相关页面和对象。

### 假设 3：后端合同只输出 string-only focus，无法表达轮数/类型/时长

- 验证方法：检查 `target.import.parse` prompt/schema、OpenAPI `TargetJobSummary`、backend parser 和 fixtures。
- 结果：确认。旧 `interviewHypotheses[]` 只能表达 focus 文案，不能承载 LLM 推断的 2~5 轮、round type/name 和 `durationMinutes`；即使前端消费该字段，也仍会保留固定卡片数量、固定 round name 和固定分钟数。

### 假设 4：UI 真理源和场景 gate 未覆盖相关页面一致性

- 验证方法：检查 `ui-design/` 原型、`docs/ui-design/`、owner plan/BDD 和 P0.016 场景资产。
- 结果：确认。旧 gate 主要覆盖 Parse 详情布局和只读语义，没有断言 Parse、Home recent rail、shared navigation context 必须共用同一个 TargetJob structured round mapper，也没有截图验收结构化轮次内容。

## 3 根因分析

- 根本原因：轮次规划没有被建模为 TargetJob 的结构化 LLM 输出对象。后端合同只保留 string-only hypotheses，前端又把“轮次名称/时长/焦点”当作 UI 静态文案处理，最终让真实 LLM 输出无法决定轮数、类型和时长。
- 涉及路径：
  - `config/prompts/target.import.parse/*`
  - `openapi/openapi.yaml`
  - `backend/internal/targetjob/parse_executor.go`
  - `frontend/src/app/screens/parse/ParseScreen.tsx`
  - `frontend/src/app/screens/home/MockInterviewCard.tsx`
  - `frontend/src/app/navigation/interviewContext.ts`
  - `ui-design/src/data.jsx`
  - `ui-design/src/screen-home.jsx`
  - `ui-design/src/screens-p0-complete.jsx`
- 为什么之前没有发现：历史测试证明了 R1-R4 卡片存在和布局正确，但没有提供差异化 structured round fixture，也没有负向断言禁止静态 fallback 在有后端数据时生效；Home recent rail、shared navigation context 和截图验收也没有被纳入同一个数据一致性 gate。

## 4 修复方案

- 将 `target.import.parse` prompt/schema 升级为 `interviewRounds[]`，要求 LLM 根据 JD、岗位级别、公司/行业性质、团队/业务上下文、招聘流程线索和同类岗位实践推断 2~5 轮。
- OpenAPI `TargetJobSummary.interviewRounds` 加 `minItems=2/maxItems=5`，并生成 Go/TS artifacts；backend parser 校验 2~5 条、合法 sequence/type、非空 name/focus 和合理 `durationMinutes`。
- 新增/收口 `buildTargetJobRoundAssumptions` shared mapper，由 `TargetJob.summary.interviewRounds[]` 派生 Parse card、Home recent rail 和 navigation context，不再补齐固定 4 轮或静态分钟数。
- UI 真理源同步：`ui-design/src/data.jsx`、`ui-design/src/screen-home.jsx`、`ui-design/src/screens-p0-complete.jsx` 使用保存的 structured rounds，并新增 contract test。
- Owner spec / plan / checklist / BDD 和 P0.016 场景资产原地修订，明确关联页面和对象必须消费同一份 TargetJob structured rounds，并要求 Playwright 截图验收。
- 替代方案是只修前端展示；已放弃，因为 string-only hypotheses 仍无法表达 LLM 推断出的轮数、类型和时长。

## 6 经验教训

- AI/LLM 输出只落库不等于已经进入用户体验；相关页面、列表卡片、route handoff 和共享对象都需要同源 mapper。
- 如果产品语义要求 LLM 推断轮数、类型和时长，后端 schema 必须结构化承载这些字段；前端不能用固定模板补齐业务对象。
- 对“看起来像 mock”的问题，先用 runtime mode、DB payload 和 provider provenance 区分数据源，再反查前端 consumer，避免误判后端。
- UI parity gate 不能只证明布局存在；对后端生成数据必须有差异化 fixture、负向断言和截图验收，禁止有数据时退回静态文案。

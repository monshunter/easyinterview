# BUG-0008: provider registry runtime bootstrap was only test-wired

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-05 | 关联 Commit: `fix(ai-provider): wire provider registry runtime bootstrap`
> 关联日志: [2026-05-05](../work-journal/2026-05-05.md)

## 1 现象

- L2 review 发现 `aiclient.New` 在非 test 环境只检查 registry/profile path 字符串，不实际读取 `AI_PROVIDER_REGISTRY_PATH` / `AI_MODEL_PROFILE_PATH`。
- `ResolveSelectedProviders` 只被单测直接调用，生产启动路径没有通过 A4 `SecretSource` 校验 active profile 选中的 provider primary/fallback secret。
- `config/ai-profiles/` 中 repo-tracked active P0 profiles 默认指向 `unit-test-stub`，而 `make lint-ai-profile-coverage` 仍返回 OK。
- profile hot reload 失败时保留旧快照，但 poller 丢弃 reload error，没有 warn 信号。

## 2 诊断过程

### 假设 1：runtime 启动路径已经在 API / worker 中完成 registry/profile materialization

- 验证方法：搜索 `aiclient.New`、`WithProfileResolver`、`ResolveSelectedProviders` 与 `AI_PROVIDER_REGISTRY_PATH` / `AI_MODEL_PROFILE_PATH` 调用点。
- 结果：排除。生产级 registry/profile materialization 没有可复用 bootstrap；`ResolveSelectedProviders` 的非测试调用缺失。

### 假设 2：profile coverage lint 已经能阻止 active profile 绑定 stub provider

- 验证方法：新增负向单测，让 `status=active` profile 指向 protocol 为 `stub` 的 `unit-test-stub` provider。
- 结果：确认。修复前 lint 返回 OK，说明 gate 只校验 provider ref 存在与 capability 对齐，没有校验 active profile 的 provider protocol 语义。

### 假设 3：profile reload 失败已有可观测信号

- 验证方法：让 profile loader poller 在初始快照后读到非法 YAML，检查旧快照和 warning 回调。
- 结果：确认。旧快照会保留，但 `Reload` error 被 poll loop 丢弃，测试和运维都拿不到失败信号。

## 3 根因分析

- 根本原因：003 plan 原有 gate 主要证明 registry/profile loader 的结构与单测 API 可用性，没有证明生产 runtime bootstrap 真实使用这些 truth sources。
- 涉及的代码路径：
  - `backend/internal/ai/aiclient/client.go`
  - `backend/internal/ai/aiclient/providerregistry/loader.go`
  - `backend/internal/ai/aiclient/profile/loader.go`
  - `scripts/lint/ai_profile_coverage.py`
  - `config/ai-profiles/*.yaml`
- 之前没有发现，是因为 checklist 1.2 / 3.1 / 3.4 的通过证据可以由 loader/provider 单测满足；coverage lint 也只验证 profile-provider 结构一致性，没有 active profile anti-stub 语义断言。

## 4 修复方案

- 选择方案：新增 `backend/internal/ai/aiclient/bootstrap` 作为生产 runtime bootstrap，统一读取 registry/profile path、校验 active profile 选中的 provider secret、拒绝非 test active stub profile，并通过 provider resolver materialize provider-ref adapters。
- 补强 `aiclient.Client`：保留测试可注入 provider map，同时支持 registry-backed `ProviderResolver` 解析 primary 与 fallback provider refs。
- 补强 profile loader：新增 `OnWarn`，poll reload 失败时保留旧快照并输出 warning。
- 补强 lint gate：`scripts/lint/ai_profile_coverage.py` 读取 provider protocol，拒绝 repo-tracked `status=active` profile 指向 `stub` provider。
- 迁移当前 active profile fixtures：P0 active defaults 从 `unit-test-stub` 切到 `default-openai-compatible`，保留 stub 仅用于单元测试 / 离线 mock fixture。

## 5 验证

- Red：`python3 -m pytest scripts/lint/ai_profile_coverage_test.py -k active_profile_uses_stub -q` 修复前失败，证明 active stub gate 缺失。
- Focused：`cd backend && go test ./internal/ai/aiclient/... -count=1` 通过。
- Focused：`python3 -m pytest scripts/lint/ai_profile_coverage_test.py scripts/lint/env_dict_test.py scripts/lint/ai_provider_terminology_test.py -q` 通过。
- Contract gates：`make lint-ai-profile-coverage`、`make lint-config`、context validation、`make docs-check`、`git diff --check` 通过。
- Global：`make lint`、`make test`、`make build` 通过。
- Negative search：`config/ai-profiles/` 中不再存在 `provider_ref: unit-test-stub` 或 `stub-chat-1` / `stub-embed-1` active defaults；剩余 stub 命中仅在测试 fixture 或 registry 测试 provider。

## 6 经验教训

- runtime contract 的完成证据必须包含真实 bootstrap / entrypoint 调用路径；只证明 loader 或 resolver 单体可用不等于生产路径已使用。
- drift gate 不能只验证结构一致性，还要编码 active/default 状态下的语义禁用项，例如 active profile 不得路由到 test-only provider。
- hot reload 的“保留旧快照”需要配套可观测信号，否则配置漂移只会被静默吞掉。

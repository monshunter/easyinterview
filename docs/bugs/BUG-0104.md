# BUG-0104: manual UAT account helper crossed into backend cmd before Mailpit boundary

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-26 | 关联 Commit: `fix(local-dev-stack): add mailpit for local auth testing (BUG-0104)`
> 关联日志: `-`

## 1 现象

- `e2e-scenarios-p0/002-manual-uat-real-provider-full-funnel` 的账号/session bootstrap 被实现为 `backend/cmd/devsession` + `backend/internal/devsession`。
- 该 helper 是 manual UAT 场景验收依赖，却出现在正式 backend cmd / internal 进程树中，容易被误认为产品 runtime 能力。
- 初始修复曾准备把账号/session bootstrap 挪到 `test/scenarios` Python helper，但这仍然绕过真实 passwordless flow；最终边界应由 `local-dev-stack` 提供 Mailpit，本地 UAT 通过 magic-link 登录，不再直接写 `sessions`。

## 2 诊断过程

### 假设 1：项目已有 Mailpit 或本地 mailbox，可直接作为账号验收入口

- 验证方法：检查 `deploy/dev-stack/README.md` 默认服务，并搜索 `deploy/dev-stack`、`backend`、`config`、`openapi` 中的 Mailpit / MailHog 引用。
- 结果：初始状态排除。默认 dev-stack 只有 Postgres / Redis / MinIO，C1 `DevMailSink` 只是进程内测试 sink，不能支撑人工 UI 收信登录。

### 假设 2：把账号 helper 放在 `backend/cmd` 是合理的 dev runtime 扩展

- 验证方法：对照 `test/scenarios/README.md`、`docs/development.md` §2 和用户反馈的场景测试边界。
- 结果：排除。该能力只服务 manual UAT 账号准备，属于 `test/scenarios/manual-uat` companion，不应进入正式 backend cmd / internal 包。

### 假设 3：把 helper 改成 `test/scenarios` Python 就足够

- 验证方法：对照用户反馈“测试验收账号也应属于 test/scenarios 范畴”以及本次新增要求“本地测试不依赖真实外部邮箱服务和真实邮箱账号”。
- 结果：排除。场景目录可以保存 runbook、材料和 shell/Python 编排，但不应通过直接 DB session bootstrap 替代被测 auth flow。账号入口应走 `startAuthEmailChallenge` -> Mailpit -> `verifyAuthEmailChallenge`。

## 3 根因分析

- 在补齐 manual UAT 账号材料时，把“真实 backend 被测进程”和“场景验收辅助依赖”混在一起，直接新增了 `backend/cmd/devsession`。
- 计划的 TDD 策略把账号 helper 写成 Go CLI，缺少 `test/scenarios` 只允许 shell/Python 场景工具的边界，也缺少“账号验收必须走真实 auth flow”的反向 gate。
- 当时 local-dev-stack 没有本地 mailbox，导致实现倾向于绕过邮件链路，而不是由 `local-dev-stack` owner 补齐 Mailpit 依赖。

## 4 修复方案

- 删除未提交的 `backend/cmd/devsession` 与 `backend/internal/devsession`，并删除后续尝试中的 `test/scenarios/manual-uat/full-funnel/scripts/bootstrap_account.py`。
- 在 `local-dev-stack/001-bootstrap` 原地追加 Mailpit revision：默认 `make dev-up` 启动 `mailpit-dev`（Web 8025 / SMTP 1025），`dev-doctor` 纳入 Mailpit `/readyz` 与 SMTP 端口探测。
- 在 backend auth 中新增 SMTP `DeliveryWriter`：`EMAIL_PROVIDER=mailpit` 时由 `cmd/api` 注册，使用 `auth_challenges` 查询收件邮箱、用 transient delivery secret 取 magic token，并投递到 Mailpit；`email_dispatch` payload 继续禁止 raw email / token / URL / 邮件正文入库。
- 修订 A4 env/config 字典、`deploy/dev-stack` 文档、`test/scenarios` 边界、manual UAT runbook/materials 与 e2e spec/plan/checklist/BDD：账号入口改为 synthetic `.example.test` 邮箱 + Mailpit magic-link。
- 增加 negative gates：`test ! -d backend/cmd/devsession && test ! -d backend/internal/devsession`、`find test/scenarios -name '*.go'` 无命中、`bootstrap_account.py` 不存在。

## 6 经验教训

- 场景验收依赖应优先落在 `test/scenarios` 或 `_shared`，不要因为需要真实后端数据就把辅助工具提升为正式 backend cmd。
- 对 manual UAT / BDD 材料要同时验证“真实 auth flow 可用”和“场景依赖没有越界进入正式 runtime”。
- `test/scenarios` 可以编排 shell/Python 场景，但不得把 direct session bootstrap 当作账号验收路径；本地账号验收缺邮箱时，应先回到 `local-dev-stack` / `backend-auth` owner 补齐 Mailpit/SMTP。

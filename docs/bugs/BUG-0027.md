# BUG-0027: targetjob L2 review exposed runtime gate and SSRF gaps

> 模块: workspace | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `fix(backend-targetjob): harden targetjob L2 runtime gates`
> 关联日志: [2026-05-08](../work-journal/2026-05-08.md)

## 1 现象

- `/plan-code-review backend-targetjob/001-targetjob-import-and-parse-bootstrap backend --fix` 复查 completed plan 时发现 BDD gate 不能代表真实运行链路。
- `urlfetch` 先解析并检查 public IP，随后默认 `http.Client` 在真实 dial 时重新走系统解析，存在 DNS rebinding / TOCTOU SSRF 窗口。
- `updateTargetJob` 的状态机校验在 service 层事务外完成，store 事务只锁 idempotency key，未锁目标 row 后重新读取当前状态。
- `E2E.P0.010` / `E2E.P0.011` / `E2E.P0.012` / `E2E.P0.013` 脚本实际运行包级 focused tests，不能证明 auth -> HTTP API -> cmd/api runtime 的真实 BDD 链路。

## 2 诊断过程

### 假设 1：URL policy check 已覆盖真实连接地址
- 验证方法：读取 `backend/internal/targetjob/urlfetch/fetcher.go` 的 `Fetch` / `checkURLPolicy` / default client 初始化。
- 结果：排除。policy check 使用注入 resolver，但真实 `http.Client.Do` 没有绑定已校验 IP，dial 阶段仍可能解析到私网或 metadata 地址。

### 假设 2：service 层状态机 preflight 足以防止非法迁移
- 验证方法：读取 `backend/internal/targetjob/service.go` 与 `backend/internal/targetjob/store.go` 的 update flow。
- 结果：排除。preflight 与 store update 不在同一事务内；不同 idempotency key 并发 PATCH 可基于旧状态通过校验，再覆盖当前状态。

### 假设 3：BDD 场景脚本已覆盖真实 API runtime
- 验证方法：读取 `test/scenarios/e2e/p0-010..013-*/scripts/trigger.sh` 与 `backend/cmd/api/main.go` wiring。
- 结果：排除。脚本只运行包级 `go test`；`cmd/api` 当前只注册 TargetJob handler/service/store，未组装 `target_import` drainer、F3 runtime client、A3 runtime client 与 `urlfetch`。

## 3 根因分析

- 根本原因：L2 前的 gate 偏向包级 focused test 与历史 checklist 状态，未把真实 dial path、事务内 row lock 和 HTTP scenario runtime wiring 作为 completed plan 的必要证据。
- 涉及的代码路径：
  - `backend/internal/targetjob/urlfetch/fetcher.go`
  - `backend/internal/targetjob/service.go`
  - `backend/internal/targetjob/store.go`
  - `test/scenarios/e2e/p0-010..013-*/scripts/trigger.sh`
  - `docs/spec/backend-targetjob/plans/001-targetjob-import-and-parse-bootstrap/*`
- 之前没有发现的原因：已有 tests 覆盖 policy precheck、合法状态迁移和包级 E2E surrogate，但没有构造 DNS rebind、stale concurrent transition 或 cmd/api runtime wiring 反证。

## 4 修复方案

- 为 default `urlfetch` transport 增加 policy-aware `DialContext`：真实拨号前重新解析目标 host、拒绝非 public IP，并拨到已校验 IP；同时关闭环境代理绕过。
- 将 `updateTargetJob` 状态机最终校验移入 `SQLStore` 事务路径，在 dedupe miss 和非 dedupe status update 时 `select ... for update` 锁定当前 row 后再校验。
- 重新打开 001 plan / checklist / BDD checklist，撤回旧 BDD completed 证据，新增 Phase 7 remediation gate，并把 `cmd/api` F3 runtime wiring 明确为 blocker。
- 将四个 TargetJob scenario trigger 标记为 proxy-only blocker，默认阻断包级 focused test 冒充真实 BDD 通过；仅允许显式设置 `ALLOW_TARGETJOB_PACKAGE_PROXY=1` 运行旧代理测试。

## 6 经验教训

- SSRF 防护必须约束最终 dial path；只在请求前做 DNS policy check 不足以防止 rebinding。
- 状态机校验必须和状态更新位于同一事务，并在 row lock 后基于当前 DB 状态判断。
- BDD gate 不能用包级 focused tests 替代真实用户链路；若只能先保留代理测试，脚本与索引必须明确 blocker，避免 completed 状态误导后续 review。

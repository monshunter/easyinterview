# BUG-0057: frontend practice event loop L2 review exposed recovery and parity gate drift

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-14 | 关联 Commit: `fix(frontend-practice): remediate practice event loop L2 findings`
> 关联日志: [2026-05-14](../work-journal/2026-05-14.md)

## 1 现象

- `$plan-code-review frontend-workspace-and-practice/002-practice-text-event-loop --fix` 复查时发现 plan 002 已处于 `completed`，但 `checklist.md` / `test-checklist.md` 仍保留 `partial` 证据注释，`bdd-checklist.md` 全部条目仍是未勾选状态。
- P0.046 / P0.047 scenario wrapper 没有执行专用错误恢复与隐私测试，缺少 `practiceErrors` / `practiceClientEventConflict` / `practiceConflict` / `practicePrivacy` 覆盖；正式 frontend 也没有 dedicated `frontend/tests/pixel-parity/practice.spec.ts`。
- `PracticeScreen` 对 append / complete 错误的分类与重试边界不足：AI timeout、409 mismatch、strict hint conflict、网络失败与 3 次失败 fallback 没有独立 runtime 断言；append 失败时输入保留与 retry 复用同一 `clientEventId` 缺少可执行证据。
- Practice runtime 仍使用旧 token 变量（如 `--ei-color-bgCard` / `--ei-color-ink*` / `--ei-mono`），Playwright computed style parity gate 会失效；AI transparency 还保留硬编码 `haiku-4.5` 文案，和隐私 / provenance 红线不一致。

## 2 诊断过程

### 假设 1：completed lifecycle 已经代表文档证据闭环

- 验证方法：直接读取 `docs/spec/frontend-workspace-and-practice/plans/002-practice-text-event-loop/{checklist.md,test-checklist.md,bdd-checklist.md}`。
- 结果：排除。主 checklist 与 test checklist 仍有 `partial` 证据，BDD checklist 虽然 Header 为 `completed`，但所有场景项未勾选。

### 假设 2：现有 Vitest / scenario wrapper 已覆盖失败恢复与隐私红线

- 验证方法：反查 `frontend/src/app/screens/practice/__tests__/`、P0.046 / P0.047 `scripts/trigger.sh` 和 `scripts/verify.sh`。
- 结果：排除。已有测试覆盖主流程与部分 hook，但 scenario trigger 没有绑定 dedicated recovery / privacy tests，无法证明 409 mismatch refresh、strict conflict warning、retry key reuse、raw text 泄漏红线。

### 假设 3：UI parity 只是缺少文档证据，不影响 runtime

- 验证方法：新增 practice pixel parity spec 后检查 DOM anchor、移动端布局、theme computed style 和截图非空；同时对 practice runtime 旧 token 做负向搜索。
- 结果：确认存在 runtime drift。多个 practice 组件仍使用旧 token 命名，需要迁移到当前 `ui-design` / app token 体系后才能让 computed style gate 稳定证明 parity。

## 3 根因分析

- 根本原因：Phase 5 close-out 把 “scenario 目录存在 / 历史 focused tests 通过 / checklist 标记 completed” 当成足够证据，没有反向验证每个 BDD wrapper 是否执行专用 runtime 断言，也没有把 `partial` / 未勾选状态作为 lifecycle 阻塞。
- 涉及的代码路径：
  - `frontend/src/app/screens/practice/PracticeScreen.tsx`
  - `frontend/src/app/screens/practice/components/*`
  - `frontend/src/app/screens/practice/__tests__/*`
  - `frontend/tests/pixel-parity/practice.spec.ts`
  - `test/scenarios/e2e/p0-046-practice-text-loop-failure-and-recovery/scripts/trigger.sh`
  - `test/scenarios/e2e/p0-047-practice-text-loop-complete-and-generating-handoff/scripts/trigger.sh`
  - `docs/spec/frontend-workspace-and-practice/plans/002-practice-text-event-loop/*`
- 之前没有发现的原因：全量 Vitest 能证明既有测试通过，但缺少 dedicated recovery / privacy / computed-style parity 测试；L2 review skill 也没有明确要求把 completed 文档里的 `partial`、未勾选 BDD 和 asset-readiness 语言视为阻塞 drift。

## 4 修复方案

- 在 `PracticeScreen` 中增加统一错误分类与 retry action 边界：AI timeout / network / 409 mismatch / strict hint conflict / session lost 分别映射到不同 UI 状态；append 失败保留输入，retry 复用同一 `clientEventId`；mismatch 触发 session refresh 与输入锁定；3 次 complete 失败后展示返回 workspace fallback。
- 去掉硬编码 AI model 文案，改为仅持有 `GenerationProvenance` 状态并以 pending profile 展示；隐私测试断言 raw answer / question / hint / provenance 不进入 URL、localStorage、console 或 handoff params。
- 新增 `practiceErrors.test.tsx`、`practiceClientEventConflict.test.tsx`、`practiceConflict.test.tsx`、`practicePrivacy.test.tsx`、`legacyNegative.test.ts` 和 `frontend/tests/pixel-parity/practice.spec.ts`，并把 P0.046 / P0.047 trigger 绑定到这些专用测试。
- 将 practice runtime 旧 CSS token 迁移到当前 token 命名，并用 pixel parity computed style / mobile layout / screenshot 非空断言固化。
- 原地修订 plan 002 checklist / test-checklist / bdd-checklist 到 v1.3，清除 `partial` / 未勾选状态，补入实际 gate 证据；同步 `plans/INDEX.md`。
- 更新 `.agent-skills/plan-code-review/SKILL.md`，把 completed 文档中的 `partial` / `pending` / 未勾选 BDD / asset-readiness 语言与未执行专用测试的 scenario wrapper 固化为 L2 阻塞 drift。

## 6 经验教训

- `completed` lifecycle 不是证据本身；任何 `partial`、`pending`、未勾选 BDD 或 asset-readiness 语言都必须在 L2 review 中升级为阻塞 finding。
- BDD scenario 目录和 INDEX 行只能证明资产存在，不能证明行为覆盖；必须读取 `trigger.sh` / `verify.sh`，确认它们实际执行专用 tests 并断言对应 failure / privacy / negative 条件。
- UI parity 不能只靠 DOM anchor；正式 frontend 迁移 design token 时必须有 computed style、responsive bounding box 与 screenshot 非空 gate，才能捕获旧 token 漂移。

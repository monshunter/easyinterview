# BUG-0041: auth user menu browser parity missed mobile overflow

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-11 | 关联 Commit: `fix(frontend-shell): close auth menu browser parity gap`
> 关联日志: `-`

## 1 现象

- 触发条件：按 `/plan-code-review frontend-shell/001-app-shell-auth-settings frontend` 复查 Phase 6 登录态与 `ui-design` parity 剩余空洞，并补上 authenticated user menu 的真实浏览器 Playwright 用例。
- 可观察错误：新增 browser-level 用例在 mobile project 下打开头像 dropdown 后，菜单左边界为 `-64.984375`，说明菜单从 viewport 左侧溢出；原有 jsdom / P0.032 只覆盖文本、状态和分流，没有捕捉真实 CSS 几何。
- 影响范围：登录态头像菜单在移动端真实浏览器布局与 `ui-design/src/app.jsx` 不一致；完整 `test:pixel-parity` 之前没有覆盖登录后的 dropdown geometry 和 logout flow。

## 2 诊断过程

### 假设 1：问题只是 ui-design CDN / golden preview 加载不稳定

- 验证方法：读取 `frontend/tests/pixel-parity/topbar.spec.ts`，把 ui-design 等待条件从 `load` / `document.querySelector` 调整为可见 `nav button`，并用 45s test timeout 单独复跑 topbar spec。
- 结果：部分历史 timeout 与 CDN/字体时序有关，但 authenticated menu 的 mobile overflow 是正式前端 CSS 几何问题，和 golden preview 加载无关。

### 假设 2：TopBar 组件逻辑缺少登录态状态流

- 验证方法：复用 Auth operation fixtures，在 Playwright 中 mock `/runtime-config`、`/me`、`/auth/email/start`、`/auth/email/verify`、`/auth/logout`，执行 login → avatar chip → dropdown → logout。
- 结果：登录态状态流可由 mocked API 驱动，logout 后也能回到非登录态；失败点集中在 mobile dropdown bounding box。

### 假设 3：移动端 CSS 仍沿用桌面右对齐策略

- 验证方法：读取 `frontend/src/app/topbar/topbar.css` 的 mobile media query，新增 Playwright geometry 断言：desktop 菜单右边与 chip 右边对齐，mobile 菜单左边与 chip 左边对齐且 `left >= -1` / `right <= viewport.width + 1`。
- 结果：确认。原 dropdown 在 mobile 下继续使用桌面 `right: 0` 对齐，chip 位于右侧窄空间时会把 220px 菜单推到 viewport 左侧之外。

## 3 根因分析

- 根本原因：Phase 6 修复只把 authenticated user menu 做到了 component/jsdom 与 P0.032 状态流层；真实浏览器 pixel gate 没有覆盖登录后 dropdown 的 bounding box 和 viewport containment。
- 涉及代码路径：
  - `frontend/src/app/topbar/topbar.css:571`
  - `frontend/tests/pixel-parity/topbar.spec.ts:75`
  - `frontend/tests/pixel-parity/topbar.spec.ts:213`
  - `test/scenarios/e2e/p0-006-ui-design-pixel-parity-gate/scripts/verify.sh`
- 为什么之前没有发现：
  - P0.032 的 jsdom 场景不计算 CSS layout，因此不能发现 mobile overflow。
  - P0.006 的 `topbar.spec.ts` 原本覆盖未登录 TopBar、语言/主题 controls 和 ui-design nav preview，没有进入 authenticated user menu。
  - E2E.P0.006 verify 脚本仍按 110 tests 口径验收，无法要求新增的登录菜单 browser parity 被执行。

## 4 修复方案

- 选择方案：在 `topbar.spec.ts` 增加 authenticated user menu browser parity 用例，用 fixture-backed mocked Auth API 驱动完整 login/logout，并在 desktop / mobile 下分别断言 dropdown 与 chip 的几何关系。
- CSS 修复：在 `@media (max-width: 720px)` 中把 `.ei-topbar-user-menu` 改为 `left: 0; right: auto;`，让移动端菜单从 chip 左侧展开并保持在 viewport 内。
- Gate 修复：P0.006 README / expected outcome / verify 脚本从 110 更新为 112 tests；frontend README、frontend-shell spec/history、001/003 owner plan/checklist 同步记录 authenticated user menu browser parity 和 Phase 6 operation matrix。
- 放弃方案：只放宽 Playwright 几何断言或只依赖 jsdom 场景。该方案无法证明移动端真实浏览器用户菜单和 `ui-design` 对齐。

## 6 经验教训

- Dropdown / popover / menu 这类浮层不能只靠 jsdom 或文本存在断言闭环；至少要有一个真实浏览器 gate 验证打开态、关闭态、bounding box、viewport containment 和关键动作。
- 当 pixel suite 新增用例后，场景 README、expected outcome 与 verify 脚本必须同步固定最新 passed count，避免完整 suite 实际扩展但 BDD gate 仍按旧口径放行。
- `ui-design` golden preview 依赖 CDN / 字体时，等待条件应锚定可见 DOM，而不是单纯等待 load 或节点存在。

# BUG-0127: host-run backend wildcard bind left resume page on stale 500

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-06-15 | 关联 Commit: `fix(local-dev-stack): bind host backend to loopback (BUG-0127)`
> 关联日志: [2026-06-15](../work-journal/2026-06-15.md)

## 1 现象

- 本地测试环境中，首次登录用户打开简历模块时，页面显示请求错误，后端响应为 500。
- 真实 API 复现中，新 synthetic 邮箱登录后，`GET /api/v1/resumes` 在 profile setup 前后均返回 500 `VALIDATION_FAILED`。
- 尝试按标准入口执行 `test/scenarios/env-redeploy.sh backend` 后，backend 启动失败，日志显示 `listen tcp :8080: bind: address already in use`。
- `netstat` 显示本机仍存在不属于 easyinterview 的 `172.18.0.6:8080` bridge listener；`lsof -nP -iTCP:8080 -sTCP:LISTEN` 未捕获该 listener，不能安全地由本仓库脚本清理。

## 2 诊断过程

### 假设 1：简历列表查询或迁移状态导致 500

- 验证方法：检查 migration 状态、当前 `resumes` 表结构、目标新用户 resume 行数，并直接执行同等 SQL 查询。
- 结果：排除。migration `version=15 dirty=false`；`resumes` 表存在当前扁平化字段；新用户 `count(*) = 0`；直接 SQL empty-list 查询可正常返回。

### 假设 2：本地 backend 没有按当前代码重启，前端命中 stale / down backend

- 验证方法：执行 `test/scenarios/env-redeploy.sh backend`，查看日志和端口监听；同时反查 8080 listener owner。
- 结果：确认。redeploy 失败于 `APP_LISTEN_ADDR=:8080` 的通配监听；Go 进程尝试绑定所有 interface，被无关 bridge listener 阻断，导致浏览器继续访问不可用或旧 backend。

### 假设 3：脚本应该杀掉所有 8080 listener

- 验证方法：比较 pidfile、`lsof` 和 `netstat` 输出。
- 结果：排除。无关 listener 不属于 `.test-output/local-dev/backend.pid`，且可能属于其它本地环境；local-dev script 不应杀掉不属于 easyinterview 的 Docker / Kind / bridge 进程。

## 3 根因分析

- 根本原因：`test/scenarios/_shared/scripts/local-dev-runtime.sh` 的 `restart_backend_runtime` 直接继承 `deploy/dev-stack/.env` 中的 `APP_LISTEN_ADDR=:8080` 启动 `go run ./backend/cmd/api`。在 Go runtime 中，`:8080` 会尝试监听所有 interface；当无关 bridge interface 已经监听 `172.18.0.6:8080` 时，本地 backend 无法启动。
- 涉及路径：
  - `test/scenarios/_shared/scripts/local-dev-runtime.sh`：host-run backend 启动前缺少 loopback listen address normalization。
  - `docs/spec/local-dev-stack/spec.md` 与 `plans/001-bootstrap`：原 Phase 8 只要求 endpoint/log/PID 可接管，没有明确通配监听地址必须收敛到 loopback。
- 为什么之前没有发现：既有 redeploy gate 只证明 backend 最终可被 `127.0.0.1:$port` 连接；没有在存在非 loopback bridge listener 的情况下约束 effective `APP_LISTEN_ADDR`，也没有静态 contract 锁定通配地址 normalization。

## 4 修复方案

- 选择方案：在 `local-dev-runtime.sh` 中新增 `backend_listen_addr`，当 `.env` 中 `APP_LISTEN_ADDR` 是 `:8080` 或 `0.0.0.0:8080` 这类通配地址时，启动 host-run backend 前导出 `APP_LISTEN_ADDR=127.0.0.1:${API_HOST_PORT:-8080}`；显式具体地址保留用户配置。
- 选择理由：本地 frontend 和场景脚本的 API base 本来就是 loopback；收敛监听面比清理未知 listener 更安全，也符合本地场景环境不接管 Kind / K8s / Helm 的边界。
- 放弃方案：杀掉所有 8080 listener。该方案可能误杀不属于 easyinterview 的本地 Docker / Kind / bridge 进程，违反本地环境边界。
- 修改文件：

## 6 经验教训

- 本地 host-run 应用的调试入口应默认只绑定 loopback；通配监听适合容器或明确公开服务，不适合作为 scenario env 的默认 host-run 行为。
- 环境脚本遇到端口冲突时，应优先判断 listener 是否属于本仓库 pidfile 或 loopback owner；不得把“杀掉所有端口占用”作为通用恢复手段。
- Redeploy gate 不能只检查“端口可连”，还要静态锁定 effective env 与监听地址，否则会漏掉 interface-level 冲突。

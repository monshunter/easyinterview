# BUG-0062: report L2 review exposed prompt schema, persistence, and privacy drift

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-05-16 | 关联 Commit: `fix(backend-review): remediate report L2 findings (BUG-0062)`
> 关联日志: [2026-05-16](../work-journal/2026-05-16.md)

## 1 现象

- L2 code review found that `report.generate` prompts requested `strengths` / `gaps`, while runtime unmarshaled `highlights` / `issues`.
- `report.question_assessment` prompts requested `dimension_scores`, while runtime expected `dimension_results`, `overall_status`, `strengths`, `gaps`, `recommended_framework`, and `review_status`.
- Store persistence allowed stale report jobs to continue when the `feedback_reports` row was no longer `generating`.
- `listTargetJobReports` could return `200` with an empty list for a target owned by another user instead of preserving target-job not-found semantics.

## 2 诊断过程

### 假设 1：F3 prompt output schema is not aligned with runtime parsing

- 验证方法：read `config/prompts/report.generate/*`, `config/prompts/report.question_assessment/*`, and `backend/internal/review/generate_report.go`; then add a preflight test that asserts required output keys.
- 结果：确认。The preflight test failed because the prompt bodies did not mention the runtime-required `highlights`, `issues`, `dimension_results`, `overall_status`, and review status fields.

### 假设 2：stale report status updates are not transactional gates

- 验证方法：add integration tests that set `feedback_reports.status='ready'` before `PersistReport` / `PersistReportFailure`.
- 结果：确认。Both calls returned nil before the fix, meaning later inserts and async job updates could proceed after a zero-row report status update.

### 假设 3：cross-user target listing is covered by existing BDD tests

- 验证方法：read `backend/cmd/api/reports_http_scenario_test.go` and `backend/internal/store/review/get_feedback_report.go`.
- 结果：排除。Existing BDD tests covered cross-user `GET /reports/{reportId}` but did not assert cross-user `GET /targets/{targetJobId}/reports`.

## 3 根因分析

- 根本原因：L2 gates covered many report-generation layers, but they under-asserted prompt output schemas, store row-count state transitions, and target ownership for report list reads.
- 涉及的代码路径：
  - `backend/internal/review/generate_report.go`: question assessment normalization now maps `score_level` into the wire `status` when the model omits status.
  - `config/prompts/report.generate/*` and `config/prompts/report.question_assessment/*`: prompt bodies now describe the actual runtime output contract.
  - `migrations/000002_seed_baseline_prompt_rubric_versions.up.sql`: seed prompt bodies and template hashes now match prompt truth source.
  - `backend/internal/store/review/persist_report.go` and `backend/internal/store/review/persist_failure.go`: report and async job updates now require one affected row inside the transaction.
  - `backend/internal/store/review/get_feedback_report.go`: target report listing now checks `target_jobs` ownership before listing reports.
  - `backend/cmd/api/reports_http_scenario_test.go`: BDD HTTP scenario now asserts cross-user target list privacy.
- 为什么之前没有发现：the original checklist relied on prompt lint, lower-level parser tests, and report read scenarios, but lacked negative schema-key, row-count rollback, and target-list ownership assertions.

## 4 修复方案

- Add prompt preflight assertions for the exact output keys consumed by report generation and question assessment runtime code.
- Update prompt Markdown, YAML hashes, and seed migration rows together so registry truth source and database bootstrap stay aligned.
- Normalize assessment `score_level` fallback to `needs_work` / `meets_bar` / `strong`.
- Require one affected row for `feedback_reports` and `async_jobs` updates in success and failure persistence transactions.
- Check target ownership before `ListTargetJobReports`, returning `REPORT_NOT_FOUND` for cross-user or missing target IDs.
- Extend HTTP scenario tests to cover cross-user target report listing and prevent report ID leakage.

## 6 经验教训

- Prompt preflight tests should assert output schema keys, not only registry resolution and safe input substitution.
- Persistence tests for state-machine updates should include zero-row update rollback cases, especially when later writes occur in the same transaction.
- Report list privacy needs target ownership assertions in addition to report ID ownership assertions.

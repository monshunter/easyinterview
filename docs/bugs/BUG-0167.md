# BUG-0167: P0.058 report evidence fixture omitted required practice reply state

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-14 | 关联 Commit: `fix(interview): close turn UX and report navigation (BUG-0167, BUG-0168, BUG-0169)`
> 关联日志: [2026-07-14](../work-journal/2026-07-14.md)

## 1 现象

- 清空本地数据并按当前 baseline migration 重建环境后，P0.058 的 report backend evidence exact test 在插入 user message 时触发 `practice_messages_client_id_check`。
- 同一测试在旧本地数据库上未暴露该约束错误，导致场景 fixture 与当前 durable reply-state schema 的漂移直到干净环境才被发现。
- 失败阻断报告失败/恢复场景验收，但不影响生产请求路径，严重度为 `medium`。

## 2 诊断过程

### 假设 1：重建后的 migration 或约束本身错误

- 验证方法：读取 baseline DDL，并运行 migration lint、当前 practice store transition tests 与真实 PostgreSQL concurrency tests。
- 结果：排除。当前约束正确要求带 `client_message_id` 的 user message 同时具备 reply status、正数 generation，并按状态约束 lease；生产 store 路径与迁移测试均通过。

### 假设 2：P0.058 报告 fixture 仍按旧 message 形态写入

- 验证方法：反查 `report_backend_evidence_e2e_test.go` 的 raw SQL insert，与 `practice_messages_client_id_check` 逐字段比较。
- 结果：确认。fixture 写入了 `client_message_id` 与 `reply_status='complete'`，但遗漏非空正数 `reply_generation`。

## 3 根因分析

- durable retry revision 给 user message 增加了联动 CHECK：带 client identity 的行必须保存完整 reply state。
- P0.058 的跨模块测试 fixture 直接写 SQL，没有复用 production store；schema 改动时只迁移了 practice owner tests，report scenario fixture 没有被同一 source contract 反查。
- 旧本地数据库仍含上一轮 DDL，先暴露了无关的历史 schema 漂移；只有执行用户授权的清库重建后，当前约束与 fixture 的真实不一致才稳定复现。

## 4 修复方案

- 在 P0.058 report backend evidence fixture 的 user message insert 中显式加入 `reply_generation=1`，使 complete reply state 与当前 baseline constraint 一致。
- 保留 raw SQL fixture，因为该场景需要跨 report/store 边界构造精确持久化证据；未通过放宽数据库 CHECK 或写入 `NULL` 绕过当前业务不变量。
- 继续以真实 PostgreSQL exact test 和 P0.058 四阶段 wrapper 作为回归门禁。

## 5 验证

- RED：干净 baseline 数据库上，四个 store subcase 均因 `practice_messages_client_id_check` 失败。
- GREEN：修复后 focused store exact test 的四个 subcase 全部通过。
- P0.058 最终 setup、trigger、verify、cleanup 全部退出 0；frontend 8 files / 82 tests、backend `internal/review`、`internal/store/review`、`internal/api/reports` 三个 exact packages 均 PASS。
- 保留的 backend evidence artifact SHA-256 为 `454dd02e68ca9d69f88c59ca0f2ac64478bc2070b86db4c4e912ee4584d7c1f3`。

## 6 经验教训

- 新增必填列或联动 CHECK 后，必须反查所有跨 owner raw SQL scenario fixtures，不能只运行 schema owner 的 store tests。
- 干净 migration rebuild 是发现旧本地 DDL 掩盖当前 fixture 漂移的必要门禁。
- 本问题可归入 PATTERNS 模式 11 的 schema/current fixture 对齐范畴；本轮不新增平行模式。

# BUG-0151: archived TargetJob import jobs retried failure cleanup

> 模块: workspace | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-09 | 关联 Commit: `fix(targetjob): terminate archived target imports (BUG-0151)`
> 关联日志: `-`

## 1 现象

- Code review 发现 `archiveTargetJob` 会把 TargetJob 软删除，但已存在的 `target_import` async job 仍可能在排队或重试后被 worker claim。
- `ParseExecutor.Handle` 读取 TargetJob 时通过 `deleted_at is null` 过滤；目标已归档后会返回 `ErrTargetJobNotFound`。
- 修复前该错误进入通用 `p.fail()`，随后 `CompleteParseFailure` 也因 `deleted_at is null` 过滤不到行而返回 `ErrTargetJobNotFound`，最终被转换成 retryable `TARGET_IMPORT_FAILED`，可能让 job 重试直到 dead。

## 2 诊断过程

### 假设 1：归档 store 会取消或完成 pending import job

- 验证方法：反查 `backend/internal/targetjob/store.go::ArchiveTargetJob`。
- 结果：排除。归档事务只更新 `target_jobs.status/deleted_at` 并写 archive dedupe marker，不取消既有 `target_import` job。

### 假设 2：parse worker 缺失目标时会终态退出

- 验证方法：反查 `backend/internal/targetjob/parse_executor.go::Handle`，并新增 focused reproducer `TestParseExecutor_MissingTargetIsTerminalWithoutFailureCleanup`。
- 结果：确认缺口。红测显示 outcome 为 `TARGET_IMPORT_FAILED`，没有返回 `TARGET_JOB_NOT_FOUND` 终态。

### 假设 3：失败清理能容忍已归档行

- 验证方法：反查 `CompleteParseFailure` SQL。
- 结果：排除。该路径执行 `delete from target_jobs where id = $1 and deleted_at is null`，归档行会返回 0 affected rows，进而返回 `ErrTargetJobNotFound`。

## 3 根因分析

- 根本原因：Phase 12 只验证了归档 handler/service/store 与 read-side soft-delete，未覆盖“归档发生在 `target_import` job 排队或重试期间”的 async lifecycle 边界。
- 涉及路径：
  - `backend/internal/targetjob/parse_executor.go`
  - `backend/internal/targetjob/pipeline_test.go`
  - `backend/internal/targetjob/store.go`
- 为什么之前没有发现：归档 focused tests 验证 API 与 soft-delete contract，parse failure tests 验证可见 failed target 的清理，两组测试没有组合出 archived/missing target 被 worker 读取的交叉场景。

## 4 修复方案

- 在 `ParseExecutor.Handle` 入口识别 `ErrTargetJobNotFound`，直接返回 non-retryable `TARGET_JOB_NOT_FOUND`。
- 该路径不调用 `CompleteParseFailure`，避免对已归档行写二次 `target.analysis.failed` 或把 cleanup failure 转成 retryable job outcome。
- 增加 focused 回归测试，断言 missing/archived target 不触发失败清理、没有 failed outbox payload，并以 terminal `TARGET_JOB_NOT_FOUND` 结束。
- 同步 `backend-targetjob/001` spec / plan / checklist，把 archive 与 queued import 的交叉 gate 固化到 owner 文档。

## 5 验证

- `go test ./backend/internal/targetjob -run TestParseExecutor_MissingTargetIsTerminalWithoutFailureCleanup -count=1`：修复前 RED，修复后 PASS。
- `go test ./backend/internal/targetjob -run 'TestParseExecutor|TestSQLStore_ArchiveTargetJob|TestService_ArchiveTargetJob|TestHandler_ArchiveTargetJob|TestSQLStore_CompleteParseFailure' -count=1` PASS。
- `go test ./backend/internal/targetjob -count=1` PASS。

## 6 经验教训

- 对软删除类 user action，除了 read-side 隐藏和 API 幂等，还要检查仍在队列中的 async job 是否会把“用户主动移除资源”误判成可重试系统失败。
- 本问题适合在 owner checklist 中保留交叉测试 gate；模式库已有异步/runtime wiring 类检查，本次不新增重复模式。

# BUG-0160: conversation simplification review exposed lifecycle, retry, score, and scenario drift

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-07-12 | 关联 Commit: `fix(practice): close conversation review regressions (BUG-0160)`
> 关联日志: -

## 1 现象

- 会话完成与仍在进行的 AI 回复并发时，完成事务可先把 session 标为 `completing` 并创建报告任务，但迟到的消息提交随后仍会插入 assistant message 并把 session 无条件改回 `running`。
- PracticeScreen 把所有本地错误的 Retry 都路由到消息发送；完成失败会误走 `send`，初始加载失败则没有可用的刷新入口，发送/加载期间 Finish 仍可点击。
- `report.generate` 没有声明候选维度分数范围，运行时却按 2/3/4 阈值计算 readiness；缺失、重复和范围外分数仍会被持久化为成功报告。
- P0.046/P0.047/P0.056/P0.058 只覆盖部分 happy-path marker，且场景契约测试仍引用简化后已删除或改写的场景与 prompt feature。

## 2 诊断过程

### 假设 1：完成事务本身会阻止迟到回复覆盖终态

- 验证方法：沿 `CompleteSession` 与 `CommitPracticeMessage` 的事务顺序反查 SQL，并用 sqlmock 模拟 completion 已先提交的状态。
- 结果：排除。消息提交只锁 user message，插入回复后无条件更新 session；没有检查当前状态，也没有检查更新行数。

### 假设 2：前端错误 banner 能根据失败操作自动选择重试函数

- 验证方法：让 loader 和 completion 分别首次返回 503，再点击同一个 Retry，并记录 generated client 方法调用。
- 结果：排除。组件只保存错误文本，没有保存错误来源；`error ? send : undefined` 把所有本地错误绑定到消息发送。

### 假设 3：provider 输出 schema 已约束报告分数语义

- 验证方法：检查 prompt、output schema、rubric payload、readiness 计算和持久化前验证，并注入缺失、重复、0.9 与 5.1 分数。
- 结果：排除。schema 只有 `number`，rubric payload 又移除了 evaluator score levels；运行时没有候选 1.0-5.0 的独立验证边界。

## 3 根因分析

- 连续会话简化保留了“AI 调用在事务外”的正确结构，但 commit 阶段没有重新声明可变 session 状态，导致 completion 与 late reply 之间缺少最终原子仲裁。
- PracticeScreen 用单一字符串合并 loader/message/completion 三种失败，却没有携带 operation source；Finish 的 disabled 条件也只覆盖 completion loading 和 completed。
- 候选报告分数与 evaluator rubric 阈值是两套量纲，prompt/schema 未显式区分，运行时又默认信任 provider 输出，因此 readiness 的阈值没有稳定输入域。
- 场景脚本和 Python contract tests 没有随着删掉 question/hint 场景与 prompt feature 同步迁移，历史 marker 因而无法证明当前失败路径。

## 4 修复方案

- `CommitPracticeMessage` 使用带 `running/waiting_user_input` 条件的 session 更新并检查 `RowsAffected`；未命中时返回 `ErrSessionConflict`，让同一事务回滚 assistant insert，服务层映射为 `PRACTICE_SESSION_CONFLICT`。
- PracticeScreen 记录 `message` / `completion` 错误来源，loader error 直接调用 `refresh`；加载、发送、暂停、完成请求和非可变 session 状态统一禁用 Finish。
- 为候选维度分数声明 1.0-5.0 schema bounds 与 prompt 语义，明确 evaluator 0.0-1.0 阈值是独立质量量纲；持久化前拒绝未知、缺失、重复和越界维度。
- 强化 P0.046/P0.047/P0.056/P0.058 的 named-test marker，并把 P0.070/P0.072 与 prompt-lint 负例迁移到当前 service/prompt 契约。
- 未采用“完成后忽略迟到回复但保留已插入 message”的方案，因为它会留下 session/report snapshot 不一致的数据。

## 6 经验教训

- 事务外调用 AI 时，reserve 与 commit 两端都必须检查业务可变状态；只在 reserve 阶段验证无法抵御跨事务终态竞态。
- 共享错误 UI 必须保留失败 operation 的身份，不能仅凭错误文案选择 retry action。
- 数字阈值进入业务派生前必须有显式单位、范围与 fail-closed 验证，特别是同一 feature 同时存在候选评分和 evaluator 评分时。
- 删除场景或 prompt feature 时，场景 registry、named-test verifier 与 tooling negative fixtures 都属于必须迁移的消费者。

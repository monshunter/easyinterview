# BUG-0046: backend upload L2 review exposed size privacy and live gate gaps

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-12 | 关联 Commit: `fix(backend-upload): harden upload privacy and live gates`
> 关联日志: `-`

## 1 现象

- L2 code review 发现 upload presign 只校验客户端声明的 `byteSize`，真实 MinIO PUT 路径没有在 register 阶段核对对象实际大小，客户端可声明小文件但上传更大的对象。
- `DELETE /api/v1/me` 创建的 `privacy_delete` job 没有在 runtime drainer 中挂入 upload deleter，已上传的 `file_objects` 行和对象存储 blob 可能在账号删除后残留。
- `createUploadPresign` 复用通用 24h idempotency TTL；当 signed URL 已过期但 idempotency 记录未过期时，重试会 replay 旧 `uploadUrl` / `expiresAt`。
- privacy deletion 中 DB hard delete 与 audit tombstone 分两步执行；若 tombstone 写入失败，`file_objects` 行已删除且不可重试补写。
- E2E.P0.033 在缺少 live `DATABASE_URL` / `OBJECT_STORAGE_*` 时仍可能因为 integration tests `t.Skip` 而被标记 PASS。

## 2 诊断过程

### 假设 1：register 阶段已用对象存储事实校验上传大小

- 验证方法：审查 `ObjectStore` interface、MinIO provider 与 `RegisterFileObject` store/service 路径，并补充 size mismatch red test。
- 结果：排除。旧路径只有 `Exists`，没有 stat size，`RegisterUploaded` 无法比较 `file_objects.byte_size` 与对象存储实际 size。

### 假设 2：privacy_delete runtime 已调用 upload deleter

- 验证方法：审查 `cmd/api` target job runtime 构建、`targetjob.Drainer` handler map 与 `backend/internal/privacy/runner`。
- 结果：排除。upload service 只在 upload route 局部构建，privacy drainer 未注册 `privacy_delete` handler。

### 假设 3：presign idempotency replay 生命周期与 signed URL 一致

- 验证方法：审查 `buildUploadRoutes` 的 idempotency middleware options 与 upload presign TTL config。
- 结果：排除。旧 middleware 使用默认 24h TTL，长于 `upload.presignTTLSeconds`。

### 假设 4：privacy deletion DB row 与 audit tombstone 具备原子性

- 验证方法：审查 `DeleteFileObjectsForUser` 中 repository 调用顺序，并用 sqlmock 补充 audit failure rollback test。
- 结果：排除。旧路径先 `HardDelete` 再 `InsertAuditTombstone`，tombstone 失败后 row 已丢失。

### 假设 5：BDD gate 会拒绝 live integration skip

- 验证方法：审查 E2E.P0.033 `trigger.sh` / `verify.sh` 与 integration-tag skip 行为。
- 结果：排除。`go test` skip 仍返回 0，verify 只看测试名，无法区分真实 live evidence 与 skip。

## 3 根因分析

- Upload completion contract 只把对象存在性作为完成条件，没有把 signed request 的 declared size 与对象存储实际 size 绑定到 register 临界区。
- Privacy delete 的局部 runner 存在，但 runtime job drainer 没有把 `privacy_delete` job type 与 upload deleter 组合，导致 `DELETE /api/v1/me` 的真实路径不完整。
- Idempotency middleware 默认 TTL 适合普通 mutation，但 presign response 携带会过期的 signed URL，需要 per-route TTL 覆盖。
- Audit tombstone 被当成删除后的附加记录，而不是 DB 删除事务的一部分，瞬时写入失败会变成不可恢复的数据治理缺口。
- Scenario scripts 把 integration-tag 测试名当作 live evidence，没有对 skip 文本和缺失 env 做失败断言。

## 4 修复方案

- 扩展 `ObjectStore` contract，新增 `Stat` / `ObjectInfo.Size`，MinIO 与 filesystem provider 返回对象实际 size；`RegisterFileObject` 在 store row lock 内调用 stat callback，missing 或 size mismatch 均返回 `VALIDATION_FAILED`，不标记 `uploaded`。
- 将 upload service 保留在 `buildUploadRoutes` 返回值中，并在 `buildTargetJobRuntime` 注册 `privacy_delete` handler，账号删除 job 通过 `privacyrunner.Runner` 调用 `DeleteFileObjectsForUser(userId)`。
- 为 idempotency middleware 增加可观测 `TTL()`，并让 upload presign route 的 idempotency TTL 使用 `upload.presignTTLSeconds`，避免过期 signed URL 被 replay。
- 新增 `HardDeleteWithAuditTombstone`，在同一 DB transaction 内写 audit tombstone 与 hard delete；audit 写入失败时 rollback，row 保留可重试。
- 强化 E2E.P0.033 scripts：缺少 live DB / object storage env 时 trigger 直接失败；verify 检测 `--- SKIP` 与 live integration skip 文本后失败。

## 6 经验教训

- Presign upload 的完成条件必须绑定对象存储事实，不应只相信客户端声明或 object exists。
- 账号删除链路必须用 runtime drainer / job handler 测试证明，而不是只证明局部 deleter 可调用。
- 会过期的 response body 不应使用比业务 TTL 更长的 idempotency replay 窗口。
- 隐私删除中的审计 tombstone 与实体删除应作为同一事务边界设计。
- BDD 场景不能用 skipped integration tests 作为 live evidence；verify 必须显式拒绝 skip。

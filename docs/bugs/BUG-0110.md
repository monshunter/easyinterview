# BUG-0110: real-provider UAT bypassed the standard scenario runner

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-27 | 关联 Commit: `fix(test): unify real-provider hybrid scenario (BUG-0110)`
> 关联日志: [2026-05-27](../work-journal/2026-05-27.md)

## 1 现象

- 用户指出真实前后端联调环境和本地场景测试环境不应割裂；测试框架的第一执行者应是 AI Agent，然后才是人工或浏览器 Agent。
- `E2E.P0.100` 真实 provider 全漏斗 UAT 曾放在 `test/scenarios/manual-uat/` companion 目录，未作为标准 `e2e` 场景登记到 `test/scenarios/e2e/INDEX.md`。
- 结果是 `$scenario-run -i E2E.P0.100` 无法统一发现、准备环境、产出 result artifact 或交接人工证据，用户体验和执行证据链被拆成两套入口。

## 2 诊断过程

### 假设 1：`E2E.P0.100` 已经登记为标准 e2e 场景，只是名称不一致

- 验证方法：读取 `test/scenarios/e2e/INDEX.md`、`test/scenarios/e2e/README.md` 与 `test/scenarios/manual-uat/`。
- 结果：排除。`E2E.P0.100` 不在 e2e INDEX 中，真实 provider UAT 资产位于独立 `manual-uat` companion 目录。

### 假设 2：独立 manual UAT 目录仍可复用统一环境 preflight

- 验证方法：读取 `test/scenarios/README.md`、`.agent-skills/scenario-run/SKILL.md` 与目标 runbook。
- 结果：排除。框架文档把人工 UAT 作为 companion 入口描述，`scenario-run` 没有要求先运行共享环境 setup/verify，也没有定义 `MANUAL_REQUIRED` 作为 hybrid 结果状态。

### 假设 3：只更新文档索引即可修复

- 验证方法：反查标准场景目录契约，包括 `README.md`、`data/`、`scripts/setup.sh`、`trigger.sh`、`verify.sh`、`cleanup.sh` 和 result artifact。
- 结果：排除。缺少可由 AI Agent 先执行的标准四段脚本、统一输出目录和缺真实凭证时的可解释结果。

## 3 根因分析

- 根本原因：真实 provider UAT 被建模为框架外人工 companion，而不是标准 scenario 的 hybrid 执行模式。
- 涉及路径：
  - `test/scenarios/manual-uat/`
  - `test/scenarios/e2e/INDEX.md`
  - `test/scenarios/README.md`
  - `test/scenarios/e2e/README.md`
  - `.agent-skills/scenario-run/SKILL.md`
  - `scripts/lint/scenario_env_contract_test.py`
  - `docs/spec/e2e-scenarios-p0/plans/002-manual-uat-real-provider-full-funnel/*`
- 之前没有发现，是因为历史验收把“人工可执行 runbook”当成真实 provider UAT 的完成形态，没有把 AI Agent first-run、共享环境 preflight 和统一 result artifact 作为场景框架不变量。

## 4 修复方案

- 选择方案：保留原 owner plan，原地追加 Phase 5，把 `E2E.P0.100` 迁入标准 `e2e` hybrid 场景，而不是新建 sibling plan。
- 新增/迁移 `test/scenarios/e2e/p0-100-real-provider-full-funnel-hybrid/`：
  - 保留 runbook、checklist、env template 与 synthetic materials。
  - 补齐 `data/seed-input.md`、`data/expected-outcome.md`。
  - 补齐 `scripts/setup.sh`、`trigger.sh`、`verify.sh`、`cleanup.sh`。
- 删除独立 `test/scenarios/manual-uat/` 入口，并在 `test/scenarios/e2e/INDEX.md` 登记 `E2E.P0.100 | hybrid | Ready`。
- 更新 `test/scenarios` README 与 `scenario-run` skill：场景运行前先执行共享环境 setup/verify；hybrid 场景由 AI Agent 先运行，缺本地真实 provider env 或浏览器证据时写出 `MANUAL_REQUIRED`，不误报 PASS 或 ERROR。
- 新增 lint contract，防止 `E2E.P0.100` 再次脱离标准 e2e 资产集。

## 6 经验教训

- 人工 UAT 也应先建模为标准场景；人工只补齐自动化无法无密钥完成的真实浏览器或真实 provider 证据。
- `MANUAL_REQUIRED` 是 hybrid 场景的有效中间结果，不应被混同为 PASS，也不应退化成框架 ERROR。
- 场景运行入口必须先证明环境已准备好；具体场景脚本只负责场景数据、执行证据和自有 cleanup。

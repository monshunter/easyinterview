# BUG-0176: settings review exposed fixture auth and failure-evidence gaps

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-15 | 关联 Commit: `fix(settings): close review regressions (BUG-0176)`
> 关联日志: -

## 1 现象

- 分支审查发现默认 fixture-backed client 在 `DELETE /me` 返回 `202` 后仍保持 authenticated；Settings 随后的 `refreshAuth()` 得到 `200`，页面无法按真实后端语义回到 signed-out Home。
- `E2E.P0.101` 使用包含完整 synthetic email 的 value-bearing Playwright matcher；一旦断言失败，list reporter 会先把值写到终端和 `trigger.log`，而 `set -euo pipefail` 会让后置 `verify.sh` 无机会拒绝该证据。
- 影响默认开发预览中的账号删除闭环，并使失败路径存在完整 synthetic account email 留存风险。

## 2 诊断过程

### 假设 1：Settings 成功分支没有刷新认证状态

- 验证方法：读取 Settings 删除状态机及现有 component tests，确认 `202` 后调用 `refreshAuth()` 并以 `/me` 401 作为 signed-out 事实。
- 结果：排除。页面逻辑符合 owner spec；缺口位于默认 fixture client 的跨 operation 状态转换。

### 假设 2：已有 P0.101 verifier 足以阻止失败日志泄漏

- 验证方法：反查 `trigger.sh` 的 `pipefail` 管道、Playwright matcher 输出及 `verify.sh` 执行顺序，并用模拟失败输出探测终端与落盘内容。
- 结果：排除。verifier 只能检查成功完成 trigger 后的日志，无法清理 reporter 已输出且导致 trigger 提前退出的敏感值。

## 3 根因分析

- `frontend/src/api/devMockClient.ts` 只为 verify、profile completion 与 logout 维护 session state，没有把成功 `DELETE /me` 建模为 session termination；页面级测试又注入了直接返回 401 的专用 mock，因此默认组合路径未被覆盖。
- `frontend/tests/e2e/auth-email-code.spec.ts` 把完整 email 直接交给 matcher 的 expected/text 参数，失败 reporter 会回显该值。
- P0.101 仅在 trigger 成功后由 `verify.sh` 搜索 email 变体，`trigger.sh` 的 reporter stream 在 `tee` 前没有脱敏边界。

## 4 修复方案

- 成功 `DELETE /me` 后将默认 dev mock auth state 切换为 signed-out，使下一次普通 `getMe` 返回 401。
- 增加 client-level verify/profile/delete/getMe 回归，以及使用真实默认 dev mock 的 mounted App 删除账号回归。
- 将 P0.101 的完整 email UI/API 校验改为只向 reporter 暴露布尔结果，并在 reporter stream 进入终端和 `trigger.log` 前同时替换原文与 URL percent-encoded 表示。
- 在 frontend-shell 原 owner spec/plan/checklist 中固化两项合同，不创建 sibling plan。

## 5 验证

- RED：新增 dev mock tests 分别观察到 post-delete `getMe` 仍 resolve、mounted App 停留 Settings；新增 scenario contract test 因缺少 pre-log redactor 失败。
- GREEN：dev mock focused tests 10/10、scenario contract tests 7/7 通过；模拟 reporter 失败流确认两种 email 表示均脱敏且原始退出码 23 由 `pipefail` 保留。
- 回归：根 `make test` 通过（Python 559 tests / 4481 subtests、Go 全包、frontend 126 files / 1003 tests）；frontend typecheck/build、owner contexts、docs/index/link、diff、core-loop lint 和旧口径负向搜索通过。

## 6 经验教训

- Stateful fixture 必须覆盖跨 operation 的业务生命周期；单个 route fixture 正确不代表默认组合行为正确，适用既有 `PATTERNS.md` Pattern 3。
- 敏感值 gate 必须在可能失败的 reporter 之前生效；成功后 verifier 只能作为补充，不能替代 pre-log redaction。
- 本问题扩展了 [BUG-0175](./BUG-0175.md) 的失败路径边界，无需新增同类 pattern。

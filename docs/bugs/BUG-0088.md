# BUG-0088: async runner review exposed scheduler and report backoff drift

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(backend-async-runner): harden scheduler and report backoff (BUG-0088)`
> 关联日志: [2026-05-22](../work-journal/2026-05-22.md)

## 1 现象

- L2 code review 指出 `Runtime.dispatch` 在 handler 执行前捕获 `now`，retryable handler 执行超过 backoff 首档时，`available_at` 会落在过去，terminal `completed_at` 也可能早于真实结束时间。
- `Runtime.Start` 的 lease loop 复用 synchronous `runOnce`，按 high-weight bucket 串行 lease + handle；持续 critical backlog 或一个长耗时 critical/default handler 会阻塞 low-priority `email_dispatch`，magic link 投递可能超过设计承诺的一个 scan 周期。
- `report_generate` 的 retryable AI failure 仍由 review service/store 通过 `AsyncJobFinalized` 绕过 kernel finalize，导致该 jobType 继续使用旧 review-store backoff 语义，而不是统一 `[30s,2m,10m,1h,6h]` shared policy。

## 2 诊断过程

### 假设 1：retry finalize 使用了 handler 后时间

- 验证方法：新增 `TestRuntime_FinalizeUsesTimestampAfterHandlerReturns`，让 handler 把 deterministic clock 从 `12:00:00` 推进到 `12:00:45` 后返回 retryable / failed outcome。
- 结果：修复前 retry `available_at` 仍基于 handler 前时间计算，terminal `completed_at` 也不使用真实完成时间，确认 timestamp 捕获位置错误。

### 假设 2：production lease loop 能在 critical job 执行中扫描 email_dispatch

- 验证方法：新增 `TestRuntime_StartDoesNotLetCriticalJobStarveEmailDispatch`，让 `report_generate` handler 持续阻塞，同时等待 `email_dispatch` handler 是否能在短时间内执行。
- 结果：修复前 `email_dispatch` 被正在运行的 critical handler 串行阻塞；说明 production `Start` 不能复用 fixed-priority synchronous `runOnce` 作为唯一 lease loop。

### 假设 3：report failure 已经走 kernel shared backoff

- 验证方法：新增 `TestGenerateHandler_NormalizesFinalizedRetryableFailureThroughKernel`，并反查 `Service.GenerateReport` / `PersistReportFailure` 的 failure path。
- 结果：修复前 retryable report failure 可携带 `AsyncJobFinalized=true`，kernel 会跳过 shared `BackoffPolicy`；review store 仍更新 `async_jobs` 并携带旧 backoff 行为。

## 3 根因分析

- `Runtime.dispatch` 在 handler 调用前复用 `now`，把 lease claim 时间误当成 finalize 时间；该缺陷只在 handler 耗时超过 backoff delay 或需要审计 terminal timestamp 时显性化。
- `Runtime.Start` 把 production scheduling 建立在 synchronous `runOnce` 上；`runOnce` 是测试/同步驱动 API，按 priority bucket 串行处理单条 job，不适合作为生产 fair scheduling loop。
- `report_generate` 迁移时保留了旧 service/store 对 `async_jobs` 的 failure finalize 兼容路径；handler 只透传 `AsyncJobFinalized`，导致 shared kernel backoff 对 report failure 没有成为唯一来源。
- 原 checklist 只覆盖 backoff table、priority bucket 与 handler 注册，没有覆盖 long-running handler starvation、fresh finalize timestamp、domain service bypass shared policy 这三类 cross-boundary invariant。

## 4 修复方案

- `Runtime.dispatch` 改为 handler 返回后重新读取 `finished := r.now()`，并用该时间计算 duration、retry `available_at`、terminal `completed_at` 与 `updated_at`。
- `Runtime.Start` 为每个 registered job_type 启动独立 lease loop；`RunOnce` 保留 synchronous priority-order 测试入口，production path 不再串行 drain fixed-priority buckets。
- `GenerateHandler` 只在成功且 service 已 finalize 时保留 `AsyncJobFinalized`；所有 failure outcome 均交回 kernel finalize。
- `Service.GenerateReport` 的 failure path 仍持久化 `feedback_reports` / outbox / audit domain state，但返回未 finalize 的 `ReportOutcome`；`Repository.PersistReportFailure` 不再更新 `async_jobs`，只根据 `Attempts` / `MaxAttempts` 决定 report 仍可 retry 时回到 `queued`，最终失败时转为 `failed`。
- 原地修订 `backend-async-runner` spec/history/plan/checklist/test-plan/test-checklist 至 v1.4 / v1.3，把 scheduler/backoff review invariant 固化为 C-21、4.18 与对应测试 gate。

## 6 经验教训

- Scheduler gate 不能只验证 priority bucket 选择；必须覆盖 long-running handler 对低优先级用户可见 job 的影响。
- Shared retry policy 的完成证据必须从 handler outcome 追到 domain service/store，确认没有 `AsyncJobFinalized` 或 domain-side SQL 绕过 kernel。
- 时间戳类 bug 需要 deterministic clock 测试覆盖 handler 执行前后时间变化，不能只看 instant handler 的 happy path。

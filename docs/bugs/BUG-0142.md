# BUG-0142: targetjob failed detail returned 500 after profile schema pruning

> 模块: workspace | 严重度: high | 状态: resolved
> 发现日期: 2026-07-08 | 关联 Commit: `fix(targetjob): align store with pruned profile schema (BUG-0142)`, `fix(targetjob): require postgres for schema drift gate (BUG-0142)`
> 关联日志: [2026-07-08](../work-journal/2026-07-08.md)

## 1 现象

- 用户在 `/parse?targetJobId=...` 页面看到「出了点问题 / 获取解析状态失败」。
- DevTools Network 显示 `GET /api/v1/targets/{targetJobId}` 返回 500，错误码为 `TARGET_IMPORT_FAILED`，前端无法读取已创建的 JD 解析状态。
- 受影响对象是已经写入 `target_jobs`、但解析任务失败且无 requirements 的 TargetJob；按当前合同，详情接口应返回 200 和 `analysisStatus="failed"`，让前端展示失败态与重试入口。

## 2 诊断过程

### 假设 1：前端 Parse 页面请求参数或鉴权态错误

- 验证方法：使用截图同款 targetJobId 和已登录本地会话直接请求 `GET /api/v1/targets/{targetJobId}`。
- 结果：排除。请求命中后端并返回 generated error envelope，说明问题位于后端详情读取路径，不是前端路由或鉴权丢失。

### 假设 2：解析失败态本身不允许读取详情

- 验证方法：检查 `target_jobs`、`target_job_sources`、`target_job_requirements`、outbox 和 async job 记录。
- 结果：排除。目标 JD 已保存原文和 source snapshot，`analysis_status='failed'`，无 requirements；该状态应可读，不应被当成导入接口失败。

### 假设 3：当前数据库 schema 与 TargetJob active SQL 不一致

- 验证方法：在本地开发 Postgres 上新增 integration red test，直接插入失败态 TargetJob 后调用 `SQLStore.GetTargetJobByUser`。
- 结果：确认。修复前测试失败为 `pq: column "profile_id" does not exist`；当前 B4 schema 已删除 `target_jobs.profile_id`，但 TargetJob store 的 select / insert / scan 和 sqlmock 列集合仍残留该列。

## 3 根因分析

- D-20/D-17 后 profile 模块退役，当前 `target_jobs` 表已无 `profile_id`。
- `backend/internal/targetjob/store.go` 仍在 `InsertTargetJob`、`GetTargetJobByUser`、列表查询、生命周期更新、parse runner 读取与通用 `scanTargetJobRow` 中引用或扫描 `profile_id`。
- 既有 sqlmock 测试同步保留了旧列，因此没有暴露真实 Postgres schema 漂移；OpenAPI / fixture / BDD harness 也没有覆盖“当前 DB schema 下失败态详情读取”的 gate。
- handler 将 store SQL error 包装成导入失败 envelope，最终前端只看到通用 500，而不是可展示的解析失败态。

## 4 修复方案

- 从 `TargetJobRecord` 和所有 active TargetJob SQL read/write/scan 路径删除 `profile_id`。
- 同步更新 `backend/internal/targetjob/store_test.go` 的 sqlmock 列集合与参数列表，避免 mock 继续模拟已退役列。
- 新增 `backend/internal/targetjob/store_integration_test.go`，用真实 Postgres schema 断言失败态 TargetJob 在无 requirements 时仍可被 `GetTargetJobByUser` 读取。
- 在 `backend-targetjob` spec / plan / checklist 中追加 C-15 和 7.11 gate，固化 `profile_id` 退役列负向搜索、真实 Postgres integration gate，以及截图同款详情接口 200 回归。
- 将 BUG-0142 的真实 Postgres integration gate 从可选 smoke 改为 fail-fast：缺少 `DATABASE_URL` 或 DB 不可达时直接失败，避免 `SKIP` 假绿掩盖 schema drift gate 没有执行。

## 5 验证

- Red 阶段：`DATABASE_URL=<local-dev-postgres-dsn> go test -tags=integration ./internal/targetjob -run TestSQLStoreIntegration_GetTargetJobByUser_AllowsFailedJobWithoutRequirements -count=1` 失败，错误为 `pq: column "profile_id" does not exist`。
- Focused green：`go test ./internal/targetjob -count=1` 通过。
- Real DB gate：`DATABASE_URL=<local-dev-postgres-dsn> go test -tags=integration ./internal/targetjob -run TestSQLStoreIntegration_GetTargetJobByUser_AllowsFailedJobWithoutRequirements -count=1` 通过。
- Skip-proof negative gate：未设置 `DATABASE_URL` 时 `go test -tags=integration ./internal/targetjob -run TestSQLStoreIntegration_GetTargetJobByUser_AllowsFailedJobWithoutRequirements -count=1 -v` 非 0，输出 `DATABASE_URL is required for BUG-0142 targetjob schema-drift integration gate`，且无 `--- SKIP`。
- Skip-proof positive gate：设置真实 local-dev Postgres `DATABASE_URL` 后同一 focused integration gate 通过。
- HTTP scenario regression：`go test ./cmd/api -run 'TestBuildTargetJobRuntimeWiresDrainerAndAIClient|TestBuildAPIHandlerMountsTargetJobRoutesBehindSessionMiddleware|TestE2EP0012HTTPParseFailureRetryableAndNonRetryable' -count=1` 通过。
- Runtime smoke：`test/scenarios/env-verify.sh` 显示 4 个本地依赖均 OK；截图同款 `GET /api/v1/targets/{targetJobId}` 返回 200，body 包含 `analysisStatus="failed"` 和空 `requirements`。
- Negative search：`rg 'profile_id|ProfileID|profileID' backend/internal/targetjob backend/cmd/api openapi/fixtures/TargetJobs openapi/openapi.yaml migrations shared` 无 active 命中。

## 6 经验教训

- 退役模块或删除 schema 列后，不能只依赖 sqlmock 或 generated contract gate；至少要有一条真实 DB-backed 读写 gate 证明 active SQL 与当前 DDL 对齐。
- 失败态业务对象仍是可读状态，详情接口测试需要覆盖“解析失败、无 parsed children、仍能展示状态”的路径，避免把 runner 失败误包装为详情读取失败。
- 被 plan / Bug 明确列为强制防线的 integration gate 不能用 `t.Skip` 处理环境缺口；缺依赖时必须 fail fast，并在验证证据中同时记录缺依赖失败与真实依赖通过。

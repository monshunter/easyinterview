# BUG-0097: prompt registry seed migration missed jd_match feature keys

> 模块: eval | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-24 | 关联 Commit: `fix(prompt-rubric): seed jd match prompt registry rows (BUG-0097)`
> 关联日志: [2026-05-24](../work-journal/2026-05-24.md)

## 1 现象

- `/plan-code-review 002-output-schema-contract --fix` 复查到 `config/prompts` / `config/rubrics` 已有 13 个 active chat feature_key，但 seed migration 静态 gate 仍只断言 11 个历史 feature_key / 22 行。
- `jd_match.recommendation` 与 `jd_match.search` 已有 prompt schema、rubric、runtime parser 与 caller 透传，但没有落入 Postgres `prompt_versions` / `rubric_versions` baseline seed rows。
- 影响范围：DB-backed prompt/rubric registry baseline 缺少 JD-Match search / recommendation 的 prompt 与 rubric rows，历史 checklist 和 seed gate 对该缺口 false-green。

## 2 诊断过程

### 假设 1：jd_match 只缺 prompt schema，不涉及 DB seed
- 验证方法：`find config/prompts -mindepth 2 -name 'v0.1.0.schema.json' -print | sort` 与 `find config/rubrics -mindepth 2 -name 'v0.1.0*.yaml' -print | sort`。
- 结果：排除。`jd_match.recommendation` / `jd_match.search` 的 schema、prompt YAML、prompt body 与 rubric YAML 均存在。

### 假设 2：seed migration 已在后续 jd_match baseline migration 中补齐
- 验证方法：`rg -n "jd_match\\.recommendation|jd_match\\.search" migrations`，并读取 `migrations/000009_jd_match_baseline.up.sql`。
- 结果：排除。`000009` 只创建 JD-Match 业务表与 async job constraint，没有 prompt_versions / rubric_versions seed rows。

### 假设 3：静态 gate 本身只验证历史固定集合
- 验证方法：读取 `backend/internal/ai/registry/db_integration_test.go`。
- 结果：确认。测试写死 11 个 feature_key 与 22 行计数；`prompt_lint.py` 只验证已出现 seed row 的 hash，无法发现完整 feature_key 缺失。

## 3 根因分析

- 根本原因：seed migration coverage test 以历史固定清单作为期望值，而不是从当前 `config/prompts` / `config/rubrics` truth source 反推出 active prompt/rubric 坐标。
- 涉及路径：
  - `backend/internal/ai/registry/db_integration_test.go`
  - `migrations/000002_seed_baseline_prompt_rubric_versions.up.sql`
  - `migrations/000009_jd_match_baseline.up.sql`
  - `config/prompts/jd_match.recommendation/*`
  - `config/prompts/jd_match.search/*`
  - `config/rubrics/jd_match.recommendation/*`
  - `config/rubrics/jd_match.search/*`
- 为什么之前没有发现：历史 plan/checklist 的 completed 状态依赖固定行数 gate；prompt lint 的 migration hash check 只遍历已经存在的 seed rows，缺少“期望行必须存在”的反向断言。

## 4 修复方案

- 选择方案：把 registry seed coverage test 改为从当前配置 truth source 生成期望坐标，并新增 `000010_seed_baseline_prompt_rubric_versions_jd_match.{up,down}.sql` 补齐 JD-Match prompt/rubric seed rows。
- 理由：该方案最小化业务代码变更，同时把漏检模式固化为 future-proof gate；后续新增 active prompt/rubric 坐标若忘记 seed，会直接红测。
- 放弃方案：继续把测试常量从 11 改成 13。该方案只能修当前缺行，不能防止下一次新增 feature_key 时再次 false-green。
- 修改文件：
  - `backend/internal/ai/registry/db_integration_test.go`
  - `migrations/000010_seed_baseline_prompt_rubric_versions_jd_match.up.sql`
  - `migrations/000010_seed_baseline_prompt_rubric_versions_jd_match.down.sql`
  - `docs/spec/prompt-rubric-registry/plans/002-output-schema-contract/{plan.md,checklist.md,context.yaml}`
  - `docs/spec/prompt-rubric-registry/plans/INDEX.md`

## 5 验证

- Red：`go test ./backend/internal/ai/registry -run TestSeedMigrationCoversBaselineFeatureKeys -count=1 -v` 失败，列出 `jd_match.recommendation|v0.1.0|{en,multi}` 与 `jd_match.search|v0.1.0|{en,multi}` 在 prompt_versions / rubric_versions 中缺失。
- Green：`go test ./backend/internal/ai/registry -run TestSeedMigrationCoversBaselineFeatureKeys -count=1 -v` → pass。
- Registry regression：`go test ./backend/internal/ai/registry -count=1` → pass。
- Static gates：`python3 scripts/lint/prompt_lint.py`、`python3 scripts/lint/rubric_lint.py`、`python3 scripts/lint/migrations_lint.py` → pass。
- Migration chain：`DATABASE_URL=postgres://easyinterview:***@localhost:5432/easyinterview?sslmode=disable make migrate-check` → pass。
- Docs gates：`validate_context.py`、`sync-doc-index.py --check`、`git diff --check` → pass。

## 6 经验教训

- 静态 migration gate 应反查当前 truth source 的完整期望集合，而不是只检查历史 row count 或已经存在的 seed rows。
- 对 completed plan 做 L2 review 时，seed / generated / migration 类 gate 必须确认“缺失也会失败”，否则 hash / count / lint 很容易只验证现有子集。
- 该模式已补充到 [PATTERNS.md](./PATTERNS.md)。

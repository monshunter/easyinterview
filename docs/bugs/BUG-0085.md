# BUG-0085: jd_match frontend plan stayed fixture-only after real backend landed

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(frontend-jd-match): close real backend gate drift (BUG-0085)`
> 关联日志: [2026-05-22](../work-journal/2026-05-22.md)

## 1 现象

- 触发条件 / 发现方式：对 `frontend-home-job-picks-and-parse/002-jd-match-recommendations` 执行 L2 code review 时，用户指出 plan 仍写 `fixture-backed transport` 与 backend handler `not-yet-implemented`，但 `backend-jobs-recommendations/001-jd-match-real-backend-baseline` 已完成 12 个真实 JobMatch endpoint。
- 可观察到的错误表现：`plan.md` 的 operation matrix、spec Out of Scope、OpenAPI generated spec 与 P0.027-P0.031 scenario docs 仍把真实 backend 当作未来 owner；frontend scenarios 只证明 fixture UI 子用例，未证明 `VITE_EI_API_MODE=real` 下 production generated client 会打到真实 backend base URL。
- 影响范围：JDMatch frontend plan 可能被误判为与真实 backend 闭环，而实际证据只覆盖 fixture/mock；P0.030 verify 还保留了旧 `toHaveScreenshot` baseline grep，与 clean-checkout pixel gate 的当前实现不一致。

## 2 诊断过程

### 假设 1：用户指出的 stale plan 事实不成立

- 验证方法：读取 `docs/spec/frontend-home-job-picks-and-parse/plans/002-jd-match-recommendations/plan.md` §3.6、owner spec、OpenAPI JobMatch tag、P0.027-P0.031 README/data。
- 结果：排除。plan matrix 12 行 backend handler 均仍为 `not-yet-implemented` 或未来 owner，scenario docs 也仍写纯 fixture-backed / no network。

### 假设 2：后端真实 endpoint 还没有完成

- 验证方法：读取 `backend/cmd/api/main.go` route mounting、`backend/cmd/api/jdmatch_live_scenario_test.go`、P0.094-P0.097 scenario scripts 与当天 work journal。
- 结果：排除。12 个 `/api/v1/jd-match/*` routes 已挂载，live scenario tests 覆盖 auth、IK replay、cross-user、search outbox privacy、AI provenance、agent scan drainer 与 privacy delete。

### 假设 3：frontend 已有 real-mode gate，但文档未更新

- 验证方法：检查 `frontend/src/api/clientFactory.test.ts` 与 `frontend/src/api/jobMatch.contract.test.ts`。
- 结果：部分确认。已有 factory test 只证明 real mode base URL，已有 JobMatch contract test 只证明 fixture transport；缺少对 12 个 JobMatch operation 在 real API mode 下 URL/auth/IK/provenance 的集中断言。

## 3 根因分析

- 根本原因：frontend plan 002 完成时 backend implementation 尚未落地，后续 backend plan 完成后未回到原 completed frontend owner plan 做 real-mode reconciliation，导致 fixture-first handoff 口径未转为真实 API 联调证据。
- 涉及路径：
  - `docs/spec/frontend-home-job-picks-and-parse/plans/002-jd-match-recommendations/plan.md`
  - `docs/spec/frontend-home-job-picks-and-parse/spec.md`
  - `openapi/openapi.yaml`
  - `test/scenarios/e2e/p0-027-*` 至 `p0-031-*`
  - `test/scenarios/e2e/p0-030-jd-match-profile-and-agent-status/scripts/verify.sh`
- 为什么之前没有发现：历史 P0.027-P0.031 gate 只要求 fixture-backed UI 子用例 PASS；后端 P0.094-P0.097 虽已证明 live backend，但没有被 frontend owner plan 作为 paired evidence 反向接入。

## 4 修复方案

- 选择的方案及理由：原地修订 completed plan 002，不新建 sibling；新增 `frontend/src/api/jdMatch.realApiMode.test.ts`，在 `VITE_EI_API_MODE=real` 下覆盖 12 个 JobMatch generated-client operation 的真实 backend base URL、cookie credentials、5 个 `Idempotency-Key` side effect、无 fixture `Prefer` header 与 `GenerationProvenance` roundtrip。
- 同步 P0.027-P0.031 trigger/verify，让每个 frontend scenario 先跑 real-mode generated-client gate，再跑 fixture-backed UI variants；fixture 子用例继续负责 DOM、auth pending action、privacy 与 failure variants。
- 更新 plan/spec/BDD/OpenAPI/generated spec 与 scenario docs，将 backend handler 状态从 future/not-yet-implemented 改为 `backend-jobs-recommendations/001` 已完成，并把 backend E2E.P0.094-P0.097 作为 paired live evidence。
- 同时修正 P0.030 verify：不再检查旧 `toHaveScreenshot` grep，改为检查当前 non-empty screenshot gate 源码锚点。

## 5 验证

- E2E.P0.027-P0.031 全部 setup→trigger→verify→cleanup PASS；每个 trigger log 均包含 `VITE_EI_API_MODE=real`、`VITE_EI_API_BASE_URL=http://localhost:8080/api/v1` 与 `jdMatch.realApiMode.test.ts`。
- Backend E2E.P0.094-P0.097 全部 setup→trigger→verify→cleanup PASS，覆盖 live Postgres route/persistence/auth/IK/privacy/AI provenance。
- `make lint-openapi` PASS；二次 `make codegen-openapi` 后 generated diff 不再变化。
- `python3 .agent-skills/sync-doc-index/scripts/sync-doc-index.py --fix-index` zero drift；`make docs-check` PASS；`git diff --check` PASS。

## 6 经验教训

- Frontend-first fixture plans 在 backend owner 完成后必须回到原 completed plan 做 real-mode reconciliation；fixture UI PASS 不能替代真实 API mode 的 production generated-client proof。
- 对于已迁移的 pixel gate，scenario verify 应检查当前 gate 的源码锚点，而不是保留旧 baseline 或旧 assertion 名称。

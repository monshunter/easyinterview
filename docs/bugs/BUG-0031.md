# BUG-0031: workspace follow-up review exposed synthetic id and fetch race drift

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-09 | 关联 Commit: `fix(frontend-workspace): harden workspace review follow-up`
> 关联日志: `-`

## 1 现象

- L2 review follow-up 发现 workspace 从 parse/home handoff 进入时会携带 synthetic placeholders，例如 `plan-${targetJobId}` 与 `resume-unbound`。
- 这些 synthetic values 被 reducer 写入 `InterviewContext` 后，workspace hooks 可能直接调用 generated client：`getPracticePlan(plan-...)`、`getResume(resume-unbound)` 或用 invalid resume id 构造 `createPracticePlan` body。
- `getTargetJob` 404/5xx 时，workspace 会落回 full workspace shell 或 stale target data，而不是缺 JD recovery UI。
- Plan Switcher 选择新 target 时，旧 `getTargetJob` 请求完成后可能覆盖新选择。
- 英文 workspace 中仍可能出现硬编码中文派生标签，如 JD block label、round fallback、status/source label。

## 2 诊断过程

### 假设 1：只是测试 fixture 使用了非真实 ID
- 验证方法：读取 `InterviewContext` route hydrate、`useWorkspacePracticePlan`、`useWorkspaceResume`、`useStartPractice` 与 OpenAPI generated client 的 path/body 输入。
- 结果：排除。问题不只是 fixture；synthetic placeholders 会进入真实 generated client path/body，违反 server-bound UUID 合约。

### 假设 2：target-job error 只影响 hook 局部状态
- 验证方法：反查 `useWorkspaceTargetJob` 返回值与 `WorkspaceScreen` empty/missing-resume 判定。
- 结果：确认。旧逻辑只记录 hook error，screen 没有把 target load failure 当作 missing target recovery path。

### 假设 3：全量测试失败只是等待条件不稳定
- 验证方法：在 full Vitest 中复现 `WorkspaceHeader` 数据偶发回到 placeholder；检查 workspace hooks 对 `AppRuntimeProvider` context 的依赖。
- 结果：确认更深层竞态。hooks 依赖整个 runtime context，`/me` 或 runtime config 状态变化会改变 context object identity，触发 workspace 数据重新置空再重拉。

## 3 根因分析

- `InterviewContext` 允许 route 层 synthetic placeholders 作为临时 UI context，但 server-bound API 调用没有统一归一化，导致 UI-only placeholder 泄漏到 generated client。
- target fetch 使用单一 in-flight guard / 未按 `targetJobId` key completion，缺少 stale completion ignore 机制。
- workspace screen 对 target load failure 与缺 target 的恢复路径没有统一处理。
- workspace 派生标签混合了硬编码中文与 locale key，新增 i18n key 后没有把 status/source/round/JD label 全部切换到 `t(...)`。
- workspace hooks 的 `useEffect` 依赖包含整个 runtime object，auth/runtime state 变化会引发不必要的 refetch 和 placeholder flicker。

## 4 修复方案

- 新增 `normalizeServerBoundId`，在 `buildCreatePlanRequest`、`useWorkspacePracticePlan`、`useWorkspaceResume`、`useWorkspaceTargetJob`、`useStartPractice` 中把 invalid/synthetic IDs 视为 absent。
- `buildCreatePlanRequest` 对 invalid `targetJobId` 明确抛错，对 invalid optional `resumeVersionId` 直接省略。
- `useWorkspaceTargetJob`、`useWorkspaceResume`、`useWorkspacePracticePlan` 使用 request sequence + cleanup ignore stale completions，并将 effect dependency 收窄为 `client + id`。
- `WorkspaceScreen` 将 target 404/5xx 纳入 empty recovery，Plan Switcher 切换时不再制造 `plan-${targetJobId}`。
- JD block labels、round fallback、status/source/prep derived labels 改为 locale key 渲染，补齐 zh/en locale entries。
- 补充 synthetic id、target stale completion、target not-found recovery 与 English no-Chinese-label 回归测试。

## 6 经验教训

- UI-only synthetic IDs 必须在 generated client boundary 之前统一归一化；route/context 可用的 placeholder 不能默认是 server-bound id。
- 异步 hook 应按输入 key 处理 stale completion，并避免依赖会因无关 runtime state 变化而频繁改变的 context object。
- L2 review follow-up 不能只修当前失败断言；full test 暴露出的 flicker 往往说明生产路径还有未被 review comment 明说的竞态。

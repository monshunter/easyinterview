# BUG-0079: frontend shell popstate left unsafe URL markers visible

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-18 | 关联 Commit: `fix(frontend-shell): scrub hostile popstate route privacy (BUG-0079)`
> 关联日志: [2026-05-18](../work-journal/2026-05-18.md)

## 1 现象

- `plan-code-review frontend-shell/004-url-addressable-routing --fix` 反查 C-12 / E2E.P0.089 时发现，`popstate` handler 只把当前 location 解析成安全 `Route` state。
- 如果浏览器历史栈中已有 hostile entry，例如 `/workspace?targetJobId=...&rawText=...#prompt` 并带 raw `history.state`，回退 / 前进后 React route state 会被过滤，但地址栏和 `window.history.state` 仍保留 raw marker。
- 影响范围是 Browser History 恢复路径；常规 App `navigate()` 和 direct-open mount 已经会 canonicalize URL。

## 2 诊断过程

### 假设 1：safe-param allowlist 没有覆盖 popstate

- 验证方法：读取 `frontend/src/app/routeUrl.ts`，确认 `parseUrlToRoute()` 会按 `ROUTE_SAFE_PARAMS` 丢弃 `rawText` / `prompt` 等字段。
- 结果：排除。allowlist 本身有效，问题不在 parser。

### 假设 2：popstate 解析后没有把浏览器历史项改写为 canonical URL

- 验证方法：读取 `frontend/src/app/routeStore.ts` 的 `popstate` effect，并新增 `AppRoutingPrivacy.test.tsx` hostile history entry 失败断言。
- 结果：确认。测试失败时 URL 仍是 `/workspace?targetJobId=tj-popstate&rawText=RAW_JD_TEXT_2c1a#RAW_AI_PROMPT_0412`。

## 3 根因分析

- `useBrowserRoute()` mount effect 会把 direct-open / hash URL 改写为 canonical URL，`navigate()` / `replaceRoute()` 也只写 canonical URL。
- 但 `popstate` handler 只执行 `parseUrlToRoute()` + `setRoute(next)`，没有比较当前 `pathname + search + hash` 与 `formatRouteUrl(next)`，也没有通过 `replaceState(null, "", canonicalUrl)` 清理当前历史项。
- 之前没有发现，是因为 E2E.P0.089 只覆盖 auth restore、jd_match restore 和 hostile direct-open，未覆盖已有历史栈 raw entry 的 browser back / forward privacy 行为。

## 4 修复方案

- 在 `frontend/src/app/routeStore.ts` 的 `popstate` handler 中，解析出安全 route 后计算 canonical URL；当当前 URL 含 unsafe query/hash 或 legacy shape 时，用 `history.replaceState(null, "", canonicalUrl)` 改写当前历史项。
- 新增 `AppRoutingPrivacy.test.tsx` hostile popstate regression，断言 URL、hash、`history.state`、storage 全 surface 无 raw marker。
- 扩展 `frontend/src/app/scenarios/p0-089-url-routing-auth-privacy.test.tsx` 与 `test/scenarios/e2e/p0-089-url-routing-auth-privacy/` wrapper 证据，将 E2E.P0.089 从 3 tests 更新为 4 tests。
- 原地修订 `frontend-shell/004-url-addressable-routing` plan/checklist/BDD 文档到 v1.2，避免 completed checklist 掩盖新发现的 privacy gate 缺口。

## 6 经验教训

- URL privacy gate 必须覆盖 direct-open、programmatic navigation、auth restore 和 browser history recovery 四类入口；只验证 parser / navigate path 不足以证明地址栏隐私安全。
- 已完成 BDD gate 的 pass count 不是永久事实；L2 review 发现新风险后，应把原 scenario wrapper 扩到能抓住该风险，再更新原 plan evidence。

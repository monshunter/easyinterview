# BUG-0158: practice phone flow and session language drifted from the real interview contract

> 模块: practice | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-11 | 关联 Commit: `fix(practice): align phone flow and session language (BUG-0158)`
> 关联日志: [2026-07-11](../work-journal/2026-07-11.md)

## 1 现象

- 电话模式仍提供“重新开始”，并同时呈现分段模式切换、通话状态和“切断”等旧控件，和真实电话中通过一个 handset 入口、中央挂断后回到同一文本会话的交互不一致。
- 电话态与文本态的会话连续性缺少完整生命周期保护：挂断、真实 speech-start 插话、TTS 完成和下一轮录音可能竞争，存在错误上报插话、继续播放或用旧 turn 回调的风险。
- 会话问题和后续问题可能表现为夹杂中英文、模板化或疑似 mock 内容；真实页面还可能显示内部 `questionIntent` 或路由占位身份，而不是后端会话与 TargetJob 的当前事实。
- 影响前端电话交互、后端文本/语音问题生成、事件持久化、OpenAPI fixtures、场景 wrapper 和真实浏览器验收。用户仍可绕过到文本模式，因此严重度为 `medium`。

## 2 诊断过程

### 假设 1：问题只来自前端旧控件

- 验证方法：反查 `ui-design/`、`docs/ui-design/`、正式 Practice 组件、i18n、场景文档和前后端正向符号，检查 restart、`callEnded`、分段切换、live chip 和可见“切断”口径。
- 结果：部分确认。旧交互不仅存在于正式组件，也被原型、计划、pixel parity 和 P0.007/P0.008/P0.009/P0.045 场景当作正向契约；只隐藏按钮会留下后端/测试/文档漂移。

### 假设 2：中英文混合只是前端 mock transport

- 验证方法：沿 generated `getPracticeSession` / `getTargetJob` 真实调用、文本 `appendSessionEvent`、语音 `createPracticeVoiceTurn`、prompt/profile、fixture 和 store 写入路径追踪问题来源，并在真实 API 模式中读取当前 session/turn/TargetJob。
- 结果：排除“只是 mock”。真实读取链路可以返回持久化数据，但文本与语音问题生成曾使用不同上下文和错误处理，客户端字段可影响输入，错误分支存在硬编码英文/罐头问题，且没有按持久化 session language 做严格校验与一次修复边界。

### 假设 3：挂断只是路由切换，不会影响会话

- 验证方法：分别测试 hang-up、真实 speech-start barge-in、partial TTS playback、TTS-ended re-arm、session A→B rerender 和异步回调晚到。
- 结果：确认存在竞态风险。旧实现没有一个共享的 `exitPhoneMode` 原子边界，也没有把 recorder/playback/callback 生命周期完整绑定到当前 call、turn 和 session。

## 3 根因分析

- Phase 6 的 restart/call-ended 电话原型被长期计划、正式 UI 和场景 gate 同时固化；产品方向变化后，owner 文档没有先原地重开并反转旧正向断言。
- 文本与语音后续问题生成分叉，缺少共享的 server-owned canonical renderer、持久化 session language 校验和“只对结构/语言无效结果修复一次”的统一规则；失败时的罐头问题掩盖了真实 AI 输出错误。
- 前端的麦克风、utterance recorder、TTS playback 和路由回调没有统一的 call/session ownership，导致挂断与 speech-start 的语义容易混淆；同路由 session 切换也可能保留旧的本地状态。
- 原有 fixture/scenario gate 偏重字段存在与历史 PASS，未同时锁定事件 metadata exact key set、原始 replay snapshot、同 session URL、真实 TargetJob 身份、语言一致性和 removed-control zero-reference。

## 4 修复方案

- 原地修订 `practice-voice-mvp/001`、`frontend-workspace-and-practice/002`、`backend-practice/002` 与 `backend-practice/003`，让当前 UI、生成、失败恢复和验证由既有 owner 继续承接。
- 在 `ui-design/` 和正式前端统一为一个 Top Bar handset；文本点击进入电话，电话 handset 与中央红色挂断共用 `exitPhoneMode`，立即停止麦克风/TTS、按已有事件结算已听/已录内容并回到同一文本会话。删除 restart、`callEnded`、分段控件、live chip 和可见“切断”正向实现。
- 增加 call-scoped recorder、VAD silence submit、TTS-ended re-arm、真实 speech-start barge-in 与 session-keyed remount；生成的 `getTargetJob` 负责公司/岗位显示，UI 不显示内部 `questionIntent` 或 fixture dialogue。
- 后端文本与语音共用 canonical question generator：只使用持久化 session/turn/plan/committed context，强制 session language；结构/parser/语言无效最多修复一次，provider/config/timeout 不修复，二次无效返回 typed error 或 `session_wait`，不再生成罐头问题。
- 收紧 voice committed-context 证据、事件 metadata whitelist、fixture replay/hash 和 P0.007/P0.008/P0.009/P0.038/P0.045/P0.046/P0.051 场景断言。
- 放弃“只删前端按钮”方案，因为它无法消除后端 restart/语言/上下文漂移；也不创建 sibling plan，避免同主题 owner 分裂。

## 6 经验教训

- 电话模式的“挂断”必须被建模为同一 practice session 内的 modality exit，而不是结束、重启或创建新会话。
- AI 会话内容不能用前端语言或请求 payload 推断；所有用户可见问题都应从持久化 session language 和 server-owned context 生成，并把 wrong-language 当作结构化输出错误处理。
- UI 简化只有在原型、正式组件、后端残留、fixture/scenario 与真实浏览器证据同时反转时才算完成；历史截图和已勾选 checklist 不能替代当前验收。

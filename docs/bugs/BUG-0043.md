# BUG-0043: resume fileless intake still required upload file object

> 模块: schema | 严重度: high | 状态: resolved
> 发现日期: 2026-05-12 | 关联 Commit: `fix(openapi): allow fileless resume intake contracts`
> 关联日志: [2026-05-12](../work-journal/2026-05-12.md)

## 1 现象

- L2 code review 发现 `RegisterResumeRequest.required` 仍包含 `fileObjectId`，`ResumeAsset.required` 也仍包含 `fileObjectId`。
- 前端 generated type 和后端 generated DTO 因此把 `fileObjectId` 生成为必填字段，阻塞 `sourceType=paste` / `sourceType=guided` 的 fileless resume intake。
- `registerResume` fixture 只有 upload 默认场景，`listResumes` 的 paste / guided assets 仍带有上传文件 ID，导致 contract gate 没能覆盖 fileless 主路径。

## 2 诊断过程

### 假设 1：OpenAPI required 列表仍沿用 upload-only 旧口径

- 验证方法：读取 `openapi/openapi.yaml`、generated Go / TS DTO、D-18 plan 与 `docs/ui-design/resume-module.md`。
- 结果：确认。schema required 和 generated artifact 均要求 `fileObjectId`，但 UI / plan 要求 upload、paste、guided 三种入口。

### 假设 2：fixtures 缺少 fileless 语义样本

- 验证方法：检查 `openapi/fixtures/Resumes/registerResume.json` 与 `listResumes.json`，并补 red tests。
- 结果：确认。新增 fixture tests 后，`paste-text` 场景缺失，paste / guided list assets 的 `fileObjectId` 不是 `null`。

### 假设 3：改为 nullable 后 generated TS 会同步允许 `null`

- 验证方法：先用 `oneOf` 跑 `openapi-diff`，再切到 `nullable: true` 并检查 generated TS。
- 结果：部分确认。`oneOf` 会被 diff gate 判为 breaking composition change；`nullable: true` 通过 diff gate，但 TS generator 起初忽略 nullable string，需要单独修复。

## 3 根因分析

- Resume Workshop additive contract 在保留上传入口时追加了 paste / guided 字段，但没有同步撤销 upload-only 时代的 `fileObjectId` required 约束。
- fixture coverage 只证明 operation 文件存在和默认场景有效，没有断言 source type 与 file object 的语义关系。
- TS / Go OpenAPI generator 的 primitive nullable 处理位于 type switch 之后，导致 `type: string, nullable: true` 被提前返回为非 nullable primitive；这让 generated consumer artifact 容易与 fixture validator 的 schema 解释不一致。

涉及路径：

- `openapi/openapi.yaml`
- `openapi/fixtures/Resumes/registerResume.json`
- `openapi/fixtures/Resumes/listResumes.json`
- `backend/cmd/codegen/openapi/render_ts.go`
- `backend/cmd/codegen/openapi/render_go.go`
- `frontend/src/api/generated/types.ts`
- `backend/internal/api/generated/types.gen.go`

## 4 修复方案

- 从 `RegisterResumeRequest` 和 `ResumeAsset` 的 required 列表移除 `fileObjectId`，并将字段标记为 `nullable: true`。
- 增加 paste / guided register fixtures，并把 list fixtures 中 fileless asset 的 `fileObjectId` 固定为 `null`。
- 修复 Go / TS OpenAPI generator 的 nullable primitive handling，确保 TS 生成 `string | null`，Go 生成指针类型。
- 新增 contract tests 覆盖 fileless intake schema、fixtures 和 generator nullable 输出。

## 6 经验教训

- Additive schema 不能只检查新增字段和 operation 数量；当新增 source discriminator 时，需要同步检查旧 required 字段是否仍强制绑定某个旧来源。
- Fixture gate 应包含至少一个非默认 source path，避免 default upload 场景掩盖 paste / guided 语义缺口。
- OpenAPI generator tests 需要覆盖项目已使用的 schema 表达方式，尤其是 `nullable: true` 与 generated TS / Go consumer type 的一致性。

# BUG-0191: account theme review exposed stale auth response and projection parity gaps

> 模块: frontend | 严重度: high | 状态: resolved
> 发现日期: 2026-07-19 | 关联 Commit: `fix(settings): harden account theme review regressions (BUG-0191)`
> 关联日志: -

## 1 现象

- 分支代码审查发现，Settings 保存账号主题时直接消费异步 `updateMe` 响应；若页面已卸载、用户已退出或当前账号已变化，迟到响应仍会调用 `refreshAuth(updated)` 并提交旧主题。
- runtime 主题投影只检查 theme 枚举和字段类型，不拒绝越界、非有限或非 closed 的 custom accent；非法 theme 会保留上一账号的确认主题，而不是 fail closed。
- dev mock 把“出现但非法”的 `displayPreferences` 当成未提供，combined request 可能先写入合法 profile 字段并返回 200，与真实后端原子拒绝语义不一致。
- frontend-shell BDD/checklist 已标记失败恢复等行为完成，但缺少 race、invalid projection、combined-invalid 与离开未保存页面的可执行测试；当前产品和 OpenAPI owner 文本也仍保留旧方案描述。

## 2 诊断过程

### 假设 1：保存失败路径会立即丢弃草稿

- 验证方法：先增加失败后保留预览并重试的 focused component test。
- 结果：排除。既有实现已保留草稿并允许重试，该测试直接通过并作为回归门禁保留。

### 假设 2：Settings 卸载会自动阻止 Promise 后续回写

- 验证方法：使用 deferred `updateMe`，在请求 pending 时卸载 Settings，再解析旧 `UserContext`。
- 结果：确认缺陷。旧实现仍调用 `refreshAuth`，说明组件卸载不会取消已注册的 Promise continuation。

### 假设 3：generated type 足以保护 runtime 与 dev mock

- 验证方法：直接向账号主题投影传入越界/额外字段，并向 dev mock 发送合法 profile + 非法 theme 的 combined request。
- 结果：排除。网络 JSON 仍需运行时校验；mock 解析失败后丢失了“字段是否出现”的信息，导致部分写入。

## 3 根因分析

- `SettingsScreen` 没有 mounted state、请求 generation 与当前 user ID 的联合 ownership guard，异步结果只由请求成功与否决定是否提交。
- `AccountThemeRuntimeSync` 分散读取 `theme/h/c`，只验证部分类型，缺少账号主题 projection 的单一 closed validator 和明确默认值。
- `devMockClient` 只保留“成功解析后的偏好值”，没有单独记录 `displayPreferences` 是否出现在请求中，因此无法区分 absent 与 present-invalid。
- Phase 16 收口依赖最终状态和历史 PASS，没有为 race、非法投影与 combined-invalid 建立反例；product/OpenAPI 当前 owner 也缺少 machine-readable semantic gate。

## 4 修复方案

- 为 Settings 主题保存增加 mounted、generation、user ID 三重 guard；卸载时使 generation 失效并恢复最近一次确认主题，迟到成功/失败结果均不再更新旧组件或认证状态。
- 在 `DisplayPreferencesProvider` 导出 closed `normalizeAccountDisplayPreferences`，统一验证 theme、exact keys、finite hue/chroma 与范围；任一非法投影整体回退 `ocean + null`。
- dev mock 在 mutation 前分别记录字段存在性与解析结果；出现但非法的 `displayPreferences` 整体返回 400，并收紧 exact keys / finite number 校验。
- 原地重开 frontend-shell、product-scope 与 OpenAPI owner，补齐 RED/GREEN、BDD、current-doc semantic tests 和真实 `E2E.P0.101` current-run 证据。

## 6 经验教训

- Promise 的类型正确不等于结果仍属于当前组件或当前用户；任何会写回 auth/runtime 的异步响应都必须显式证明 ownership 仍有效。
- closed OpenAPI schema 在网络边界、fixture transport 与 dev mock 中仍需要运行时 fail-closed；验证失败必须保留 absent 与 present-invalid 的区别。
- BDD/checklist 的历史勾选不能替代当前可执行反例。跨层功能收口至少要覆盖 race、invalid projection、原子拒绝和 active owner 文本漂移。

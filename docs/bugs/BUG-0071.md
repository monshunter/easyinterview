# BUG-0071: frontend debrief record modes drifted from ui-design parity

> 模块: debrief | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-17 | 关联 Commit: `fix(frontend-debrief): restore record mode ui parity (BUG-0071)`
> 关联日志: [2026-05-17](../work-journal/2026-05-17.md)

## 1 现象

- 用户在 `localhost:5173` 复盘页面截图中指出，复盘记录 Step 0 的文本模式与语音模式和 `ui-design` 存在较大差异。
- 文本模式只显示扁平的 `AI 引导问题` 面板和简单问题列表，缺少 `ui-design/src/screens-p1-depth.jsx::DebriefFullScreen` 中的当前引导卡、已记录问题卡片列表、活动问题详情编辑区和右侧 `整体感受` card。
- 语音模式显示 `语音复盘集成中，敬请期待` 类占位状态，缺少原型中的 `AI 语音复盘 · 连续对话` intro card、话题列表、麦克风 CTA 和右侧 `整体感受` card。
- 第一轮修复后，语音 intro card 已出现，但 `开始语音复盘对话` 按钮未接前端状态；点击后不会进入 `ui-design` 中的连续对话 / 实时提取卡片画面。
- 影响范围为 `frontend-debrief/001-debrief-screen-and-handoff` 的用户可见 Step 0 复盘记录入口，属于 UI parity drift。

## 2 诊断过程

### 假设 1：正式前端没有复刻 Step 0 的记录工作区层级

- 验证方法：读取 `docs/ui-design/user-flow.md`、`docs/ui-design/ui-architecture.md`、`ui-design/src/screens-p1-depth.jsx` 中 `DebriefFullScreen` / `GuidedDebriefRecord` / `VoiceDebriefRecord`，对照 `frontend/src/app/screens/debrief/*` 实现。
- 结果：确认。正式实现缺少 `1fr + 320px` record workspace、右侧 vibe check、文本模式 detail grid 与语音 intro card。

### 假设 2：现有 pixel parity gate 只验证宽泛锚点，未覆盖模式内结构

- 验证方法：先向 `frontend/tests/pixel-parity/debrief.spec.ts` 添加文本/语音模式结构断言，然后在修复前运行 focused Playwright。
- 结果：确认。新增断言在修复前失败，缺少 `debrief-record-workspace` 与 `debrief-voice-intro-card`。

### 假设 3：旧语音占位口径仍可见或残留

- 验证方法：修复后反向搜索 `ei-debrief-voice__placeholder`、`debrief-voice-not-implemented`、`语音复盘集成中`、`Voice debrief integration coming soon`。
- 结果：确认已清理。旧占位 copy 和测试锚点不再存在于 frontend 源码与 pixel parity tests 中。

### 假设 4：语音 intro CTA 只渲染，没有接入原型态状态机

- 验证方法：新增 Playwright 断言，点击 `debrief-voice-start` 后必须出现 `debrief-voice-chat`、`debrief-voice-live-extract`、3 张 `debrief-voice-extracted-card` 和 `debrief-voice-end-review`。
- 结果：确认。修复前用例失败，提示找不到 `debrief-voice-chat`；补齐本地连续对话状态后通过。

## 3 根因分析

- 根本原因：正式前端 Step 0 只落地了复盘记录流程的基础交互状态，没有把 `ui-design` 的 mode-specific DOM 层级、布局锚点和语音 CTA 状态转移固化进可执行 parity gate。
- 涉及路径：
  - `frontend/src/app/screens/debrief/DebriefScreen.tsx`
  - `frontend/src/app/screens/debrief/components/GuidedDebriefRecord.tsx`
  - `frontend/src/app/screens/debrief/components/VoiceDebriefRecord.tsx`
  - `frontend/src/app/screens/debrief/debrief.css`
  - `frontend/tests/pixel-parity/debrief.spec.ts`
- 为什么之前没有发现：已有 Playwright parity 只覆盖 Step 0 broad anchors、viewport overflow、theme computed values 与 screenshot smoke，没有断言文本/语音模式内部必须出现的 workspace、vibe check、card list、active card、voice intro card 和 topic list。

## 4 修复方案

- 选择方案：在现有复盘实现中原地对齐 `ui-design` Step 0，而不是重建新页面或保留旧占位入口。
- 修改内容：
  - 新增 `DebriefVibeCheck`，把 `整体感受` card 放入文本/语音共享的 record workspace 右侧。
  - 重构 `GuidedDebriefRecord` 的文本模式层级，补齐当前引导卡、源信息、确认/跳过/编辑动作、已记录卡片列表与活动问题详情编辑区。
  - 重构 `VoiceDebriefRecord` 的语音 intro card，补齐麦克风图标、话题列表、启动 CTA 与提示文案，并删除旧 `coming soon` 占位口径。
  - 补齐 `VoiceDebriefRecord` 的本地连续对话原型态：点击启动后进入聊天线程、显示正在收尾状态、实时抽取 3 张候选卡片、支持暂停、结束进入确认并写入 `voice_extracted` 复盘记录。
  - 扩展 `debrief.css`，按 `ui-design` 的 desktop `1fr + 320px` 与 mobile stacked 行为补齐布局。
  - 扩展 Playwright parity gate，覆盖文本/语音模式关键 DOM 锚点、desktop/mobile 几何关系、语音 start click 状态和旧语音占位回归。
- 放弃方案：只改 CSS 或截图高度。这无法恢复缺失的 record workspace 结构，也无法防止语音模式继续退化成占位态。

## 6 经验教训

- 用户可见 UI parity gate 不能只验证页面级锚点；对 mode/tab 状态必须断言 mode-specific DOM、核心布局关系和旧占位口径零残留。
- 本问题符合 `docs/bugs/PATTERNS.md` 模式 4 的 pixel parity false-green 方向；暂不修改模式库，后续若继续出现模式内 UI parity 漏断言，可将 BUG-0071 追加到模式 4。

# BUG-0116: auth profile setup and scenario evidence drift escaped frontend-shell review

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-28 | 关联 Commit: `fix(frontend-shell): close auth profile L2 gaps (BUG-0116)`
> 关联日志: [2026-05-28](../work-journal/2026-05-28.md)

## 1 现象

- L2 code review 发现 `AuthLoginScreen` 仍会把 legacy `returnTo` 放进 `startAuthEmailChallenge` request body，和当前 Phase 9 “发码只提交 email”的单入口登录契约冲突。
- `auth_profile_setup` 调用 `completeMyProfile` 后会直接恢复 pendingAction；如果后端刷新后的 `/me.profileCompletionRequired` 仍为 true，页面也会提前离开资料补全流。
- `E2E.P0.102` wrapper 只证明 Go package 执行过，未校验 focused test 的 `--- PASS` 名称；修复过程中还暴露 forbidden marker 以 `---` 开头时 `grep` 缺少 `--`，导致验证脚本打印 grep 错误但仍退出 0。

## 2 诊断过程

### 假设 1：单入口登录实现仍沿用旧 `returnTo` request body

- 验证方法：读取 `docs/spec/frontend-shell/plans/001-app-shell-auth-settings/plan.md` Phase 9.2、`AuthLoginScreen.tsx` 与 `AuthScreens.test.tsx`，并先把测试改成断言发码 body 只有 email。
- 结果：确认。Red 阶段 `AuthLoginScreen` 测试失败，实际调用包含 `{ email, returnTo }`。

### 假设 2：资料补全恢复动作没有等待后端确认

- 验证方法：读取 Phase 9.4、`AuthProfileSetupScreen.tsx` 与 `App.tsx` 的 `completeMyProfile` adapter，新增 `profileCompletionRequired=true` 时不得导航的 focused test。
- 结果：确认。Red 阶段页面没有显示错误状态，直接调用 `onNavigate` 恢复 pendingAction。

### 假设 3：P0.102 wrapper 证据不足以证明 focused backend tests 真正执行

- 验证方法：读取 `test/scenarios/e2e/p0-102-auth-gated-interview-routes/scripts/trigger.sh` / `verify.sh`，检查 trigger log 断言粒度，并运行四段场景脚本。
- 结果：确认。wrapper 缺少 runner marker、Go `-v` 输出和 named PASS 断言；修复后首次 verify 还暴露 `grep -Fq "$forbidden"` 对 `--- FAIL:` 这类 pattern 的选项解析缺陷。

## 3 根因分析

- 根本原因：Phase 9 将注册/登录合并为单入口后，部分旧 `returnTo` 兼容路径仍被测试固化为当前行为；资料补全页的组件契约只返回 `Promise<void>`，没有把刷新后的 `/me.profileCompletionRequired` 结果作为恢复业务 route 的硬门禁。
- 场景 wrapper 的根因是验收脚本只检查 package / file 粗粒度 marker，缺少 runner marker、具体测试名和 no-test/skip/fail 负向断言；shell grep pattern 未覆盖以 `-` 开头的固定字符串。
- 涉及的代码路径：
  - `frontend/src/app/auth/AuthLoginScreen.tsx`
  - `frontend/src/app/auth/AuthProfileSetupScreen.tsx`
  - `frontend/src/app/App.tsx`
  - `frontend/src/app/auth/AuthScreens.test.tsx`
  - `frontend/src/app/auth/AuthVisual.test.tsx`
  - `test/scenarios/e2e/p0-102-auth-gated-interview-routes/scripts/trigger.sh`
  - `test/scenarios/e2e/p0-102-auth-gated-interview-routes/scripts/verify.sh`
- 之前没有发现的原因：既有 focused tests 仍接受 stale `returnTo` body，P0.102 wrapper 只证明 runner 大致执行，没有反查 named PASS 和 shell option edge case。

## 4 修复方案

- 选择的方案：把 `AuthLoginScreen` 的 generated API 调用收紧为 `{ email }`，并在进入 `auth_verify` 前剔除 legacy `returnTo`，只保留 safe pendingAction params。
- `AuthProfileSetupScreen` 的 `onCompleteProfile` 改为返回 `{ profileCompletionRequired }`；App adapter 在 `completeMyProfile` 后刷新 auth context 并返回后端结果，组件只在 false 时恢复 pendingAction，否则停留在错误状态。
- P0.102 trigger 使用 Go `-v` 并输出 scenario runner marker；verify 检查 runner marker、目标 Vitest 文件、每个 backend focused test 的 `--- PASS` 名称、package `ok` 行和 forbidden markers，且所有 `grep` 固定字符串都用 `--` 终止选项。
- 考虑过但放弃的替代方案：保留 `returnTo` request body 作为兼容。当前项目尚未上线且 active spec 明确只允许 email 发码，继续兼容会让 stale route param 进入 auth API 契约。

## 6 经验教训

- Auth flow 改版后，旧 `returnTo` / register-purpose / displayName-before-verify 这类兼容路径必须同时做实现负向搜索和测试负向断言，不能只依赖新 happy path。
- 对“先补全资料再恢复业务动作”的契约，组件 callback 类型必须携带后端确认结果；`Promise<void>` 过于宽松，容易让 UI 在后端状态仍未达标时继续导航。
- 场景 wrapper 不能只检查文件名或 package 名；必须校验 runner marker、目标 test 名称、PASS/ok marker，并对 no-test、skip、fail 和 shell option edge case 建立负向断言。

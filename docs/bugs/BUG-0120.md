# BUG-0120: D-20 resume gates missed non-current fixture keys and stale scenario scripts

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-06-14 | 关联 Commit: `fix(test): close resume d20 gate drift (BUG-0120)`
> 关联日志: [2026-06-14](../work-journal/2026-06-14.md)

## 1 现象

- `/plan-code-review backend-resume/002-tailor-runs-and-save-v1 backend --fix` 闭环时发现，D-20 resume flatten 的 gate 仍可被旧测试/旧 fixture key 漏过。
- `make validate-fixtures` 原先只证明 schema / operation 结构一致，未递归拒绝 fixture request / response 内的 non-current resume key。
- P0.078 / P0.080 场景脚本仍引用已删除 live-store 测试名，P0.077 / P0.078 / P0.080 场景说明仍描述专属 tailor run / suggestion table，而当前 D-20 contract 已收敛到 `async_jobs` + `ai_task_runs`。

## 2 诊断过程

### 假设 1：fixture validator 没有扫 non-current key

- 验证方法：为 `scripts/lint/validate_fixtures.py` 增加 request / response key-level negative check 后运行 validator 单测。
- 结果：确认。新增测试立即暴露 `suggestDebriefQuestions.scenarios.default.request.body` 中仍有 non-current key；修复后 validator 单测和 `make validate-fixtures` PASS。

### 假设 2：场景脚本仍在证明旧 tailor store contract

- 验证方法：搜索 P0.077 / P0.078 / P0.080 scenario scripts、README 和 expected data 中的旧 live-store 测试名与专属 tailor 表语义。
- 结果：确认。P0.078 / P0.080 脚本使用已删除测试名，场景说明仍将 tailor run 和 suggestions 描述为专属表持久化。

### 假设 3：owner 文档仍可能把历史 Phase 1-9 当成当前 gate

- 验证方法：反查 `backend-resume/002` plan / checklist / BDD docs，确认是否存在 D-20 current matrix 和 current scenario checklist。
- 结果：确认。Phase 10 已存在，但缺少足够明确的 current-vs-historical 执行边界；已补齐 D-20 current operation matrix、BDD checklist 和 current gate note。

## 3 根因分析

- 根本原因：D-20 schema / operation deletion 后，验证集中在 OpenAPI operation count、runtime route 和部分 backend tests，未把 fixture payload key、scenario script test-name drift 和 BDD semantic drift 纳入同一个 owner-level hardening gate。
- 涉及路径：
  - `scripts/lint/validate_fixtures.py`
  - `scripts/lint/validate_fixtures_cli_test.py`
  - `openapi/fixtures/Debriefs/suggestDebriefQuestions.json`
  - `test/scenarios/e2e/p0-077-resume-tailor-async-dispatch-and-ready/`
  - `test/scenarios/e2e/p0-078-resume-tailor-failure-and-retry/`
  - `test/scenarios/e2e/p0-080-resume-versions-privacy-legacy/`
  - `docs/spec/backend-resume/plans/002-tailor-runs-and-save-v1/`
- 之前没有发现，是因为 scenario PASS 只检查脚本退出码和部分历史 markers，fixture validator 也没有递归检查 non-current key literal。

## 4 修复方案

- 在 fixture validator 增加 recursive non-current-key scan，拒绝 `resumeAssetId` / `resumeVersionId` 出现在 fixture request / response payload 中。
- 修正 Debriefs fixture，将旧 resume version key 改为当前 `resumeId`。
- 更新 P0.077 / P0.078 / P0.080 scenario docs/data/scripts 到当前 `async_jobs` + `ai_task_runs` tailor contract，并修正 live-store 测试名。
- 更新 `backend-resume/002` plan / checklist / BDD docs，明确 Phase 1-9 仅为历史 baseline，当前验收以 Phase 10 + D-20 BDD checklist 为准。
- 清理 runtime 注释中的 non-current literal，避免 D-20 zero-reference gate 被注释残留误报。

## 6 经验教训

- 删除 API / schema key 时，fixture validator 必须递归扫描 payload key，而不仅验证 schema-valid。
- 场景脚本的 test-name markers 应该与当前 code truth source 绑定；旧测试名要作为 negative grep gate 固化。
- owner plan 承接历史重塑时，必须明确 current-vs-historical gate 边界，避免历史完成项被误用为当前证据。

# BUG-0114: auth registration treated display name as account identity boundary

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-27 | 关联 Commit: `fix(auth): enforce email identity semantics (BUG-0114)`
> 关联日志: `-`

## 1 现象

- 用户澄清登录邮箱和注册邮箱必须是同一个账号标识，注册邮箱就是后续登录邮箱。
- 旧实现和文档仍混有 mail-link / displayName 注册口径，未把 email uniqueness 作为注册与登录共享的不变量。
- 重复注册同一邮箱在真实场景中可能进入发码/rate-limit 分支，不能稳定表达“邮箱已注册，请直接登录”，也不能证明 displayName 不会覆盖原账号。

## 2 诊断过程

### 假设 1：只是前端注册页没有传 displayName

- 验证方法：反查 `AuthRegisterScreen`、`AppAuthDispatch` 与 generated `AuthEmailStartRequest`。
- 结果：确认前端原先没有稳定传递 `purpose=signup` 与 displayName，且 verify 页仍保留 magic-link token callback 口径；但这只是表层问题。

### 假设 2：后端缺少 signup/login purpose 分流和 email unique gate

- 验证方法：反查 auth service/store/handler、`users.email` 唯一约束、challenge 记录字段和重复注册路径。
- 结果：确认后端需要把 email 作为唯一账号标识：signup 创建未注册邮箱，login 只允许已注册邮箱；重复 signup 必须在 start 阶段返回 409，不创建 challenge、不发送 code。

### 假设 3：BDD 场景未覆盖同一邮箱注册后再登录

- 验证方法：运行 P0.101 real frontend/backend/Mailpit 场景并检查 trigger/verify 断言。
- 结果：确认需要新增真实 same-email lifecycle：register -> logout -> login same email -> duplicate register rejected before code。

## 3 根因分析

- 认证契约把 passwordless challenge 当作通用邮件验证，没有显式携带 `purpose=login|signup` 与 registration displayName。
- 注册与登录没有在 service 层共享 email identity invariant，重复注册可以落入 challenge/rate-limit 行为，而不是稳定的账号唯一性错误。
- 前端和场景 gate 之前更关注 mail-link callback 可用性，缺少“同一邮箱注册后就是登录邮箱”和“displayName 不唯一”的端到端断言。

## 4 修复方案

- OpenAPI 增加 `AuthEmailStartRequest.purpose` 与 `displayName`，verify token schema 收紧为 6 位数字 code，并重新生成 Go/TS generated artifacts。
- 后端将 challenge code 改为 6 位数字、TTL 改为 5 分钟，challenge 持久化 displayName；signup verify 创建唯一 email 用户，login verify 读取既有 email 用户；重复 signup 在 start 阶段返回 409 且不创建 challenge / 不发送 code。
- 前端注册页提交 `purpose=signup` + trimmed displayName，登录页提交 `purpose=login`；重复注册错误停留在 register 并显示 inline 状态；TopBar 删除样例身份 fallback。
- P0.101 改为 email-code same-email lifecycle，并断言重复注册不发新邮件、displayName 不被覆盖、日志不泄露 raw code/session。

## 6 经验教训

- Auth plan 必须把 account identity invariant 写成 service-level gate，而不是只依赖数据库唯一索引。
- 涉及注册/登录双入口时，BDD 必须覆盖同一邮箱跨入口生命周期与重复注册负向路径。
- displayName、昵称、语言偏好等 profile 字段不得参与账号唯一性判断；测试命名和场景数据应显式区分 identity 字段与 profile 字段。

# BUG-0066: debrief scenario wrappers were missing or under-asserted

> 模块: test | 严重度: high | 状态: resolved
> 发现日期: 2026-05-17 | 关联 Commit: `fix(test): close debrief scenario wrapper evidence (BUG-0066)`
> 关联日志: [2026-05-17](../work-journal/2026-05-17.md)

## 1 现象

- `backend-practice/004-derived-plans-debrief` 的 `E2E.P0.070-073` 在 `test/scenarios/e2e/INDEX.md` 中标记为 `Ready`，但对应目录只有 README 和 data 文件，缺少场景框架要求的 `scripts/setup.sh` / `trigger.sh` / `verify.sh` / `cleanup.sh`。
- `backend-debrief/001` 的 `E2E.P0.060-064` 已有 wrapper，但 `verify.sh` 主要检查测试名和 `PASS` 字样，没有显式拒绝 `FAIL` / `no tests to run`，也没有逐项要求 `--- PASS` 与 package-level `ok`。
- `backend-debrief/001` completed checklist 中还保留一条 `pending` 证据注释；`backend-practice/004` bdd-plan 中 `E2E.P0.073` 的验证入口测试名也与实际 Go test 名称不一致。

## 2 诊断过程

### 假设 1：场景目录是否完整
- 验证方法：读取 `test/scenarios/README.md`、`test/scenarios/e2e/README.md`，并列出 `p0-070` 到 `p0-073` 目录内容。
- 结果：确认框架要求每个场景目录至少有四段脚本；`E2E.P0.070-073` 缺少整个 `scripts/` 目录。

### 假设 2：既有 backend-debrief wrapper 是否足以证明 BDD gate
- 验证方法：读取 `p0-060` 到 `p0-064` 的 `trigger.sh` / `verify.sh`。
- 结果：`trigger.sh` 通过 `set -euo pipefail` 保留了 runner 失败状态，但 `verify.sh` 缺少 no-op / failure 反查和 package `ok` 断言，无法独立证明 focused Go test 没有 false-green。

### 假设 3：文档证据是否与 runtime artifact 对齐
- 验证方法：反查 `backend-debrief/001` 与 `backend-practice/004` 的 checklist、BDD checklist 和 bdd-plan。
- 结果：发现 completed checklist 中仍有 `pending` 注释，004 BDD 证据只引用 Go test 命令而不是四段脚本，且 073 的测试名写成旧名称。

## 3 根因分析

- 004 补充 derived debrief plan 时把 Go HTTP scenario test 当成了完整 BDD 场景证据，没有同时创建场景框架要求的 wrapper 资产。
- 当时的 `scripts/lint/scenario_script_contract_test.py` 只覆盖 backend-practice/003 的 P0.048-P0.051，没有覆盖 backend-debrief/001 或 backend-practice/004 的新增场景。
- completed 文档证据依赖历史勾选，缺少对 `Ready` scenario 是否真的具备可执行 wrapper 的反查。
- 涉及路径：
  - `test/scenarios/e2e/p0-060-*` 到 `p0-064-*`
  - `test/scenarios/e2e/p0-070-*` 到 `p0-073-*`
  - `scripts/lint/scenario_script_contract_test.py`
  - `docs/spec/backend-debrief/plans/001-debrief-record-and-analysis/checklist.md`
  - `docs/spec/backend-practice/plans/004-derived-plans-debrief/`

## 4 修复方案

- 为 `E2E.P0.070-073` 新增四段脚本：setup 记录场景输出目录，trigger 执行精确 Go HTTP scenario 并显式保留 `go test` exit status，verify 断言 `=== RUN` / `--- PASS` / package `ok` 并拒绝 `FAIL` / `no tests to run`，cleanup 幂等清理 setup marker。
- 加固 `E2E.P0.060-064` 既有 verify：补齐目标测试开始/通过、package-level `ok`、失败/no-op 反查。
- 扩展 `scenario_script_contract_test.py`，把 backend-debrief 060-064 和 backend-practice 070-073 纳入脚本契约测试。
- 更新 backend-debrief / backend-practice plan 与 BDD 文档证据，清除 completed 文档中的 `pending` 漂移，并修正 073 验证入口测试名。

## 6 经验教训

- `Ready` 场景不能只代表 Go test 存在；必须有 repo-defined wrapper 资产，并由 lint/test 反查 `trigger.sh` 与 `verify.sh` 是否能证明 runner 执行、pass marker、package ok 和 no-op rejection。
- L2 review 需要把 scenario wrapper 当作独立证据 artifact；只读 Go test body 会漏掉场景框架契约漂移。

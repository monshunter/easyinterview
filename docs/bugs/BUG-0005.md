# BUG-0005: report follow-up CTAs returned to setup instead of starting sessions

> 模块: review | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-02 | 关联 Commit: `fix(ui-design): start report follow-up sessions directly`
> 关联日志: [2026-05-02](../work-journal/2026-05-02.md)

## 1 现象

- 用户指出报告页的 `复练当前轮` 和 `进入下一轮` 都应直接进入面试，而不是回到面试前准备页再让用户二次点击。
- 静态 UI 报告页顶部和详情页 CTA 之前都把用户带回 `workspace`，导致“复练 / 下一轮”动作和“返回面试前确认”混在一起。
- 影响范围是 `ui-design` 报告闭环：用户从报告继续练习时，会被带离报告后续动作的直接执行路径。

## 2 诊断过程

### 假设 1：只有按钮文案有歧义，路由已经进入面试

- 验证方法：检查 `ui-design/src/screen-report.jsx` 中 `goReplay`、`goNextRound`、详情页 CTA 和 `requestAuth` pending action。
- 结果：排除。CTA 的执行函数和 pending action 都指向 `workspace`，不是 `practice`。

### 假设 2：直接进入面试需要保留报告上下文 payload

- 验证方法：检查 `ReportDashboard` 中 `replayContext`、`nextRoundContext` 和 `InterviewContext` 贯通规则。
- 结果：确认。复练需要保留 `sourceSessionId`、`replayItems`、`evidenceGaps`；下一轮需要保留同一 `targetJobId`、`resumeVersionId` 并切换 `roundId`。

### 假设 3：设计文档仍把报告后续动作定义为回到规划页

- 验证方法：检索 `docs/ui-design` 中 `Mock Interview Plan(same round)`、`创建下一轮面试规划`、`准备下一轮` 等表述。
- 结果：确认。`report-dashboard.md`、`user-flow.md`、`ui-architecture.md` 和 `module-practice-review.md` 都有需要同步的旧口径。

## 3 根因分析

- 根本原因是报告页 CTA 的动作语义没有和返回动作分离：`复练当前轮` / `进入下一轮` 是创建并进入新面试 session 的动作，但实现沿用了“返回面试前确认”的 `workspace` 路由。
- 涉及的代码路径：
  - `ui-design/src/screen-report.jsx`: 报告 CTA、详情页本轮复练按钮、下一轮按钮、pending action route。
  - `ui-design/ui-design-contract.test.mjs`: 之前没有断言报告后续动作必须进入 `practice`。
  - `docs/ui-design/*.md`: 多处文档仍描述为回到或创建面试规划。
- 之前没有发现，是因为原契约测试只验证了报告区分两条路径，没有验证两条路径的目标 route 和 sessionId 语义。

## 4 修复方案

- 将报告页 `复练当前轮` 和 `进入下一轮` 的执行 route 改为 `practice`，并把 `requestAuth` 的 `pendingAction.route` 同步改为 `practice`。
- 将复练 sessionId 设为当前轮 replay session，将下一轮 sessionId 设为 next-round start session，避免继续使用准备页语义的 `prep` 后缀。
- 保留 `返回面试前确认` 作为唯一回 `workspace` 的显式返回动作。
- 更新 `docs/ui-design` 中报告、用户流程、架构、认证入口和模块文档，明确两条报告 CTA 都直接进入对应面试 session。
- 增加契约测试，阻止 `route: "workspace"`、`prepare-next-round` 和 `*-prep` 回归到报告 CTA。

## 6 经验教训

- 报告页后续动作必须同时校验“文案区分路径”和“目标 route 符合动作语义”，不能只看按钮是否存在。
- 所有带登录拦截的 CTA 都要同时检查直接执行路径和 pending action 恢复路径，否则未登录用户会进入不同流程。
- 复练、下一轮、返回准备页三类动作应在文档和测试中保持互斥，避免用户被迫二次确认。

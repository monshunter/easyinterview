# BUG-0132: home resume selector and recent mocks drifted from compact shortcut UX

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-06 | 关联 Commit: `fix(frontend-home): use resume dropdown and cap recent mocks (BUG-0132)`
> 关联日志: [2026-07-06](../work-journal/2026-07-06.md)

## 1 现象

- 用户截图反馈 Home 的“选择已有简历”把所有 ready 简历平铺成静态卡片/按钮列表，而不是紧凑下拉框。
- Home “最近模拟面试”在 12+ fixture 下展示多行卡片，占据首页过多空间；当前产品语义只需要展示 3 个最近快捷入口。
- Recent mocks 区域缺少“更多”入口，用户无法从首页快捷跳转到完整“模拟面试”列表页。

## 2 诊断过程

### 假设 1：Phase 8 的“选择已有简历”允许用平铺列表表达

- 验证方法：对照用户截图反馈、owner spec / plan、`docs/ui-design/` 与 `ui-design/src/screen-home.jsx`，并新增 `HomeResumeSelection.test.tsx` 控件类型断言。
- 结果：排除。首页已经是“模拟面试”新建规划快捷方式，简历选择应是表单控件而不是把所有简历平铺在页面上；红灯显示旧实现的 `home-resume-select` 是 `DIV`，不是 `SELECT`。

### 假设 2：Recent mocks 保留 12 张展示更符合原 plan

- 验证方法：对照用户反馈和当前首页信息架构，更新 `HomeRecentMocks.test.tsx` 的 twelve-plus 断言为只渲染 3 张最近卡片，并要求 `home-recent-more` 跳转 `workspace`。
- 结果：排除。旧实现仍按历史 12-card cap 展示，红灯显示页面渲染 12 张卡片而不是 3 张。

### 假设 3：只改正式前端即可闭环

- 验证方法：检查 UI 真理源、P0.014 / P0.015 场景说明、owner plan / BDD gate 与 pixel parity gate。
- 结果：排除。该偏差来自 Phase 8 UI contract 表述不够具体，必须同步修订静态原型、docs/ui-design、owner spec/plan/BDD、scenario docs 与正式前端测试。

## 3 根因分析

- Phase 8 只固化了“首页必须选择已有简历”，但没有把控件形态锁为 dropdown / combobox；正式 `HomeScreen` 因此用平铺按钮列表满足了数据绑定，却偏离了首页紧凑表单语义。
- Phase 2 历史 checklist 和 P0.014 gate 保留了 `listTargetJobs` 最多 12 条的列表展示口径，没有随着“Recent mocks 只是首页快捷入口”的新产品语义改为 3-card cap。
- 测试覆盖了“有简历可选”和“recent mocks 排序”，但缺少 `SELECT` 控件类型、旧平铺按钮负向断言、3-card cap 和“更多”跳转断言。

## 4 修复方案

- 更新 `ui-design/src/screen-home.jsx`：将 Home resume selection 改为 native dropdown，Recent mock interviews 只渲染 `recentJobs.slice(0, 3)`，超过 3 条时显示“更多”并跳 `workspace`。
- 更新正式 `HomeScreen`：`home-resume-select` 使用 native `select` / combobox，保留未选择 ready 简历时禁用 import；recent mocks 继续按 `updatedAt desc` 排序后只展示 3 张，`home-recent-more` 通过受保护路由进入 `workspace`。
- 更新 i18n、Home focused tests、pixel parity test、UI contract test、owner spec / plan / BDD 和 P0.014 / P0.015 scenario docs。
- 把 checklist Phase 9 与 BDD gate 写回已验证证据，并将 plan lifecycle 恢复为 `completed`。

## 6 经验教训

- 当用户指定“选择”类控件时，plan 和测试必须锁定控件类型，避免用平铺按钮列表满足数据语义但破坏页面密度。
- 首页 recent 区域如果是快捷入口而不是完整列表，scenario gate 应断言显示上限和“更多”去向，不能沿用历史 pageSize / 12-card fixture 口径。
- UI screenshot feedback 要回写 owner spec / plan / BDD，而不是只修正式前端；否则下一轮 parity 或场景测试会继续接受旧布局。

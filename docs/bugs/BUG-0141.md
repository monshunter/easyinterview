# BUG-0141: home resume picker hid readable existing resumes

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-08 | 关联 Commit: `fix(resume): close source preview and picker regressions (BUG-0139, BUG-0140, BUG-0141)`
> 关联日志: [2026-07-08](../work-journal/2026-07-08.md)

## 1 现象

- 用户在 Home 新建模拟面试入口看到 `选择已有简历` 下拉框不可点击，并显示 `还没有可用简历`。
- 同一账号的简历列表中已经存在 3 份简历，用户无法把已有简历绑定到 JD 导入流程，只能被错误引导去新建简历。
- 影响范围覆盖 Home JD quick-start 入口，并可能在 Parse 确认页复现相同的可选简历过滤问题。

## 2 诊断过程

### 假设 1：Home 没有调用 `listResumes`

- 验证方法：读取 `frontend/src/app/screens/home/HomeScreen.tsx` 和 `HomeResumeSelection.test.tsx`。
- 结果：排除。Home 已在 authenticated runtime 下调用 `runtime.client.listResumes`，并用 native `<select data-testid="home-resume-select">` 渲染选项。

### 假设 2：真实返回的简历存在，但被前端过滤掉

- 验证方法：新增 red test，让 `listResumes` 返回 3 条非归档且已有 `parsedTextSnapshot` / `originalText` 的简历，但 `parseStatus` 分别为 `failed`、`queued`、`processing`。
- 结果：确认。旧实现只接受 `parseStatus === "ready"`，测试中 select 保持 disabled，且空态继续显示。

### 假设 3：只有 Home 有过滤漂移

- 验证方法：读取 `frontend/src/app/screens/parse/ParseScreen.tsx`。
- 结果：排除。Parse 确认页也复制了同一 `parseStatus === "ready"` 判断，存在同类回归风险。

## 3 根因分析

- Home 和 Parse 各自维护了本地 `isSelectableResume`，只允许 `parseStatus === "ready"` 的简历进入选择列表。
- 当前 Resume 详情合同已经允许 `failed` 或 `queued` / `processing` 但已有可读正文快照的简历展示正文；这些简历对用户来说已经是可识别、可选择的已有简历。
- 之前测试只覆盖 ready fixture 和 empty fixture，没有覆盖“列表中存在非归档且已有可读正文，但 parseStatus 尚未 ready”的状态组合。

## 4 修复方案

- 新增 `frontend/src/app/interview-context/selectableResume.ts`，集中定义 `isSelectableInterviewResume`。
- 判断规则改为：非归档且未删除，并且 `parseStatus === "ready"` 或已有 `parsedTextSnapshot` / `originalText` / non-empty `structuredProfile`。
- Home 和 Parse 改为复用同一 helper，避免两个入口再次漂移。
- 在 `frontend-resume-workshop/002-create-flow` plan/checklist/BDD 中补充 Phase 8 和 E2E.P0.084，固化 Home existing resume picker 回归门禁。

## 6 经验教训

- 用户可选择的业务资产不要只看单一状态枚举；当产品合同已经承认可读正文快照为有效展示证据时，选择器也要复用同一可读性判断。
- Home 与 Parse 这类连续流程入口不应复制过滤规则；共享 helper 更容易让测试覆盖同一业务不变量。

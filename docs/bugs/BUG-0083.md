# BUG-0083: jd-match runtime bypassed A3/F3 and missed search/privacy contracts

> 模块: schema | 严重度: high | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(backend-jobs): harden jd-match runtime contracts (BUG-0083)`
> 关联日志: `-`

## 1 现象

- `/plan-code-review backend-jobs-recommendations/001-jd-match-real-backend-baseline --fix` 深审时，`main.go` 仍以 `stubJDMatchAI{}` / `generatorAIAdapter{}` 启动 JD-Match runtime。
- F3 `jd_match.recommendation` / `jd_match.search` prompt 没有把 `jobMatchId` 写成必填输出，但 runtime search handler 需要通过 ID join 既有 `jd_match_recommendations`。
- `searchJobs` 只写 `jd_match_search_runs`，没有发射 B3 `jd_match.search.completed` outbox event。
- `DeleteJobMatchDataForUser` 在 cmd/api runtime 下不是同一事务，plan 中的失败回滚证据不能成立。

## 2 诊断过程

### 假设 1：JD-Match production runtime 已复用 targetJobRuntime.AI.Client

- 验证方法：读取 `backend/cmd/api/main.go` 与 `backend/cmd/api/jdmatch_runtime.go`。
- 结果：排除。`main.go` 直接传入 `stubJDMatchAI{}` 与 `generatorAIAdapter{}`，导致生产启动路径绕过 A3 AIClient 与 F3 registry。

### 假设 2：F3 prompt 与 runtime search join 契约一致

- 验证方法：读取 `config/prompts/jd_match.{recommendation,search}/v0.1.0*.md` 与 `backend/internal/jdmatch/handler/search.go`。
- 结果：排除。prompt 只要求推荐字段集，未要求保留 `jobMatchId`；handler 只能使用 AI 返回的 ID 列表投影已有 recommendation。

### 假设 3：search/outbox/privacy 已由 live scenario 覆盖

- 验证方法：读取 `TestJDMatchHTTPScenario`、outbox assertions 和 privacy delete helper。
- 结果：部分确认。live scenario 覆盖 search run 与 IK replay，但缺少 `jd_match.search.completed` outbox 检查、output_invalid 不新增 search_run 检查，以及事务 rollback 单测。

## 3 根因分析

- 根本原因：post-reopen completion 主要补齐 route / drainer / BDD wrapper 证据，但没有继续反查 production composition、F3 prompt body 与 B3/B4 failure semantics 是否与 spec D-9 / D-10 / D-12 / D-15 一致。
- 涉及路径：`backend/cmd/api/main.go`、`backend/cmd/api/jdmatch_runtime.go`、`backend/internal/jdmatch/handler/search.go`、`backend/internal/jdmatch/generators/recommendation.go`、`config/prompts/jd_match.*`、`docs/spec/backend-jobs-recommendations/plans/001-jd-match-real-backend-baseline/*`。
- 之前没有发现：focused gates 证明 handler 与 wrapper 可执行，但没有断言 production runtime 必须使用 real A3/F3 adapter，也没有把 prompt `jobMatchId`、search completed event、transaction rollback 写成可执行 tests。

## 4 修复方案

- 新增 `buildJDMatchAIAdapters(loader, db, targetJobRuntime.AI.Client)`，用 F3 registry Resolve `jd_match.search` / `jd_match.recommendation`，再通过 A3 `AIClient.Complete` 调用对应 model profile。
- `main.go` 改为使用真实 JD-Match A3/F3 adapters；stub 仅保留给单元 smoke tests。
- F3 JD-Match prompt 明确要求保留 `jobMatchId`，并刷新 4 个 prompt `template_hash`；新增 `TestBackendJDMatchF3Preflight`。
- `searchJobs` 成功后发射 privacy-safe `jd_match.search.completed` outbox event；output_invalid 映射为 `AI_OUTPUT_INVALID` 且不新增 search_run。
- cmd/api privacy delete 进入单一 SQL transaction，并新增 commit / rollback 单测。

## 6 经验教训

- Completed backend plan 的 L2 review 必须反查 production composition，不只看 handler/unit/scenario green。
- AI feature_key 不能只检查 registry 文件存在；prompt body、runtime parser 和 downstream join key 必须有同一条 preflight 测试覆盖。
- 对 B3/B4 cross-owner additive，事件发射与 transaction rollback 都必须有 runtime-level tests，不能只在 service-level dependency test 中证明。

# BUG-0101: parse target switch kept stale preview

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-24 | 关联 Commit: `fix(frontend-home): reset parse state on target switch (BUG-0101)`
> 关联日志: [2026-05-24](../work-journal/2026-05-24.md)

## 1 现象

- `ParseScreen` 已经进入 parsed preview 后，如果同一 mounted component 收到新的 `route.params.targetJobId`，页面会继续显示旧 TargetJob 的 title/company/requirements。
- `getTargetJob(newTargetJobId)` 可以返回新的 ready TargetJob，但 ready response 只写入 `pendingReadyJob`；hydration effect 要求 `stage === "loading"`，因此旧 preview 状态下不会 hydrate 新数据。
- 影响范围是 SPA 内同一路由参数切换、history 导航或父组件 rerender 不重建 `ParseScreen` instance 的路径；初始进入 parse 和 re-parse flow 不受影响。

## 2 诊断过程

### 假设 1：route `targetJobId` 切换会重新挂载 ParseScreen

- 验证方法：反查 App route render 与 `ParseScreen` 使用方式，并在 `ParseFlow.test.tsx` 中用同一个 component rerender 不同 `targetJobId`。
- 结果：排除。App 路由不会给 `ParseScreen` 加 key；同一路由参数变化会复用 mounted instance，因此内部 React state 必须主动 reset。

### 假设 2：ready response hydration effect 会覆盖旧 preview

- 验证方法：复现 `target-a` preview 后 rerender 到 `target-b`，让 mock API 返回新的 `analysisStatus=ready`，断言 `parse-loading-step-0` 出现且旧 title 消失。
- 结果：排除。红灯显示 DOM 仍停留在 `Senior Frontend Engineer` 旧 preview，`parse-loading-step-0` 不存在。

### 假设 3：修复只需在 targetJobId 变化时重置 Parse 本地状态

- 验证方法：新增 target switch regression，修复后重跑 focused parse flow、parse suite、frontend build 与 P0.015/P0.016 scenario wrappers。
- 结果：确认。`targetJobId` 变化时清空旧 preview/edit/pending/error state 并回到 loading，可以保留 ready-response loading gate、failed state、confirm handoff 与 re-parse 行为。

## 3 根因分析

- BUG-0099 修复把 ready TargetJob 延迟到 loading demo 完成后 hydrate，但没有处理 `targetJobId` 变化时旧 `stage` 仍是 `preview` 的路径。
- `ParseScreen` 的 editable state、hit toggles、error state、pending ready state 与 `targetJobId` 没有建立 reset 边界。
- 既有 `ParseFlow.test.tsx` 覆盖 initial mount ready、queued/processing polling、failed、re-parse 和 unmount cleanup，但没有覆盖 same mounted route-param switch。

## 4 修复方案

- 在 `ParseScreen` 中抽出稳定 `targetJobId`，避免 effect dependency 直接依赖 `route.params?.targetJobId` 表达式。
- 新增 `useEffect([targetJobId, _mockStage, _mockTargetJob])`：非 mock mode 下，当 `targetJobId` 变化时清理 polling timeout、旧 TargetJob、editable fields、hit toggles、error/confirm error、pending ready job、loading completion 和 step，并把 `stage` 置回 `loading`。
- `getTargetJob` polling effect 使用稳定 `targetJobId`；ready response 继续进入 loading gate，待 tick 完成后 hydrate 新 TargetJob。
- 新增 `ParseFlow.test.tsx` rerender regression，断言旧 preview 清空、新 loading DOM 出现、loading gate 后新 title 渲染。

## 5 验证

- Red gate：新增 `ParseFlow.test.tsx` same-route target switch regression 后，旧实现失败，`parse-loading-step-0` 不存在且旧 title 仍在 DOM。
- `pnpm --filter @easyinterview/frontend test src/app/screens/parse/ParseFlow.test.tsx`：7 tests PASS。
- `pnpm --filter @easyinterview/frontend test src/app/screens/parse`：5 files / 28 tests PASS。
- `pnpm --filter @easyinterview/frontend build`：PASS（仅保留既有 Vite chunk size warning）。
- `test/scenarios/e2e/p0-015-jd-import-and-parse/scripts/trigger.sh` → `verify.sh`：PASS，含 ready-response loading browser gate screenshotBytes markers。

## 6 经验教训

- SPA route params 变化不等同于 component remount；屏幕级 state 必须把 owner identity（如 `targetJobId`）作为 reset boundary。
- 已有 initial mount 和 re-parse 测试不能替代 same mounted rerender regression；route-param 切换应显式断言旧 DOM 消失。
- 对 ready-response loading gate 的后续修复，需要同时检查 pending state 与当前 stage 的交叉条件，避免新 gate 只覆盖首次进入路径。

# BUG-0139: resume detail rendered Markdown as plain text and PDF AI failure kept non-Markdown snapshot

> 模块: materials | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-07 | 关联 Commit: `fix(resume): close source preview and picker regressions (BUG-0139, BUG-0140, BUG-0141)`
> 关联日志: [2026-07-08](../work-journal/2026-07-08.md)

## 1 现象

- 用户复核简历详情页时发现正文只是展示文本内容，`**bold**` / `[link](url)` 等 Markdown 语义没有由正式 Markdown 引擎渲染。
- PDF 解析后进入 AI 输出异常或 provider 失败路径时，失败快照仍可能是抽取后的普通文本，不满足“后续统一 UI 渲染 Markdown 文档”的合同。
- 影响范围覆盖简历详情预览、dev fixture 截图验收、PDF 上传解析失败后的可读兜底正文。

## 2 诊断过程

### 假设 1：前端已使用 Markdown 引擎但样式不可见

- 验证方法：检查 `frontend/src/app/screens/resume-workshop/components/ResumePreviewTab.tsx` 与 focused test。
- 结果：确认旧实现不是正式 Markdown 引擎，只是有限解析标题和列表；inline strong/link 没有 DOM 语义。

### 假设 2：Prompt 没有要求 LLM 输出 Markdown

- 验证方法：读取 `config/prompts/resume.parse/v0.1.0.md`、prompt schema 与后端 decoder。
- 结果：排除。Prompt 已要求 `markdownText` 保留顺序、结构、措辞与事实，仅归一化为 Markdown；后端 success path 也要求并持久化 `markdownText`。

### 假设 3：AI 失败路径没有 Markdown 兜底

- 验证方法：新增 PDF 抽取成功但 AI 返回 invalid JSON 的 focused Go 测试。
- 结果：确认。失败路径虽保存了抽取文本，但没有 Markdown 化章节和列表，用户仍会看到非统一格式正文。

## 3 根因分析

- 前端详情预览停留在手写 Markdown 子集解析，缺少 `react-markdown` / GFM 引擎，无法正确渲染 inline strong/link 等常见 Markdown 语义。涉及代码路径：`frontend/src/app/screens/resume-workshop/components/ResumePreviewTab.tsx:1`。
- 后端 success path 已走 LLM `markdownText`，但 failure path 在 AI provider/output 错误后没有把已抽取 PDF 文本转换为 Markdown fallback。涉及代码路径：`backend/internal/resume/jobs/parse.go:420`。
- 之前的测试只覆盖正文可见与标题/列表形态，没有覆盖 inline Markdown DOM，也没有覆盖 PDF 抽取成功但 AI 输出失败的兜底格式。

## 4 修复方案

- 前端引入 `react-markdown` + `remark-gfm`，用正式 Markdown renderer 渲染 `parsedTextSnapshot`，并开启 `skipHtml` 避免把快照中的 HTML 当作可信 DOM 注入。
- 后端保留 success path 的 LLM Markdown 合同：AI prompt 继续要求 `markdownText`，decoder 要求字段存在且非空，success 持久化 `markdownText`。
- 后端补齐 failure path：在 PDF/文本抽取已成功但 AI provider/output 失败时，用可读 Markdown fallback 写入 `parsedTextSnapshot`，包含标题、章节和技能类列表转换。
- dev fixture 改为包含标题、列表、strong、link 的 Markdown 样本，保证浏览器截图和 fixture parity 测试能证明 renderer 生效。

## 6 经验教训

- Markdown 渲染不能用“文本看起来接近”验收；必须断言 DOM 中存在 `strong` / `a` / heading / list 等语义元素。
- AI success contract 和 failure fallback contract 要分开测。Prompt 要求 `markdownText` 只能覆盖 success path，AI 失败后的快照也需要可读、可渲染的 Markdown 兜底。
- 本地浏览器验收要先确认前端运行模式。`VITE_EI_API_MODE=real` 下应走真实 Mailpit 登录；fixture 视觉验收应显式启动 mock-mode 备用端口，避免把真实后端鉴权失败误判成前端问题。

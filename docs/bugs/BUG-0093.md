# BUG-0093: Ready e2e scenarios were missing required data assets

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-23 | 关联 Commit: `fix(test): add missing scenario data assets (BUG-0093)`
> 关联日志: [2026-05-23](../work-journal/2026-05-23.md)

## 1 现象

- `/scenario-env verify` 后的场景结构扫描发现 9 个 `Ready` e2e 场景缺少 `data/seed-input.md` 或 `data/expected-outcome.md`。
- 受影响范围包括 `E2E.P0.014`-`E2E.P0.017` 和 `E2E.P0.065`-`E2E.P0.069`。
- 这些场景的 README 与脚本已存在，`INDEX.md` 也标记为 `Ready`，但不满足 `test/scenarios/README.md` 定义的完整场景契约。

## 2 诊断过程

### 假设 1：场景环境仍在使用旧 Kind 入口导致误报

- 验证方法：读取 `test/scenarios/README.md`、`test/scenarios/e2e/README.md`，并执行本地 runner 结构检查。
- 结果：排除。当前场景框架只保留本地 runner 契约，不默认要求 Kind / K8s / Helm；缺口来自场景目录资产不完整。

### 假设 2：缺失仅是未使用的文档资产，不影响场景执行

- 验证方法：对比 `test/scenarios/README.md` 的场景契约，读取 9 个场景的 README 和脚本。
- 结果：排除。`data/seed-input.md` 和 `data/expected-outcome.md` 是每个场景目录的必需资产，用于固定输入、期望输出和后续审查证据。

### 假设 3：补齐资产后场景真实 runner 仍可能失败

- 验证方法：补齐资产后串行执行 `E2E.P0.014`、`E2E.P0.015`、`E2E.P0.016`、`E2E.P0.017`、`E2E.P0.065`、`E2E.P0.066`、`E2E.P0.067`、`E2E.P0.068`、`E2E.P0.069` 的 `setup -> trigger -> verify -> cleanup`。
- 结果：排除。`scenario-run-20260523T073633Z` 中 9 个场景全部 PASS，`failed=0`，`errored=0`。

## 3 根因分析

- 之前的 owner gate 修复聚焦在 wrapper 是否真正执行 runner、verify 是否反查日志，但没有把 `Ready` 场景的必需资产清单纳入同一轮结构 preflight。
- 场景目录的 README、脚本和 `INDEX.md` Ready 状态容易让审查误以为场景完整，缺少 `data/` 资产时不会被 `bash -n`、`docs-check` 或单纯 runner pass 自动暴露。
- 该缺口位于 `test/scenarios/e2e/p0-014-*`、`p0-015-*`、`p0-016-*`、`p0-017-*`、`p0-065-*`、`p0-066-*`、`p0-067-*`、`p0-068-*`、`p0-069-*`。

## 4 修复方案

- 为 9 个受影响场景补齐 `data/seed-input.md` 和 `data/expected-outcome.md`。
- 每个 `seed-input.md` 固定场景输入、fixture、real backend overlay 或 gate 前置条件。
- 每个 `expected-outcome.md` 固定用户可见结果、runner 证据、privacy / legacy negative gate 和 cleanup 期望。
- 更新 `docs/bugs/PATTERNS.md` 模式 4，把 Ready/Verified 场景资产完整性纳入 runner gate 检查清单。

## 5 验证

- 场景结构检查：`scenario_dirs=87`，`missing_files=0`。
- 新增资产检查：`new_asset_files=18`，`new_asset_whitespace=PASS`。
- shell 语法检查：`scenario_script_count=350`，`scenario_scripts_bash_n=PASS`。
- 文档检查：`make docs-check` 通过，零 drift。
- 本地依赖检查：`make dev-doctor` 通过，`postgres-dev` / `redis-dev` / `minio-dev` 全部 OK。
- targeted 场景运行：`.test-output/runs/scenario-run-20260523T073633Z/summary.json` 显示 `passed=9`、`failed=0`、`errored=0`。

## 6 经验教训

- `Ready` / `Verified` 场景审查不能只看 README、脚本和 runner pass；必须先检查完整资产清单。
- 场景结构 preflight 应覆盖 `README.md`、`scripts/setup.sh`、`scripts/trigger.sh`、`scripts/verify.sh`、`scripts/cleanup.sh`、`data/seed-input.md`、`data/expected-outcome.md`。
- 后续可把这次临时结构扫描固化为 repo-tracked helper，供 `/scenario-env verify` 和 `/scenario-run` 在执行前复用。

# BUG-0018: frontend shell L2 review exposed route and auth wire drift

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-07 | 关联 Commit: `fix(frontend-shell): remediate app shell L2 findings`
> 关联日志: [2026-05-07](../work-journal/2026-05-07.md)

## 1 现象

- `/plan-code-review frontend-shell/plans/001-app-shell-auth-settings --fix` 反查当前 UI / 产品真理源与正式前端实现时，发现 `voice` 仍在正式 route normalization 中作为 alias 映射到 `practice`。
- `auth_verify` 页面提交验证码后，App 层调用 generated `verifyAuthEmailChallenge` 时没有把验证码 token 传入必填 query。
- 登录 / 注册页临时字段 `email` / `displayName` 会随 pendingAction decode 进入业务 route params。

## 2 诊断过程

### 假设 1：`voice` alias 是符合 D1 checklist 的兼容行为

- 验证方法：读取 `docs/spec/product-scope/spec.md`、`docs/ui-design/removed-modules-and-scope.md`、`ui-design/src/app.jsx` 与 `frontend/src/app/normalizeRoute.ts`。
- 结果：排除。当前产品和 UI 真理源要求删除 `voice` route alias，语音面试只能通过 `practice` 显式参数进入；正式前端仍保留 alias 是实现漂移。

### 假设 2：verify token 已由 generated client 或 fixture 自动处理

- 验证方法：读取 `openapi/openapi.yaml`、`frontend/src/api/generated/client.ts` 与 `frontend/src/app/App.tsx`，并补 focused test 捕获 verify 请求 URL。
- 结果：排除。OpenAPI 要求 `token` 为必填 query；generated client 只会拼接调用者传入的 `opts.query`，App 层原实现未传入。

### 假设 3：pendingAction 恢复只包含业务上下文

- 验证方法：读取 `AuthLoginScreen` / `AuthRegisterScreen` route params 透传逻辑与 `decodePendingActionRoute` reserved key 列表，并补 focused test 断言恢复后的 route params 不含 auth-only 字段。
- 结果：确认。`email` / `displayName` 不是 reserved key，会泄漏到业务 route params。

## 3 根因分析

- D1 checklist 的旧 route 拦截项把 `voice` 与其他旧 alias 一并列入“映射到当前 route 或 Home”，但后续产品 / UI 真理源已明确删除 `voice` alias，测试没有反向锁住“`voice` 必须 fallback home 且 normalizeRoute 不含 `voice:`”。
- App 层 auth dispatch 测试只覆盖 `startAuthEmailChallenge` 的网络 wire，没有覆盖 `verifyAuthEmailChallenge` 的必填 query。
- pendingAction 测试只断言 interview context key 被保留，没有断言认证页临时字段被剥离。

## 4 修复方案

- 删除 `frontend/src/app/normalizeRoute.ts` 中的 `voice` alias，并更新 route normalization / App normalization / scope focused tests 与 `frontend/README.md` handoff。
- 修改 `frontend/src/app/App.tsx`，将 `AuthVerifyScreen` 提交的 token 传入 `verifyAuthEmailChallenge({ query: { token } })`。
- 将 `email` / `displayName` 纳入 pendingAction decode reserved keys，确保它们只停留在认证页 params，不传播到业务 route params。
- 原地修订 `frontend-shell/001-app-shell-auth-settings` plan/checklist 到 v1.2，追加并完成 L2 remediation checklist 项。

## 6 经验教训

- L2 review 必须从当前产品 / UI 真理源反向提取不变量，不能只按历史 checklist 的旧措辞接受实现。
- generated client 调用测试需要覆盖必填 query/path/header 参数，而不仅是 operationId 是否被调用。
- pendingAction 恢复测试需要同时断言“必须保留的业务上下文”和“必须剥离的 auth-only 字段”。

# BUG-0070: async job polling and voice playback context were contract-only at runtime

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-17 | 关联 Commit: `fix(runtime): wire job polling and voice playback context (BUG-0070)`
> 关联日志: [2026-05-17](../work-journal/2026-05-17.md)

## 1 现象

- `checkpoint/015` review 发现 frontend debrief 已通过 generated client 调用 `getJob` 轮询生成任务，但真实 `cmd/api` 未挂载 `GET /api/v1/jobs/{jobId}`，进入真实后端会停在 404。
- Practice voice turn 返回的 TTS `audioRef` 是 `voice-turn://...` opaque ref，frontend 直接传给 `new Audio()` 时浏览器无法播放。
- 下一轮 voice turn 没有从已持久化的 `follow_up_generated` / playback events 还原 committed context，barge-in 后的下一轮 prompt 可能丢失用户已听到的 assistant 上下文。

## 2 诊断过程

### 假设 1：OpenAPI / generated client 已存在但真实 route 未接入

- 验证方法：反查 `openapi/openapi.yaml`、generated client 与 `backend/cmd/api/main.go` 的 mux 注册。
- 结果：确认 `getJob` contract 与 client 存在，但 `cmd/api` 未注册对应 route，frontend 真实 polling 会走空路径。

### 假设 2：voice playback ref 可以被浏览器直接消费

- 验证方法：追踪 `voiceTTSChunk` response 到 frontend `Audio` 构造路径。
- 结果：排除。`voice-turn://...` 不是浏览器可播放 URL，且仓库没有对应 resolver route。

### 假设 3：committed context builder 已自动进入生产链路

- 验证方法：追踪 `BuildCommittedVoiceContext` 的调用点和 `CreatePracticeVoiceTurn` chat payload 构造。
- 结果：排除。builder 只在局部测试中被覆盖，生产 service 未从 store 加载 event-derived context；frontend barge-in 也缺少 partial `tts_chunk_played` 证据。

## 3 根因分析

- 根本原因：review 前的完成证据停在 contract、fixture、generated client 和局部 domain helper，未把 frontend consumer、真实 `cmd/api` route、浏览器可播放 media source、store replay 和下一轮 AI payload 串起来做 artifact-level 反查。
- 涉及路径：
  - `backend/cmd/api/main.go:168`：真实 API server 原先未构建和挂载 jobs route。
  - `backend/internal/practice/voice_turn_service.go:173`：chat payload 原先只使用请求内 context，未从 store 加载已提交播放上下文。
  - `backend/internal/practice/voice_turn_service.go:360`：TTS response 原先返回 opaque `voice-turn://...` ref。
  - `frontend/src/app/screens/practice/hooks/usePracticeVoicePlayback.ts:153`：barge-in 原先只发送 interruption event，缺少已播放文本长度事件。
- 为什么之前没有发现：既有测试覆盖了 generated operation、fixture、domain helper 和 fake Audio 行为，但没有断言真实 route 挂载、HTTP response `audioRef` 的浏览器可播放性，以及 barge-in 后下一轮 prompt 的 event-derived context。

## 4 修复方案

- 新增 jobs domain / API / store，并在 `cmd/api` 挂载 `GET /api/v1/jobs/{jobId}`；store 按 `async_jobs.resource_type` 回查 owner resource，避免跨用户读取任务状态。
- TTS HTTP response 返回 `data:audio/...;base64,...`，确保 frontend `Audio` 可直接播放；持久化事件仍写入 `voice-turn://...` summary，避免把音频数据持久化进 session event payload。
- `CreatePracticeVoiceTurn` 在请求未携带 committed context 时，从 store 加载最新 voice turn 和后续 playback events，通过 `BuildCommittedVoiceContext` 注入下一轮 chat payload。
- Frontend barge-in 前先按播放进度估算并上报 partial `tts_chunk_played`，再发送 `barge_in_detected`。

## 6 经验教训

- 对 frontend 已消费的异步 polling operation，code review gate 必须覆盖 OpenAPI -> fixture -> generated client -> backend handler/store -> `cmd/api` route -> focused test。
- 对 voice / media contract，response 中给浏览器消费的 ref 必须可播放，或必须存在同步落地的 resolver route；持久化路径要单独验证不写入原始音频数据。
- 对 barge-in committed context，必须同时验证 frontend event evidence、store replay 和下一轮 AI payload，不能只测 builder 函数。

# BUG-0072: practice voice fixture audio refs used mock-only scheme

> 模块: schema | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-17 | 关联 Commit: `fix(practice-voice): align fixture audio refs (BUG-0072)`
> 关联日志: [2026-05-17](../work-journal/2026-05-17.md)

## 1 现象

- `practice-voice-mvp/001-cascaded-stt-llm-tts` 已在 backend service 层修复 HTTP response `ttsChunks[].audioRef`，要求返回浏览器可播放 `data:audio/...;base64,...`。
- 但 `openapi/fixtures/PracticeSessions/createPracticeVoiceTurn.json` 的 default scenario 仍返回 `fixture-audio://practice-voice/default/chunk-001`。
- 前端 dev fixture client 可拿到 voice turn response，却无法把该 mock-only scheme 交给浏览器播放器使用，mock 与真实 backend 语义再次分叉。

## 2 诊断过程

### 假设 1：真实 backend voice turn service 仍返回不可播放 ref

- 验证方法：运行 practice voice backend focused tests。
- 结果：排除。`go test ./internal/practice ./internal/api/practice ./internal/store/practice ./cmd/api -count=1` 通过，service 已覆盖 response data URL 与 persisted opaque summary 的分离边界。

### 假设 2：fixture-backed frontend mock 仍持有旧 scheme

- 验证方法：读取 `createPracticeVoiceTurn` fixture 并扩展 `frontend/src/api/devMockClient.test.ts`，断言 `ttsChunks[0].audioRef` 必须匹配 `data:audio/...;base64,...`。
- 结果：确认。Red phase 收到 `fixture-audio://practice-voice/default/chunk-001`。

### 假设 3：现有 fixture validator 能拦截该漂移

- 验证方法：检查并运行 `scripts/lint/validate_fixtures.py`。
- 结果：排除。既有 validator 只覆盖 schema、scenario、provenance、privacy 和 UUIDv7，不检查 `createPracticeVoiceTurn` 的 media playback 语义。

## 3 根因分析

- 根本原因：BUG-0070 修复了真实 service 的 browser-playable `audioRef`，但没有把同一语义下沉到 OpenAPI fixture 和 fixture validator。`audioRef` 的 OpenAPI 描述还允许 “fixture handle”，导致 schema-valid 的 mock-only 值能继续进入 generated fixture client。
- 涉及路径：
  - `openapi/fixtures/PracticeSessions/createPracticeVoiceTurn.json`：default response 使用 `fixture-audio://...`。
  - `openapi/openapi.yaml`：`PracticeVoiceTTSChunk.audioRef` 描述未区分 HTTP response playback ref 与 persisted opaque summary。
  - `scripts/lint/validate_fixtures.py`：缺少 practice voice audioRef semantic gate。
  - `frontend/src/api/devMockClient.test.ts`：原测试只确认 voice turn fixture 可消费，未断言 media ref 可播放。
- 为什么之前没有发现：验证停在真实 backend service 与 frontend voice controller mock 行为，未把 fixture-backed dev client 作为浏览器播放契约的一部分做断言。

## 4 修复方案

- 将 `createPracticeVoiceTurn` default fixture 的 TTS chunk 改为 `audio/wav` data URL，并同步 `byteLength`。
- 更新 OpenAPI `PracticeVoiceTTSChunk` 描述：HTTP response 必须返回 data URL 或 resolver URL；持久化 session event summary 必须改写为 opaque `voice-turn://...`。
- 扩展 `validate_fixtures.py`，专门拒绝 `createPracticeVoiceTurn` response 中的 mock-only audioRef scheme。
- 扩展 frontend dev fixture client 测试，断言 generated fixture client 返回的 voice TTS chunk 可被浏览器播放器消费。

## 6 经验教训

- Voice/media contract 的 mock fixture 不能只 schema-valid；任何给浏览器直接消费的 media ref 都必须被 semantic lint 覆盖。
- 真实 backend 修复了 response/persistence 分离后，要同步 fixture-backed dev client，否则本地 UI preview 仍可能表现为“接口有返回但播放无效”。

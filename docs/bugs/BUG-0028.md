# BUG-0028: targetjob review exposed error envelope and parse runtime drift

> 模块: workspace | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `fix(backend-targetjob): align targetjob review contracts`
> 关联日志: [2026-05-08](../work-journal/2026-05-08.md)

## 1 现象

- `/plan-code-review backend-targetjob/001-targetjob-import-and-parse-bootstrap backend --base-rev 49a55c0c0c45b94f931ff013856232b3b36f0463` 发现 Phase 7 历史 PASS 仍遗漏多处当前契约漂移。
- TargetJob handler 错误响应返回 legacy `{"errors":[...]}`，不符合 B1/B2 generated `ApiErrorResponse`。
- `listTargetJobs.pageInfo.pageSize` 返回 0，违反 generated `PageInfo` 与 fixture 契约。
- `ParseExecutor` 直接携带 prompt 文本，且 `APP_ENV=test` 的 `cmd/api` runtime 使用通用 stub 时无法产出 JSON parse content。
- AI 输出中 requirement 全部被过滤或存在非法 kind / label / evidenceLevel 时仍可能进入成功路径并把 `requirementCount=0` 写成 ready。
- BDD/handoff 文案曾把包级 go-test proxy 与真实 HTTP BDD 证据混在一起，导致 checklist 状态容易误读；后续 7.10 已把 p0-010..013 迁移为 `cmd/api` HTTP scenario harness。

## 2 诊断过程

### 假设 1：OpenAPI generated envelope 与 handler 输出漂移

- 验证方法：读取 `backend/internal/api/generated/types.gen.go`、`openapi/openapi.yaml` 与 `backend/internal/targetjob/handler.go`，再补 handler focused tests 反查 JSON envelope。
- 结果：确认。generated contract 要求 `ApiErrorResponse.error`，handler 仍写 legacy `errors` 数组。

### 假设 2：分页 pageSize 只在 store 层 clamp，service 没回填 wire envelope

- 验证方法：读取 `backend/internal/targetjob/service.go` 与 `backend/internal/shared/types/http_dto.go`，补 service tests 覆盖默认、非法、显式和超大 pageSize。
- 结果：确认。store 使用有效 page size 查询，但 service 只写 `nextCursor/hasMore`。

### 假设 3：parse runtime 的 TDD fake 与 cmd/api runtime 不是同一条链路

- 验证方法：对照 `backend/internal/targetjob/e2e_scenario_test.go` fake AI、`backend/cmd/api/main.go` A3 bootstrap runtime 与 `backend/internal/ai/aiclient/providers/stub/stub.go` generic stub response。
- 结果：确认。包级测试注入 JSON fake，cmd/api test runtime 走通用 stub 的 `stub:<profile>:<seed>` 文本，真实 drainer 成功路径无法用该输出闭合。

### 假设 4：BDD 证据状态文案仍保留历史完成口径

- 验证方法：读取 plan/checklist、`backend/internal/targetjob/doc.go`、`test/scenarios/e2e/INDEX.md` 与 p0-010..013 scenario scripts。
- 结果：确认。部分文案把 package proxy output 描述成 scenario evidence，真实 `auth -> HTTP API -> cmd/api drainer` gate 仍未落地。

## 3 根因分析

- 根本原因是 L2 remediation 只验证了局部结构和包级 focused tests，没有把 generated wire shape、cmd/api runtime wiring、semantic AI output validation 与 BDD evidence truth source 一起反查。
- 涉及路径：
  - `backend/internal/targetjob/handler.go`
  - `backend/internal/targetjob/service.go`
  - `backend/internal/targetjob/parse_executor.go`
  - `backend/cmd/api/main.go`
  - `backend/internal/targetjob/doc.go`
  - `test/scenarios/e2e/p0-010..013-*`
  - `docs/spec/backend-targetjob/plans/001-targetjob-import-and-parse-bootstrap/*`
- 之前没有发现，是因为历史 gates 主要证明包级测试通过，未断言 generated error envelope、`PageInfo.pageSize`、test runtime parse fixture 与 scenario evidence validity。

## 4 修复方案

- 将 TargetJob handler 错误输出统一为 generated `ApiErrorResponse`，并补 focused handler tests 断言无 legacy `errors` key。
- 在 service 层回填有效 `pageInfo.pageSize`，并补默认、非法、显式和 clamp tests。
- 扩展 F3 `PromptResolution` contract bridge，让 `ParseExecutor` 从 registry metadata 组装 messages，不再直接携带 prompt body。
- 新增 `DeterministicParseAIClient`，仅在 `APP_ENV=test` 的 `cmd/api` runtime 拦截 `target.import.parse` 并返回 schema-valid JSON，非目标 feature 全部委托 A3 runtime。
- 严格校验 AI requirements：非法 kind、空 label、非法 evidenceLevel 或无有效 requirement 均走 `AI_OUTPUT_INVALID`，不写 ready 成功。
- 对齐 plan/checklist、doc.go、scenario INDEX、README 和 verify output：先阻断包级 proxy 冒充 BDD PASS，再通过 7.10 新增 `backend/cmd/api` HTTP scenario harness；p0-010..013 `verify.sh` 输出 `method=cmd-api-http` / `validBddEvidence=true`。

## 6 经验教训

- L2 review 修复不能只看包级 PASS；必须把 generated contract、runtime wiring、scenario evidence validity 和 semantic failure cases纳入同一个 remediation checklist。
- BDD 场景 output 必须用机器可读字段表达真实性；当前 p0-010..013 固定为 `method=cmd-api-http` + `validBddEvidence=true`，避免后续把 TDD evidence 当成真实 BDD PASS。
- `APP_ENV=test` 的成功路径需要 deterministic runtime fixture 时，fixture 必须只作用于明确 feature key，避免破坏 dev / staging / prod fail-closed 语义。

# BUG-0129: core-loop pruning left stale profile and codegen test gates

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-06-30 | 关联 Commit: `fix(core-loop): close pruning gate drift (BUG-0129)`
> 关联日志: [2026-06-30](../work-journal/2026-06-30.md)

## 1 现象

- core-loop module pruning 删除 debrief/profile 后，review 发现标准 gate 仍为红灯。
- `python3 scripts/lint/ai_profile_coverage.py --repo-root .` 缺 `debrief.generate.default`、`debrief.suggest_questions.default`、`debrief.voice.extract.default`、`debrief.voice.tts.default`、`profile.update.default`。
- `cd backend && go test ./cmd/codegen/openapi` 仍断言 43-operation 注释，`PYTHONPATH=. python3 -m pytest -q scripts/lint/openapi_diff_test.py` 仍期望 43 operations。
- `cd backend && go test ./cmd/codegen/events` 仍期望旧 debrief-era job alignment 和 16 schema files。
- `pnpm --filter @easyinterview/frontend test src/lib/events/envelope.test.ts` 仍期望 3 个 envelope fixture，但 `debrief.completed` fixture 已删除。

## 2 诊断过程

### 假设 1：AI profile catalog 删除不完整
- 验证方法：运行 `python3 scripts/lint/ai_profile_coverage.py --repo-root .`，读取 `scripts/lint/ai_profile_coverage.py` 的来源规则。
- 结果：排除 catalog 本身。缺失 profile 来自 active A3 §4.5 Product/UI AI Capability Catalog 和 F3 §3.1.1 feature_key 表仍保留 debrief/profile active rows。

### 假设 2：OpenAPI generated artifacts 未同步
- 验证方法：读取 `backend/internal/api/generated/server.gen.go` 与 `backend/cmd/codegen/openapi/run_test.go`。
- 结果：部分排除。generated artifact 已是 35-row comment，失败来自测试仍断言 43-row comment；Python diff test 也仍用旧 expectedOperations。

### 假设 3：event generator 输出不稳定
- 验证方法：读取 `backend/internal/shared/jobs/jobs.go`、`backend/cmd/codegen/events/run_test.go` 和生成 schema 目录断言。
- 结果：排除生成器问题。当前 generated jobs/events 已删除 debrief job/event，schema count 为 14；测试仍保留 16 count 和旧对齐字符串。

### 假设 4：frontend envelope fixture 删除破坏 runtime parser
- 验证方法：读取 `shared/events/__fixtures__/envelopes.json` 与 `frontend/src/lib/events/envelope.test.ts`。
- 结果：排除 parser 问题。fixture 集当前只有 `target.import.requested` 与 `report.generated` 两条，测试固定长度未同步。

## 3 根因分析

- D-22 模块裁剪更新了主要 truth source 和 generated artifacts，但没有把所有 gate consumer 作为必须同步对象逐一反查。
- AI profile coverage gate 的来源是 active spec 文档，不只是 `config/ai-profiles.yaml`，因此删除 catalog entry 后还必须删除 A3/F3 中对应 active rows。
- OpenAPI/event/frontend 测试中存在固定历史数量和旧字符串断言；这些断言在删除合同后必须和生成物一起更新，否则 `make lint` / `make test` 会继续失败。
- 之前的 checklist final-gate evidence 未能阻止 stale consumer drift，说明删除类计划需要在 owner checklist 中固化这些 direct-consumer focused gates。

## 4 修复方案

- 更新 A3 spec/history：Product/UI AI Capability Catalog 删除用户画像信号更新与 Debrief 文本 / 问题 / 语音 / 分析 rows，F3 coverage 口径改为 9 active feature_key。
- 更新 F3 spec/history 与 config README：§3.1.1 删除 `debrief.generate` / `debrief.suggest_questions`，当前 prompt/rubric/eval/profile coverage 改为 9 key / >=36 eval cases。
- 更新 OpenAPI tests：Go generator test 改为 35-row comment，Python diff test expectedOperations 改为 35。
- 更新 event codegen tests：job alignment 字符串匹配当前 gofmt 输出，schema count 改为 14。
- 更新 frontend envelope test：fixture count 改为 2。
- 原地修订 `product-scope/001-core-loop-module-pruning` plan/checklist 到 v1.2，把 review-focused 与 aggregate gates 写回 owner evidence。

## 6 经验教训

- 删除模块时不能只看配置 / generated 主体是否干净；所有固定数量断言、active spec-derived lint、fixture-count tests 都是 direct consumers。
- 对 contract deletion，owner checklist 必须列出并运行 direct consumer focused gates，再用 `make lint` / `make test` 聚合验证兜底。

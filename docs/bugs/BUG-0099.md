# BUG-0099: parse ready response skipped the ui-design loading demo

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-24 | 关联 Commit: `fix(frontend-home): preserve parse loading demo (BUG-0099)`
> 关联日志: [2026-05-24](../work-journal/2026-05-24.md)

## 1 现象

- 用户从 Home 上传 JD、粘贴 JD 或 URL 导入后，如果 `getTargetJob` 首次返回 `analysisStatus=ready`，Parse 屏会直接进入 parsed preview。
- 这跳过了 `ui-design/src/screens-p0-complete.jsx::ParseScreen` 中定义的 4 步 loading 演示流程，用户看不到“解析中”动画节奏。
- 影响范围覆盖 fixture-backed frontend flow 和真实 backend 已较快完成解析时的 import→parse 主路径；失败态仍能直接进入 failed state。

## 2 诊断过程

### 假设 1：ui-design 只要求在 backend 未 ready 时显示 loading

- 验证方法：读取 `ui-design/src/screens-p0-complete.jsx::ParseScreen` 与 `docs/ui-design/` 相关说明，并反查 owner spec 的 D-2 决策。
- 结果：排除。`ui-design` 原型从 `loading` 初始阶段按 `[600, 900, 800, 700]` tick 推进，约 3200ms 后进入 preview；正式前端应复刻这段用户可见解析演示，而不是把 ready response 当成跳过条件。

### 假设 2：正式前端实现已有 ready-response loading gate

- 验证方法：读取 `frontend/src/app/screens/parse/ParseScreen.tsx` 与 `ParseFlow.test.tsx`，并先跑 focused test。
- 结果：排除。旧实现一旦 `job.analysisStatus === "ready"` 就立即 hydrate parsed data 并 `setStage("preview")`；旧测试也断言 ready response “immediately” 显示 preview，反而固化了回归。

### 假设 3：修复只需延迟 preview，不影响 polling、failed、re-parse、confirm

- 验证方法：用 fake timers 增加 ready-response loading regression test，并重跑 parse suite、frontend full test、build 与 E2E.P0.015 wrapper。
- 结果：确认。把 ready job 暂存到 `pendingReadyJob`，等 loading demo 完成后再 hydrate preview，可以保留 queued/processing polling、failed state、re-parse 重新轮询和 confirm flow。

## 3 根因分析

- `frontend/src/app/screens/parse/ParseScreen.tsx:137` 旧逻辑把 `analysisStatus=ready` 视作可立即进入 preview 的 runtime 状态，没有把 `ui-design` loading demo 当成必须完成的用户可见阶段。
- `frontend/src/app/screens/parse/ParseFlow.test.tsx` 旧 ready test 使用“立即 preview”作为期望，导致测试和当前设计真理源方向相反。
- Owner spec 的 D-2 原先强调状态机来源和“不假装正在调用 LLM”，但没有明确“即使首次 ready 也必须展示 loading 演示”的边界，容易被实现理解为 ready 即可跳过 loading。

## 4 修复方案

- 在 `ParseScreen` 中引入 `loadingStepTicks` / `loadingPreviewDelay`，按 `ui-design` `[600, 900, 800, 700] + 200ms` 节奏驱动 loading demo。
- `getTargetJob` 返回 ready 时只写入 `pendingReadyJob`；等 loading demo 标记完成后再 hydrate title/company/location 并切到 preview。
- `Re-parse` 重置 pending ready job、loading completion、step 与 poll nonce，保证重新解析也先展示 loading。
- 修订 parse 相关测试：ready response 必须先看到 `parse-loading-step-0`，3200ms 后才出现 `parse-basics-title`；同步 preview 相关测试 helper，避免因为新 loading gate 误判。
- 原地修订 `frontend-home-job-picks-and-parse` spec/plan/checklist/BDD checklist/context 与 INDEX，把 BUG-0099 的 regression gate 固化到 owner plan。

## 6 经验教训

- UI parity 不只包含最终 preview DOM；用户可见的过渡阶段、tick 节奏和进入时机也必须被测试锁定。
- 当 backend fixture 或真实 runtime 很快返回 ready 时，测试必须覆盖“ready response does not skip required UI transition”这类边界。
- 对已 completed 的 frontend plan 做回归修复时，必须把设计真理源的新边界写回 owner spec/BDD gate，否则测试可能继续固化旧行为。

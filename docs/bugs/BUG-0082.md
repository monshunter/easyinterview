# BUG-0082: jd-match BDD closure treated smoke wrappers as completed gates

> 模块: test | 严重度: high | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(backend-jobs): harden jd-match runtime contracts (BUG-0082, BUG-0083)`
> 关联日志: `-`

## 1 现象

- `/plan-code-review 001-jd-match-real-backend-baseline --fix` 复查时，plan / checklist / bdd-plan / bdd-checklist 已被标记为 `completed`。
- 主 checklist 6.5-6.8 写入 `bddChecklist=complete`，但 `bdd-checklist.md` 中 E2E.P0.094-097 全部执行项仍是 `[ ]`。
- `test/scenarios/e2e/INDEX.md` 仍把 E2E.P0.094-097 标为 `scaffold` / `Pending`。
- 4 个场景 wrapper 都只运行同一个 `TestJDMatchHTTPScenario` smoke，未验证各自 BDD 矩阵要求的 DB、IK、fixture、search AI variant、drainer、privacy rollback 证据。

## 2 诊断过程

### 假设 1：BDD checklist 已完成但 INDEX 未同步

- 验证方法：读取 `bdd-checklist.md` 与 `test/scenarios/e2e/INDEX.md`。
- 结果：排除。BDD checklist 正文仍全未勾选，INDEX 仍为 scaffold / Pending。

### 假设 2：cmd/api scenario 覆盖了所有 BDD 断言

- 验证方法：读取 `backend/cmd/api/jdmatch_live_scenario_test.go`、运行 `go test -list`、读取四个 scenario `trigger.sh` / `verify.sh`。
- 结果：排除。测试源码声明自己是 smoke，不是完整 12-route + IK replay + drainer scenario；`go test -list` 没有 `TestBuildJDMatchRuntime` / `TestJDMatchAgentScanDrainerScenario`；wrapper 只检查 `--- PASS: TestJDMatchHTTPScenario` 与 raw email。

### 假设 3：只是本地缺少 live DB 导致误判

- 验证方法：运行 `bash test/scenarios/e2e/p0-094-jd-match-profile-and-recommendations-list/scripts/trigger.sh` 和 `verify.sh`。
- 结果：排除。live Postgres 可达时 wrapper 可以通过，但它只证明 smoke 断言，不证明 BDD checklist 要求的完整证据。

## 3 根因分析

- 根本原因：Phase 6 close-out 把 smoke wrapper pass 当作 BDD-Gate pass，并把 unchecked BDD checklist / Pending scenario INDEX 误写成 completed 证据。
- 涉及路径：`docs/spec/backend-jobs-recommendations/plans/001-jd-match-real-backend-baseline/checklist.md`、`bdd-checklist.md`、`test/scenarios/e2e/p0-094..097-*/scripts/*`、`backend/cmd/api/jdmatch_live_scenario_test.go`。
- 之前没有发现：历史 close-out 只看 `TestJDMatchHTTPScenario` PASS 和 raw email grep，没有执行 artifact-level 反查：`go test -list`、BDD checklist body、scenario INDEX 状态和 wrapper 断言覆盖。

## 4 修复方案

- 将 plan / checklist / bdd-plan / bdd-checklist 从 completed v1.1 退回 active v1.2。
- 重新打开 5.5-5.8、6.1-6.2、6.5-6.8、6.12-6.13，并逐项写入 L2 reopened 证据。
- 将 plans/INDEX row 从 Completed 移回 Active，`context.yaml` `specVersion.to` 对齐当前 spec v1.2。
- 将原交付复盘报告改为 L2 Reopen Assessment，避免继续传播 false-green 结论。
- 后续同一 owner plan 内补齐 `TestBuildJDMatchRuntimeWiresRoutesDrainerAndLifecycle`、`TestJDMatchAgentScanDrainerScenario`、`TestJDMatchFixtureParity` 与 4 个 E2E scenario wrapper `setup -> trigger -> verify` 真实 PASS 证据。
- 4 个 `verify.sh` 追加 package-level `ok github.com/monshunter/easyinterview/backend/cmd/api` 断言，并拒绝 `--- FAIL` / package `FAIL` / skip / no-test / raw email，防止同类 false-green 复发。

## 5 验证

- `make docs-check` PASS。
- `python3 .agent-skills/implement/shared/scripts/validate_context.py --context docs/spec/backend-jobs-recommendations/plans/001-jd-match-real-backend-baseline/context.yaml --docs-root docs --target backend` PASS。
- `git diff --check` PASS。
- `cd backend && go test ./...` PASS。
- `cd backend && DATABASE_URL='postgres://user:***@localhost:5432/easyinterview?sslmode=disable' go test ./cmd/api -run '^(TestBuildJDMatchRuntimeWiresRoutesDrainerAndLifecycle|TestJDMatchRoutesRequireSessionOnAllRoutes|TestJDMatchHTTPScenario|TestJDMatchAgentScanDrainerScenario|TestJDMatchFixtureParity)$' -count=1 -v` PASS。
- E2E.P0.094 / P0.095 / P0.096 / P0.097 `setup.sh -> trigger.sh -> verify.sh` 全部 PASS。

## 6 经验教训

- Completed plan 的 BDD-Gate 必须同时核对主 checklist、BDD checklist、scenario INDEX、runner body、wrapper trigger/verify 和 `go test -list`。
- 只检查 `--- PASS` 或 raw email grep 的 wrapper 不足以证明行为场景；verify 必须拒绝 skip / no tests，并断言场景专属 runtime evidence。

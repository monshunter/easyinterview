# BUG-0047: upload presign runtime path was only partially wired

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-12 | 关联 Commit: `fix(backend-upload): wire presign runtime contracts`
> 关联日志: `-`

## 1 现象

- L2 code review 发现 `POST /api/v1/uploads/presign` 未挂载到真实 API router，实际请求返回 `404`，无法进入 session middleware、idempotency middleware 或 upload handler。
- `CreateUploadPresign` 只在 handler fake service 中被证明，真实 upload service 没有创建 pending `file_objects` 记录，也没有调用 ObjectStore presign。
- `RegisterFileObject` 在显式事务外调用 `LockForRegister`，再用单独事务 `MarkUploaded`，导致 row lock、对象存在性检查和状态更新不在同一个临界区。
- MinIO integration smoke 在配置存在时仍会无条件 skip，无法证明 provider SDK presign、signed headers、PUT 上传和 object existence。
- E2E.P0.033 场景脚本只覆盖 fake handler wrapper 与已有 service/register 测试，未断言真实 route 挂载、真实 presign service 或事务化 register 行为。

## 2 诊断过程

### 假设 1：presign API 已通过 `cmd/api` router 挂载

- 验证方法：新增并运行 `backend/cmd/api` focused test，请求 `POST /api/v1/uploads/presign`。
- 结果：排除。红测返回 `404`，确认 router 未挂载 upload presign 路径。

### 假设 2：handler fake service 覆盖了真实 `CreateUploadPresign` 行为

- 验证方法：为 `backend/internal/upload/service` 增加红测，要求 presign 创建 pending `file_objects` 行、生成 object key、调用 ObjectStore presign 并返回上传契约。
- 结果：排除。红测编译失败，真实 service 中不存在 `CreateUploadPresign` 与对应 input type。

### 假设 3：`RegisterFileObject` 已经在 row lock 内检查对象存在并标记 uploaded

- 验证方法：为 `backend/internal/upload/store` 增加红测，使用 sqlmock 断言 `SELECT ... FOR UPDATE`、对象存在性 callback、`UPDATE ... SET status = uploaded` 在同一事务内执行。
- 结果：排除。红测编译失败，store repository 没有事务化 `RegisterUploaded` 方法；旧 service 路径的 lock 与 update 分离。

### 假设 4：MinIO smoke 会在 live env 配置齐全时实际运行

- 验证方法：读取 `backend/internal/upload/objectstore/minio_smoke_test.go`。
- 结果：排除。测试在 env 检查后仍无条件 `t.Skip`，无法在真实 MinIO 环境中执行 presign/PUT/existence gate。

## 3 根因分析

- `createUploadPresign` 的 handler contract、runtime route mounting 和 service implementation 没有同测，导致 fake handler PASS 掩盖真实 API 缺口。
- Upload service interface 先支持 register path，presign path 留在 handler fake 中，缺少 “handler -> real service -> store -> object store” 的 focused contract test。
- Store 层只有分散的 `LockForRegister` / `MarkUploaded` 测试，没有事务边界断言，导致行锁临界区被 service 组合时破坏。
- Scenario gate 只证明已有测试能跑，没有包含负向 grep 或 real path focused tests 来防止 fake-only gate。
- MinIO smoke 的 provider SDK 引入后没有移除历史 defer skip，导致 live env 就绪时仍无法补齐真实 provider 验证。

## 4 修复方案

- 在 `backend/cmd/api/main.go` 构建 upload routes，挂载 `POST /api/v1/uploads/presign`，并接入 session middleware、idempotency middleware、SQL repository、upload service 和 configured ObjectStore。
- 在 `backend/internal/upload/service/register.go` 实现 `CreateUploadPresign`，统一校验 user/purpose/fileName/contentType/byteSize/TTL，创建 pending `file_objects` 记录，并返回真实 ObjectStore presign response。
- 在 `backend/internal/upload/store/file_objects.go` 新增事务化 `RegisterUploaded`，确保 row lock、object existence callback 和 uploaded update 在同一事务内完成，并保持 uploaded 状态的幂等返回。
- 调整 MinIO provider 使用 signed headers presign，并让 `minio_smoke_test` 在 `OBJECT_STORAGE_*` env 齐全时实际执行 presign、HTTP PUT、Exists 与 expired presign rejection。
- 强化 E2E.P0.033 scenario scripts，要求 service/store/cmd API focused tests 都被执行，防止后续退回 fake-only gate。

## 6 经验教训

- Handler fake tests 不能替代真实 runtime route 和 real service tests；新增 API 必须至少有一条 `cmd/api` route mounting test。
- Service interface 新增业务路径时，必须同时证明 persistence side effect 与 provider call，而不是只证明 handler envelope。
- 涉及 row lock 的状态机应在 store 层提供事务化方法，并用 sqlmock/集成测试断言锁、外部检查与 update 的顺序。
- Scenario trigger/verify 脚本应断言 real-path focused tests 已执行，避免只跑 wrapper 或历史 helper。
- Live provider smoke 引入 SDK 后不能保留无条件 skip；允许因 env 缺失 skip，但 env 齐全时必须实际运行。

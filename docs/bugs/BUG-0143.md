# BUG-0143: interview plan list cards lacked visual hierarchy and concise content

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-08 | 关联 Commit: `fix(workspace): close plan list UX and context regressions (BUG-0143, BUG-0144, BUG-0145)`
> 关联日志: [2026-07-09](../work-journal/2026-07-09.md)

## 1 现象

- 用户在 `/workspace` 一级 `面试` 入口截图中看到面试规划列表区域几乎没有卡片样式，两条规划表现为松散文本列。
- 后续截图反馈显示卡片虽然有容器，但仍展示 `手动输入` / 来源类型 / 目标语言等低价值字段，`进入规划` 使用二级按钮样式，卡片和页面背景层次仍不够清楚。
- 影响范围是无上下文 `workspace` 的面试规划列表 landing；当前规划详情、数据拉取和计划打开行为不受影响。
- 视觉问题会让用户误以为列表未完成、页面样式丢失，或把导入技术元信息误读为规划重点。

## 2 诊断过程

### 假设 1：列表只缺少卡片结构

- 验证方法：读取 `ui-design/src/screen-workspace.jsx` 与 `frontend/src/app/screens/workspace/WorkspaceScreen.tsx` 的 `WorkspacePlanList`。
- 结果：确认。列表项虽然有 `article`，但缺少明确的 body/footer 分区和可验证的卡片视觉断言。

### 假设 2：浏览器里卡片边框样式没有真实生效

- 验证方法：新增 Playwright computed style 断言，检查 plan-list card 的 border、box-shadow、body/footer padding 与 footer border。
- 结果：确认。首次运行 `workspace plan-list cards keep visible card affordance` 时，`borderTopWidth` 实际为 `0px`。

### 假设 3：数据或 fixture 导致列表渲染异常

- 验证方法：`WorkspaceEmptyState.test.tsx` 显式 mock `listTargetJobs` fixture body，并断言不调用 `getTargetJob`。
- 结果：排除。列表数据可正常进入 UI，问题集中在样式与视觉合同。

### 假设 4：卡片内容层级仍然把低价值导入字段放大

- 验证方法：根据用户截图反查 `WorkspacePlanList`，检查 body/footer 是否仍消费 `workspace.planList.cardMeta`、`job.sourceType` 和 `job.targetLanguage`。
- 结果：确认。Phase 8 后 body 仍显示 `{source} · {language}`，footer 又重复显示 source，且 `进入规划` 仍是 secondary button。

## 3 根因分析

- `WorkspacePlanList` 使用了从静态原型命名习惯迁移来的 CSS 变量名，例如 `--ei-color-bgCard`、`--ei-color-rule`、`--ei-color-ink*`。
- 正式前端主题 token 的真实变量名是 kebab-case，例如 `--ei-color-bg-card`、`--ei-color-rule-strong`、`--ei-color-fg-primary`。
- 浏览器遇到未定义 CSS variable 时会让对应属性无效，导致卡片背景、边框和文字颜色等样式没有按预期生效。
- Phase 7 的既有测试只验证了列表和卡片 anchor 存在、可导航，没有验证 card computed style、body/footer 结构和视觉 affordance。
- Phase 8 的卡片合同只要求“有卡片容器和分区”，没有明确禁止 source/language/`手动输入` 等导入元信息进入卡片，也没有要求 `进入规划` CTA 与主题 accent 保持一致。

## 4 修复方案

- 在 `docs/spec/frontend-workspace-and-practice/spec.md`、owner plan/checklist、BDD plan/checklist 和 `docs/ui-design/module-job-workspace.md` 中补充面试规划列表卡片化合同。
- 更新 `ui-design/src/screen-workspace.jsx`，让静态原型的 plan list 使用独立卡片容器、轻阴影、body/footer 分区和 footer 操作区。
- 更新 `frontend/src/app/screens/workspace/WorkspaceScreen.tsx` 中 `WorkspacePlanList`，改用正式 kebab-case 主题 token，并补齐卡片 body/footer testid。
- 更新 `WorkspaceEmptyState.test.tsx` 与 `frontend/tests/pixel-parity/workspace.spec.ts`，用 jsdom 和 Playwright computed style 锁定卡片背景、边框、阴影、分区和响应式几何。
- 更新 `test/scenarios/e2e/p0-018-workspace-default-render/scripts/verify.sh`，反查 card body/footer anchor 与 elevation token，避免只检查列表存在。
- Phase 9 follow-up：移除 no-context card 的 `workspace.planList.cardMeta`、`job.sourceType` 和 `job.targetLanguage` 展示；footer 只保留主题 accent `进入规划` CTA；elevation 升级为既有 `--ei-shadow-elev2` token，footer 背景回到 `bg-card`，避免大块浅色 band 分散注意力。
- Phase 9 follow-up：更新 jsdom、Playwright 与 E2E.P0.018 verify，明确拒绝 source/language 元信息和 secondary button 样式回流。

## 6 经验教训

- 从 `ui-design` 原型迁移到正式前端时，样式变量不能直接沿用原型对象字段命名；必须映射到正式 `themes.css` 的 kebab-case token。
- UI parity 对用户可见卡片/列表不能只验 DOM anchor 和截图非空；应至少断言关键 computed style 和分区 bounding box。
- 对刚新增的一级入口 landing，BDD verify 应反查视觉锚点和 style token，避免“有数据、有文字、无样式”的假绿。
- 卡片化不等于把所有可用字段都放进去；列表卡片应只保留用户继续决策需要的信息，导入来源和目标语言这类技术元信息应留在详情或导入链路。

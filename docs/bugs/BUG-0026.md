# BUG-0026: targetjob L2 review exposed import parse contract drift

> 模块: workspace | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `fix(backend-targetjob): remediate import parse L2 findings`
> 关联日志: [2026-05-08](../work-journal/2026-05-08.md)

## 1 现象

- `/plan-code-review backend-targetjob/001-targetjob-import-and-parse-bootstrap backend --fix` 发现 TargetJob import / parse bootstrap 已通过历史 checklist，但当前代码仍存在语义漂移。
- `manual_form` idempotency dedupe 在事务外先查再写，存在同 user / same idempotency key 并发重复导入窗口。
- A3 call metadata 没有携带 F3 `data_source_version`，削弱 parse evidence 与 prompt / rubric / model profile 的追溯闭环。
- AI / F3 失败路径把上游错误文本写入 `async_jobs.last_error` / failure outcome，可能把 prompt、response 或 provider secret 片段暴露给持久化错误证据。
- TargetJob handler 仍保留旧 Phase stub fallback，缺 service 时返回 `NOT_IMPLEMENTED` / 501，和当前四个 operation 已实施状态不一致。

## 2 诊断过程

### 假设 1：幂等 race 已由 async_jobs partial unique index 覆盖
- 验证方法：读取 `backend/internal/targetjob/store.go` 的 `ImportTargetJob` 路径和 store sqlmock tests。
- 结果：排除。manual_form 写入的是 terminal succeeded marker，不受 active runner partial unique index 保护；旧实现先查 dedupe 再开始事务，仍有并发窗口。

### 假设 2：A3 metadata 已由 promptVersion / rubricVersion 覆盖
- 验证方法：检查 `backend/internal/ai/aiclient/payload.go` 的 `CallMetadata` 与 `ParseExecutor` 构造 payload 的路径。
- 结果：排除。`dataSourceVersion` 不在 payload contract 中，happy-path test 也没有断言该字段。

### 假设 3：失败错误消息已统一走 privacy redaction
- 验证方法：检查 `ParseExecutor` F3 resolve / AI call / persistence failure path，并新增错误文本包含 prompt、response、provider secret 的 redline test。
- 结果：确认缺口。部分失败路径使用原始 `err.Error()` 作为 failure message，需要在写入 outcome 和 store 前统一映射为 code-based safe summary。

### 假设 4：旧 handler stub 只是测试辅助，不影响当前状态
- 验证方法：负向搜索 `NOT_IMPLEMENTED` / `not yet implemented` / phase stub，并检查 `NewHandler()` 缺 service 路径。
- 结果：确认缺口。虽然 `cmd/api` 已注入真实 service，活跃代码仍保留已完成 operation 的旧 stub fallback 和测试。

## 3 根因分析

- 根本原因：L2 审查前的 gate 更偏向结构与历史 PASS，未把并发 dedupe 原子性、AI evidence metadata 完整性、failure message 安全摘要和旧 phase stub 零残留固化为可执行断言。
- 涉及代码路径：
  - `backend/internal/targetjob/store.go`
  - `backend/internal/targetjob/parse_executor.go`
  - `backend/internal/ai/aiclient/payload.go`
  - `backend/internal/targetjob/handler.go`
  - `backend/internal/targetjob/doc.go`
- 之前没有发现的原因：既有 tests 覆盖正常 import / parse / BDD path，但没有覆盖 manual_form 并发 dedupe 锁定点、A3 metadata field presence、raw upstream error redaction 和 completed-phase stub negative search。

## 4 修复方案

- 在 `ImportTargetJob` 事务内执行 dedupe 查询，并通过 `pg_advisory_xact_lock(hashtext($1))` 锁定 same user / same idempotency key 的 import dedupe key，关闭 manual_form race window。
- 在 A3 `CallMetadata` 中加入 `dataSourceVersion`，并由 `ParseExecutor` 从 F3 `PromptResolution` 传入；happy-path fake AI client 捕获 payload 并断言 metadata 完整。
- 为 parse failure path 增加 safe failure message 映射，按 B1 error code 输出摘要，并补充 prompt / response / provider secret redline regression test。
- 删除 handler `NOT_IMPLEMENTED` fallback；缺 service 现在返回 `TARGET_IMPORT_FAILED` 500 配置错误，并用测试断言旧 stub 不再暴露。
- 同步 `backend/internal/targetjob/doc.go` 的 BDD handoff 状态，移除旧的 unchecked / parked 文案。

## 6 经验教训

- Import idempotency 的 race 保护必须和 dedupe read / write 位于同一事务边界；terminal marker path 不能假设 runner active unique index 会保护它。
- AI evidence metadata tests 应断言每个 provenance 字段都进入下游 call payload，而不是只断言 prompt / rubric / profile 的部分字段。
- Failure evidence 只能持久化 code-based safe summary；上游错误字符串即使已经被 wrapper 包装，也不能直接进入 async job 或 outbox 证据。
- Completed-phase stub / placeholder text 需要进入负向搜索 gate，防止已实现 operation 仍暴露历史 scaffold 行为。

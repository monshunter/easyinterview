# BUG-0119: resume flatten review left runtime and data retention regressions

> 模块: schema | 严重度: high | 状态: resolved
> 发现日期: 2026-06-14 | 关联 Commit: `fix(schema): close resume flatten review regressions (BUG-0119)`
> 关联日志: [2026-06-14](../work-journal/2026-06-14.md)

## 1 现象

- Reviewer 对 D-20 resume flatten / D-17 JD Match removal patch 做 L2 review 后发现，核心 runtime 与 contract gates 仍未闭环。
- `/api/v1/jobs/{jobId}` ownership 查询仍引用已删除或重命名的 resume run / asset 表，migration 后 job polling 会在 PostgreSQL planning 阶段失败。
- OpenAPI diff gate 仍使用 60 operation 的旧 baseline，而当前 contract 是 43 operations。
- Resume display-name-only PATCH 会把既有 structured profile 写成 `{}`；隐私删除会留下 resume tailor payload / result；保存已接受 rewrite 后刷新仍显示旧 bullet。
- 对历史合法数据执行 migration 时，`guided` resume rows 和 JD Match async job rows 会在重新添加收窄后的 check constraint 时失败。

## 2 诊断过程

### 假设 1：schema collapse 没有同步到 async job ownership 查询

- 验证方法：读取 `backend/internal/store/jobs/store.go` 与 D-20 migration，确认 query 分支是否仍依赖 `resume_tailor_runs` / `resume_assets`。
- 结果：确认。job store 的 `resume_asset` / `resume_tailor_run` owner check 没有随 schema collapse 改到 `resumes` 和 `async_jobs.payload.resumeId`。

### 假设 2：OpenAPI baseline 与当前 contract 不一致

- 验证方法：运行 `make openapi-diff`，对照 `openapi/openapi.yaml`、baseline 与 `openapi/diff-config.yaml` inventory。
- 结果：确认。当前 OpenAPI 是 43 operations，但 baseline 与 README / diff config 仍保留旧计数，diff gate 报 breaking deletions。

### 假设 3：resume update / privacy delete / rewrite save 流程存在数据保留或数据丢失风险

- 验证方法：反查 handler/service/store 的 PATCH 字段语义、privacy runner 删除顺序，以及 frontend preview/copy 使用的 structured profile 投影。
- 结果：确认。PATCH 未区分字段缺席与空对象；privacy runner 删除 resume 前未先清理 payload 内关联 resumeId 的 async jobs；frontend 只保存 sidecar，没有改写被投影消费的 bullets。

### 假设 4：migration 对历史合法行缺少 cleanup

- 验证方法：构造带 `jd_match_agent_scan` / `jd_match_search` async jobs 与 `source_type='guided'` resumes 的 pre-migration rows，再应用目标 migration。
- 结果：确认。若不先删除或改写这些 rows，收窄后的 check constraint 会拒绝历史合法数据。

## 3 根因分析

- 根本原因：D-20 resume flatten 和 D-17 JD Match removal 删除了旧 schema / operation，但 review 前的 gate 主要覆盖 route、fixture 和场景表层，未完整反查 job ownership query、privacy payload retention、PATCH 局部更新语义、frontend persisted projection 与 valid legacy row migration。
- 涉及路径：
  - `backend/internal/store/jobs/store.go`
  - `backend/internal/resume/service.go`
  - `backend/internal/resume/store/assets.go`
  - `backend/internal/privacy/runner/store.go`
  - `frontend/src/app/screens/resume-workshop/components/ResumeDetailView.tsx`
  - `migrations/000014_drop_jd_match_module.up.sql`
  - `migrations/000015_resume_flatten.up.sql`
  - `openapi/baseline/openapi-v1.0.0.yaml`
- 之前没有发现，是因为 schema deletion 后的负向搜索没有覆盖 runtime owner query 和 privacy cleanup SQL；OpenAPI diff baseline 没有在同一 patch 重新冻结；migration contract test 缺少历史合法行 cleanup 顺序断言。

## 4 修复方案

- 将 job polling owner query 改为基于 `resumes` owner 与 `async_jobs.payload->>'resumeId'` 判断，不再依赖 dropped tables。
- 将 resume parse job resource type 恢复为 contract 允许的 `resume_asset`，保持 OpenAPI enum、RegisterResume response 与 polling query 一致。
- 为 resume PATCH 增加 `StructuredProfileSet` 字段存在语义，store 条件更新 structured profile，避免 display-name-only PATCH 清空既有 profile。
- 在隐私删除前清理与用户 resumes 相关的 `resume_asset` / `resume` / `resume_tailor_run` async jobs，包括 payload 内 resumeId 关联。
- 前端保存 accepted rewrite 时直接改写 persisted structured profile bullets，移除不会被 projection 消费的 `acceptedRewrites` sidecar。
- 重新冻结 OpenAPI baseline，并同步 `openapi/diff-config.yaml` 与 `openapi/README.md` 的 43 operation inventory。
- 在 migration 添加收窄约束前删除 non-current JD Match async jobs，并将 `guided` resumes 改写为允许的 `paste` source。

## 6 经验教训

- 删除表、operation 或 feature module 时，owner query、privacy cleanup、payload JSON 引用和 migration cleanup order 必须进入同一轮负向搜索。
- PATCH 局部更新必须显式区分字段缺席、字段置空和字段设置默认值，避免 display-only 修改造成结构化数据丢失。
- OpenAPI breaking-change baseline 是 contract gate 的一部分，operation 删除与 baseline freeze 必须同 patch 完成。
- Migration 收窄 check constraint 前，应先为历史合法值写入 cleanup 或转换，并用 contract test 固化顺序。

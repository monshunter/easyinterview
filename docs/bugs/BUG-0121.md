# BUG-0121: debrief suggestions ignored resumeId in real backend context

> 模块: debrief | 严重度: medium | 状态: resolved
> 发现日期: 2026-06-14 | 关联 Commit: `fix(debrief): wire resume context for suggestions (BUG-0121)`
> 关联日志: [2026-06-14](../work-journal/2026-06-14.md)

## 1 现象

- `/plan-code-review backend-debrief/001-debrief-record-and-analysis backend --fix` 要求闭环 `suggestDebriefQuestions` 从 fixture 的 `resumeId` contract 到真实 backend handler、fixture parity 和 scenario gate。
- OpenAPI fixture `openapi/fixtures/Debriefs/suggestDebriefQuestions.json` 已使用 `resumeId`，handler generated request 也有 `ResumeId`，但真实 store `Repository.GetSuggestionContext` 只读取 `target_jobs`，未按 `(user_id, resume_id)` 查询扁平 `resumes.structured_profile`。
- P0.063 scenario wrapper 只跑 service/API 层旧测试，未证明真实 store、cmd/api route 和 fixture `resumeId` parity，因此 Debriefs owner 看起来完成但实际停在 mock / 文档声明层。

## 2 诊断过程

### 假设 1：OpenAPI / fixture 仍是旧 `resumeVersionId`
- 验证方法：检查 `openapi/openapi.yaml`、generated artifacts 和 `openapi/fixtures/Debriefs/suggestDebriefQuestions.json`。
- 结果：排除。当前 contract 与 fixture 已使用 `resumeId`。

### 假设 2：handler 没有把 `resumeId` 传入 domain service
- 验证方法：读取 `backend/internal/api/debriefs/handler.go` 和新增 `TestSuggestDebriefQuestions_MapsResumeIDToService`。
- 结果：部分排除。handler 已读取 generated `body.ResumeId` 并传入 `domain.SuggestQuestionsRequest.ResumeID`，但之前缺少测试锁定。

### 假设 3：store 层没有使用 `resumeId`
- 验证方法：新增 Red tests `TestStoreGetSuggestionContext_LoadsResumeStructuredProfile` 与 `TestStoreGetSuggestionContext_CrossUserResumeNotFound`。
- 结果：确认。Red 失败显示 `ResumeSummary` 为空，missing / cross-user resume 未返回 prerequisite error。

## 3 根因分析

- 根因是 D-20 `resumeVersionId` -> `resumeId` contract rename 只在 OpenAPI / fixture / generated request 层完成，Debriefs backend store 没有同步实现真实 `resumes.structured_profile` 查询。
- 相关路径：
  - `backend/internal/store/debrief/store.go`
  - `backend/internal/api/debriefs/handler.go`
  - `backend/internal/debrief/service.go`
  - `test/scenarios/e2e/p0-063-debrief-suggest-questions/scripts/trigger.sh`
  - `test/scenarios/e2e/p0-063-debrief-suggest-questions/scripts/verify.sh`
- 之前没有发现是因为 P0.063 gate 只证明 service/API mock 行为，没有把 store SQL、cmd/api route 装载、fixture `resumeId` 和 no-`resumeVersionId` 负向断言纳入同一个 scenario wrapper。

## 4 修复方案

- 在 `Repository.GetSuggestionContext` 中，当 request 带 `ResumeID` 时，按 `resumes.id = $1 AND resumes.user_id = $2 AND deleted_at IS NULL` 查询 `structured_profile`，写入 `SuggestionContext.ResumeSummary`。
- 增加 store Red/Green tests，覆盖 structured_profile 注入与 cross-user / missing resume fail-closed。
- 增加 service/API/cmd-api tests，分别锁定 resume structured_profile 进入 prompt、generated `resumeId` 映射到 domain request、真实 cmd/api route 挂在 session middleware 后。
- 强化 E2E.P0.063 trigger/verify：运行 store/service/API/cmd-api focused tests、`make validate-fixtures`，并检查 fixture 含 `resumeId` 且不含 `resumeVersionId`。
- 原地更新 backend-debrief/001 Phase 8 plan/checklist/test/BDD 文档与 history，恢复 owner plan `completed`。

## 6 经验教训

- Fixture parity is not enough for backend owner closure. For cross-layer API contract changes, the scenario wrapper must prove fixture shape, generated request mapping, real handler/service/store behavior, route mounting, and stale-field negative gates together.
- This fits existing patterns 4 and 5: completed checklist state can hide under-asserted runner gates, and domain behavior must be reverse-audited from the production caller and persistence path.

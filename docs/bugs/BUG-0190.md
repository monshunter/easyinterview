# BUG-0190: active session recovery could hang and finalize a stale snapshot

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-07-19 | 关联 Commit: `fix(practice): harden active session recovery (BUG-0190)`
> 关联日志: -

## 1 现象

- 对已有 `queued` session 的 fresh-key 恢复每 100ms 读取一次数据库，只依赖 caller context 退出；原启动 worker 在 reservation 后失联时，HTTP 请求可以无限等待并持续轮询。
- 对已有 `running` session 的恢复先读取 snapshot，再完成新 idempotency key；并发 completion 可以在两步之间把 session 推进到终态，导致 succeeded key 保存过时的 `running` 响应。
- 若只在恢复超时后把 queued session 标记为 failed，原启动 worker 仍可能迟到提交 opening 事务并把该 session 重新推进为 running。

## 2 诊断过程

### 假设 1：HTTP caller 会提供可靠的全链路 deadline

- 验证方法：反查 API server timeout、frontend start request 和 `recoverReservedSessionStart` 的退出条件。
- 结果：排除。恢复循环只有 caller context 与固定 100ms timer；当前 server/request 和 frontend start path 都不能保证在 orphaned queued start 上触发 deadline。

### 假设 2：恢复最终化的 running 校验与 idempotency update 已在同一锁顺序内

- 验证方法：逐行审查 `CommitSessionStartRecovery`，并用 SQL mock 要求先执行 session `FOR UPDATE`。
- 结果：排除。事务使用普通 `selectSessionForUser` 读取 session/messages，之后才更新 idempotency record；session row 在 snapshot 与 key finalization 之间没有锁。

### 假设 3：queued 超时可以直接复用现有失败路径而不增加提交 fence

- 验证方法：反查 `FailSessionStart` 与 `CommitSessionStart` 的 update predicate，并在真实 PostgreSQL 中模拟 timeout 先提交、原 worker 后提交。
- 结果：排除。两个 update 都未要求当前状态为 queued；没有 fence 时迟到 worker 可以复活已失败 session。

## 3 根因分析

- Phase 9 只定义“queued 等到 running 或 context 取消”，没有持久化 owner 的最长恢复时间、orphan 收敛语义和轮询负载上限，因此进程失联路径没有终止条件。
- `CommitSessionStartRecovery` 在同一事务内读取和写 key，但未锁定会被 completion 修改的 session row，事务边界没有形成业务线性化点。
- `CommitSessionStart` / `FailSessionStart` 仅以 session identity 更新状态，没有 `status='queued'` compare-and-set；恢复超时与原 worker 的竞态缺少唯一胜者。
- 原 focused/integration tests 覆盖恢复 happy path、不同 key 并发和 caller cancellation，但没有覆盖“原 starter 永不推进”“completion 在 snapshot 后终态化”“timeout 后原 starter 迟到”的负向顺序。

## 4 修复方案

- queued recovery 从持久化 `updated_at` 起最多等待 35 秒，采用 100ms 起步、1 秒封顶的指数退避；超时通过现有失败事务把 session 与当前 recovery key 原子写为 retryable `AI_PROVIDER_TIMEOUT`。
- `CommitSessionStartRecovery` 在读取 status/messages 与最终化 key 前先 `SELECT ... FOR UPDATE` 锁定 session row，使 recovery 与 completion 按数据库锁顺序线性化。
- `CommitSessionStart` 与 `FailSessionStart` 都增加 user + `status='queued'` fence，并要求 idempotency record 仍为当前 user/operation 的 pending row；未影响恰好一行时返回 typed conflict 并回滚整个事务。
- service 将迟到原 commit 的 store conflict 映射为 `PRACTICE_SESSION_CONFLICT`；timeout failure 若输给 running transition，则重新读取并完成正确的恢复响应。
- OpenAPI 与迁移不变；修复原地回填 backend-practice/001 Phase 9 的 spec/plan/test/BDD gate。

## 5 验证

- Deterministic service tests 证明永久 queued 只按 `100ms, 200ms, 400ms, ...` 有界等待，最终写 retryable timeout，且 prompt/AI/opening recovery finalization 均为零；caller cancellation 继续不修改状态。
- SQL mock tests 要求 recovery finalization 先锁 session row，并要求 original commit/failure update 包含 queued fence。
- 真实 PostgreSQL integration 持有 session row lock 后先提交 completion，恢复随后返回 conflict 且 recovery key 保持 pending；timeout 先提交时，迟到原 worker 返回 conflict，message/event/outbox/audit 全部为零。
- practice API/domain/store 全量单测与全部 store integration 通过；根 `make test` 通过 Python 584 tests / 4583 subtests、Go 全包与 frontend 127 files / 1035 tests。
- `make build`、`make lint`、`make codegen-check`、OpenAPI diff、fixtures、docs/context/index 与 `git diff --check` 全部通过；OpenAPI diff 为 0 breaking / 0 additive / 0 informational。

## 6 经验教训

- 轮询恢复必须同时定义时间边界、负载退避和持久化终态；caller context 不是 orphaned server workflow 的生命周期 owner。
- “同一事务”不等于并发安全。响应 snapshot 引用可变业务行时，必须先锁 owner row，再校验状态、读取投影并最终化幂等响应。
- 任何 stale-work recovery 都需要成对的 late-worker fence；只写 timeout failure 而不限制原 commit，会把失败状态重新复活。
- 并发测试应覆盖至少两个相反提交顺序，并断言失败事务的全部副作用回滚，而不只检查最终 session status。

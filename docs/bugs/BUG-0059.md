# BUG-0059: appendSessionEvent hint replay returned stored errors and hint snapshots incorrectly

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-15 | 关联 Commit: `fix(backend-practice): preserve append event replay snapshots`
> 关联日志: `-`

## 1 现象

- L2 reviewer 指出 SQL-backed replay strict `hint_requested` finalized error 时，repository 同时返回零值 `ReplayResult` 与 `ReplayError`；service 先处理 `ReplayResult`，导致原本应重放的 409 被误返回为 200 风格空结果。
- 同一 turn 上使用不同 `clientEventId` 连续请求多个 assisted hint 后，旧 replay path 会从可变的 `practice_turns.hint_text` 重建 response；后一次 hint 覆盖该列后，重放第一次 client event 会返回后一次 hint。
- 影响范围是 `appendSessionEvent` 的 body `clientEventId` 幂等契约：同一个 `(session_id, client_event_id)` 重试必须重放原始成功或错误结果，不得被后续事件或零值解码污染。

## 2 诊断过程

### 假设 1：service 会优先返回 repository 中记录的 finalized error

- 验证方法：新增 `TestAppendSessionEventReplayReturnsStoredErrorBeforeResult`，构造同时包含 `ReplayResult` 与 `ReplayError` 的 reservation replay。
- 结果：确认失败。修复前 `AppendSessionEvent` 先检查 `ReplayResult`，返回 replay=true 的空成功结果，没有返回原始 conflict error。

### 假设 2：redacted event payload 可安全承担 show_hint replay source

- 验证方法：检查 `selectSessionEventReplay` 与 P0.048 assisted hint scenario；先请求 hint A，再用另一个 `clientEventId` 请求 hint B，最后 replay hint A。
- 结果：确认失败。为避免 `practice_session_events.payload` 保存 hint 明文，旧实现从 `practice_turns.hint_text` 回填 hint；该列是 turn-level mutable snapshot，不是 event-level replay snapshot。

### 假设 3：只修 service 优先级即可恢复完整 replay 契约

- 验证方法：对 store payload marshalling 与 SQL repository replay 增加单元测试，覆盖 payload redaction 与 replay snapshot 分离。
- 结果：排除。严格错误 replay 和多 hint success replay 是两个不同持久化问题；需要同时修 service error 优先级和 DB replay snapshot source。

## 3 根因分析

- 根本原因：003 hint 隐私修复把 event payload redaction 与 replay source 混在同一字段上处理；为避免 `payload` 泄漏 hint 明文，replay 成功路径改为从业务表 `practice_turns.hint_text` 重建，但该表列会被同一 turn 的后续 hint 覆盖，不具备 per-event 不变性。
- 严格错误路径的根因是 `appendSessionEvent` service 假设 `ReplayResult` 与 `ReplayError` 互斥，未考虑 SQL sanitized error payload 解码时会产生零值 result。
- 涉及的代码路径：
  - `backend/internal/practice/append_session_event_service.go:145`
  - `backend/internal/store/practice/append_event.go:278`
  - `backend/internal/store/practice/append_event.go:527`
  - `backend/internal/store/practice/append_event.go:648`
  - `migrations/000001_create_baseline.up.sql:222`
- 之前没有发现的原因：P0.048 只断言 event payload 不泄漏 hint 与单次 replay 可用，没有覆盖“同一 turn 多个不同 hint event 后 replay 第一个 event”；strict 409 replay 测试没有模拟 SQL repository 同时返回零值 result 与 stored error 的形态。

## 4 修复方案

- `AppendSessionEvent` 先返回 `reservation.ReplayError`，再处理 `ReplayResult`，保证 finalized error replay 保持原始 409 语义。
- `practice_session_events` 在 pre-launch baseline 新增 nullable `replay_payload jsonb`，把 redacted event payload 与 client-visible replay snapshot 分离。
- assisted success finalize 时，`payload` 继续保存不含 hint 明文的 event payload；`replay_payload` 保存同 `(session_id, client_event_id)` replay 专用的 result snapshot。
- error finalize 时只写 sanitized error envelope，并将 `replay_payload` 置空，避免错误路径保存请求 payload 或空 success result。
- `selectSessionEventReplay` 优先用 `replay_payload` 的 event-level snapshot 重放成功结果，不再从可变的 `practice_turns.hint_text` 重建 hint。

## 6 经验教训

- 幂等 replay 的 source 必须绑定到原始 event，而不能从 mutable domain snapshot 重建；任何按 `clientEventId` 承诺 replay 的 response 都需要 event-level snapshot 或可证明不可变的来源。
- replay error 与 replay success 不应只靠零值 struct 是否存在区分；service 层应显式优先返回 stored error。
- 隐私红线与 replay 正确性是两个不同约束：`payload` 可保持 redacted，但 replay snapshot 仍需要独立字段、独立访问边界和针对多事件覆盖的回归测试。

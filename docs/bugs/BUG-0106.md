# BUG-0106: privacy delete cleanup completed without removing UAT account identity

> 模块: platform | 严重度: medium | 状态: investigating
> 发现日期: 2026-05-26 | 关联 Commit: `-`
> 关联日志: `-`

## 1 现象

- `E2E.P0.100` cleanup 按材料走真实产品路径：`DELETE /api/v1/me` 返回 `202`，创建 `privacy_delete` job。
- backend runner 将 privacy request 标记为 `completed`，async job 标记为 `succeeded`。
- 但随后查询 `users`，synthetic UAT 邮箱 `manual-uat-full-funnel@example.test` 对应用户行仍存在，且 `deleted_at` 为空。

## 2 诊断过程

### 假设 1：privacy_delete runner 没有执行
- 验证方法：查询 `privacy_requests` 与 `async_jobs`。
- 结果：排除。privacy request `019e6435-0a9c-7e09-876c-8f3f2109b190` 为 `completed`，job `019e6435-0a9c-7e0d-b785-bd2e1677126c` 为 `succeeded`。

### 假设 2：cleanup 材料使用了错误入口
- 验证方法：检查 `test/scenarios/manual-uat/full-funnel/materials/account.md` 与浏览器执行路径。
- 结果：排除。入口是文档指定的 `DELETE /api/v1/me` 产品路径，没有使用 broad DB delete 或直接 session table writes。

### 假设 3：privacy_delete runner 当前只删除部分 domain 数据
- 验证方法：阅读 `backend/internal/privacy/runner/delete_handler.go`。
- 结果：确认。runner 当前调用 upload/profile/jd_match deleter，并标记 request completed；没有删除、匿名化或 tombstone `users.email` / `users.deleted_at` 的实现。

## 3 根因分析

- 初步根因是 `privacy_delete` job 的完成语义与 cleanup/account deletion 期望不一致：job success 只代表部分 domain deleter 成功，不代表用户身份行已清除或匿名化。
- 涉及路径：
  - `backend/internal/auth/service.go` 创建 privacy delete handoff 并撤销 session。
  - `backend/internal/privacy/runner/delete_handler.go` 执行 upload/profile/jd_match cleanup 后完成 request。
  - `users` 表身份数据未被 hard delete、soft delete 或 anonymize。
- 之前没有发现，是因为 manual UAT cleanup gate 只检查 job/request terminal status，没有反查 UAT 邮箱或用户身份数据残留。

## 4 修复方案

- 待 owner 设计确认：`privacy_delete` 应该 hard-delete user、soft-delete user，还是 anonymize email 并设置 `deleted_at`。
- 修复时需要补充 runner/handler 级测试：privacy request/job succeeded 后，UAT 用户身份数据不再以原邮箱可查询；同时保留必要审计 tombstone 且不泄露邮箱。
- cleanup 材料需要在修复后明确“completed”的语义，避免把 job terminal state 误当成账号已清除。

## 5 验证

- 当前仅验证了缺陷存在：
  - `DELETE /api/v1/me` 返回 `202`。
  - privacy request `completed`。
  - privacy job `succeeded`。
  - `select id,email,deleted_at from users where email='manual-uat-full-funnel@example.test';` 仍返回原邮箱且 `deleted_at` 为空。
- 修复验证待补。

## 6 经验教训

- Cleanup gate 不能只看 async job/request terminal state；必须反查 cleanup 目标的关键残留字段。
- 隐私删除类 runner 的 success 语义需要覆盖身份数据、domain data、session/token、audit tombstone 的最小闭环，而不是只覆盖已接入的 domain deleter。

# BUG-0111: scenario env review follow-ups exposed stale env and evidence gates

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-27 | 关联 Commit: `fix(test): close scenario env review follow-ups (BUG-0111)`
> 关联日志: `-`

## 1 现象

- Review 指出 `test/scenarios/env-redeploy.sh frontend|all` 构建前没有加载 `deploy/dev-stack/.env`，Vite build 可能把 frontend real-mode artifact 回退到生产默认 `/api/v1`。
- Review 指出 `test/scenarios/env-setup.sh --with-migrations` 启动 dev-stack 后，migration 仍只看调用者进程 env 或硬编码 5432 默认值，无法遵守单一 dev-stack env 文件契约。
- Review 指出 `E2E.P0.100` hybrid 场景只要旧 `evidence.md` 含 provider/profile/model/task-run 四个 marker 就会写入 `PASS`，且没有在 PASS 前扫描 evidence 脱敏红线。
- Review 指出 `p0-100` README 的 Owner plan 相对链接少了一层 `../`，点击会解析到不存在的 `test/docs/...`。

## 2 诊断过程

### 假设 1：frontend redeploy build 已经通过 scenario README 手动 source `.env`

- 验证方法：读取 `test/scenarios/env-redeploy.sh` 和 `test/scenarios/e2e/p0-100-real-provider-full-funnel-hybrid/README.md` 的 frontend real-mode 启动说明。
- 结果：排除。README 手动步骤要求 `set -a; . ../deploy/dev-stack/.env; set +a`，但共享 `env-redeploy.sh` 只执行 `pnpm --filter @easyinterview/frontend build`。

### 假设 2：migration 命令会从 dev-stack compose env 自动继承 `DATABASE_URL`

- 验证方法：读取 `test/scenarios/env-setup.sh`、`deploy/dev-stack/README.md` 与本地栈的 host-run 边界。
- 结果：排除。`env-setup.sh` 先调用 `make dev-up` / `make dev-doctor`，随后 migration 分支只设置 `DATABASE_URL="${DATABASE_URL:-postgres://easyinterview:dev@localhost:5432/easyinterview?sslmode=disable}"`，没有加载 `deploy/dev-stack/.env`。

### 假设 3：hybrid evidence 的 PASS 分支已经证明当次运行和脱敏

- 验证方法：读取 `p0-100` 的 `setup.sh`、`trigger.sh`、`verify.sh`。
- 结果：排除。`setup.sh` 不清理 `evidence.md`；`trigger.sh` 只检查四个宽泛 marker；`verify.sh` 只扫描 `trigger.log`，不扫描 evidence 文件。

### 假设 4：README Owner plan 链接只影响文档可读性，不影响执行

- 验证方法：用 `check_md_links.py` 对目标场景目录做相对链接检查。
- 结果：确认但仍需修复。链接不影响脚本执行，但会让 owner plan 追溯失败，削弱后续 review / handoff 定位。

## 3 根因分析

- 根本原因：BUG-0110 的 single-env-source 修复只覆盖了场景 `trigger.sh` 和 README 手动 runbook，没有把共享环境脚本 `env-setup.sh` / `env-redeploy.sh` 一并纳入 contract gate。
- 次级根因：Hybrid PASS 语义只表达“有人写过 evidence marker”，没有把“本轮运行生成”和“脱敏红线通过”作为 PASS 的必要条件。
- 涉及路径：
  - `test/scenarios/env-redeploy.sh`
  - `test/scenarios/env-setup.sh`
  - `test/scenarios/e2e/p0-100-real-provider-full-funnel-hybrid/scripts/setup.sh`
  - `test/scenarios/e2e/p0-100-real-provider-full-funnel-hybrid/scripts/trigger.sh`
  - `test/scenarios/e2e/p0-100-real-provider-full-funnel-hybrid/scripts/verify.sh`
  - `test/scenarios/e2e/p0-100-real-provider-full-funnel-hybrid/README.md`
  - `scripts/lint/scenario_env_contract_test.py`
- 之前没有发现，是因为现有 contract test 只检查了 env 文件名、hybrid 注册和基本脚本结构，没有覆盖构建时 Vite env、migration env 派生、evidence 当次性或 evidence 脱敏扫描。

## 4 修复方案

- `env-redeploy.sh` 的 frontend 分支先 source `deploy/dev-stack/.env`，并要求 `VITE_EI_API_MODE` / `VITE_EI_API_BASE_URL` 存在后再运行 frontend build。
- `env-setup.sh --with-migrations` 先 source `deploy/dev-stack/.env`；如果 `DATABASE_URL` 缺失或仍是模板默认值，则从 `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` / `POSTGRES_HOST_PORT` 派生带 URL escape 的本地连接串。
- `p0-100 setup.sh` 每轮写入 `RUN_ID` 并清理旧 `evidence.md`。
- `p0-100 trigger.sh` 要求 evidence 包含本轮 `run_id`，并在写入 PASS 前扫描 `AI_PROVIDER_API_KEY`、auth secret、session cookie、magic-link token、prompt / response body 和 `sk-...` 形态。
- `p0-100 verify.sh` 在 result 为 PASS 时重复扫描 evidence 文件，并要求 `trigger.log` 含 redacted evidence PASS marker。
- 修正 P0.100 README Owner plan 链接，并在 README 说明 `run_id` evidence 契约。
- 扩展 `scripts/lint/scenario_env_contract_test.py`，覆盖以上 review follow-up gate。

## 6 经验教训

- “单一 env 文件”契约必须覆盖所有消费入口，包括 setup、migration、build 和 scenario trigger；只修 runbook 或单一脚本会留下非默认端口 / 账号的漂移面。
- Hybrid 场景的 `PASS` 需要 run-scoped evidence marker 与脱敏扫描；只检查摘要 marker 会让陈旧证据或未脱敏证据 false-green。
- Scenario README 的 owner plan 链接应纳入目标目录 link check，不能只依赖 `docs-check` 扫描 `docs/`。

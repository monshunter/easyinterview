# BUG-0012: AI client L2 review exposed tools, streaming, and observability drift

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-06 | 关联 Commit: `fix(ai-provider): remediate tools streaming stt L2 findings`
> 关联日志: `-`

## 1 现象

- `$plan-code-review ai-provider-and-model-routing/002-tools-streaming-and-stt --fix` 复扫发现工具调用、流式终态和观测标签三个已完成 checklist 路径仍存在运行时语义缺口。
- Stub provider 在带工具的请求中仍只返回确定性文本，没有回放 deterministic tool calls，无法作为工具路由和函数调用契约的可执行替身。
- `Client.Stream` 直接转发 provider 的 final `done.Meta`，绕过 canonical meta merge，导致 provider 返回局部 meta 时缺少 profile、provider、model、capability、route 等字段。
- Observability wrapper 暴露了 `ProfileResolver` 配置，但 failure / stream labels 仍可能落到 `unknown`，历史 gate 没有证明 resolved profile 会进入指标标签。

## 2 诊断过程

### 假设 1：stub provider 已经覆盖 tools replay

- 验证方法：增强 `TestStubCompleteWithToolsIsDeterministic`，断言带 tools 的 `Complete` 返回 deterministic `ToolCalls`、`FinishReason=tool_calls` 和脱敏后的 `ToolInvocations` meta。
- 结果：确认缺口。修复前 focused test 失败，返回的 `ToolCalls` 为空。

### 假设 2：stream done meta 会在 client 层补齐 canonical fields

- 验证方法：新增 `TestStream_PartialDoneMetaIsCanonicalMerged`，并强化既有 stream done 测试，要求 final `done.Meta` 带有 profile、provider、model、capability 与 route 字段。
- 结果：确认缺口。修复前 partial done meta 中 `ModelProfileName`、`Provider`、`ModelID` 等 canonical fields 为空。

### 假设 3：observability ProfileResolver 会用于 pre-dispatch failure 和 stream labels

- 验证方法：新增 pre-dispatch failure、stream done、stream error 三个 focused tests，断言 metrics labels 使用 resolver 返回的 provider/model/profile/capability。
- 结果：确认缺口。修复前 labels 仍记录为 `unknown` 或没有 enriched sample。

## 3 根因分析

- 根本原因：002 plan 的历史 gate 主要证明了类型、接口和基本 happy path 存在，没有把 provider replay、client canonical meta merge 和 observability label enrichment 作为同一运行时契约闭环验证。
- 涉及路径：
  - `backend/internal/ai/aiclient/providers/stub/stub.go`
  - `backend/internal/ai/aiclient/providers/stub/stub_test.go`
  - `backend/internal/ai/aiclient/client.go`
  - `backend/internal/ai/aiclient/aiclient_test.go`
  - `backend/internal/ai/aiclient/observability/decorator.go`
  - `backend/internal/ai/aiclient/observability/decorator_test.go`
  - `docs/spec/ai-provider-and-model-routing/plans/002-tools-streaming-and-stt/plan.md`
  - `docs/spec/ai-provider-and-model-routing/plans/002-tools-streaming-and-stt/checklist.md`
- 为什么之前没有发现：既有测试偏向结构和最小输出断言，没有反查当前 active spec 中“工具调用可回放、流式 meta 可审计、profile route 可观测”的运行时不变量。

## 4 修复方案

- 在 stub provider 中将 tools / tool_choice 纳入 deterministic seed；当请求允许工具调用时生成 deterministic tool calls，并填充 tool invocation meta，避免把 prompt 明文写入 arguments。
- 在 `Client.Stream` 中包装 provider stream，对 terminal `done.Meta` 重新执行 canonical merge；merge 失败时返回 `AI_OUTPUT_INVALID` stream error。
- 在 observability decorator 中新增 meta enrichment 路径，让 complete / embed / transcribe failure 与 stream done / error labels 从 call metadata 和 `ProfileResolver` 补齐 provider、model、profile、capability、route 与 profile version。
- 原地修订 002 plan/checklist，新增并完成 2.4、3.4、5.4 L2 remediation gate，并同步 plans INDEX。
- 未选择另建 sibling bugfix plan，因为当前主题已有 owner plan，按深度重校对门禁应由原 plan 原地承接。

## 6 经验教训

- AI provider routing 的 L2 gate 不能只断言接口可调用，还要断言工具调用、stream terminal meta 和 observability labels 三个运行时证据面同时保留 canonical profile route。
- Stub provider 是契约测试替身时必须回放结构化工具调用，否则工具链测试会退化为纯文本 completion。
- Stream wrapper 不能信任 provider 的局部 meta 已经完整；client 层需要在统一出口补齐或拒绝不合法 meta。

# BUG-0024: home JD import L2 review exposed privacy and gate drift

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `fix(frontend-home): remediate jd import L2 findings`
> 关联日志: `-`

## 1 现象

- `frontend-home-job-picks-and-parse/001-home-jd-import-and-parse` 已标记 completed，但 L2 code review 发现实现仍偏离当前 spec 与 `ui-design` parity gate。
- Home recent jobs 在 hook 层提前截断 12 条，导致 `updatedAt desc` 排序后可能丢掉最新岗位。
- 未登录导入 JD 的 pending action 会把 source payload 放入 route params，存在把 JD 原文或 source URL 序列化到路由状态的隐私风险。
- Parse loading footer 暴露 provider/model/prompt hash 口径，和当前“backend-managed / provenance redacted”要求不一致。
- 场景与工程 gate 也存在漂移：P0.006 wrapper 仍断言旧的 `48 passed`，`backend/internal/platform/secrets` 包被 `.gitignore` 忽略，`make codegen-check` 暴露 targetjob event/job 裸 literal。

## 2 诊断过程

### 假设 1：历史 checklist completed 和旧 PASS 足以证明当前实现闭环

- 验证方法：直接读取 active spec/plan/checklist、`ui-design` 源码、前端实现、场景脚本、generated contracts，并重跑当前 gate。
- 结果：排除。历史 PASS 没覆盖当前 L2 语义：recent 排序、pending import privacy、provider-neutral parse footer、P0.006 68-test wrapper 和 generated event/job constant 都需要重新验证。

### 假设 2：问题只在前端单元测试层

- 验证方法：运行 focused Vitest、全量 Vitest、Playwright pixel parity、scenario wrapper、`make build`、`make codegen-check`。
- 结果：排除。问题跨前端 runtime、场景 verify 脚本、根级构建跟踪和 backend targetjob contract lint。

### 假设 3：`make build` 通过即可说明后端 secrets package 已被纳入交付

- 验证方法：检查 `git status --short` 与 `git check-ignore -v backend/internal/platform/secrets/*`。
- 结果：排除。`secrets/` 通用 ignore 规则会忽略该 Go source package，导致本地构建修复无法进入版本控制。

## 3 根因分析

- L2 review 之前的 gate 更偏结构数量和旧脚本 PASS，没有把 current spec 的 privacy / generated-client / provider-neutral 语义转成负向断言。
- Home recent jobs 的分页与展示排序边界混在 hook 内处理，提前 slice 破坏了“先按 `updatedAt desc` 排序再取 12 条”的契约。
- D1 pending action 可序列化 route params，但 JD import source 是敏感 payload；实现没有引入 opaque pending id 来隔离 URL/localStorage 可见状态。
- P0.006 scenario verify 把 Playwright 总数硬编码为旧 D2/D3 数量，新增 home/parse/jd_match parity specs 后没有同步 wrapper 断言。
- `.gitignore` 的通用 `secrets/` 规则没有区分真实密钥目录与 `backend/internal/platform/secrets` 源码包。
- targetjob backend 使用了 event/job 字符串 literal 和注释中的 quoted literal，触发 generated-contract lint。

## 4 修复方案

- Home recent jobs：hook 只请求 `pageSize=12` 并保持数组防御，HomeScreen 负责按 `updatedAt desc` 排序后取 12 条；新增/修订测试锁定 13 条 fixture 最新项保留。
- JD import pending action：新增 SPA 会话内存 pending import store，只在 route params 放 `pendingImportId`，登录恢复后回 home 消费 payload 自动导入；测试负向断言 JD 原文/source URL 不进入 route params、localStorage 或日志。
- Parse footer：保留 `ui-design` 3 行 footer DOM 结构，但文案改为 provider-neutral metadata；补充负向测试禁止 `claude`、`haiku`、`prompt@` 等口径。
- 场景 gate：P0.006 verify 改为当前 `68 passed`，并要求 home/parse/jd_match parity specs 出现在 trigger log。
- 构建/契约：放行 `backend/internal/platform/secrets/**` 的 Git 跟踪，补齐 `EnvSecretSource` package；targetjob event/job 写入改用 generated `events` / `jobs` constants。
- 文档：原地修订 owner spec/plan/checklist/BDD checklist，完成后恢复 plan 状态为 `completed` 并同步 INDEX。

## 6 经验教训

- Completed checklist 只能作为线索，L2 review 必须重新执行 current truth source 的语义 gate，尤其是 privacy、generated contract、scenario wrapper 和 UI parity 断言。
- 对敏感 payload 的 pending action 只能传 opaque handle；任何可序列化 route/localStorage 状态都应默认视为可见面。
- Scenario verify 不能只硬编码历史测试总数；新增 spec 后必须同步断言具体 spec marker，避免“本体 PASS、wrapper FAIL”在收口阶段才暴露。
- 通用 `.gitignore` 规则要避免误伤源码包；新增目录名包含 `secrets` / `config` 等词时必须用 `git check-ignore -v` 验证。

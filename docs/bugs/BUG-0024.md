# BUG-0024: targetjob detail omitted parsed summary provenance

> 模块: workspace | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `feat(backend-targetjob): complete import parse bootstrap handoff`
> 关联日志: [2026-05-08](../work-journal/2026-05-08.md)

## 1 现象

- `E2E.P0.010` 的 red test 在 `GET /targets/{id}` 后失败，错误为 `detail missing parsed summary provenance: <nil>`。
- Parse executor 已经把 requirements、`summary`、`fit_summary`、`latest_parse_job_id` 和 `target.parsed` 写入 store，但 generated `TargetJob` response 中 `summary` / `fitSummary` 仍为空。
- 影响范围是 TargetJob 详情和列表 response 的 evidence surface；后端持久化成功，但 frontend 无法获得 P0 要求的 summary provenance。

## 2 诊断过程

### 假设 1：parse executor 没有持久化 summary / fit_summary
- 验证方法：检查 `backend/internal/targetjob/parse_executor.go` 的成功路径与 store mock 记录。
- 结果：排除。parse executor 成功路径会把解析出的 summary / fit summary 传给 store，并更新 `analysis_status='ready'`。

### 假设 2：store record 已包含 JSON，但 API mapper 没有映射
- 验证方法：检查 `backend/internal/targetjob/service.go` 中 `TargetJobRecord` 到 generated `api.TargetJob` 的转换。
- 结果：确认。mapper 只设置了基础字段和 requirements，没有把 `rec.Summary` / `rec.FitSummary` 解码到 generated response。

### 假设 3：既有 tests 已覆盖 generated response provenance
- 验证方法：检查 `backend/internal/targetjob` 现有 focused tests。
- 结果：确认缺口。既有测试覆盖了解析写入和 outbox，但没有在 generated API detail/list response 上断言 summary provenance。

## 3 根因分析

- 根本原因：持久化层和解析层已经演进到写入 summary / fit summary JSON，但 service response mapper 没有同步承接新增字段。
- 涉及代码路径：
  - `backend/internal/targetjob/service.go`
  - `backend/internal/targetjob/e2e_scenario_test.go`
- 之前没有发现的原因：早期 tests 偏向 store / pipeline side effects，缺少从 import → parse → generated detail response 的端到端断言。

## 4 修复方案

- 在 `service.go` 的 TargetJob mapper 中解码 `rec.Summary` 与 `rec.FitSummary`，并设置 generated `api.TargetJob.Summary` / `api.TargetJob.FitSummary`。
- 增加 `decodeTargetJobSummary`、`decodeTargetJobFitSummary`、`hasMaterializedJSON`、`hasGenerationProvenance` helpers，避免空 JSON 或缺少 provenance 的对象被误展示。
- 新增 `TestE2EP0010TextImportParseReady` 断言详情 response 包含 parsed summary provenance 与 fit summary provenance。

## 5 验证

- Red: `go test -v ./internal/targetjob -run TestE2EP0010TextImportParseReady -count=1` 失败于 missing summary provenance。
- Green: `go test -v ./internal/targetjob -run TestE2EP0010TextImportParseReady -count=1` 通过。
- 回归: `go test ./internal/targetjob ./internal/targetjob/urlfetch ./cmd/api -count=1` 通过。
- BDD: `.test-output/runs/targetjob-20260508-001/e2e/E2E.P0.010/result.json` 生成并通过。

## 6 经验教训

- API mapper tests 必须覆盖 generated response 的 evidence fields，而不只验证 store side effects。
- 当 plan 要求 frontend handoff 时，后端 focused tests 应至少有一条从 persistence 到 generated API response 的完整断言。

# BUG-0084: jd-match L2 follow-up missed privacy runner and agent-scan context wiring

> 模块: schema | 严重度: high | 状态: resolved
> 发现日期: 2026-05-22 | 关联 Commit: `fix(backend-jobs): close jd-match L2 follow-up (BUG-0084)`
> 关联日志: [2026-05-22](../work-journal/2026-05-22.md)

## 1 现象

- `/plan-code-review backend-jobs-recommendations/001-jd-match-real-backend-baseline --fix` follow-up review 发现 `privacy_delete` handler 仍只删除 upload file objects，未调用 backend-profile candidate profile 删除和 JD-Match 5 表 privacy delete。
- `jd_match_agent_scan` 调用 `RunRecommendationGenerator` 时只传 `userID` / `agentScanID`，导致 F3 `jd_match.recommendation` prompt 中的 `candidateProfile` 和 `jobsPool` 为空上下文。
- JDMatch handler 的错误响应返回裸 `ApiError`，且 `AI_OUTPUT_INVALID` 因 HTTP 502 被标记为 retryable=true。
- `.claude/scheduled_tasks.lock` 误进入版本控制，包含本地 session / pid 状态。

## 2 诊断过程

### 假设 1：privacy delete 已通过 JDMatch service 实现自动接入 runner

- 验证方法：读取 `backend/internal/privacy/runner/delete_handler.go` 与 `backend/cmd/api/main.go` 的 `privacy_delete` handler 注册。
- 结果：排除。JDMatch service 有 `DeleteJobMatchDataForUser`，但 `privacyrunner.NewPrivacyDeleteHandler` 只拿到 `Requests` 与 `UploadFiles`。

### 假设 2：agent_scan drainer 会在 cmd/api 层补齐 generator context

- 验证方法：读取 `backend/internal/jdmatch/jobs/agent_scan.go` 与 `backend/cmd/api/jdmatch_runtime.go::runJDMatchAgentScan`。
- 结果：排除。`RunRecommendationGeneratorInput` 支持 `CandidateProfileJSON` / `JobsPoolJSON`，但调用方没有填充。

### 假设 3：JDMatch error contract 已跟其他 backend handler 对齐

- 验证方法：对比 upload/practice/debrief handler 的 `ApiErrorResponse` envelope，并运行 `TestSearchJobsInvalidOutput`。
- 结果：排除。JDMatch 使用裸 `ApiError`，客户端无法按 B2 envelope 读取 `error.code`，且 retryable 来自 status>=500 而不是 shared registry。

## 3 根因分析

- 根本原因：上一轮 L2 hardening 关注 production AI adapter、F3 prompt、search outbox 和 transaction rollback，但没有把已经存在的 domain service 反查到上游 runtime caller，也没有断言 AI payload 内容非空。
- 涉及路径：`backend/internal/privacy/runner/delete_handler.go`、`backend/cmd/api/main.go`、`backend/cmd/api/jdmatch_runtime.go`、`backend/internal/jdmatch/jobs/agent_scan.go`、`backend/internal/jdmatch/handler/handler.go`、`.claude/scheduled_tasks.lock`。
- 之前没有发现：live scenario 只断言 drainer upsert / outbox / privacy helper 函数结果，没有捕获 generator payload，也没有通过 `/api/v1/me` async privacy job 证明 domain deleter 已注册。

## 4 修复方案

- 扩展 `PrivacyDeleteHandlerOptions`，允许注入 profile 与 JDMatch domain deleter；`cmd/api` 通过 `privacyDeleteRuntimeHooks` 在 startup composition 中把 backend-profile service 与 JDMatch privacy function 接入 `privacy_delete` runner。
- `AgentScanDeps` 新增 `CandidateProfile` 与 `JobsPool` builders；cmd/api 复用 `BuildJobMatchProfile` 与 `jd_match_recommendations` compact pool，确保 generator payload 含 runtime candidate profile 与 `jobMatchId`。
- JDMatch `writeAPIError` 改为返回 `api.ApiErrorResponse{Error: ...}`，retryable 使用 `sharederrors.CodeRegistry`。
- 删除 `.claude/scheduled_tasks.lock`，并在 `.gitignore` 忽略该本地锁文件。

## 6 经验教训

- L2 follow-up 不应只验证 service 已存在；必须从 production caller 反查到 domain deleter、AI payload、error envelope 等 runtime composition 入口。
- AI prompt contract 的字段存在不等于 payload 有值；drainer/live tests 应捕获传给 AI adapter 的 context JSON。
- 本地 agent lock / session 文件应作为 local tool state 统一 ignore，避免把机器状态提交到仓库。

# BUG-0054: backend-practice event loop L2 review exposed payload, status, and BDD gate drift

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-13 | 关联 Commit: `fix(backend-practice): remediate event loop L2 findings`
> 关联日志: [2026-05-13](../work-journal/2026-05-13.md)

## 1 现象

- `plan-code-review 002-event-loop-and-completion --fix` 复查当前代码与 plan/spec 时发现 `answer_submitted` 只校验 `payload.turnId`，没有按 spec §2.1 要求校验 `payload.answerText`。
- `completePracticeSession` repository 在没有既有 report/job 时允许任意 user-scoped session 进入 `completing` 并创建 queued report/job，没有限制源状态。
- backend-practice 002 BDD 编号使用了 `E2E.P0.034` / `E2E.P0.035`，而 `test/scenarios/e2e/INDEX.md` 已把这两个编号分配给 backend-resume。

## 2 诊断过程

### 假设 1：`answer_submitted` 的 payload 校验已在 router 层覆盖

- 验证方法：读取 `backend/internal/practice/append_session_event_service.go` 与 `backend/internal/practice/session_event.go`，对照 `docs/spec/backend-practice/spec.md` §2.1 的 event payload contract。
- 结果：排除。service 在 reservation 后只校验当前 `turnId`，router 可以继续处理缺失 `answerText` 的输入，导致空答案进入状态机。

### 假设 2：完成接口的幂等 replay 已经等价于状态机 guard

- 验证方法：读取 `backend/internal/store/practice/complete_session.go` 的事务顺序，对照 plan D-35 与 spec C-11 / C-27。
- 结果：部分确认。既有 `feedback_reports` + `async_jobs` 的 D-35 replay 应先返回原 report/job；但没有既有 report/job 时，`queued` / `completing` / `failed` / `cancelled` 不应创建新的 queued report/job。

### 假设 3：BDD 编号只在 Go test 名里使用，不会与 `test/scenarios` 资产冲突

- 验证方法：读取 `docs/spec/backend-practice/plans/002-event-loop-and-completion/bdd-plan.md` 与 `test/scenarios/e2e/INDEX.md`。
- 结果：排除。编号空间是全局 `E2E.P0.xxx` 语义空间，即使本 plan 使用 `cmd/api` Go HTTP scenario tests，也不能复用已分配给 backend-resume 的编号。

## 3 根因分析

- 根本原因：L2 review 前的完成证据偏重 happy path / replay path，没有把 required payload 字段、非法状态转移、全局 BDD 编号唯一性纳入可执行 gate。
- 涉及的代码路径：
  - `backend/internal/practice/append_session_event_service.go`
  - `backend/internal/store/practice/complete_session.go`
  - `backend/cmd/api/practice_http_scenario_test.go`
  - `scripts/lint/backend_practice_legacy.py`
  - `docs/spec/backend-practice/plans/002-event-loop-and-completion/*`
- 之前没有发现的原因：历史 checklist 已标记 completed，且 `backend_practice_legacy.py` 只覆盖 retired vocabulary / dispatcher 反查，没有解析 BDD 编号分配与 scenario index 的交叉唯一性。

## 4 修复方案

- 在 `AppendSessionEvent` service 中对 `answer_submitted` 增加非空 `payload.answerText` 校验；缺失时返回 `VALIDATION_FAILED`，只完成 reservation，不触发 AI、不写 event。
- 在 SQL repository 的 `CompleteSession` 中保留 D-35 replay 优先级；没有既有 report/job 时，只允许 `running` / `waiting_user_input` / `completed` 创建 queued report/job，其它 session status 返回 `ErrSessionConflict`。
- 同步 `cmd/api` fake store 与 HTTP scenario，补非法状态 complete 返回 409 的用户可见断言。
- 把 backend-practice 002 BDD 编号迁移为 `E2E.P0.038` ~ `E2E.P0.043`，并在 `backend_practice_legacy.py` 增加编号碰撞与 Go HTTP scenario test 入口反查。
- 修订原 002 plan / checklist / test-plan / BDD 文档并 bump 到 v1.1。

## 6 经验教训

- Event append 类 API 的 required payload 字段必须在 service reservation 后、状态机 route 前校验，不能只依赖 handler decode 或 router branch。
- Completion 类 API 需要同时验证 replay 优先级与非法源状态；幂等 replay 不是状态机 guard 的替代品。
- BDD 编号空间是全局语义空间；即使场景实现形态是 Go HTTP scenario tests，也必须与 `test/scenarios/e2e/INDEX.md` 做碰撞检查。

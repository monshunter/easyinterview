# BUG-0171: report generation rejected valid context at a stale hardcoded byte limit

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-07-14 | 关联 Commit: `fix(runtime): centralize content size limits (BUG-0171)`
> 关联日志: [2026-07-14](../work-journal/2026-07-14.md)

## 1 现象

- 报告生成页对一份约 62,397-byte 的真实 framed input 显示“本次材料与对话过长”，并要求用户缩短输入后重新开始。
- 请求在 AI provider 调用前被拒绝，虽然当前 `report.generate.default` profile 的 context window 为 1,000,000 tokens。
- 同类内容大小限制分散在配置、业务服务、provider adapter 与前端本地常量中，存在继续漂移的风险。

## 2 诊断过程

### 假设 1：模型上下文或 TPM 限制拒绝请求

- 验证方法：读取当前 profile catalog、loader 和 report 调用路径，并核对 provider admission 之前的返回位置。
- 结果：排除。失败发生在 provider 调用前；TPM 是吞吐提示，不是单请求 context hard cap。

### 假设 2：报告业务层保留了过时的本地大小常量

- 验证方法：追踪 report framed-input 校验并用 62,397-byte 输入复现；再以 917,504 / 917,505-byte 内存边界测试观测 provider 调用次数。
- 结果：确认。旧 48,000-byte package hardcode 提前拒绝了合法输入。

### 假设 3：修订默认值后各 consumer 仍可能使用独立口径

- 验证方法：审计 HTTP、upload/resume、TargetJob、Practice、Report、AI provider 与 RuntimeConfig 前端 consumer，并执行旧常量负向搜索。
- 结果：确认存在跨层漂移，需要统一 typed defaults 与显式注入，而不是只提高 report 单点常量。

## 3 根因分析

- Report 业务层使用独立的 48,000-byte framed-input 常量，没有消费统一运行时配置。
- 配置缺少完整的 typed content-limit contract；前端和 provider adapter 也保留了各自默认值，使单点修改无法保证跨层一致。
- 历史边界测试依赖提交的 `input-*.json`，既放大仓库体积，也没有直接锁定“limit 接受、limit+1 在下游调用前拒绝”的业务副作用。

## 4 修复方案

- 建立统一 `ContentLimits` typed defaults、YAML override 与 fail-fast validation，并将 HTTP、Resume、TargetJob、Practice、Report 和四个 provider adapter 改为注入配置。
- 将 report 默认 framed-input 上限修订为 917,504 bytes；62,397 与 917,504 bytes 各允许一次 provider admission，917,505 bytes 返回 terminal `REPORT_CONTEXT_TOO_LARGE` 且 provider calls 为 0。
- RuntimeConfig 只公开五个前端 preflight 所需字段；内部 HTTP/report/provider/profile 上限不公开。
- 删除 `backend/internal/review/testdata/report-boundary/input-*.json`，改用 deterministic in-memory helper 构造 exact byte payload；仅保留 current-schema zh/en output worst-case fixtures。
- 未采用“只把 48,000 改成更大 package 常量”，因为该方案无法消除其他 consumer 的重复真理源。

## 5 验证

- Report focused tests：62,397 / 917,504 / 917,505-byte 内存边界、canonical frame round-trip 和 provider call/no-call 全部通过。
- BDD：E2E.P0.010、P0.015、P0.046、P0.056、P0.058、P0.081 当前串行证据通过；P0.058 PostgreSQL 证据确认 oversized failure 持久化且 provider calls 为 0。
- 全量 backend `go test ./... -count=1`、相关 package `-race`、frontend 126 files / 1,018 tests 与 production build 通过。
- OpenAPI 37 operations / 10 tags、fixture、diff、OPENAPI-006 exact audit 与 52 个 diff-wrapper tests 通过。

## 6 经验教训

- 内容大小限制应由一个 typed configuration owner 持有，consumer 只接收注入值；公开 runtime projection 必须保持最小化。
- 边界测试优先在内存构造 exact bytes，并同时断言下游调用次数和持久化副作用；没有必要提交真实大输入文件。
- 调高默认值后，所有历史“oversized”测试样本必须重新核对是否仍真正越界，否则会悄悄进入不同业务路径。

# BUG-0171: report generation rejected valid context at a stale hardcoded byte limit

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-07-14 | 关联 Commit: `fix(runtime): centralize content size limits (BUG-0171)`
> 关联日志: [2026-07-14](../work-journal/2026-07-14.md)

## 1 现象

- 报告生成页对一份约 62,397-byte 的 framed input 显示“本次材料与对话过长”，要求用户缩短输入后重新开始。
- 请求在 AI provider 调用前被拒绝，而当前 `report.generate.default` profile 的 context window 为 1,000,000 tokens。
- 同类限制分散在配置、业务服务、provider adapter 与前端本地默认值中，单点修改不能保证一致。

## 2 诊断过程

### 假设 1：模型上下文或 TPM 限制拒绝请求

- 读取当前 profile catalog、loader 和 report 调用路径，核对失败发生位置。
- 结果：排除。请求尚未到 provider；TPM 是吞吐提示，不是单请求 context hard cap。

### 假设 2：报告业务层保留过时本地常量

- 追踪 framed-input guard 与失败发生位置，确认 observed 62,397-byte request 在 provider admission 前被旧常量拒绝；该尺寸只作为诊断证据，不重建为测试材料。
- 结果：确认。旧 48,000-byte package hardcode 在统一配置之前提前拒绝合法输入。

### 假设 3：只提高 report 常量即可消除问题

- 审计 HTTP、Resume、TargetJob、Practice、Report、provider adapter 与 RuntimeConfig consumer。
- 结果：排除。多处生产默认值会继续漂移，必须建立单一 typed configuration owner 并显式注入。

## 3 根因

- Report 使用独立的 48,000-byte framed-input 常量，没有消费统一运行时配置。
- 内容限制缺少完整 typed default、YAML override、invalid/cross-field validation 与公开投影边界。
- 测试设计复制了同一配置合同：多个 package 和场景反复验证默认数值、default/default+1，并为此构造大 payload 或提交 `input-*.json`。
- `test/scenarios/e2e` 曾包装 Go/Vitest 等代码测试，造成“场景通过”与真实 API/UI 端到端流程混淆。

## 4 修复

- 建立统一 `ContentLimits` typed defaults、YAML override 与 fail-fast validation，并向 HTTP、Resume、TargetJob、Practice、Report 和 provider adapter 注入解析后的值。
- A4 将 Report framed-input 默认值设为 917,504 bytes；A3 独立锁定六个 active profile `max_tokens >= 16384` 与 report 1,000,000-token context window。不使用 bytes + tokens 的跨单位公式宣称两者容量等价，TPM 也不参与单请求裁决。
- RuntimeConfig 只公开五个前端 preflight 字段；前端删除本地默认 fallback，内部 HTTP/report/provider/profile 上限不公开。
- 删除 `backend/internal/review/testdata/report-boundary/input-*.json`，不再构造默认大小或 default+1 材料；仅保留与输入 byte guard 无关的当前 output-schema zh/en fixtures 及 manifest。
- 配置合同只在 A4 typed loader/validator owner 保留一组 default、override、invalid、cross-field 测试；consumer 仅以小型注入值测试 provider call/no-call、错误映射或持久化原子性等非平凡业务分支。
- E2E 不验证配置传播或默认边界；只有真实 frontend/backend API/UI 流程可进入 `test/scenarios/e2e`。

## 5 验证合同

- A4 owner contract：默认值、完整 override、代表性非法值和跨字段约束。
- Report focused guard：用小型注入上限验证 admitted frame 调用 provider、overflow 在 provider/repair 前失败且持久化 `REPORT_CONTEXT_TOO_LARGE`；不重建 62,397-byte 历史输入或任何默认尺寸材料。
- A3 profile contract：typed loader 覆盖 report defaults/override/invalid，canonical coverage lint 要求六个 active profile `max_tokens >= 16384`，并锁定 report context window 为 1,000,000；不发起真实 provider 请求。
- 负向检查：`report-boundary/input-*.json`、package-local defaults、前端本地 fallback 与配置测试型 E2E 引用为零。
- 全量单测由仓库根 `make test` 一次性承接 backend `go test ./...` 与 frontend 全量 tests；focused PASS 只用于开发反馈，不能替代全量回归。
- 真实 E2E 与配置验证分开报告，E2E shell 不调用 `make test`、Go test、Vitest、lint 或 build。
- 当前验证已通过：根 `make test`（backend 全部 package、frontend 125 files / 998 tests、UI prototype 65 tests、Python/Skill 542 tests / 5,122 subtests）、`make lint-config`、active-profile coverage lint、49 个 plan context 与 20 组 BDD 成对审计；未运行真实 E2E。

## 6 经验教训

- 配置业务逻辑的最小充分证据是“一个 typed owner contract + 必要的小值 consumer test”，不是每层复制 default/default+1。
- 大小限制无需真实大文件或默认大小 payload；只有文件格式解析本身需要真实文件时，才由相应 parser owner 保留最小 fixture。
- 全量单测与 E2E 是并列 gate：根 `make test` 负责代码回归，E2E 负责真实 API/UI 用户流程，二者不能互相包装或替代。

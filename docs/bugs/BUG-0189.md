# BUG-0189: repeated practice start conflicted with the existing active session

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-07-18 | 关联 Commit: `fix(practice): recover active sessions on repeated start (BUG-0189)`
> 关联日志: [2026-07-18](../work-journal/2026-07-18.md)

## 1 现象

- 用户启动面试后没有发送回答，直接关闭面试窗口；服务端会话继续保持 `running`。
- 用户之后从正式入口对同一 plan 再次点击“开始面试”，接口返回 `PRACTICE_SESSION_CONFLICT`，无法回到原会话。
- 一个账号如果多个 plan 都曾留下活动会话，每个 plan 的启动入口都会分别遇到同样的问题，看起来像“所有面试都失效”。

## 2 诊断过程

### 假设 1：关闭浏览器会取消服务端面试会话

- 验证方法：查询受影响账号的 `practice_sessions`，并对照关闭窗口前后的 lifecycle 状态。
- 结果：排除。浏览器关闭没有会话 lifecycle 命令，三个近期 plan 各自保留一个 `running` session。

### 假设 2：活动会话唯一索引异常阻止了合法启动

- 验证方法：反查 `migrations/000004_practice_session_active_unique.up.sql`、`ReserveSessionStart` 和真实 PostgreSQL 错误映射。
- 结果：部分确认。唯一索引正确阻止重复活动会话；真正缺口是 fresh idempotency key 的启动路径只尝试创建新 session，没有把同 user/plan 的 queued/running session 作为可恢复结果。

### 假设 3：恢复原会话需要重新生成 opening message

- 验证方法：核对原 session 的 message/event/outbox/audit 事实以及启动响应所需 snapshot。
- 结果：排除。原 session 已有完整 opening 事实，恢复只需返回相同 session snapshot 并完成当前幂等记录，重新调用 AI 会产生重复问题和审计副作用。

## 3 根因分析

- `ReserveSessionStart` 在 fresh idempotency key 下先预留幂等记录，再无条件尝试插入新的 queued session；同 user/plan 已存在 queued/running session 时，活动会话唯一索引冲突被统一映射成 `PRACTICE_SESSION_CONFLICT`。
- 服务层只区分 same-key replay 和 fresh create，没有“恢复已有活动 session”的 reservation 分支（`backend/internal/practice/session_starter.go:157`）。
- Store 缺少 user/plan 范围的并发串行化和 recovery finalization，因此不同启动 key 并发时也不能安全地共享同一活动 session（`backend/internal/store/practice/create_plan.go:267`）。
- 既有测试覆盖 same-key 幂等、唯一索引和新建 opening，但没有覆盖 fresh-key repeated start 对既有 queued/running session 的恢复语义。

## 4 修复方案

- 在 idempotency reservation 后增加 user/plan-scoped transaction advisory lock；在同一事务内先选择现有 queued/running session，仅当不存在活动 session 时才插入新 session，继续保留 partial unique index 作为最终并发护栏。
- 为 reservation 增加 `RecoverSession` 分支：running session 直接恢复；queued session 等待原启动完成后再恢复，context 取消时不调用 AI、不写完成响应。
- 新 recovery finalization 只把既有 running session 的精确 snapshot 写入当前 idempotency record，不新增 session、opening message、AI task、lifecycle event、outbox 或 audit fact。
- 保留 same-key replay、fingerprint mismatch、pending conflict、cross-user/plan isolation 和 fresh create opening 语义。

## 5 验证

- Service focused tests 覆盖 running recovery、queued 等待、context cancel 和零 opening AI 副作用。
- Store 真实 PostgreSQL integration 覆盖 running/queued recovery、different-key 并发、same-key replay/pending/mismatch、cross-user plan 隔离和零重复事实，输出 `active-session-start-recovery=PASS`。
- 根 `make test` 通过 Python 584 tests / 4583 subtests、Go 全包与 frontend 127 files / 1035 tests；`make build`、OpenAPI diff、fixture、docs 与索引 gate 通过。
- 后端重部署后，使用 Chrome 从 Workspace 正式入口启动受影响 plan，页面进入原 `sessionId=019f751a-b64b-7e01-b607-3c99372beff7`，未再出现 `PRACTICE_SESSION_CONFLICT`。
- PostgreSQL 前后对比保持 sessions/messages/events/outbox/audit/AI tasks 为 `1/1/1/1/1/0`，只把该 session 的 succeeded idempotency records 从 1 增加到 2。

## 6 经验教训

- 浏览器窗口生命周期不能充当服务端业务会话生命周期；重复 command 必须根据服务端当前事实提供可恢复语义。
- 唯一索引是最终一致性护栏，不应把“已有合法活动资源”一律暴露成用户冲突；command owner 需要在写入前定义 recover/create 决策。
- 新幂等 key 不等于新的业务事实。恢复已有资源时，只应完成该 key 的精确响应快照，禁止重复 AI 和 lifecycle side effects。
- 本次恢复、并发和零副作用门禁已固化到原 backend-practice owner plan，无需新增通用 Bug pattern。

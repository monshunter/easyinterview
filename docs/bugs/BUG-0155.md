# BUG-0155: change-intake overcounted generic terms and ignored exact scenario owners

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-10 | 关联 Commit: `fix(change-intake): prioritize exact owner evidence (BUG-0155)`
> 关联日志: -

## 1 现象

- Resume Create prototype 的 “Save and open” 问题被 high-confidence 推荐到通用 listing/detail owner，而当前 create-flow owner 仅排第二。
- P0.037 异步 DOM 问题曾被通用 `getResume` / report 关键词误导，场景 README 中的显式 Owner 没有进入 matcher 证据。
- 两次都需要人工读取 active plan 或 scenario README 才能避免原地修订错误 subject。

## 2 诊断过程

### 假设 1：候选 context 缺少 discovery metadata

- 验证方法：比较 Resume Workshop 001/002 context 的 aliases、keywords、packages、routes 和 APIs。
- 结果：部分确认。Create Flow 缺少 prototype Save-and-open / waiting-detail action vocabulary，但已有 create/direct-detail 关键词仍被通用 owner 的重复词压过。

### 假设 2：status tie-break 足以让当前 active owner 胜出

- 验证方法：检查 score 构成并用真实 create-flow 查询做只读评分实验。
- 结果：排除。相同 `resume` / `detail` / `workshop` token 会在同一字段的多个 value 中反复累加，旧 active bonus 仅为 1，无法纠正数十分的虚假差距。

### 假设 3：场景 ID 已被普通 tokenizer 识别为强归属证据

- 验证方法：用 `P0.037 getResume report polling remains loading` 查询 matcher，并检查 reason 列表。
- 结果：排除。旧实现只扫描 plan contexts，不读取匹配场景 README 的 Owner 链接。

## 3 根因分析

- `score_discovery_values` 按 value 累加 partial overlap，同一 query token 在一个字段内可被重复计分，metadata 数量多的通用 owner 获得结构性优势。
- tokenizer 保留 `and/of` 等低信息量词，也不识别 `created/opening/returns` 等常见 action 词形，进一步偏向名称较长的 context。
- matcher 没有把精确 `E2E.P*.NNN` / `P*.NNN` 映射到场景 README Owner，因此最直接的仓库归属证据缺席。
- Create Flow context 未记录本次真实 UI action vocabulary，脚本无法区分 create-to-detail 与通用 detail/listing 问题。

## 4 修复方案

- 将同一 discovery 字段的 partial overlap 改为 unique token 集合计分，并让 exact hit 只提供该字段的最低精确分，不再按重复 value 累加。
- 删除低信息量英文停用词，增加有限的 `ing` / `ed` / plural action 词形变体；status 只保留 3/1 分的有界 tie-break。
- 查询含精确 scenario ID 时，按仓库内 `test/scenarios/**/README.md` 的 Owner 链接给对应 plan 100 分确定性证据，并在 reasons 中输出 `scenarioOwner=E2E.*`。
- 补全 `frontend-resume-workshop/002-create-flow/context.yaml` 的 prototype Save-and-open / waiting-ready discovery 词。
- 更新 change-intake Skill 说明，要求显式 Owner 高于通用 API/route overlap，且 high confidence 与当前 active owner 冲突时仍需 artifact-level 复核。

## 5 验证

- Focused change-intake RED：旧 9 项通过，新增 5 项全部失败；补充 stopword-only edge RED 后为 1 fail / 9 pass。
- GREEN：matcher + Skill contract 15/15，Python compile 和 Skill quick validation 通过。
- 真实 create 查询 high-confidence 命中 `frontend-resume-workshop/002-create-flow`，score 66 对 listing owner 47。
- 真实 P0.037 歧义查询 high-confidence 命中 `frontend-resume-workshop/001-listing-routing-and-detail-readonly`，score 109，首条 reason 为 `scenarioOwner=E2E.P0.037`。
- Product/Create Flow contexts、`git diff --check` 和 pruning surface 通过，`real_residuals=0`。

## 6 经验教训

- Discovery metadata 的 value 数量不能天然转化为置信度；重复词必须在字段边界内去重。
- BUG ID、scenario Owner 等仓库显式边应优先于通用词法相似度，status 只能处理语义接近的候选。
- Skill 脚本的排序规则必须同时有 hermetic fixture 和真实 repo query，避免只证明算法单元而未证明当前 owner 数据。

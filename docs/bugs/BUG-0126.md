# BUG-0126: resume flatten follow-ups exposed display, duplicate, rollback, and lint drift

> 模块: schema | 严重度: medium | 状态: resolved
> 发现日期: 2026-06-15 | 关联 Commit: `fix(review): close resume flatten follow-up gaps (BUG-0126)`
> 关联日志: [2026-06-15](../work-journal/2026-06-15.md)

## 1 现象

- Reviewer 指出 D-20 resume flatten 后续 patch 仍留下 4 个 P2 correctness gap。
- 新 upload / paste resume 创建路径未写入 `display_name`，导致 API/UI 读取新建简历时 `displayName` 为空。
- `duplicateResume` 无法区分省略 `structuredProfile` 与显式传入空对象 `{}`，显式清空草稿会被错误替换为源简历 profile。
- `000015_resume_flatten.down.sql` 一步回滚后重建的 `resume_tailor_runs` 缺少 migration 000008 已加入的 provenance columns，version-14 代码会在 rollback 后读写失败。
- `scripts/lint` pytest suite 仍保留已退役 resume / JD-Match contract 期望，导致 repo lint 测试失败。

## 2 诊断过程

### 假设 1：新建 flat resume 的 display name 只在 API DTO 层丢失

- 验证方法：读取 `backend/internal/resume/store/assets.go` 的 `CreateWithParseJob` insert 与 parse completion update path，并补充 store focused test。
- 结果：确认。insert 只写 `title`，parse completion 不会回填 `display_name`，因此持久化层会返回空 display name。

### 假设 2：duplicate request 缺少 field-presence contract

- 验证方法：读取 handler / service / store 之间的 request type，新增 handler/service/store 对显式 `{}` 的 red test。
- 结果：确认。service 以 `len(in.StructuredProfile) > 0` 判断是否覆盖 profile，导致 `{}` 与 omitted field 被合并为同一种语义。

### 假设 3：rollback table shape 回到了过旧 schema

- 验证方法：读取 `migrations/000015_resume_flatten.down.sql` 与 migration contract test，新增 down migration contract assertion。
- 结果：确认。rollback 重建 `resume_tailor_runs` 时遗漏 `language`、`feature_flag`、`data_source_version`。

### 假设 4：lint pytest fixture 仍绑定退休契约

- 验证方法：运行 `python3 -m pytest scripts/lint -q` 复现，并逐项读取失败的 lint tests。
- 结果：确认。`conventions_yaml_test`、`openapi_diff_test`、`prompt_lint_test`、`rubric_lint_test` 仍期待已退役 sections、operation count 或 JD-Match prompt/rubric surfaces。

## 3 根因分析

- `CreateWithParseJob` 在新增 required API/UI 字段后未把 `display_name` 初始化写入 `resumes` insert path。
- `DuplicateResumeRequest` 没有携带 `structuredProfile` 字段存在标记，service 只能用 map 长度推断用户意图。
- `000015` down migration 手工重建 table 时复制了更早基线结构，没有以 post-000014 schema 作为 rollback truth。
- Retired contract 收窄只更新了 source inventory，未同步 scripts/lint pytest suite 的 fixtures 和负向断言。
- 涉及路径：
  - `backend/internal/resume/store/assets.go`
  - `backend/internal/resume/service.go`
  - `backend/internal/resume/handler/duplicate.go`
  - `migrations/000015_resume_flatten.down.sql`
  - `scripts/lint/*`

## 4 修复方案

- 在 `CreateWithParseJob` insert 中把 `display_name` 初始化为 title，保证 flat resume 创建后 API/UI 有稳定名称。
- 为 duplicate request 增加 `StructuredProfileSet` presence flag，并从 handler 传递到 service / store；显式 `{}` 持久化为空对象，省略字段才复制源 profile。
- 把 `000015` down migration 的 `resume_tailor_runs` 重建 schema 修正为 post-000014 shape，恢复 `language`、`feature_flag`、`data_source_version`。
- 更新 scripts/lint pytest fixtures 到当前 5.1-5.13 conventions 和 43-operation OpenAPI inventory，并对 retired JD-Match prompt/rubric keys 增加负向断言。
- 考虑过只在 API/UI 层做 fallback display name，但放弃：该方案不能修复持久化数据形态，也不能防止后续消费者读到空 display name。

## 6 经验教训

- 新增 required response field 时，必须同时检查 create、parse completion、duplicate/copy、PATCH 和 list/detail readback path。
- JSON object fields 需要区分 omitted 与 explicit empty object 时，handler 应传递 field-presence flag，不能用 map length 代表用户意图。
- Down migration 手工重建 table 时，应以“回滚到的目标版本”schema 为准，而不是复制最早 baseline。
- Retired contract 删除应同时更新 positive fixture、operation inventory 和 negative lint expectations；本次已用 tests 固化，无需新增通用 `PATTERNS.md` 条目。

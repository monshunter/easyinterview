# BUG-0002: secrets-config completed plan missed runtime binding drift

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-04-30 | 关联 Commit: `fix(secrets-config): remediate L2 review findings`
> 关联日志: `-`

## 1 现象

- `/plan-code-review secrets-and-config/001-bootstrap --fix` 的 L2 review 发现已完成 plan 中仍有 4 个实现漂移点。
- `cmd/worker` 在 prod + 完整 env 注入时仍无法通过 `loader.Validate()`，因为 worker 未加载 canonical secret bindings。
- `validator.go` 只校验 `AI_GATEWAY_API_KEY`，未校验 `AI_GATEWAY_BASE_URL`，导致缺 gateway URL 的 prod API dump-config 可通过。
- `make lint-config` 输出 code-side keys 数量为 1 仍成功，说明 env dictionary gate 没有覆盖 binding map 字面量。
- PostHog-backed runtime-config 首次请求依赖 cold `Snapshot()`，导致 public flag 可能为空；同时 `cmd/api` 未把 `feature-flags.yaml` 的 public allowlist 传给 PostHog provider。

## 2 诊断过程

### 假设 1：worker 没有读取 secret env

- 验证方法：在 prod 环境注入 dummy `SESSION_COOKIE_SECRET` / `AI_GATEWAY_API_KEY` / `POSTHOG_PROJECT_API_KEY` 等 env 后运行 worker config 路径。
- 结果：确认。worker 仅绑定 `WORKER_LISTEN_ADDR`，没有 `SecretBindings`，因此 validator 仍报告多项 missing required secret。

### 假设 2：AI gateway base URL 缺失未进入 validator

- 验证方法：注入完整 prod dummy secret 但省略 `AI_GATEWAY_BASE_URL`，运行 API config dump 和 focused validator test。
- 结果：确认。缺 base URL 仍可通过 validation。

### 假设 3：env_dict 只扫描直接 `os.Getenv`

- 验证方法：构造 Go `EnvBindings` 字面量声明 `DATABASE_URL`，但在 spec / `.env.example` 中省略该 key，运行 `env_dict_test.py`。
- 结果：确认。修复前脚本报告 code-side keys 为 0/1 且测试失败。

### 假设 4：runtime-config 首次请求没有触发 PostHog evaluation

- 验证方法：用 `httptest` mock `/decide` 返回 public 与 operator-only flag，直接调用 `BuildRuntimeConfig`。
- 结果：确认。修复前 cold snapshot 为空，public flag 未投影。

## 3 根因分析

- `cmd/api` 与 `cmd/worker` 各自手写 binding map，worker 路径遗漏了 secret bindings；没有单一 code-side env dictionary 真理源。
- `validator.go` 将 AI gateway 校验拆成 API key secret 检查，遗漏了非 secret 的 base URL required check。
- `env_dict.py` 只用正则提取 `os.Getenv` / `os.LookupEnv` 调用，没有把 `EnvBindings` / `SecretBindings` 字面量纳入 code-side declaration。
- `FeatureFlagClient` 把 runtime-config projection 需要的 `Snapshot()` 混入业务接口，但 PostHog 的 no-arg snapshot 只返回 last-known-good，无法在 cold start 按当前 `FlagContext` evaluate。
- `cmd/api` 构造 PostHog provider 时没有读取本地 feature flag baseline 中的 `public` visibility metadata。

## 4 修复方案

- 新增 `config.LoadCanonical`、`DefaultEnvBindings()`、`DefaultSecretBindings()`，让 API / worker 复用同一 canonical env dictionary。
- `cmd/worker` 改为通过 `loadWorkerConfig` 使用 canonical loader；新增 worker prod focused test。
- `validator.go` 增加 `AI_GATEWAY_BASE_URL` required value check；新增 missing-base focused test。
- `env_dict.py` 增加 env-shaped Go string literal 扫描，覆盖 binding map；新增 pytest regression。
- 将业务 `FeatureFlagClient` 收回为 D-4 的 `IsEnabled` / `Variant` 两方法，新增独立 `SnapshotProvider` 供 runtime-config 使用。
- `PostHogFlagProvider.Snapshot(ctx)` cold call 触发 `/decide` evaluation；`cmd/api` 从 `feature-flags.yaml` 加载 public allowlist 后传入 PostHog provider。

## 6 经验教训

- 完成态 checklist 的 L2 review 要重点检查「verification 说通过」与「真实代码路径」是否一致，尤其是 entrypoint / worker / tooling gate 这类非业务路径。
- 对 env/config 这类跨 entrypoint 契约，应优先沉淀单一 code-side declaration，避免每个 command 手写 binding map。
- lint 脚本的正向成功输出必须能说明覆盖面；code-side keys 数量异常小应作为 review 信号。

# BUG-0037: jd_match L2 review exposed detail fetch and auth resume drift

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-10 | 关联 Commit: `fix(frontend-jd-match): remediate jd match L2 findings`
> 关联日志: `-`

## 1 现象

- 触发条件 / 发现方式：`/plan-code-review 002-jd-match-recommendations --fix` 对已完成 plan 做 L2 semantic reconcile 时发现当前实现与 plan/spec 不闭环。
- 可观察到的错误表现：Recommended tab 选中卡片后未以 `getJobRecommendation(jobMatchId)` 作为详情主数据源，`JDDetail` 可能只复用列表摘要；未登录用户完成登录后，`jd_match_action` pending action 只携带 route params 但没有实际一次性恢复执行。
- 影响范围：影响 `jd_match` Recommended 详情可信度、Save/Unsave/Dismiss/Confirm interview 登录恢复，以及 Search tab 未登录 Run/Create saved search 的 opaque payload 隐私闭环。

## 2 诊断过程

### 假设 1：详情接口已由列表加载间接覆盖

- 验证方法：新增并运行 `JDMatchDetailFetch.test.tsx` 与 `useJobRecommendation.test.tsx`，断言选中卡片必须调用 generated `getJobRecommendation(jobMatchId)`，并用详情专属标题证明不是列表摘要复用。
- 结果：确认。红测暴露现有实现缺少 `useJobRecommendation` hook 与 `JDMatchScreen` 详情查询接线。

### 假设 2：pendingAction 已能由 `auth_login` 自动恢复

- 验证方法：新增并运行 `JDMatchAutoResume.test.tsx`，覆盖 Recommended 的 `save` / `confirm_interview` 以及 Search 的 `run_search` / `create_saved_search` 登录后一次性自动恢复。
- 结果：确认。原实现只生成 pending action params，没有消费 params 并重新触发 action 的恢复逻辑。

### 假设 3：BDD/scenario gate 已覆盖上述语义

- 验证方法：审查并补强 P0.027/P0.028/P0.030/P0.031 scenario `trigger.sh` / `verify.sh`，补充 detail fetch、auth auto-resume、opaque Search payload、P0.015/P0.016 parse regression、pixel parity 新 gate 断言。
- 结果：确认。旧 gate 主要覆盖历史结构与 PASS marker，未强制证明这两个关键语义。

## 3 根因分析

- 根本原因：plan 002 首次完成时的实现和场景门禁更偏向 DOM/列表/按钮结构闭环，没有把“详情必须来自 detail operation”和“pending action 必须登录后自动执行”固化为不可绕过的红测与 scenario verify。
- 涉及的代码路径：`frontend/src/app/screens/jd_match/JDMatchScreen.tsx`、`RecommendedTab.tsx`、`SearchTab.tsx`、新增 `useJobRecommendation.ts`、新增 `pendingJdMatchActionState.ts`，以及 P0.022/P0.023/P0.025/P0.026 scenario scripts。
- 为什么之前没有发现：历史 checklist 和 scenario PASS 被当作线索，但没有包含 detail operation 调用计数、详情专属数据断言、auth auto-resume 一次性执行断言和 opaque payload 负向断言。

## 4 修复方案

- 选择的方案及理由：在原 plan 002 内原地 remediation，新增 `useJobRecommendation(selectedId)` 并让 `JDDetail` 优先使用 detail response；新增 SPA 内存态 `pendingJdMatchActionState` 保存 Search query/label，仅在 route params 传 `pendingJdMatchActionId`；`JDMatchScreen` 从 route params 一次性恢复 Recommended/Search action。
- 考虑过但放弃的替代方案：不新增 detail hook、继续用 list summary 渲染会违反 plan 的 operation matrix；把 Search query/label 放入 route params 能简化恢复但违反隐私红线。
- 修改的文件列表：`frontend/src/app/screens/jd_match/*` 相关组件、hook 与测试，`frontend/tests/pixel-parity/jd_match.spec.ts`，P0.022/P0.023/P0.025/P0.026 scenario scripts，plan 002 的 spec/plan/checklist/BDD 文档和索引。

## 6 经验教训

- L2 review 的 PASS 证据必须证明语义不变量，而不是只证明 DOM 结构或历史场景仍能运行。
- 对于“列表 + 详情”的 UI，详情视图必须有专门的 operation 调用计数和详情专属字段断言。
- 对于 auth pending action，gate 必须同时覆盖 pending params 最小化、登录后一次性恢复执行、重复触发防护和隐私字段负向搜索。

# BUG-0010: mock contract runtime gate missed registry and stale route count

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-05 | 关联 Commit: `fix(mock-contract): harden runtime drift gates`
> 关联日志: `-`

## 1 现象

- `/plan-code-review mock-contract-suite --fix` L2 review 发现 `lint-mock-contract` 只执行 fixture coverage、OpenAPI inventory 和 mock boundary lint，没有执行 `scripts.mock_contract.fixture_registry_test`。
- `scripts/mock_contract/fixture_registry.py` 已是 Phase 1 operation registry 的关键 artifact，但 `context.yaml` discovery packages 未包含 `scripts/mock_contract`，后续 review / issue intake 容易漏扫。
- Go generated server 注释仍写旧 `36-row table`，而当前 B2 / mock contract truth source 已收敛为 12 tag / 34 operation。
- 前端 mock transport 已支持 `Prefer: example=<scenario>` 和 unknown scenario fail-loudly，但 focused Vitest 只覆盖 default response。

## 2 诊断过程

### 假设 1：`lint-mock-contract` 已覆盖 operation registry metadata test

- 验证方法：读取 `Makefile` 的 `lint-mock-contract` target，并运行 `make -n lint-mock-contract`。
- 结果：排除。target 只调用 `validate-fixtures`、`lint-openapi` 和 `mock_runtime_boundary.py`，没有执行 `scripts.mock_contract.fixture_registry_test`。

### 假设 2：旧 `36-row` 只存在历史文档，不影响 generated artifact

- 验证方法：搜索 `36-row` / `36 operations`，并读取 `openapi/templates/go/server.tmpl` 与 `backend/internal/api/generated/server.gen.go`。
- 结果：排除。模板和生成物都保留旧注释，当前 generated server artifact 与 34 operation truth source 文案不一致。

### 假设 3：前端 named scenario / unknown scenario 行为已有回归测试

- 验证方法：读取 `frontend/src/api/mockTransport.test.ts`。
- 结果：排除。测试只覆盖 default typed response；named scenario selection 和 unknown scenario fail-loudly 没有直接断言。

## 3 根因分析

- 根本原因：Phase 4 gate 只把 fixture coverage、OpenAPI inventory、prototype import boundary 和 retired token negative search 接入本地 lint，遗漏了 Phase 1 新增的 registry metadata unit test；生成模板注释也没有从 operation inventory 动态派生数量。
- 涉及的代码和文档路径：
  - `Makefile`
  - `scripts/mock_contract/fixture_registry.py`
  - `scripts/mock_contract/fixture_registry_test.py`
  - `openapi/templates/go/server.tmpl`
  - `backend/internal/api/generated/server.gen.go`
  - `frontend/src/api/mockTransport.test.ts`
  - `docs/spec/mock-contract-suite/plans/001-fixture-backed-mock-runtime/context.yaml`
- 之前没有发现，是因为已有 gates 能证明 34 fixtures / OpenAPI inventory / boundary lint 通过，但不能证明 registry metadata test 被聚合执行，也没有测试生成注释中的 operation count。

## 4 修复方案

- 将 `scripts.mock_contract.fixture_registry_test` 接入 `make lint-mock-contract`，并用 `scripts/lint/makefile_dry_run_test.py` 锁定 target wiring。
- 将 Go server 模板中的 route table 数量改为 `{{len .Operations}}` 动态渲染，并新增 codegen test 断言 generated server 使用 `34-row` 且不保留 `36-row`。
- 扩展前端 mock transport test，覆盖 `Prefer: example=authenticated` named scenario 和 unknown scenario fail-loudly。
- 将 `scripts/mock_contract` 加入 mock-contract plan context discovery packages，避免后续 owner review 漏掉 registry artifact。
- 原地修订 mock-contract 001 plan / checklist，新增并完成 Phase 4.3 L2 remediation 项。

## 5 验证

- Red：`python3 -m unittest scripts.lint.makefile_dry_run_test.MakefileDryRunTest.test_lint_wires_mock_contract_gate -v` 修复前失败，提示缺少 `scripts.mock_contract.fixture_registry_test`。
- Red：`cd backend && go test ./cmd/codegen/openapi -run TestRun_ApiErrorInnerObjectAndResponseEnvelope -count=1` 修复前失败，提示生成物缺少 `34-row`。
- Existing behavior note：`pnpm --filter @easyinterview/frontend test src/api/mockTransport.test.ts` 在新增 named scenario / unknown scenario 测试后已通过，说明前端行为存在但此前缺少回归断言。
- Focused：`python3 -m unittest scripts.mock_contract.fixture_registry_test -v`、`cd backend && go test ./cmd/codegen/openapi ./internal/api/mockruntime -count=1`、`pnpm --filter @easyinterview/frontend test src/api/mockTransport.test.ts` 均通过。
- Contract gates：`make lint-mock-contract`、context validation、`make docs-check`、`git diff --check` 均通过。
- Codegen drift：`make codegen-check` 在当前未提交补丁下会停在预期 generated diff；使用临时 Git index 将当前补丁作为基线后，`make codegen-check` 通过，证明 codegen 幂等。

## 6 经验教训

- 新增 owner-owned helper / registry 后，聚合 gate 必须直接执行对应 focused test；只执行下游 lint 不足以证明 helper artifact 被持续验证。
- generated artifact 中的 inventory 数量不应手写，必须由 codegen data 动态渲染或完全删除。
- already-working behavior 也需要 focused regression test，否则 L2 review 只能从实现推断契约，无法把契约固定为 future gate。

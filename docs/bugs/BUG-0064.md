# BUG-0064: report replay handoff reused source sessions and pixel gate was false-green

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-05-16 | 关联 Commit: `fix(frontend-report): harden replay handoff and pixel gate (BUG-0064)`
> 关联日志: [2026-05-16](../work-journal/2026-05-16.md)

## 1 现象

- L2 review 发现 ReportDashboard 的「复练当前轮」/「进入下一轮」CTA 直接导航到 `practice`，只携带 `sourceSessionId` / replay params，没有新 `sessionId`。
- 已登录用户从 report CTA 进入 practice 时，`InterviewContext` 会保留来源 session，导致 practice 重新加载已完成的 source session；未登录 pending action 恢复后则可能进入缺 session 状态。
- `E2E.P0.059` 场景只运行 i18n / legacy negative gate，verify 只确认 Playwright spec 文件存在，不能证明 pixel parity 真正执行。

## 2 诊断过程

### 假设 1：Report CTA 直接进入 practice 会复用旧 session

- 验证方法：读取 `useReplayCtaHandlers.ts`、`InterviewContext.tsx` 与 `PracticeScreen.tsx` 的 route/context 合并逻辑，并把 `ReplayCta.test.tsx` 改为期待 workspace auto-start + `startPracticeSession`。
- 结果：确认。红测显示点击 CTA 后 `startPracticeSession` 调用次数为 0，DOM 落到 `practice-session-lost`。

### 假设 2：现有 workspace auto-start 已能承接 report replay payload

- 验证方法：读取 `WorkspaceScreen.navigateToPractice` 与 `useStartPractice`，确认 route params 会保留 replay / next-round fields，并由 workspace owner 创建 practice plan/session。
- 结果：确认。只需 report CTA 导航到 `workspace` 并携带 `autoStartPractice=1`，即可复用现有 owner 边界。

### 假设 3：P0.059 场景没有真实执行 Playwright

- 验证方法：读取 `p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/{trigger,verify}.sh`。
- 结果：确认。trigger 未调用 `test:pixel-parity`，verify 仅检查 `generating.spec.ts` / `report.spec.ts` 文件存在。

## 3 根因分析

- Report owner 把 replay / next-round handoff 建模成 direct `practice` route params，但 practice route 需要一个 fresh sessionId；当 route 缺新 sessionId 时，context sync 会保留旧 `ctx.sessionId`。
- Pending action 仍恢复到 direct practice route，绕过了 workspace 既有的 `autoStartPractice` session creation owner。
- P0.059 的 scenario wrapper 把 Playwright 作为人工/外部执行项，只把 spec 文件存在性当成自动化证据，导致 false-green。

涉及代码路径：

- `frontend/src/app/screens/report/useReplayCtaHandlers.ts`
- `frontend/src/app/auth/__tests__/pendingActionReplayPractice.test.ts`
- `frontend/src/app/screens/report/__tests__/ReplayCta.test.tsx`
- `test/scenarios/e2e/p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/trigger.sh`
- `test/scenarios/e2e/p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/verify.sh`

## 4 修复方案

- Report replay / next-round CTA 改为导航到 `workspace`，携带原 replay payload 与 `autoStartPractice=1`；未登录 pending action 也恢复到 `workspace`。
- 更新 CTA 测试，断言登录态会经 workspace owner 调用 `startPracticeSession`，未登录态只进入 auth pending action，不直接 start。
- P0.059 trigger 增加 frontend build 与 Playwright `generating.spec.ts` / `report.spec.ts` 执行；verify 检查 Playwright run marker、两个 spec path 和 passed marker。
- 修正 P0.059 setup metadata，并同步 frontend-report-dashboard spec / plan / BDD / test docs 到 v1.1。

## 5 验证

- `pnpm --filter @easyinterview/frontend test src/app/auth/__tests__/pendingActionReplayPractice.test.ts src/app/screens/report/__tests__/ReplayCta.test.tsx`
- `E2E.P0.057` setup → trigger → verify → cleanup
- `bash test/scenarios/e2e/p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/setup.sh && bash test/scenarios/e2e/p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/trigger.sh && bash test/scenarios/e2e/p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/verify.sh && bash test/scenarios/e2e/p0-059-report-pixel-parity-i18n-and-legacy-negative/scripts/cleanup.sh`
- P0.059 trigger 内执行 `pnpm --filter @easyinterview/frontend build` 与 Playwright 14 tests，结果 `14 passed`.

## 6 经验教训

- Report replay handoff 必须验证 fresh session creation，而不是只验证路由离开 report。
- Scenario verify 不能把 spec 文件存在性当成执行证据；必须检查 trigger log 中的实际 runner marker 与 pass marker。

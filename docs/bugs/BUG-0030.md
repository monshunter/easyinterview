# BUG-0030: prompt registry L2 review exposed provenance and no-op gate drift

> 模块: eval | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-09 | 关联 Commit: `fix(prompt-rubric-registry): remediate provenance L2 findings`
> 关联日志: `-`

## 1 现象

- `/plan-code-review prompt-rubric-registry --fix` 的 L2 review 发现 `GenerationProvenance.modelId` 写入 registry `ModelProfileName`，而计划 §8.1.2.4 明确要求它来自 A3 resolved `AICallMeta.ModelID`。
- `ParseExecutor` 传给 A3 的 `CallMetadata` 缺少 `FeatureFlag`，且没有填充 B4 `TaskRun` context，导致 `ai_task_runs.feature_flag` 无法证明来自 registry resolution，真实 writer 路径也没有被 targetjob 测试覆盖。
- checklist 4.8 记录的 `go test ./backend/internal/targetjob -run TestParseExecutorAITaskRuns -race` 当时实际返回 `[no tests to run]`，但 checklist 仍记录为全绿。
- `openapi/fixtures/TargetJobs/getTargetJob.json` 的 TargetJobs provenance 示例沿用 `model-profile:target.import.default`，与 `modelId` 表示实际模型 id 的 contract 不一致；fixture validator 也只允许 `model-profile:*`。
- checklist / plan 对 4.7 integration gate 和 5.2 `make migrate-check` 的描述过度乐观：当前 repo 中 4.7 是 static SQL parse gate，且本地未配置 `DATABASE_URL` 时 live DB migrate-check 不能记为绿色 DB gate。

## 2 诊断过程

### 假设 1：ParseExecutor 丢弃了 A3 returned meta

- 验证方法：读取 `backend/internal/targetjob/parse_executor.go`，对照 plan §8.1.2.4 的 `modelId` 字段来源说明，并补 `TestParseExecutor_HappyPath` provenance 断言。
- 结果：确认。旧实现调用 `p.ai.Complete` 时丢弃第二返回值，并把 `resolution.ModelProfileName` 写入 summary / fitSummary provenance。

### 假设 2：targetjob 没有把 F3 feature flag 和 B4 task context 传进 A3

- 验证方法：读取 `aiclient.CallMetadata` / `AITaskRunContext` / observability writer，并补 `TestParseExecutorAITaskRuns` 通过 observability wrapper 写入 in-memory `AITaskRunWriter`。
- 结果：确认。旧 metadata block 只有 feature key、prompt/rubric version、language、data source version；`FeatureFlag` 和 `TaskRun` 都为空。

### 假设 3：fixture validator 把旧 profile id 当成 `modelId` 唯一合法形态

- 验证方法：补 `validate_fixtures_test.py::test_validator_accepts_provider_neutral_fixture_model_id`，输入 `fixture-model:target-import-parse`。
- 结果：确认。旧 validator 报错要求 `model-profile:<name>`，不能表达 provider-neutral actual model id。

### 假设 4：completed checklist 记录了不可执行或环境未满足的 gate

- 验证方法：实际运行 `go test ./backend/internal/targetjob -run TestParseExecutorAITaskRuns -race`、读取 `db_integration_test.go`，并运行 `make migrate-check`。
- 结果：确认。原 `TestParseExecutorAITaskRuns` 不存在；`db_integration_test.go` 是 static SQL parse；`make migrate-check` 在本地只跑到 migration lint 后因缺 `DATABASE_URL` 失败。

## 3 根因分析

- 根本原因是 Phase 3/4 的 cross-layer gate 只验证了 shape 和局部 decorator writer，没有把 targetjob ParseExecutor 的真实 A3 meta 返回值、B4 `TaskRun` context、OpenAPI fixture validator 与 checklist 命令真实性放在同一个回归面里。
- 涉及路径：
  - `backend/internal/targetjob/parse_executor.go`
  - `backend/internal/targetjob/pipeline_test.go`
  - `backend/internal/targetjob/parse_executor_cross_layer_test.go`
  - `backend/cmd/api/targetjob_http_scenario_test.go`
  - `openapi/fixtures/TargetJobs/getTargetJob.json`
  - `scripts/lint/validate_fixtures.py`
  - `docs/spec/prompt-rubric-registry/plans/001-baseline/{plan.md,checklist.md}`
- 之前没有发现，是因为历史 checklist 把不存在的 focused test 和缺少 `DATABASE_URL` 的 live DB gate 记成绿，同时 fixture validator 的旧命名规则掩盖了 `modelId` 与 `model_profile_name` 的层次差异。

## 4 修复方案

- `ParseExecutor` 捕获 `AIClient.Complete` 返回的 `AICallMeta`，用 `ModelID` 写入 summary / fitSummary `GenerationProvenance.modelId`。
- `ParseExecutor` 的 `CallMetadata` 显式携带 `FeatureFlag=coalesceFlag(resolution.FeatureFlag)`，并填充 B4 `TaskRun` context（`jd_parse` / `target_job` / `targetJobID`）。
- 新增/增强 targetjob tests：`TestParseExecutor_HappyPath` 断言 provenance model id 与 feature flag；`TestParseExecutorMetadataCarriesF3Triple` 断言 metadata task context；`TestParseExecutorAITaskRuns` 通过 observability wrapper 验证真实 writer row。
- `cmd/api` invalid-output scenario AI stub 返回合法 A3 meta，确保该测试继续覆盖 invalid output，而不是 metadata 缺失。
- fixture validator 接受 `model-profile:*` 和 `fixture-model:*` 两类 provider-neutral id，继续拒绝 vendor/model token；TargetJobs fixture 改用 `fixture-model:target-import-parse`。
- 原 plan/checklist 升到 v1.2，追加 Phase 6 remediation，修订 4.7 / 4.8 / 5.2 gate evidence，并恢复 `completed` lifecycle。

## 6 经验教训

- L2 remediation 不能只断言 provenance shape；当字段语义跨层时，测试必须同时覆盖 source-of-truth field、actual returned meta、wire fixture 示例和 validator contract。
- checklist 中的 `go test -run <Name>` 必须确认至少一个测试被执行；`[no tests to run]` 不能作为绿色证据。
- live DB gate 必须区分 static migration lint 与需要 `DATABASE_URL` 的真实 migration run；缺环境变量时只能记录 blocker，不能写成全绿。

# BUG-0161: practice timing and report progression ignored structured round context

> 模块: frontend | 严重度: high | 状态: resolved
> 发现日期: 2026-07-12 | 关联 Commit: `fix(frontend): align round timing and progression (BUG-0161)`
> 关联日志: `-`

## 1 现象

- 面试规划显示当前轮为 60 分钟，但 Practice Top Bar 的总时长始终显示 `25:00`。
- 在报告页点击“进入下一轮”后，创建的新练习仍回到第一轮，而不是 TargetJob 结构化面试轮次中的紧邻后一轮。
- 末轮、未知轮次、重复派生 ID、TargetJob 加载/失败以及重复点击没有统一的 fail-closed 与幂等 UI 边界。
- 影响：用户无法信任练习时间，也无法沿面试规划推进核心模拟面试流程；下一轮没有可用绕过方式。

## 2 诊断过程

### 假设 1：后端 PracticePlan 没有预算字段

- 验证方法：检查 OpenAPI、generated client、PracticePlan fixture 和 create/get operation。
- 结果：排除。`CreatePracticePlanRequest.timeBudgetMinutes` 与 `PracticePlan.timeBudgetMinutes` 已存在；问题是前端创建路径写入固定预算，Practice 页面也没有读取当前 plan。

### 假设 2：报告页可以只根据当前 roundId 推导下一轮

- 验证方法：反查 `handoff.ts`、Report TargetJob 加载和 structured round mapper。
- 结果：排除。旧实现维护固定 `ROUND_ORDER` 与默认下一轮回退，未读取当前 TargetJob 的实际轮次序列，因此无法正确处理不同轮数、类型和末轮边界。

### 假设 3：重复点击由后端 idempotency 自动消除即可

- 验证方法：让 create plan 请求保持 in-flight 并连续点击 CTA，记录调用次数。
- 结果：排除。两个前端调用会各自生成 idempotency batch，旧实现会发起两次 plan/session 流程；需要 CTA 层同步锁。

## 3 根因分析

- `BUG-0148` 已把轮次建模为 `TargetJob.summary.interviewRounds[]`，但后续运行时消费者仍各自保留固定值或固定顺序：plan 创建默认 30 分钟、Practice Top Bar 固定 25 分钟、Report 下一轮使用固定 ladder。
- 共享 start-practice 只验证 target/resume，复用旧 plan 时没有校验 round duration snapshot；Practice 页面没有把 `planId` 解析为当前 plan budget。
- 报告页没有以 TargetJob round load state 作为下一轮可用性边界，也没有同步 in-flight guard。
- 既有测试主要覆盖“能创建 session/能渲染 CTA”，没有覆盖任意正整数预算、预算不匹配旧 plan、末轮/未知/重复 ID 和双击并发。

## 4 修复方案

- 新增共享 `resolveTargetJobRoundContext`，以 ordered structured rounds 同时解析当前轮与紧邻后一轮；重复 ID、未知轮次和缺失数据统一 fail closed。
- shared start-practice 先读取 TargetJob，使用当前轮 `durationMinutes` 创建 PracticePlan；只有 target/resume/time budget 全部匹配时才复用现有 baseline plan。
- PracticeScreen 读取当前 PracticePlan，将任意正整数 `timeBudgetMinutes` 格式化为 Top Bar 总时长；读取缺失/失败显示 `--:--`，不伪造默认预算。
- Report 下一轮 payload 只接受 resolver 返回的 immediate successor；末轮/未知/loading/failure 时禁用 CTA。
- 在 replay/next-round 共用 handler 增加同步 in-flight 锁，任一启动过程禁用两个 CTA，重复点击最多创建一个 plan/session。
- 同步 `ui-design/`、owner spec/plan/checklist、P0.021/P0.045/P0.057 场景和正式 README，删除固定下一轮旧口径。

## 6 经验教训

- 结构化业务数据完成建模后，必须建立 producer → shared resolver → persistence snapshot → UI/report consumer 的完整矩阵；只修展示 mapper 仍会留下运行时断链。
- 时间预算属于 PracticePlan 的创建时快照，显示层应读取持久化计划，不应再次从 UI 文案或固定常量推断。
- “下一轮”是有边界的 successor 操作；末轮、未知/重复 identity 和 load failure 都必须显式 fail closed，不能回退到第一轮或当前轮。
- UI idempotency 不能只依赖每次调用新生成的后端 idempotency key；用户重复点击需要同步前端锁。

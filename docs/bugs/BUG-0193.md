# BUG-0193: custom theme controls overlaid the preset selector

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-19 | 关联 Commit: `fix(settings): preserve custom theme hierarchy (BUG-0193)`
> 关联日志: [2026-07-19](../work-journal/2026-07-19.md)

## 1 现象

- 在 Settings Appearance 选择“自定义”后，Ocean / Plum / Custom 一级主题按钮从可见界面中消失，用户无法回退到预定义主题。
- hue/chroma 面板看起来替换了一级选择器，而不是在一级选择器下方按需展开。
- 原生灰色 range 轨道不能直观表达完整色相范围或当前 hue 下的彩度方向。

## 2 诊断过程

### 假设 1：选择 Custom 时组件状态删除了一级主题按钮

- 验证方法：读取 `SettingsScreen` 的条件渲染和 Ocean / Plum click handler，并用 component DOM 检查 Custom 激活后的节点。
- 结果：排除。一级按钮始终存在，选择预定义主题也会正确执行 `setCustomAccent(null)`；状态机具备回退能力。

### 假设 2：Appearance 横向构图把一级和二级内容放进同一布局区域

- 验证方法：检查 page-scoped CSS，并在 Chrome 对 theme options、custom panel 与 editor 读取 bounding box。
- 结果：确认。`.ei-settings-theme-options`、`.ei-settings-custom-accent` 和错误提示同时被分配到 `grid-column: 2; grid-row: 1 / span 2`，Custom 面板覆盖了一级按钮。

### 假设 3：只移除 custom panel 的 grid row 即可稳定修复

- 验证方法：对照 desktop/mobile 阅读顺序、错误提示和 Save 操作列，评估无共同 parent 时的自动网格放置。
- 结果：排除。一级、二级和错误状态需要一个明确的 theme editor owner 才能保证固定顺序；继续依赖兄弟节点自动放置会保留后续回归空间。

## 3 根因分析

- Settings 截图构图重排时，把互斥假设错误地应用于实际会同时存在的一级选择器与条件二级编辑器，导致两个兄弟节点共享同一 grid area。
- 既有 component tests 证明节点存在和主题请求预算，但 jsdom 不执行真实布局，无法发现“DOM 存在但被覆盖”的可操作性缺陷。
- Phase 18 的 Chrome 验收覆盖 Settings 总体构图，但没有固定 Custom 展开态的一级/二级 bbox 关系，因此 collapsed/default 状态的截图无法暴露问题。

## 4 修复方案

- 在 `SettingsScreen` 新增常驻 `.ei-settings-theme-editor` 分组，由它依次拥有一级主题选项、条件 Custom 面板和保存错误。
- Appearance grid 只放置 theme editor parent；parent 内部使用正常文档流纵向堆叠，Save 继续留在独立操作列。
- hue range 使用完整光谱渐变；chroma range 通过 CSS custom property 跟随当前 hue，从低彩中性色渐变到高彩色，并保留原生 range 的键盘、focus 和值域语义。
- 原地修订 frontend-shell spec/plan/checklist/BDD 与 UI design，固定一级常驻、二级按需展开和预定义主题可逆回退合同。

## 6 经验教训

- 条件 UI 的“节点存在”不等于“用户可操作”；涉及 grid/flex overlap 的回归必须在展开态检查真实 bbox、层级和点击路径。
- 一级选择与二级编辑应由明确 parent owner 按正常文档流组织，不能把多个同时存在的兄弟节点分配到同一 grid area。
- 色彩 range 的轨道本身应表达调节维度；不增加数值或 preview 面板，也能通过完整 hue 与 hue-aware chroma 渐变提升可理解性。

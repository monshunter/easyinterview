# BUG-0182: report retries repeatedly selected a terminal assistant message as evidence

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-07-16 | 关联 Commit: `fix(report): harden regeneration and local diagnostics (BUG-0182, BUG-0183)`
> 关联日志: -

## 1 现象

- 同一场面试已经结束，但初次生成、人工重新生成和后续自动重试都可能稳定失败；另一轮报告使用相同 provider/profile 却能成功。
- Provider HTTP 调用和 JSON schema 校验均成功，最终仍被报告语义校验拒绝；旧界面只显示失败，用户无法从同一报告恢复或查看面试记录。
- 影响报告生成主链路，且简单增加重试次数不能绕过，因此阻塞核心复盘功能。

## 2 诊断过程

### 假设 1：更换 AI client 后出现接口或输出 schema 不兼容

- 验证方法：检查 DEBUG raw request/response、provider finish metadata、schema validation status 与相邻成功报告。
- 结果：排除。失败调用均完成 provider/schema 层，成功报告证明同一 client/profile 可以生成合法报告。

### 假设 2：输入被截断、损坏，或上一轮模型输出污染后续 prompt

- 验证方法：比较连续失败请求的结构、长度与 hash，核对消息角色、冻结 transcript 和每次修复请求组成。
- 结果：排除。请求只包含 system + 当前 user payload，不追加上一轮模型输出；冻结 transcript 完整，连续请求结构稳定，也没有控制标记进入系统指令。

### 假设 3：语义修复提示不能纠正 evidence anchor

- 验证方法：把语义 validator 坐标与 transcript 角色逐项对应，比较合法 user 序号和模型反复选择的 anchor。
- 结果：确认。合法候选是 user 序号 `2/4/6`，末尾 `7` 是尚未回答的 assistant 消息；模型连续把 `7` 当证据，validator 均返回 `not_user_message`。旧修复只提供内部 path/code，没有允许序号和可执行意图，因此重试稳定复现同一错误。

## 3 根因分析

- 冻结 transcript 以 assistant 终止，末尾问题在语义上很显眼，但不能作为候选人回答证据；报告 prompt 没有明确列出可信 user anchor。
- `backend/internal/review/generate_report.go` 的旧修复信息只传模糊 validator 坐标，没有说明“证据必须引用哪些 user 消息、当前字段应删除还是重写、跨字段约束是什么”。
- 自动重试和人工重试都复用同一确定性输入与无效修复合同；它们增加了调用次数，却没有增加纠正信息，所以失败不是随机波动。
- 原测试覆盖 schema invalid 和若干独立语义错误，但没有覆盖“终止于未回答 assistant 消息 + 多轮 `not_user_message` 修复”的组合。

## 4 修复方案

- 从服务端已验证 transcript 推导严格递增、去重的 candidate user 序号 allowlist，并把它作为修复约束；assistant 序号永远不能被提升为证据。
- 将所有可达语义问题归入结构、anchor、evidence、readiness、action、text 六类明确意图；同一响应出现多类问题时合并提示，给出具体保留、删除或重写要求。
- 对修复坐标执行受限 JSON path、code/path 兼容、正整数序号、非空 allowlist 校验；未知或不安全坐标在调用 provider 前 fail closed。修复请求不附带上一轮模型输出，也不回显原始 transcript 或内部 path。
- 新增同 report ID 的原子重新生成 API、幂等/并发边界和 ReportsScreen 恢复动作；失败、生成中与成功状态都保留独立的面试记录入口。
- DEBUG raw capture 写盘失败改为进程可见的脱敏 WARN，捕获失败仍不改变 AI 业务结果，便于后续定位而不泄露原文。

## 5 验证

- Focused Go tests 覆盖六类修复意图、组合问题、未知坐标、literal marker escaping、终止 assistant 和可信 user allowlist。
- 真实 PostgreSQL integration tests 证明并发不同幂等键最多创建一个 active job，冻结输入不变；非 failed、超限、active old job 与跨用户均零写入。
- 对原失败报告执行同 ID 真实 provider 重试后到达 ready；前两次明确修复缺失 evidence，第三次通过，最终只引用合法 user anchor。
- Chrome 插件在真实 10900/10901 环境验证 failed → generating → ready，并确认重新生成、生成进度和面试记录动作按状态正确共存。
- 最终根 `make test` 通过：Python 584 tests / 4583 subtests、Go 全包、frontend 126 files / 1026 tests；`make build`、context/docs/index/diff gates 同步通过。

## 6 经验教训

- 结构化输出的 retry 不能只传 validator code/path；必须把错误翻译成模型可执行的业务意图和可信值域。
- 如果同一输入多次稳定失败，应优先检查确定性 repair contract，而不是继续增加相同重试。
- 排查 prompt 污染时应同时证明消息组成、上一轮输出是否携带、冻结输入 hash 与角色边界，不能只看最终错误码。

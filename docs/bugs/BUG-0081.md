# BUG-0081: backend profile L2 review exposed privacy rollback and scenario evidence drift

> 模块: materials | 严重度: high | 状态: resolved
> 发现日期: 2026-05-21 | 关联 Commit: `fix(backend-profile): close profile L2 evidence drift (BUG-0081)`
> 关联日志: [2026-05-21](../work-journal/2026-05-21.md)

## 1 现象

- `backend-profile/001-candidate-profile-and-experience-cards` 已标记 completed，但 L2 review 发现 P0.091 / P0.092 / P0.093 场景资产仍残留旧 P0.081 / P0.082 / P0.083 文本与 `.test-output` 路径。
- C-14 legacy negative grep 会命中 `backend/internal/profile/handler/update_test.go` 中的 `"growth-stage SaaS"`，说明测试 fixture 仍含当前设计已弃用的 `growth` 口径。
- `make codegen-check` 仍会因 `resume_parse` source type 与 `debrief_generate` naked literal 失败，导致已完成计划的最终 gate 不能稳定重跑。
- 隐私删除链路缺少失败路径回滚与失败审计证据；如果删除 experience cards 后 profile 删除失败，原实现没有事务性保证和可验证的 failure audit tombstone。

## 2 诊断过程

### 假设 1: 场景目录重命名后内部文本和产物路径未同步

- 验证方法：对 P0.091 / P0.092 / P0.093 场景目录、`profile_http_scenario_test.go` 和场景脚本做旧 ID 负向搜索。
- 结果：确认。目录名已是 P0.091-093，但 README、data、setup、trigger、verify、cleanup 与 Go 注释仍包含旧 P0.081-083 或旧 output path。

### 假设 2: Completed checklist 的 legacy negative gate 没有用当前代码事实重跑

- 验证方法：执行 `rg -n 'mistake|growth|drill|experiences|star' backend/internal/profile/`。
- 结果：确认。命中来自测试 fixture，而不是产品代码，但仍会使 C-14 gate 无法作为零残留证明。

### 假设 3: 隐私删除只覆盖 happy path，缺少事务失败恢复断言

- 验证方法：检查 `DeleteCandidateProfileForUser` service/store 路径、audit tombstone writer 和 P0.093 场景脚本。
- 结果：确认。service 只调用非事务 store 删除；store 的 audit writer 只支持 success metadata；P0.093 只检查成功删除和 PII redaction，没有模拟 profile delete failure。

### 假设 4: `make codegen-check` failure 全部是无关遗留

- 验证方法：分别检查 `backend/internal/profile/profile.go` 的 `resume_parse` 语义和 `frontend/src/api/devMockClient.ts` 的 `debrief_generate` job comparison。
- 结果：部分确认。`resume_parse` 是 `experience_cards.source_type` 允许值，不是 job dispatch；`debrief_generate` 是真实 naked job literal，需要改为共享 job 常量。

## 3 根因分析

- 基线实施阶段完成了场景目录编号改名，但没有把旧 ID / 旧 output path 当成零残留 gate 纳入最终负向搜索。
- C-14 只写了 legacy term 目标，却没有保证测试 fixture 文案也避开 non-current product vocabulary。
- 隐私删除验收只证明了 success tombstone 和 PII redaction，没有用数据库失败注入验证 rollback 与 failure audit。
- `make codegen-check` 被历史 pre-existing failure 干扰后没有在 L2 close-out 中强制恢复为完整 PASS，导致 backend profile 完成态仍依赖局部解释。

## 4 修复方案

- 统一修正 P0.091 / P0.092 / P0.093 场景 README、data、setup、trigger、verify、cleanup 和 `profile_http_scenario_test.go` 中的旧 ID 与 output path，并用旧 ID 负向搜索确认零残留。
- 将 handler 测试 fixture 中的 `"growth-stage SaaS"` 改为 `"scale-stage SaaS"`，让 C-14 negative grep 对 `backend/internal/profile/` 返回 0 命中。
- 新增 `Repository.DeleteCandidateProfileForUserWithAudit`，在一个事务中完成 count、cards delete、profile delete、success audit；失败时 rollback 并写入 redacted failure audit metadata。
- 为 privacy delete failure path 增加 Postgres integration test，使用触发器强制 profile delete 失败并断言 profile/cards 保留、failure audit 存在且不含 raw content。
- 将 P0.093 scenario trigger/verify 扩展为运行并检查新的 failure rollback integration gate。
- 在 event lint 中为 profile `resume_parse` source type 增加窄 allowlist 与单元测试，并把 frontend dev mock 的 `debrief_generate` 比较改为共享 job 常量。
- 将原 plan/checklist/BDD 文档升至 1.2 completed，新增 Phase 6 remediation 项并同步 `context.yaml` 与 plans INDEX。

## 6 经验教训

- Completed plan 的 L2 close-out 必须重新跑旧 ID / 旧 path / non-current term 的负向搜索，不能只相信目录名、checklist 状态或历史 PASS。
- 隐私删除类流程需要同时覆盖 success audit、failure audit、rollback 和 PII redaction；只测成功删除会漏掉最重要的恢复路径。
- 对于 `make codegen-check` 这类复合 gate，L2 修复的闭环标准应是恢复完整 target PASS，而不是继续保留“无关遗留 failure”的解释。

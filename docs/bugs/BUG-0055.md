# BUG-0055: backend-practice event loop trusted client follow-up state

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-14 | 关联 Commit: `fix(backend-practice): harden event loop replay contracts`
> 关联日志: [2026-05-14](../work-journal/2026-05-14.md)

## 1 现象

- `/plan-code-review backend-practice/002-event-loop-and-completion --fix` 复查发现 `answer_submitted` 分支通过客户端 `payload.followUpCount` 决定 `ask_follow_up` / `ask_question`，而不是读取 DB 中的 `practice_turns.follow_up_count`。
- OpenAPI 已将 `occurredAt` / `clientCompletedAt` 标为 required，但 handler 接受缺失字段并由 service 默认成 server time。
- D-35 replay 文档要求 `async_jobs(job_type='report_generate', dedupe_key=sessionId)`，实际 replay 查询只按 `resource_id` + `job_type` 找 job。
- `E2E.P0.038` / `E2E.P0.039` 历史 gate 与 BDD 文档不一致：三分支与 5 kind exhaustive 语义没有被当前测试完整证明。

## 2 诊断过程

### 假设 1：状态机已使用 server-owned follow-up state

- 验证方法：反查 `SessionEventService.handleAnswerSubmitted`、`selectAppendSessionContext` 与 HTTP scenario。
- 结果：排除。`handleAnswerSubmitted` 读取 `payload.followUpCount`；repository latest-turn 查询没有选取 `practice_turns.follow_up_count`；P0.038 通过客户端 `"followUpCount": 1` 直达 `ask_question`。

### 假设 2：required timestamp 已由 generated request type 强制

- 验证方法：读取 `openapi/openapi.yaml` required 字段，并添加 handler red tests。
- 结果：排除。generated Go 字段为空字符串时 handler 使用 optional parser 接受，service 默认填入 server time。

### 假设 3：D-35 replay 查询已绑定 session-level dedupe key

- 验证方法：读取 `complete_session.go` 与 sqlmock expectation。
- 结果：排除。查询没有 `j.dedupe_key=sessionId` 与 `j.resource_type='feedback_report'` 条件，测试也只期待 user/session/job_type 三个参数。

## 3 根因分析

- 根本原因：L2 remediation 仅修复了缺失 answerText 与 status guard，未把 spec Q-5 的 server-owned `follow_up_count` 反向落实到 domain/store 数据模型，也未用 BDD gate 验证“客户端 payload 不能驱动状态机”。
- 涉及代码路径：
  - `backend/internal/practice/session_event.go`
  - `backend/internal/store/practice/append_event.go`
  - `backend/internal/api/practice/session_event_handlers.go`
  - `backend/internal/store/practice/complete_session.go`
  - `backend/cmd/api/practice_http_scenario_test.go`
- 为什么之前没有发现：历史 BDD gate 只证明测试通过，没有证明场景文本中的三分支、5 kind exhaustive、required timestamp 与 D-35 dedupe lookup 都按当前真理源执行。

## 4 修复方案

- 将 `TurnRecord` 扩展为携带 `FollowUpCount`，repository latest-turn 查询读取 `practice_turns.follow_up_count`。
- `answer_submitted` 状态机改用 DB follow-up count，忽略客户端 `payload.followUpCount`；首次答案进入 `follow_up_requested` 且不发 `practice.turn.completed`，turn 到 `assessed` 后才一次/turn 发 outbox。
- handler 对 `occurredAt` / `clientCompletedAt` 缺失返回 `422 VALIDATION_FAILED`，不再默认 server time。
- D-35 replay 查询增加 `resource_type='feedback_report'` 与 `dedupe_key=sessionId`。
- 补强 `E2E.P0.038` / `E2E.P0.039` / `E2E.P0.040` 与对应 unit tests，并把 plan/test/BDD 文档推进到 v1.2。

## 6 经验教训

- 对事件状态机 review，不能把客户端 payload 中的派生状态字段当作 server truth；必须从 DB truth source 反查 domain input。
- BDD gate 的文本如果写了三分支、5 kind exhaustive、一次/turn outbox，HTTP scenario 必须逐项断言，不应只保留历史 PASS。
- OpenAPI required 字段需要 handler 负向测试；generated type 编译通过不等于 runtime 必填校验存在。

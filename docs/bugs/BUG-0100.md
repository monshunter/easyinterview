# BUG-0100: parse confirm handoff dropped workspace context fields

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-24 | 关联 Commit: `fix(frontend-home): preserve parse confirm context (BUG-0100)`
> 关联日志: [2026-05-24](../work-journal/2026-05-24.md)

## 1 现象

- `E2E.P0.016` 的 scenario wrapper 与 jsdom component tests 均通过，但 browser-level route/context audit 发现 Parse Confirm handoff 只把 `targetJobId` / `jdId` / `planId` / `resumeVersionId` / `roundId` 写入 workspace route。
- `frontend/src/app/navigation/interviewContext.ts` 已定义完整 7 字段 workspace context：`targetJobId` / `jobId` / `jdId` / `planId` / `resumeVersionId` / `roundId` / `roundName`。
- 缺失的 `jobId` 与 `roundName` 会让下游 Workspace 无法稳定依赖统一的 interview context contract；未登录 Confirm 的 `pendingAction.params` 同样漏字段。

## 2 诊断过程

### 假设 1：Parse Confirm 可以只携带 workspace 最小字段

- 验证方法：反查 owner spec/plan 的 TargetJob view-model mapping、`interviewContextFromTargetJob` helper、Home recent card handoff 与 Workspace route hydration 口径。
- 结果：排除。Home card handoff 和 plan §3.7 已锁定 7 字段 deterministic defaults；Parse Confirm 应复用同一 helper，而不是手写子集。

### 假设 2：已有 E2E.P0.016 已覆盖浏览器 route query

- 验证方法：读取 `test/scenarios/e2e/p0-016-parse-confirm-to-workspace/scripts/{trigger,verify}.sh` 与 trigger log。
- 结果：排除。旧 gate 只跑 real-mode generated client 与 ParseEdit/AuthGate jsdom tests；`verify.sh` 检查宽泛 PASS marker，没有 browser click、URL query、DOM state 或 screenshot marker。

### 假设 3：修复只需统一 params helper，不影响 updateTargetJob 或 loading flow

- 验证方法：先把 ParseEdit/AuthGate 断言扩展为 7 字段，确认红灯集中在 `jobId` / `roundName`；再修复 `ParseScreen.handleConfirm` 并跑 focused Vitest、build、Playwright 和 P0.016 wrapper。
- 结果：确认。复用 `interviewContextFromTargetJob(targetJob)` 后，已登录 navigate 与未登录 pendingAction 同时恢复完整 context，其他 parse 流程不变。

## 3 根因分析

- `ParseScreen.handleConfirm` 手写 workspace params，而不是复用 `interviewContextFromTargetJob` 这一 owner helper。
- 已有 `ParseEdit.test.tsx` / `ParseAuthGate.test.tsx` 只检查 5 字段子集，未与 plan §3.7 的 7 字段 mapping 做 exact set compare。
- `E2E.P0.016` scenario verify 没有浏览器层 URL query 和 DOM marker，因此 jsdom PASS 掩盖了正式 route handoff drift。

## 4 修复方案

- `frontend/src/app/screens/parse/ParseScreen.tsx` 引入并复用 `interviewContextFromTargetJob(targetJob)`，生成单一 `workspaceParams`。
- 已登录 Confirm 成功后执行 `navigate({ name: "workspace", params: workspaceParams })`；未登录 Confirm 执行 `requestAuth({ type: "confirm_interview", route: "workspace", params: workspaceParams })`。
- 扩展 `ParseEdit.test.tsx` 与 `ParseAuthGate.test.tsx`，锁定 7 字段 exact context。
- 在 `frontend/tests/pixel-parity/parse.spec.ts` 增加 Playwright browser gate：打开 `/parse?targetJobId=...`，mock generated API，点击 Confirm，断言 `/workspace` query 7 字段完整并捕获 `workspace-missing-resume` screenshot。
- `E2E.P0.016` trigger/verify/README/expected-outcome 同步 browser route/context gate；owner spec/plan/checklist/BDD/context 原地修订。

## 5 验证

- Red gate：扩展后的 `ParseEdit.test.tsx` / `ParseAuthGate.test.tsx` 先失败，错误集中在缺失 `jobId` / `roundName`。
- `pnpm --filter @easyinterview/frontend test src/app/screens/parse/ParseEdit.test.tsx src/app/screens/parse/ParseAuthGate.test.tsx`：2 files / 13 tests PASS。
- `pnpm --filter @easyinterview/frontend build`：PASS（仅保留既有 Vite chunk size warning）。
- `pnpm --filter @easyinterview/frontend exec playwright test tests/pixel-parity/parse.spec.ts --grep "confirm navigates to workspace missing-resume with complete interview context"`：2 tests PASS，desktop/mobile 均输出 P0.016 contextKeys + screenshotBytes marker。
- `test/scenarios/e2e/p0-016-parse-confirm-to-workspace/scripts/setup.sh` → `trigger.sh` → `verify.sh` → `cleanup.sh`：PASS。

## 6 经验教训

- route/context handoff 不应在每个 screen 内手写字段集合；已有 owner helper 时，生产代码和测试都应反查 helper 的 exact output。
- 用户可见 route handoff 的 BDD wrapper 必须检查浏览器 URL query、目标 route DOM state 与 screenshot marker，不能只依赖 component-level navigation spy。
- 对 completed frontend plan 做 post-pass hardening 时，scenario `verify.sh` 需要检查目标测试名、context key marker 和 screenshot bytes，避免宽泛 PASS 继续形成 false-green。

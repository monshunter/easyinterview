# BUG-0197: practice interviewer adopted the candidate resume employer identity

> 模块: practice | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-21 | 关联 Commit: `fix(practice): ground interviewer employer identity (BUG-0197)`
> 关联日志: [2026-07-21](../work-journal/2026-07-21.md)

## 1 现象

- 在真实面试会话中，面试官把候选人简历里的任职公司当成自己的所属公司，并以该公司 HR / 招聘方身份继续对话。
- 当 TargetJob 未给出可靠公司名、简历又包含明确公司经历时，模型缺少“匿名目标岗位必须保持公司中立”的身份边界，容易用更显眼的简历公司补全招聘方身份。
- 错误身份一旦进入 assistant conversation history，后续轮次还可能把先前自称当作上下文事实并继续沿用。

## 2 诊断过程

### 假设 1：TargetJob 与 Resume 在 practice runtime 中发生字段错接

- 验证方法：反查 session start/send 的 prompt render 路径、TargetJob/round 投影、resumeContext 和 conversationHistory 组装，并检查现有 focused tests。
- 结果：排除。运行时分别传入 TargetJob、Resume 与 round，未发现把简历公司写入目标公司字段的代码路径；缺口发生在模型如何解释这些并列上下文。

### 假设 2：现有 prompt 已明确限制面试官雇主来源，但模型没有遵守

- 验证方法：读取 active `practice.session.chat/v0.2.0` prompt、rubric 和全量 Practice eval cases，检查是否存在招聘方身份来源、匿名目标公司和历史错误身份纠偏规则。
- 结果：排除。v0.2 只约束候选人事实 grounding，未声明面试官雇主的唯一来源；rubric 只有追问、深度和语言维度，既有 eval 也没有 TargetJob-versus-Resume 身份冲突用例。

### 假设 3：只在 prompt 中补一句禁止冒充即可稳定防回归

- 验证方法：按 owner 链路审查 prompt version、rubric dimension、eval registry、activation preflight、DB migration 与 rollback contract。
- 结果：排除。只改 prompt 无法让离线 gate 识别身份回归，也不能保证 dev registry 与 PostgreSQL 同时切换到受验证的 immutable pair；需要 prompt、rubric、eval、激活和 rollback 一起闭环。

## 3 根因分析

- `practice.session.chat/v0.2.0` 把 TargetJob、Resume 和 conversation history 一并提供给模型，却没有定义面试官招聘方身份的证据优先级。模型可以把 Resume 公司从“候选人履历”错误提升为“面试官雇主”。
- 对匿名、泛化或无法可靠识别公司名的 TargetJob，没有要求省略公司名；模型因此倾向于从简历里选择一个具体公司补全自然话术。
- conversation history 中的 assistant 内容只被禁止作为候选人事实来源，没有同样禁止其作为面试官身份来源，导致错误自称可能自我延续。
- v0.2 rubric/eval 没有独立 `role_identity` 维度和 named-target、anonymous-target、Resume-company impersonation、history-correction 四类 case，因此旧 gate 对该缺陷没有检测能力。

## 4 修复方案

- 新建 immutable `practice.session.chat/v0.3.0` prompt/schema pair：persisted TargetJob/round 是面试官雇主与招聘方角色的唯一来源；Resume 公司只能表示候选人经历；TargetJob 匿名或模糊时不得报公司名；assistant 历史自称不构成身份事实，发现错误时必须回到 TargetJob 边界。
- 新建 v0.3 rubric，将 `role_identity` 作为独立维度并赋予 0.4 权重；11 个 Practice case 全部锁定 v0.3 exact coordinate，并增加四类身份冲突/纠偏用例。
- 用显式 owner markers 保护激活 preflight，避免普通文字或失败说明误触发；registry cache、resolver、Practice caller 与 evalkit tests 同步要求 active v0.3，同时保留 exact v0.2 rollback coordinate。
- 通过 `000023_activate_practice_interviewer_identity_v030` 原子切换 practice prompt/rubric；down 恢复 v0.2，并证明 report prompt/rubric 不受影响。
- 未采用“在后端删除简历公司”方案：这会破坏对候选人真实经历的追问能力，也没有解决 assistant history 自我延续问题。

- 真实 DeepSeek provider 使用 exact v0.3 做 5 次诊断 completion：3 次匿名目标、1 次实名目标、1 次历史错误自称纠偏，5/5 均返回有效 JSON 且没有把 Resume 公司认作面试官雇主。实名目标使用正确招聘方，匿名目标保持面试官身份中立。live judge 因既有 JSON 解析错误不可用，未计为通过证据。

## 6 经验教训

- 多来源 prompt 不应只约束“事实是否真实”，还要为每类身份/事实声明唯一 owner、冲突优先级和缺失时的降级话术。
- assistant history 是延续对话所需的上下文，但不是自证事实；身份与候选人事实都必须回到 persisted owner，而不能由模型自己的旧输出闭环证明。
- Prompt 行为修复必须同时拥有 rubric dimension、强弱样例、exact version activation 和 rollback gate；只改文字无法提供稳定的回归保护。
- 真实 provider completion 可以证明运行模型是否遵守关键边界，但 judge/parser 失败必须单独报告为 unavailable，不能用 completion 成功替代 judge PASS。

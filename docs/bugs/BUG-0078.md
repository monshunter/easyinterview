# BUG-0078: frontend debrief dev mock flow was stuck in analysis

> 模块: debrief | 严重度: high | 状态: resolved
> 发现日期: 2026-05-18 | 关联 Commit: `fix(frontend-debrief): repair dev mock debrief flow (BUG-0078)`
> 关联日志: [2026-05-18](../work-journal/2026-05-18.md)

## 1 现象

- 用户在默认前端 mock 模式进入 `debrief` 后，Step 0 可提交复盘记录，但 Step 1 一直显示 `AI 分析中...`。
- 因 Step 1 没有拿到 completed debrief，Step 2 `复盘面试`无法访问，前端默认 mock 流程无法跑通。
- 影响范围是 Vite dev fixture-backed mock；真实 backend opt-in 模式不由本 bug 判断。

## 2 诊断过程

### 假设 1：正式前端页面未实现

- 验证方法：读取 `frontend/src/app/App.tsx`、`frontend/src/app/screens/debrief/DebriefScreen.tsx`、`DebriefAnalysisStep.tsx`、`DebriefReplayPlan.tsx` 和计划 `frontend-debrief/001-debrief-screen-and-handoff`。
- 结果：排除。正式 `debrief` route 已接入 `DebriefScreen`，Step 1/Step 2 组件和 handoff 逻辑存在。

### 假设 2：默认前端 dev mock 不应调用真实 backend

- 验证方法：读取 `docs/development.md` §2、`frontend/README.md` §2.3/2.4、`frontend/src/api/clientFactory.ts` 和 `frontend/src/api/devMockClient.ts`。
- 结果：确认。Vite dev 默认应使用 fixture-backed mock；真实 backend 只通过 `VITE_EI_API_MODE=real` 显式启用。

### 假设 3：fixture-backed async job 场景没有推进

- 验证方法：对比 `openapi/fixtures/Debriefs/createDebrief.json`、`openapi/fixtures/Jobs/getJob.json`、`frontend/src/api/mockTransport.ts` 和 `useDebriefPolling`。
- 结果：确认。`createDebrief` default 返回 `debrief_generate` job id `01918fa0-0000-7000-8000-00000000b400`，但 `getJob` default fixture 是 generic `report_generate/running` job id `01918fa0-0000-7000-8000-00000000b200`。`createFixtureBackedFetch` 无显式 `Prefer` 时始终选 default，导致 polling 永远看不到 debrief succeeded。

## 3 根因分析

- 根本原因：`createDevMockClient()` 只实现了 `/me` auth 状态推进，没有为跨 operation 的 async job / derived practice handoff 建立 dev mock 状态映射。
- 涉及代码路径：
  - `frontend/src/api/devMockClient.ts`：缺少 `POST /debriefs -> GET /jobs/{jobId}` 的 stateful scenario 选择。
  - `openapi/fixtures/Jobs/getJob.json`：缺少可被 dev mock 自动选择的 debrief succeeded job 场景。
  - `frontend/src/app/screens/debrief/DebriefScreen.test.tsx`：此前只有 stubbed client happy path，没有用真实 `createDevMockClient()` 覆盖默认 dev 预览路径。
- 为什么之前没有发现：计划收口测试证明了组件逻辑和 stubbed polling 能跑通，但没有覆盖正式 bootstrap 使用的 fixture-backed dev mock 的跨 operation 状态机。

## 4 修复方案

- 在 `openapi/fixtures/Jobs/getJob.json` 增加 `debrief-succeeded` 场景，匹配 `createDebrief` default 返回的 debrief job。
- 在 `frontend/src/api/devMockClient.ts` 记录 `POST /debriefs` 返回的 debrief job id；后续无显式 `Prefer` 的 `GET /jobs/{jobId}` 自动注入 `Prefer: example=debrief-succeeded`。
- 同步修复 replay handoff 的默认 mock：`goal='debrief'` 的 `createPracticePlan` 自动选择 `debrief-derived` fixture，随后对应 `startPracticeSession` 自动选择 `debrief-derived-first-question` fixture。
- 补充 `frontend/src/api/devMockClient.test.ts` 和 `frontend/src/app/screens/debrief/DebriefScreen.test.tsx` 回归测试。
- 原地修订 `frontend-debrief/001-debrief-screen-and-handoff` spec/plan/checklist/test-checklist，固化默认 dev mock flow gate。

## 6 经验教训

- 默认 dev mock 的完成标准不能只验证单 operation fixture 覆盖率；跨 operation 的 async job 必须有状态推进测试。
- 用户可见主流程应至少有一条用 `createDevMockClient()` 的 full-flow regression，覆盖真实 Vite dev bootstrap 的 mock 语义。
- Completed checklist 的历史 PASS 不能替代当前默认 mock 体验验证。

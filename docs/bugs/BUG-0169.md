# BUG-0169: Closeout contract gates drifted from current source ownership

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-14 | 关联 Commit: `fix(interview): close turn UX and report navigation (BUG-0167, BUG-0168, BUG-0169)`
> 关联日志: [2026-07-14](../work-journal/2026-07-14.md)

## 1 现象

- 提交前运行根级 `make test` 时，4 个 Python contract tests 失败：practice conversation contract 仍断言旧的 13 events / 26 tables；P0.044/P0.046 仍各自解析 Vitest summary；P0.059 script contract 仍要求旧的 parity 顺序、trigger marker 与 pass 文案。
- 当前 source truth、共享 verifier 与场景脚本本身已经收敛，失败来自防漂移测试仍复制历史数量或旧实现细节。
- 这些误报阻断同 owner 范围的提交验收，但没有产品运行时失败，因此严重度为 `medium`。

## 2 诊断过程

### 假设 1：当前事件、迁移或场景实现缺少应保留的合同

- 验证方法：直接解析 `shared/events.yaml`、baseline migration、P0.044/P0.046/P0.059 trigger/verify 脚本，并运行相关 focused tests。
- 结果：排除。当前 truth source 明确为 12 个事件、25 张表；P0.044/P0.046 已调用共享 frontend/backend verifier；P0.059 当前 Reports/Report/Generating parity runner 与 16 个 Playwright tests 均通过。

### 假设 2：contract tests 复制了已变化的数量或被共享层接管的逻辑

- 验证方法：逐项比较失败断言与当前 exact set、共享 `frontend-real-backend-verify.sh` 的职责及 P0.059 当前 marker。
- 结果：确认。旧测试使用裸数量、要求 scenario-local Vitest summary parser，并匹配已经被当前实现替换的字符串。

## 3 根因分析

- practice conversation contract 只断言事件/表总数，没有列出允许集合；合法删除旧资产后，计数失败无法表达真正的 current-scope 边界。
- P0.044/P0.046 的 source contract 没有把共享 verifier 作为唯一 runner summary owner，导致 scenario-local parser 删除后测试仍要求重复实现。
- P0.059 contract test 没有与当前脚本 marker 同步，且缺少 `reports.spec.ts` 这一当前规划报告列表 parity owner。
- 之前的 focused implementation gates 不运行根级 Python 聚合，漂移直到 `/work-journal` 的全量 `make test` 才暴露。

## 4 修复方案

- 将事件和 baseline table 裸数量替换为 exact-name sets，同时保留 removed-scope 负向断言。
- P0.044/P0.046 contract 只要求调用共享 verifier，并显式禁止恢复 scenario-local `Test Files|Tests` summary parser。
- 将 P0.059 contract 对齐当前 `formal/prototype/diff`、Reports/Report/Generating trigger marker、Playwright pass marker和 `reports.spec.ts` inventory。
- 保留完整根级测试和三个场景的当前源码执行作为交付门禁，不放宽 runner 失败检测。

## 5 验证

- Focused Python：practice conversation 7/7、共享 verifier/场景 contract 8/8、P0.059 contract 2/2 全部通过；P0.044/P0.046 `bash -n` 通过。
- 根级 `make test` 通过：UI source contract 62/62、Python 590 tests / 5181 subtests、全部 Go packages、frontend 121 files / 977 tests。
- P0.044 当前源码 run `cd8c378d-6fcc-4045-a00f-c9129873e511` 与 P0.046 run `dfc68a8f-41de-46e5-9b0c-1aec3fbb67fb` 均完成 setup/trigger/verify/cleanup；共享 source SHA-256 为 `3e644ae013ee2159937e4853c8f3f32a3f2bd1f1351fd6fe74d9b66aa2ea11d2`，隔离 PostgreSQL residual 为 0。
- P0.059 当前源码完成 10 个聚焦 Vitest 文件 / 137 tests、production build、16 个 desktop/mobile Playwright parity tests，并通过 verify/cleanup。

## 6 经验教训

- 防漂移测试应断言允许的 exact set，而不是只维护数量；这样新增和删除都能给出可解释差异。
- 共享 verifier 接管职责后，scenario contract 必须验证委托关系和禁止重复 parser，不能继续复制共享实现。
- `/work-journal` 的根级聚合测试是发现跨 owner contract drift 的必要最终门禁，不能用 focused PASS 替代。

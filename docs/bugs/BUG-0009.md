# BUG-0009: dev-stack profile catalog drift escaped lint gates

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-05 | 关联 Commit: `fix(ai-provider): harden profile catalog drift gates`
> 关联日志: `-`

## 1 现象

- L2 review 发现 `deploy/dev-stack/.env.example` 仍将 `AI_MODEL_PROFILE_PATH` 指向旧目录 `config/ai-profiles/`，且没有暴露 `AI_PROVIDER_REGISTRY_PATH`。
- `docs/spec/product-scope/spec.md` 的 active owner matrix 仍把旧 profile directory 写成 AI model profile 的可执行真理源。
- `make lint-config` 与 `make lint-ai-profile-coverage` 均能通过，说明现有 gate 没有覆盖 deploy env profile catalog path，也没有校验 Product/UI capability catalog 与 `config/ai-profiles.yaml` 的语义一致性。

## 2 诊断过程

### 假设 1：A4 env dict lint 已覆盖 dev-stack AI catalog path

- 验证方法：在修复前运行 `make lint-config`，并检查 `deploy/dev-stack/.env.example` 中的 AI env keys。
- 结果：排除。lint 只证明 root `.env.example`、spec env 字典、代码读取和 provider registry env refs 对齐；dev-stack env example 中的旧 profile directory 没有被拦截。

### 假设 2：profile coverage lint 已覆盖 Product/UI capability 与 catalog capability 的语义一致性

- 验证方法：新增负向单测，让 A3 Product/UI 表声明 `practice.followup.default` 为 `chat`，但 `config/ai-profiles.yaml` 将该 profile 写成 `embed`。
- 结果：确认缺口。修复前 lint 返回 OK；修复后 `ai_profile_coverage.py` 报 `Product/UI capability mismatch`。

### 假设 3：旧目录口径只存在于历史记录，不影响 active truth source

- 验证方法：对 deploy/config/code/active specs 与当前 003 plan 执行 `config/ai-profiles/` / `AI_MODEL_PROFILE_PATH=config/ai-profiles/` 负向搜索。
- 结果：确认 active drift。Product Scope owner matrix、Repo Scaffold active spec 与 dev-stack env example 都存在当前语义下的旧目录口径。

## 3 根因分析

- 根本原因：003 catalog consolidation 把 loader、profile catalog 和 root env 字典迁移到 `config/ai-profiles.yaml`，但没有把 deploy env example 与 active product/repo owner docs 纳入同一个 semantic drift gate。
- 涉及的代码和文档路径：
  - `deploy/dev-stack/.env.example`
  - `deploy/dev-stack/README.md`
  - `docs/spec/product-scope/spec.md`
  - `docs/spec/repo-scaffold/spec.md`
  - `docs/spec/local-dev-stack/spec.md`
  - `scripts/lint/ai_profile_coverage.py`
  - `scripts/lint/ai_profile_coverage_test.py`
- 之前没有发现，是因为既有 gate 偏向结构覆盖：profile 是否存在、provider ref 是否存在、root env 字典是否闭环；缺少 deploy-specific canonical path 和 Product/UI capability-to-profile 语义断言。

## 4 修复方案

- 在 `scripts/lint/ai_profile_coverage.py` 中新增 dev-stack canonical AI env 检查，要求 `AI_PROVIDER_REGISTRY_PATH=config/ai-providers.yaml` 与 `AI_MODEL_PROFILE_PATH=config/ai-profiles.yaml`。
- 在同一 lint 中解析 A3 `Product/UI AI Capability Catalog`，校验每个默认 profile 的 catalog capability 与文档声明一致；多 capability / 多 profile 行按表格顺序匹配，避免集合顺序造成误报。
- 更新 `deploy/dev-stack/.env.example` 和 `deploy/dev-stack/README.md`，让本地栈显式使用单文件 provider registry / profile catalog。
- 更新 Product Scope、Repo Scaffold、Local Dev Stack active spec 与 history，移除旧 profile directory 作为 active truth source 的描述。
- 原地 reopen A3 003 plan/checklist，新增并完成 5.7 remediation item，验证通过后恢复 `completed` 并同步 plans INDEX。

## 5 验证

- Red：新增 `test_fails_when_dev_stack_env_uses_legacy_profile_directory` 与 `test_fails_when_product_ui_capability_disagrees_with_catalog`，修复前 `python3 -m pytest scripts/lint/ai_profile_coverage_test.py -q` 出现 2 个预期失败。
- Focused：`python3 -m pytest scripts/lint/ai_profile_coverage_test.py -q` 通过。

- Backend focused：`cd backend && go test ./internal/ai/aiclient/... ./internal/platform/config/... ./cmd/worker -count=1` 通过。
- Negative search：deploy/config/code/active specs 与当前 003 plan/checklist 作用域中仅保留 003 remediation 文本对旧目录口径的负向描述。

## 6 经验教训

- catalog migration 的完成 gate 不能只覆盖 runtime loader 与 root env 字典，还必须覆盖部署样例和 active owner matrix，否则后续本地启动路径会重新引用 retired layout。
- profile coverage lint 应同时验证“存在性”和“语义一致性”：Product/UI capability、F3 feature key、profile catalog capability 三者必须闭环。
- 后续类似目录到单文件的迁移，应在 plan verification 中写明 active-scope negative search 的具体路径族和允许的历史例外。

# BUG-0061: report runtime only mounted read routes

> 模块: review | 严重度: high | 状态: resolved
> 发现日期: 2026-05-16 | 关联 Commit: `fix(backend-review): wire report runner runtime (BUG-0061)`
> 关联日志: [2026-05-16](../work-journal/2026-05-16.md)

## 1 现象

- L2 code review found that `cmd/api` mounted report read routes but did not build or start the report generation runtime.
- In the real API process, `async_jobs(job_type='report_generate')` rows could remain queued because no report runner or reaper was started.
- Unit/domain tests passed because they exercised the runner/service/store directly, not the production runtime composition path.

## 2 诊断过程

### 假设 1：report routes are mounted but generation workers are not started

- 验证方法：read `backend/cmd/api/main.go` startup path and the report route builder.
- 结果：确认。The startup path used a read-only report route helper and started `targetJobRuntime` plus `resumeRuntime`, but no report runner/reaper.

### 假设 2：review service silently degrades when registry or AI wiring is absent

- 验证方法：read `backend/internal/review/service.go` and compare with the real route helper wiring.
- 结果：确认。The previous route helper constructed `review.Service` with only a repository, so report generation would not have F3 registry, AI client, or `ai_task_runs` writer.

### 假设 3：existing tests already catch production runtime composition

- 验证方法：search `cmd/api` tests for report runtime composition assertions.
- 结果：排除。Existing tests covered route auth middleware and lower-level review behavior, but did not assert that `cmd/api` builds and starts report runner/reaper with AI-backed service.

## 3 根因分析

- 根本原因：the report generation implementation closed domain/store/handler tests but missed the production runtime composition boundary in `cmd/api`.
- 涉及的代码路径：
  - `backend/cmd/api/main.go`: API startup now builds `reportRuntime`, mounts its routes, starts it, and shuts it down.
  - `backend/cmd/api/main.go`: `buildReportRuntime` now wires F3 registry, observed AI client, `ai_task_runs` writer, service, runner, reaper, and handler together.
  - `backend/cmd/api/main_test.go`: runtime composition tests now prevent read-only report wiring from regressing.
- 为什么之前没有发现：the original test checklist used many lower-level and scenario gates, but lacked a `cmd/api` composition gate for background report processing.

## 4 修复方案

- Add `reportRuntime` with `Routes`, `Start`, and `Shutdown` methods.
- Replace read-only `buildReportRoutes` usage with `buildReportRuntime` in `main`.
- Wire report generation to F3 registry, AI observability, `storeai.TaskRunWriter`, review repository, runner, reaper, and report handler.
- Fail fast when the report runtime is built without an AI client.
- Add `TestBuildReportRuntimeWiresRoutesRunnerReaperAndAI` and `TestBuildReportRuntimeRejectsMissingAIClient`.
- Update plan/test checklist lifecycle to include the runtime wiring L2 gate.

## 6 经验教训

- Feature plans with background processing must include a production runtime composition gate in addition to domain/service/store tests.
- Completed checklists should not rely on historical phase evidence alone; L2 closure must verify the actual process entry point that will run in development and production.

# BUG-0013: backend auth L2 review exposed session contract drift

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-06 | 关联 Commit: `fix(backend-auth): remediate passwordless L2 findings`
> 关联日志: [2026-05-06](../work-journal/2026-05-06.md)

## 1 现象

- `/plan-code-review backend-auth/001-passwordless-session-bootstrap --fix` 复扫发现 C1 passwordless session 交付虽然 focused tests 通过，但真实运行与跨用户安全语义仍存在漂移。
- 主要表现包括：`cmd/api` 只挂 anonymous `/runtime-config`，未挂 C1 Auth routes；`DELETE /me` idempotency dedupe key 未按用户隔离；session cookie 未设置 Secure；challenge rate-limit SQL 只统计 pending；logout revoke 失败仍返回 204。
- 后续 review 继续发现三个 runtime 级缺口：`cmd/api` 在 dev 默认配置下会用空 `AUTH_CHALLENGE_TOKEN_PEPPER` / `SESSION_COOKIE_SECRET` 构造 auth service；cookie-bearing logout 遇到 session resolver / store error 时被 optional-session middleware 静默当成匿名 logout 返回 204；并发 logout 下 `TouchSession` 的 `sql.ErrNoRows` 被归类成 500，而不是已撤销会话的幂等清 cookie 路径。
- 影响范围覆盖 backend auth runtime wiring、session cookie、privacy delete handoff、rate-limit 和 logout error path。

## 2 诊断过程

### 假设 1：包级 handler 测试不能证明真实 API runtime wiring

- 验证方法：读取 `backend/cmd/api/main.go` 与 `backend/internal/auth/*_test.go`，对比 checklist 1.3 / 4.1。
- 结果：确认。`cmd/api` 只注册 `/api/v1/runtime-config`，`SessionResolver` 为 nil；auth handler 仅在包级测试和 BDD 内存测试中手动拼接。

### 假设 2：idempotency 只按 raw header key 去重

- 验证方法：读取 `SQLStore.CreatePrivacyDeleteHandoff` 查询条件，并新增两用户同 key 的 SQL store red test。
- 结果：确认。原查询只过滤 `job_type` + `dedupe_key`，没有 user scope，且 active dedupe unique index 会放大跨用户碰撞风险。

### 假设 3：现有 gates 未覆盖 ADR-Q1 cookie / rate-limit / logout error 语义

- 验证方法：读取 ADR-Q1、B2 OpenAPI、backend-auth checklist，并新增 cookie policy、rate-limit SQL、logout revoke failure red tests。
- 结果：确认。原测试未断言 Secure；SQL expectation 未覆盖 consumed challenge；logout handler 忽略 service error。

### 假设 4：真实 runtime builder 未验证 C1 auth secrets

- 验证方法：读取 A4 secrets-and-config spec、`config/dev.yaml` 与 `backend/cmd/api/main.go`，新增 `buildAuthService` 空 secret red test。
- 结果：确认。dev 默认文件保留 secret 空值，`loader.Validate()` 只在 staging/prod 强制 auth secret，runtime builder 未在构造 `PasswordlessService` 前 fail-fast。

### 假设 5：logout optional-session middleware 吞掉非认证类 resolver error

- 验证方法：通过真实 `cmd/api` route 构造 cookie-bearing logout，请 store 在 `GetSessionByHash` 返回数据库错误。
- 结果：确认。旧 middleware 对 optional operation 的任意 `ResolveSession` error 都直接进入 handler，导致 handler 没有 current session 并返回 204。

### 假设 6：logout 并发撤销 race 的 touch zero-row 被误归类

- 验证方法：构造 `ResolveSession` 已读到 active session，但 `TouchSession` 返回 `sql.ErrNoRows` 的 middleware red test，并分别覆盖 optional logout 与 required protected endpoint。
- 结果：确认。旧 service 直接透传 `sql.ErrNoRows`，optional logout 返回 500；但该状态表示会话已被并发 revoke，应该按 revoked auth-state 处理。

## 3 根因分析

- 根本原因是 Phase 1-5 的实现证据偏向包级 focused test 和内存 BDD，缺少真实 runtime wiring、跨用户隔离、SQL query shape 与 error-path semantic gate。
- 涉及代码路径：
  - `backend/cmd/api/main.go`: runtime route / auth service wiring。
  - `backend/internal/auth/service.go`: session touch lost race 的错误归类。
  - `backend/internal/auth/session_middleware.go`: optional-session resolver error 分类。
  - `backend/internal/auth/store.go`: privacy delete dedupe 与 challenge count SQL。
  - `backend/internal/auth/handler.go`: session cookie policy 与 logout error response。
- 之前没有发现，是因为 BDD 场景直接组合 handler / middleware，没有经过 `cmd/api` 入口；idempotency 测试只覆盖同一用户重复 key；cookie 与 logout failure 只测 happy path 或缺 session；后续 review 前没有针对 dev 空 secret builder、optional-session 非认证错误分类和 revoke/touch TOCTOU race 的 route / middleware 级负例。

## 4 修复方案

- 在原 plan/checklist 追加 Phase 6 L2 remediation，按 `/tdd` 红绿修复 5 个 finding。
- `DELETE /me` dedupe key 改为 user-scoped hash，并使用 B3 generated job constants，避免裸 job literal drift。
- `cmd/api` 新增 auth service bootstrap 和 `buildAPIHandler`，挂载 auth start / verify / logout / `/me` / `DELETE /me` / session-aware `/runtime-config`。
- Auth handler 新增 `CookiePolicyForAppEnv`，默认/prod/staging Secure=true，dev/test 可显式降级；verify/logout/deleteMe 共用该 policy。
- `CountRecentChallenges` 移除 pending-only 过滤；logout revoke 失败返回 redacted B1 error envelope 并清 cookie。
- `buildAuthService` 在创建 dev mail dispatcher 前检查 `AUTH_CHALLENGE_TOKEN_PEPPER` / `SESSION_COOKIE_SECRET`，缺失时返回明确 init error 且不构造 C1 runtime。
- `SessionMiddleware` 将 optional-session error 分为已知 auth-invalid（invalid / expired / revoked，可继续幂等 handler）和 resolver/store 内部错误（返回 B1 error envelope）。
- `ResolveSession` 将 `TouchSession` 返回的 `sql.ErrNoRows` 归一为 `ErrSessionRevoked`，让并发 logout 已撤销会话走 optional-session 幂等路径，同时 protected endpoint 仍返回 B1 `AUTH_UNAUTHORIZED`。

## 6 经验教训

- Runtime wiring 必须有入口级测试；包级 handler BDD 不能替代真实 `cmd/api` route / middleware 挂载证据。
- Idempotency tests 必须覆盖跨主体同 key 场景，尤其是隐私删除、账号删除和异步 job handoff。
- 安全属性和错误路径要作为语义 gate 写入 checklist，不能只依赖 OpenAPI / ADR 文本。
- Runtime builder 也需要安全负例：即使上游 config validator 有环境分支，真正构造 cryptographic/auth service 的入口必须拒绝空 secret。
- Optional-session 中间件需要区分“认证状态无效”和“系统无法确认认证状态”，后者不能按匿名请求成功收口。
- Session 解析路径还需要覆盖 touch/renew 与 revoke 之间的 TOCTOU race；store zero-row 不一定是系统错误，也可能是已完成注销的认证状态。

# BUG-0187: resume parse polling flashed the generic loading state

> 模块: materials | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-18 | 关联 Commit: `fix(frontend): stabilize resume parse polling (BUG-0187)`
> 关联日志: -

## 1 现象

- 简历详情处于 queued/processing 时，本应持续显示“正在解析简历”等待态。
- 每次 250ms 后台轮询开始时，页面短暂切换为“正在加载简历…”，请求返回后又切回解析等待态，形成整块内容反复闪现。
- 等待图标容器还使用循环 `scale(1 → 1.06 → 1)`，在边框和 SVG 上叠加轻微亚像素抖动。

## 2 诊断过程

### 假设 1：等待图标的缩放动画导致全部抖动

- 验证方法：检查 `screens.css` keyframes，并在真实 Chrome 中连续采样图标、标题和说明的 bounding box。
- 结果：只解释图标自身的轻微抖动，不能解释“正在加载简历…”反复闪现。

### 假设 2：后台轮询复用了首次加载状态

- 验证方法：为 `ResumeDetailView` 构造首轮 processing、第二轮 pending Promise；观察第二个 `getResume` 尚未返回时的 DOM。
- 结果：确认。旧实现立即移除 `resume-detail-parse-waiting` 并显示通用 loading。

## 3 根因分析

- `useResume` 用 `reloadSeq` 同时驱动首次读取、用户 retry 和后台 poll，但每次 effect 都无条件调用 `setLoading(true)`、`setData(null)`。
- 因此即使已经取得同一 Resume 的 processing 数据，下一轮请求发出时仍会销毁稳定等待态。
- 既有测试只覆盖 polling 最终到 ready 和调用次数，没有把第二轮请求保持 pending，遗漏了请求间隙的可见状态。

## 4 修复方案

- 用 ref 保存最后一份成功 Resume；仅首次读取或 Resume 身份变化时清空 data 并显示通用 loading。
- 同一 pending Resume 的后台 poll 保留现有 data 和解析等待 DOM，终态到达后再替换。
- 移除等待图标循环 scale，使用 opacity/box-shadow 呼吸效果，并在 reduced-motion 下停止动画。
- 不改变 API、轮询间隔、持久化、解析任务或上传链路。

## 5 验证

- TDD RED 精确复现第二轮 pending 请求期间的 loading flash；GREEN 后 Resume Workshop CSS/详情组件 17 tests 与 frontend typecheck 通过。
- 真实 Chrome 连接正式 frontend/backend，使用本地合成 Resume 的 processing 状态连续采样 40 次（50ms 间隔，覆盖多个 250ms poll）：40/40 均为“正在解析简历”，通用 loading 闪现 0 次。
- 同一 Chrome 会话对图标、标题、说明连续采样 10 帧：三者几何值各只有 1 个唯一值，图标 computed transform 恒为 `none`；opacity/box-shadow 持续变化。
- 测试 Resume 已恢复为 ready，随后自动切换到正式只读详情。

## 6 经验教训

- 后台刷新不应复用首次加载的空白态；已有可用数据时，应保留当前投影直到新响应完成。
- 轮询测试不仅要断言最终状态和请求次数，还要显式冻结中间 Promise，验证请求间隙的用户可见 DOM。

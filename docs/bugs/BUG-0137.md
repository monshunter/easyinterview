# BUG-0137: resume creation kept generic names and obsolete parse steps

> 模块: materials | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-07 | 关联 Commit: `fix(resume-workshop): read full upload PDFs (BUG-0137)`
> 关联日志: `-`

## 1 现象

- 用户上传或粘贴简历后，列表和详情仍显示通用名称“粘贴的简历”，无法区分真实简历内容。
- 第一轮修复后，列表名称仍可能直接取 raw resume 第一行（例如 Markdown `# 姓名 | 岗位`），或上传详情标题显示 PDF 文件名，而不是 LLM 根据简历生成的业务名称。
- 创建流程仍进入解析动画页和预览确认页，和当前“上传后直接打开只读详情”的产品诉求冲突。
- 详情页使用 structured projection 渲染正文，导致原始简历信息大量丢失；用户看到的不是简历原文。
- 上传 PDF / DOCX 后详情页可能只显示文件名或空白正文，因为后端 parse path 未提取文件可读正文。
- 第二轮修复后，真实上传 PDF 仍可能一片空白或显示乱码：浏览器生成的 PDF 大小为 554631 bytes，而 parse job 只读取前 256KiB，导致 PDF parser 缺少尾部 xref / 字体映射；扩大读取预算后，旧 literal fallback 又会把 PDF binary / literal 乱码当成非空正文；同时如果 LLM 输出失败，已抽取原文也不会写入 `parsed_text_snapshot`。

## 2 诊断过程

### 假设 1：前端列表和详情直接信任注册时的通用 title
- 验证方法：检查 `mapResumeToUiSource` 和创建流 title 生成逻辑。
- 结果：确认。`mapResumeToUiSource` 使用 `displayName || title`，`deriveDefaultTitle("paste")` 仍返回通用名称，旧数据和解析前状态都会暴露“粘贴的简历”。

### 假设 2：创建流仍保留旧 parse/preview 状态机
- 验证方法：搜索 `ResumeParseFlow`、`ParsingStage`、`PreviewStage`、`ResumePreviewConfirm` 和场景脚本。
- 结果：确认。`ResumeCreateFlow` 注册后进入 parsing/preview 分支，P0.081-084 场景也仍把旧 DOM 当成正向期望。

### 假设 3：详情页正文来自 structuredProfile 而非原文
- 验证方法：检查 `ResumePreviewTab` 和 detail 场景断言。
- 结果：确认。详情正文由 `buildResumePreview` 投影 structuredProfile，未优先渲染 `parsedTextSnapshot` / `originalText`。

### 假设 4：第一轮命名兜底仍把 raw 第一行 / 文件名当作简历名
- 验证方法：检查 `PasteTab`、`create/util/title.ts`、`mapResumeToUiSource` 和 `store/assets.go`。
- 结果：确认。`PasteTab` 使用 `derivePasteTitle(rawText)` 提交第一行，adapter 继续把 `title` 和 `firstContentLine()` 放进可见名称链路，store 创建 queued resume 时把 `in.Title` 写入 `display_name`。

### 假设 5：upload parse 没有文件正文提取
- 验证方法：检查 `backend/internal/resume/jobs/parse.go` 的 upload source 分支。
- 结果：确认。upload 分支读取 object bytes 后直接 `string(body)`，PDF / DOCX 二进制不会产生可读 `parsed_text_snapshot`。

### 假设 6：真实 PDF 被读取预算截断，且失败路径不保留已抽取正文
- 验证方法：查看本地 dev DB 中失败 resume / async job / file_object；从 MinIO 取回实际 PDF 并用 `pdftotext -layout` 验证可读；对比 object size 与 parse handler 读取预算；重排 job 观察失败状态。
- 结果：确认。失败 resume `019f3bfb-7d9d-76d2-a82d-08bcbac225ce` 对应 PDF object 为 554631 bytes，旧 `defaultMaxResumeInputBytes` 只有 262144 bytes；`pdftotext` 能提取正文，说明文件不是扫描件。扩大读取预算后旧 fallback 曾写入 3285 字符乱码并误判 ready；加入 `pdftotext` + 可读性 gate 后，同一 PDF 在 LLM output invalid 时仍持久化 3083 字符可读中文 snapshot。

## 3 根因分析

- `frontend-resume-workshop` 已切换到 flat/read-only 产品合同后，创建流和详情投影仍保留早期“解析草稿 -> 预览确认 -> 保存”的二步模型。
- 第一轮修复把“原文首行 / 文件名兜底”误当成可识别名称策略；真正合同应是 backend LLM-derived `displayName` 优先，frontend 仅可用 LLM/结构化 headline 或中性“名称生成中”占位，不能把 raw resume 第一行或来源文件名作为简历名。
- backend upload parse 缺少 PDF / DOCX / Markdown / text 的正文提取层，导致 `parsed_text_snapshot` 和 AI prompt input 可能来自二进制 bytes 或文件名上下文。
- 真实 PDF 读取预算过小，parser 只拿到文件头部片段，缺少尾部 xref / ToUnicode / 字体映射时会提取失败；旧 literal fallback 没有可读性校验，会把 PDF binary / literal 乱码当作成功正文；且 `parsed_text_snapshot` 只在 AI 结构化成功后写入，导致 LLM 输出失败时原文预览也随之丢失。
- 场景 gate 和 UI contract test 仍锁定旧解析动画、确认保存和 structured preview，导致旧行为没有被负向拦截。

## 4 修复方案

- 简化 `ResumeCreateFlow`：上传或粘贴注册成功后直接导航到 `resume_versions?resumeId=<id>`，删除解析动画、预览确认和 create-flow 内 `updateResume` 相关组件与测试。
- 更新 paste 标题生成：只提交中性来源 title（`粘贴文本` / `Pasted text`），raw resume 第一行不进入 request title 或可见名称。
- 更新 resume adapter 和详情：过滤通用名称，移除 `title` / raw first line 命名兜底，只从 LLM-derived `displayName` / structured headline 派生名称；解析前显示中性“名称生成中”。
- 更新后端 store：queued 创建不再把注册 `title` 写入 `display_name`，ready 后只由 parse success 写入 LLM-derived `display_name`。
- 补强 backend parse displayName 派生：过滤更多通用名称，并支持 root-level headline/title/summary。
- 增加 upload 文件正文提取：PDF 通过 PDF parser + literal fallback，DOCX 读取 `word/document.xml`，Markdown / text 保持 UTF-8 正文；AI prompt input 与 `parsed_text_snapshot` 使用同一可读正文。
- 将 resume parse upload object 默认读取预算提升到 8MiB，并用 554631 bytes 真实失败样本大小作为单元测试下限，防止真实浏览器生成 PDF 被头部截断。
- PDF 抽取优先使用 `pdftotext -layout - -`，并给 Go parser / literal fallback 增加可读性 gate；unreadable PDF fixture 必须在进入 AI 前失败，不能写入乱码 snapshot。
- 将 `CompleteParseFailure` 扩展为可写入已抽取的 `parsed_text_snapshot`；AI provider / AI output validation 失败时仍保留原始正文，详情页不再依赖结构化解析成功才显示原文。
- 前端 adapter 过滤与来源 `title` 相同的上传文件名 `displayName`，文件名只作为 `sourceName` 展示，不作为简历业务名称。
- 同步 docs/spec、docs/ui-design、ui-design contract test、P0.037/P0.081-P0.084 场景资产和 checklist。

## 6 经验教训

- 当产品合同明确删除中间确认页时，必须同时删除 runtime 组件、locale 文案、CSS、UI contract、scenario wrapper 和 expected-outcome；只改入口跳转会留下 false-green gate。
- 简历名称不能把来源描述、raw resume 第一行或文件名当业务名称；可见名称必须来自 LLM-derived `displayName` / LLM structured headline，未生成前使用中性占位。
- 上传文件白名单必须配套正文提取 gate；只验证 upload/register 成功不足以证明详情能显示原始简历正文。
- PDF 解析 gate 必须使用接近真实上传大小的样本或至少断言 object 读取预算；小型合成 PDF 只能证明 parser 分支存在，不能证明真实简历不会被截断。
- PDF 解析 gate 不能只断言 snapshot 非空；必须断言可读性并拒绝 PDF literal / binary 乱码进入 prompt 和 snapshot。
- 原文预览应与 LLM 结构化成功解耦：一旦已经抽取出正文，就应先保证只读详情可展示，再处理结构化名称和 profile。

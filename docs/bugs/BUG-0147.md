# BUG-0147: failed JD parse entered interview list and workspace kept route context

> 模块: workspace | 严重度: high | 状态: resolved
> 发现日期: 2026-07-09 | 关联 Commit: `fix(workspace): enforce parse failure admission and route purity (BUG-0147)`
> 关联日志: [2026-07-09](../work-journal/2026-07-09.md)

## 1 现象

- JD 解析失败后，失败 TargetJob 仍以 `analysisStatus="failed"`、空 title / company / requirements 的形式进入面试规划列表。
- 从规划详情或带历史参数的 `/workspace?targetJobId=...` 进入顶栏“面试”时，页面继承目标岗位上下文并显示「缺少目标岗位 ID」或「JD 解析失败」等详情页错误。
- `workspace` 作为列表页不应拥有 `targetJobId / planId / resumeId / autoStartPractice` 参数语义。

## 2 诊断过程

### 假设 1：后端失败解析仍把 TargetJob 当作可继续规划资产保留

- 验证方法：读取 `CompleteParseFailure`、`UpdateTargetJobAnalysisFailure`、TargetJob list/get store 查询和 HTTP scenario。
- 结果：确认。失败路径写出 `target.analysis.failed` 后保留 `target_jobs.analysis_status='failed'`；列表查询没有 read-side 排除 failed 资产。

### 假设 2：workspace 仍承接详情和启动面试上下文

- 验证方法：检查 `INTERVIEW_CONTEXT_ROUTES`、`WorkspaceScreen` 卡片导航、workspace hooks/modal/start-practice 文件和 P0.018 场景。
- 结果：确认。历史 workspace 详情页、resume/plan hooks、`autoStartPractice` 与合成 `plan-${id}` / `jd-${id}` 参数仍在 owner 边界内，导致列表页从 URL / stale context 派生详情状态。

## 3 根因分析

- 上游准入错误：解析失败被持久化成 TargetJob 可读资产，而不是只留下 async job/outbox 诊断证据。
- 读侧防线不足：`listTargetJobs` / `getTargetJob` 没有排除 failed TargetJob，导致历史脏数据暴露给前端。
- 前端 owner 边界漂移：`workspace` 同时承担列表、详情、resume/plan 切换和 session start，违背当前 RESTful resource 与 route owner 分工；`parse` 应拥有详情和启动合同，`workspace` 只负责集合列表。

## 4 修复方案

- `CompleteParseFailure` 在同一事务写出 `target.analysis.failed` 后删除失败 `target_jobs` 行，依赖 FK cascade 清理 source / requirements；legacy failure update 入口也改为删除。
- `GetTargetJobByUser` / `ListTargetJobsForUser` 排除 `analysis_status='failed'`；即使调用方传 `analysisStatus=failed`，也不返回失败资产。
- `workspace` 查询只请求 `analysisStatus=ready`，并防御过滤 failed / non-ready / 空 title 记录。
- `workspace` 从 `INTERVIEW_CONTEXT_ROUTES` 移除，进入时清空 stale interview context；卡片只导航到 `parse`，并且只携带真实 `targetJobId`、可选真实 `planId/resumeId`。
- URL codec 的 `workspace` safe params 改为空集合，legacy `/workspace?targetJobId=...&planId=...&autoStartPractice=1` 会 canonicalize 为 `/workspace`。
- `parse` 与 report owner 直接调用 shared `startPracticeFromParams` 创建 / 启动 practice session，不再通过 workspace `autoStartPractice` 中转。

## 6 经验教训

- Failure diagnostics and user-facing assets must be different persistence contracts. A failed parse can keep async job/outbox evidence, but it must not become a planning asset.
- Collection routes should stay collection routes. `workspace` should not infer item detail context from URL params; item detail belongs to `parse`.
- Scenario gates should assert both write-side admission and read-side defensive filtering, otherwise historical dirty rows can re-enter the UI after the write bug is fixed.

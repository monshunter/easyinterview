# BUG-0023: jobs_test referenced removed embedding_upsert constant after capability cleanup

> 模块: schema | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `fix(event-outbox): drop stale embedding job reference in jobs_test`
> 关联日志: [2026-05-08](../work-journal/2026-05-08.md)

## 1 现象

- 现象: `go test ./internal/shared/jobs/...` 编译失败，报错 `undefined: JobTypeEmbeddingUpsert`，导致 `make test` 在该包不通过；其他通过 `./...` 聚合执行的 Go test 命令也会被该包失败牵连。
- 触发条件: 任意运行 backend Go test aggregate（`go test ./...` / `make test` / IDE 全量 test build）的场景。
- 影响范围: 单包构建失败（`backend/internal/shared/jobs`），间接阻断仓库聚合测试结果；产品功能未受影响，但隐藏了 backend-targetjob/001 §6.3 active-scope 负向搜索（embedding capability 必须 0 命中）的潜在违规。

## 2 诊断过程

### 假设 1：`embedding_upsert` 仅是 generated 漏生成

- 验证方法: `grep` `shared/jobs.yaml` 中 `embedding`，`grep` 整仓 `EmbeddingUpsert`。
- 结果: `shared/jobs.yaml` 已无该 entry；`backend/internal/shared/jobs/jobs.go` 已不导出 `JobTypeEmbeddingUpsert`；唯一悬空引用在 `backend/internal/shared/jobs/jobs_test.go:46`。truth source 已正确 — 排除“仅 generated 漏生成”。

### 假设 2：cleanup commit 删除时漏同步消费侧测试

- 验证方法: `git log --oneline --all -- shared/jobs.yaml`，定位最近一次删除 `embedding_upsert` 的 commit。
- 结果: `9a30023 refactor(ai-provider): remove retrieval placeholders and use deepseek` 在 2026-05-08 删除 embedding capability/profile/job/migration/pgvector，commit message 显式声明同步 A3/B1/B3/B4/F3/A2 docs/generated artifacts/work journal，但实际未把 B3 拥有的 `internal/shared/jobs/jobs_test.go` 中早先于 2026-04-30 commit `0af4a82 test(event-outbox): assert final job mappings` 添加的 `JobTypeEmbeddingUpsert` 断言一并清理。确认。

### 假设 3：是否需要重启 B3 / A3 已 completed plan

- 验证方法: 对照 backend-targetjob/001 plan §6.3 active-scope 负向搜索条款（embedding/rerank capability 必须 0 命中）。
- 结果: 该悬空引用本就是当前 active plan 的反向搜索目标项之一，归入当前 plan 顺手清理即可，无需把 B3 001-bootstrap 或 A3 004-cascaded-speech-provider-foundation（均已 completed）回退为 active 仅为一行删除。

## 3 根因分析

- 根本原因: cleanup commit `9a30023` 在多文件多目录范围内删除 embedding 能力时，commit message 列出的 owner 同步清单不包含 “generated package 的 test consumer”。具体到 `backend/internal/shared/jobs/jobs_test.go:46` 的断言数组 `{JobTypeSourceRefresh, JobTypeEmbeddingUpsert, JobTypeEmailDispatch}`，作者只更新了 yaml/spec/generated `.go` 主体，没有意识到测试文件还硬编码了一个枚举元素。
- 涉及代码: `backend/internal/shared/jobs/jobs_test.go:46`。
- 为什么之前没有发现:
  - `make codegen-check` 只比对 generated artifacts 与 truth source，不跑 Go test build；
  - `make lint`（包含 `make lint-events`）校验 `events.yaml/jobs.yaml` 一致性，不感知 Go test 文件；
  - `make test` 会暴露该错误，但 `9a30023` 落地后第一份后续 work-journal（2026-05-08 12:24 entry）已记下 “make lint 全绿（除预存 `JobTypeEmbeddingUpsert` 问题）”，意味着 `make test` 未被作为 cleanup commit 的强制 gate；
  - 该常量来源于 2026-04-30 commit `0af4a82` 添加的 internal-only subset 断言，距离 `9a30023` 8 天，跨主题导致回归。

## 4 修复方案

- 选择方案: 在 `jobs_test.go:46` 的内部 only 断言数组中删除 `JobTypeEmbeddingUpsert`，保留对 `JobTypeSourceRefresh` 与 `JobTypeEmailDispatch` 仍是 internal-only 的断言。
- 替代方案放弃理由:
  - “恢复 `embedding_upsert` 常量” 与 backend-targetjob/001 §6.3 active-scope 反向搜索冲突，且产品已转向 deepseek 不再需要 embedding；
  - “重启 B3 001-bootstrap 为 active” 仅为一行测试删除引入大量 plan/header/INDEX 同步成本，且违反 §4.4「禁止悬空原地修订」精神。
- 修改文件:
  - `backend/internal/shared/jobs/jobs_test.go`

## 5 验证

- 焦点测试: `go test ./internal/shared/jobs/ -count=1 -v` → `TestBuildEmailDispatchPayload` / `TestGeneratedJobMappings` 全部通过。
- 回归 gates: `make codegen-check`、`make validate-fixtures`、`make lint-events` 均为 OK。
- 主题反向 grep: `grep -rn 'JobTypeEmbeddingUpsert\|embedding_upsert\|embedding\.upsert\|EmbeddingUpsert'` 在 `--include='*.go'` `*.ts` `*.yaml` `*.json` `*.tmpl` `*.sql` 中已 0 命中（仅历史 work journal / report 中保留事件追溯说明）。
- 关联 commit: `cfd88e1 fix(event-outbox): drop stale embedding job reference in jobs_test`。

## 6 经验教训

- 通用化检查模式: 跨 owner 的能力 cleanup commit 必须把 “generated package 的 test consumer” 显式列入清理清单，不能只看 truth source 与 generated 主体；可加入 `/plan-code-review --fix` 对 cleanup 类提交的 reverse-grep 默认套件。
- 流程建议: `make codegen-check` 完成后应额外跑触达包的 `go test ./...` 子集，以避免 generated 主体清理但 test 没跟上。
- 工具建议: 在 `lint-events` 之外补一条 `lint-jobs-consumers` 静态检查，确保 `internal-only` 等 enum 集合断言均能在 `jobs.yaml` 找到对应 entry。

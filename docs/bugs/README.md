# Bug 记录规范

> 本文件定义 Bug 知识库的记录规范，供 Agent 和开发者在创建 Bug 记录时参考。

## 1 建档触发条件

满足任一条件时，应创建独立 Bug 记录：

- 需要两步及以上诊断推理才能定位根因
- 影响多个组件、多个环境或引发回归风险
- 需要沉淀为可复用检查模式（适合进入 `PATTERNS.md`）

可不建独立 Bug 记录的情况：

- 明显 typo / 配置笔误，且无额外排查成本
- 仅格式修复、无行为变化
- 可在 work-journal 一句话说明清楚且无复发风险

## 2 命名规范

- 文件名：`BUG-NNNN.md`，四位数字，从 0001 递增
- ID 分配：读取 `INDEX.md` 中最大 ID + 1，写入前再次校验目标文件不存在

## 3 模块枚举

`模块` 字段必须使用以下枚举值（不使用展示名如 `Workspace`、`Frontend`）：

| 枚举值 | 说明 |
|--------|------|
| `workspace` | 目标岗位工作台与 JD 导入 |
| `practice` | 模拟面试编排与练习会话 |
| `review` | 报告、逐题复盘、错题本 |
| `materials` | 简历改写、材料工坊、渐进式画像 |
| `debrief` | 真实面试复盘 |
| `eval` | Prompt/Rubric、离线评估与质量治理 |
| `frontend` | Web 前端 |
| `platform` | 配置、存储、鉴权、异步任务等平台能力 |
| `schema` | API / 数据模型 / 持久化契约 |
| `test` | 测试框架 |

## 4 严重度定义

| 值 | 说明 |
|----|------|
| `critical` | 系统不可用或数据损坏 |
| `high` | 核心功能受阻，无临时解决办法 |
| `medium` | 功能异常但有绕过方式 |
| `low` | 边缘场景或外观问题 |

## 5 状态定义与约束

| 值 | 说明 |
|----|------|
| `open` | 已发现，尚未开始诊断 |
| `investigating` | 正在诊断中 |
| `resolved` | 已修复并验证 |

**状态约束**：

- `open` / `investigating`：`关联 Commit` 可为 `-`
- `resolved`：必须补齐「根因分析」「修复方案」「验证」，且 `关联 Commit` 不能为 `-`

## 6 必填最小集

- 标题、模块、严重度、状态、发现日期
- 「现象」「诊断过程」至少各 1 条有效内容

## 7 脱敏规范

Bug 记录必须避免写入敏感信息：

- 不记录真实 token、密码、私钥、Cookie、完整连接串
- 需要示例时使用掩码（如 `ghp_***`、`postgres://user:***@host/db`）
- 日志和命令输出仅保留定位问题所需最小片段

# BUG-0051: getResume not-found fixture used undocumented error code

> 模块: schema | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-13 | 关联 Commit: `feat(backend-resume): close resume baseline verification`
> 关联日志: [2026-05-13](../work-journal/2026-05-13.md)

## 1 现象

- 在 backend-resume Phase 5 mock-first 收口时，`getResume` 只有 handler 200/404 行为测试，缺少 `openapi/fixtures/Resumes/getResume.json` 的 default / not-found fixture parity 测试。
- `getResume.not-found` fixture 使用 `error.code="NOT_FOUND"`，但当前 shared error enum / generated `ApiErrorCode` 不声明 `NOT_FOUND`。
- 如果不修复，frontend/mock-first consumer 可能继续依赖未声明错误码，而真实 backend handler 返回 `TARGET_JOB_NOT_FOUND`。

## 2 诊断过程

### 假设 1：真实 handler 使用了错误码
- 验证方法：读取 `backend/internal/resume/handler/get.go` 与 `backend/internal/shared/errors/codes.go`。
- 结果：排除。handler 对 `resume.ErrNotFound` 返回 `TARGET_JOB_NOT_FOUND`，且 shared enum 只声明该错误码，不声明 `NOT_FOUND`。

### 假设 2：fixture 与 shared error enum 漂移
- 验证方法：读取 `openapi/fixtures/Resumes/getResume.json` 并反查 `openapi/openapi.yaml` `ApiErrorCode` enum。
- 结果：确认。fixture 使用的 `NOT_FOUND` 不在当前 OpenAPI/shared error code 集合内。

### 假设 3：Phase 5 gate 未覆盖 getResume not-found fixture parity
- 验证方法：搜索 `TestGetResumeFixtureParity` 与 `FixtureParity` 覆盖。
- 结果：确认。已有 `registerResume` / `listResumes` fixture parity，缺少 `getResume` default / not-found parity。

## 3 根因分析

- 根本原因：resume additive fixture 中的 not-found 场景继承了旧通用 `NOT_FOUND` 口径，但 backend/shared error code 仍以现有 B1 枚举为真理源；backend-resume 早期 checklist 未要求 `getResume` fixture parity 测试，因此漂移没有在 Phase 1-4 暴露。
- 涉及路径：
  - `openapi/fixtures/Resumes/getResume.json`
  - `backend/internal/resume/handler/get.go`
  - `backend/internal/resume/handler/get_test.go`
  - `openapi/openapi.yaml`
- 为什么之前没有发现：mock-first parity 已覆盖 register/list，但 getResume not-found 只做了行为断言，没有把 fixture error body 纳入 regression gate。

## 4 修复方案

- 将 `openapi/fixtures/Resumes/getResume.json` 的 `not-found` 场景改为 `TARGET_JOB_NOT_FOUND` + backend handler 当前消息。
- 新增 `TestGetResumeFixtureParity`，覆盖 `default` 与 `not-found` 两个场景，并对 not-found response 做 error envelope parity。
- 将 P0.034 场景 trigger / verify 纳入 `TestGetResumeFixtureParity`，防止后续 fixture 漂移绕过 BDD gate。

放弃方案：

- 新增通用 `NOT_FOUND` error code：会扩大 B1 shared error enum，本次 backend-resume/001 不应擅自改变全局错误码契约。
- 修改 backend handler 返回 `NOT_FOUND`：会与当前 OpenAPI enum / shared generated code 冲突。

## 6 经验教训

- Any operation with fixture success and not-found variants needs explicit parity coverage for both status and error envelope, not only success DTO shape.
- When a fixture uses an error code, verify the literal is present in `shared/conventions.yaml` / generated `ApiErrorCode` before treating the fixture as current truth.

# BUG-0172: malformed report conversation ID reached the UUID database query

> 模块: review | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-15 | 关联 Commit: `feat(report): implement readonly conversation record (BUG-0172, BUG-0173)`
> 关联日志: [2026-07-15](../work-journal/2026-07-15.md)

## 1 现象

- 只读面试记录页面的 `reportId` 来自可篡改 URL；空值会被隐藏为未找到，但非空普通字符串、`tmp_` ID 或非 UUIDv7 会继续进入 repository。
- 生产 PostgreSQL 的 `feedback_reports.id` 是 UUID；非法字符串可能触发数据库类型错误并被 API 映射为 500，而不是设计要求的隐藏 404。
- 影响范围限于 report-owned conversation read；正常 UUIDv7 报告读取不受影响。

## 2 诊断过程

### 假设 1：前端已经阻止非法 reportId 请求

- 验证方法：反查 route normalization、`ReportConversationScreen` 与 generated client 调用。
- 结果：排除。前端会对非法响应 fail closed，但仍可能向后端发送非空非法 locator，后端必须独立防御。

### 假设 2：后端 service ingress 已验证 UUIDv7

- 验证方法：读取 `GetReportConversation` service 与项目共享 `idx.RequireServerID` 契约，并增加 repository spy 回归测试。
- 结果：确认缺口。Red 测试证明 `report-1` 会到达 repository 且返回空成功结果。

## 3 根因分析

- `GetReportConversation` 只对 `userID` / `reportID` 做 trim 和空值检查，没有使用项目已有的 server-ID ingress guard。
- repository SQL 直接把字符串参数与 UUID 列比较，因此非法客户端 locator 的错误语义被数据库驱动决定。
- 原有测试覆盖空 identity、跨用户、消息投影损坏和顺序异常，但 service 测试使用普通字符串假 ID，没有覆盖用户可控 locator 的类型边界。

## 4 修复方案

- 在 `backend/internal/review/get_feedback_report_service.go` 查询 repository 前调用 `idx.RequireServerID(reportID)`。
- 将非法、`tmp_` 和非 UUIDv7 locator 统一映射为 `ErrReportNotFound`，保持 hidden-404 语义且零 repository / DB read。
- 保持 OpenAPI、handler、store 和持久化结构不变；不新增兼容层或本地 UUID validator。
- 在 `backend/internal/review/list_target_job_reports_service_test.go` 使用 canonical UUIDv7 正例，并增加三类非法 locator 的 Red/Green 回归。

## 5 验证

- Red：focused service test 证明 `report-1` 穿透 repository。
- Green：同一测试证明普通字符串、`tmp_` ID、UUIDv4 均返回 `ErrReportNotFound` 且 repository 未被调用。
- 相邻回归：`backend/internal/review`、`backend/internal/api/reports`、`backend/internal/store/review`、`backend/cmd/api` 全部通过。
- 最终根 `make test` 通过（backend 552 tests / 5415 subtests，frontend 126 files / 1018 tests）；P0.099 current-run Chrome 真实环境回归也已 PASS。

## 6 经验教训

- URL/path 中的服务端资源 ID 必须在 service/handler ingress 使用共享 ID guard；不能把 PostgreSQL UUID cast 当作输入校验。
- 单测中的普通字符串假 ID 会掩盖生产类型边界；至少一个 service 正例应使用 canonical UUIDv7，并保留非法 locator 的 no-read 断言。

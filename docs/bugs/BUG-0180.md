# BUG-0180: production email review exposed delivery lifecycle and PID ownership gaps

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-07-16 | 关联 Commit: `fix(platform): harden smtp lifecycle and pid ownership (BUG-0180)`
> 关联日志: `-`

## 1 现象

- 分支代码审查发现 `email_dispatch` handler 丢弃 runner context，SMTP 仅限制建连时间；服务端在 greeting、TLS、AUTH、DATA 或 QUIT 阶段停滞时，job handler 与 graceful shutdown 都可能长期阻塞。
- SMTP DATA 已得到最终成功响应后，QUIT 失败仍返回 retryable error，runner 重试会再次发送同一验证码。
- delivery secret 写入 Redis 失败发生在 challenge 入库之后，失败请求会留下可计入 rate-limit 的 `auth_challenges` 记录。
- `dev-container-up` 只凭 pidfile 与 `kill -0` 判断进程归属；PID 被系统复用时可能终止不属于 easyinterview 的进程。

## 2 诊断过程

### 假设 1：现有 timeout 覆盖完整 SMTP 会话

- 验证方法：反查 runner context 传递链、`SMTPDeliveryWriter` 的 Redis/DB/SMTP 调用和 `net/smtp` 会话阶段，并用建连后不发送 greeting 的本地 TCP server 写 RED test。
- 结果：排除。旧实现使用 `context.Background()`，且 `net.Dialer.Timeout` 只约束建连；建连后的读取不会响应 runner cancel。

### 假设 2：DATA 成功后的 QUIT 失败可以安全重试

- 验证方法：用最小 SMTP server 接受完整 DATA、返回 `250 queued`，收到 QUIT 后直接断开，并观察 transport 返回值。
- 结果：排除。邮件已被服务器接受，旧实现仍把 QUIT EOF 映射为 retryable failure，存在重复投递窗口。

### 假设 3：pidfile 中的存活 PID 足以证明进程归属

- 验证方法：启动独立 `sleep` 进程，将其 PID 写入临时 backend pidfile，调用 `_stop_host_runtimes`。
- 结果：排除。RED test 观测到无关进程收到 `SIGTERM`。

## 3 根因分析

- `backend/internal/auth/email_dispatch_handler.go:83` 的旧签名忽略 runner context，`backend/internal/auth/smtp_delivery_writer.go` 又为 Redis、SQL 与 SMTP 使用后台 context；网络 I/O 生命周期未绑定 job 生命周期。
- `backend/internal/auth/smtp_delivery_writer.go:251` 的旧成功边界把 QUIT 当作投递事务的一部分，而 SMTP DATA 最终 `250` 响应才是消息被接受的边界。
- `backend/internal/auth/service.go:148` 的旧副作用顺序先创建 challenge、后写 delivery secret，且 challenge 创建失败没有 secret 补偿路径。
- `deploy/dev-stack/Makefile:50` 的旧 stop 逻辑只验证 PID 存活，不验证当前命令是否仍属于对应 repo-managed role。
- 原测试覆盖 TLS 配置、发送失败与 pidfile 字符串存在性，但缺少停滞会话取消、accepted-once、跨存储失败顺序和 PID 复用的真实进程测试。

## 4 修复方案

- 将 `context.Context` 纳入 `DeliveryWriter`、challenge email lookup 与 SMTP send 合同，由 runner handler 贯穿 Redis、SQL 和 transport。
- 为完整 SMTP 会话设置 30 秒上限与连接 deadline；runner cancel 立即推进 deadline，隐式 TLS 使用 context-aware handshake。
- DATA 最终成功后将 QUIT 视为 best-effort cleanup，不再返回 retryable error。
- 先写 delivery secret，再创建 challenge；Redis Put 失败时不留 challenge，challenge 创建或 enqueue 失败时用独立的 2 秒补偿 context 删除 secret。
- `_stop_host_runtimes` 在 TERM/KILL 前读取 `ps command`，仅接受当前 backend/frontend 启动命令；PID 非法、不匹配或不可读时只删除 pidfile。
- 未采用仅延长 dial timeout 或仅检查 `kill -0` 的方案，因为两者都没有覆盖对应生命周期/归属边界。

修改范围：

- `backend/internal/auth/{mail.go,email_dispatch_handler.go,smtp_delivery_writer.go,service.go}`
- `backend/internal/auth/*_test.go`
- `deploy/dev-stack/Makefile`
- `scripts/lint/scenario_env_contract_test.py`
- `docs/spec/backend-auth/`、`docs/spec/local-dev-stack/`

## 6 经验教训

- 异步 handler 必须把 runner context 传到所有下游 I/O；仅有 dial timeout 不等于完整协议会话有界。
- 重试边界必须以远端副作用的提交点定义；SMTP DATA 最终成功后不能因为连接清理失败重复业务操作。
- 跨存储写入必须先安排不可见/可过期资源，再发布会影响限流或消费的事实，并为后续失败提供有界补偿。
- pidfile 只能证明“曾经拥有”；发送信号前还必须校验当前进程身份，并用真实无关进程做负向测试。

# BUG-0068: debrief real backend flows were blocked by mock-only contract drift

> 模块: debrief | 严重度: high | 状态: resolved
> 发现日期: 2026-05-17 | 关联 Commit: `fix(frontend-debrief): repair real backend debrief flows (BUG-0068)`
> 关联日志: [2026-05-17](../work-journal/2026-05-17.md)

## 1 现象

- L2 review found the debrief happy path could pass fixture-backed tests while failing against the real runtime.
- Step 0 AI/manual entries submitted blank `myAnswerSummary`, so `createDebrief` would be rejected by the backend validation contract.
- Step 2 replay forwarded a completed mock session ID or navigated without any `sessionId`, so the practice route could not start a usable debrief practice session.
- The mock-session picker called generated `listPracticeSessions`, but the real backend did not register or implement `GET /api/v1/practice/sessions`.
- JD selection filtered `status=ready` instead of `analysisStatus=ready`; resume re-selection skipped the asset list and could clear the selected asset.
- When suggestions failed or returned empty, manual entry was not reachable, leaving the user unable to submit any debrief input.

## 2 诊断过程

### 假设 1：frontend entry mapper violated backend createDebrief requirements
- 验证方法：read `useSubmitDebrief.ts`, `GuidedDebriefRecord.tsx`, and the debrief create contract behavior called out by review.
- 结果：确认。Step 0 entries only carried `questionText` and `source`; `entriesToQuestionInputs` filled `myAnswerSummary` with an empty string.

### 假设 2：replay CTA reused source session context instead of creating a debrief session
- 验证方法：trace `DebriefScreen.handleStartReplay` into practice route params and compare with practice loader requirements.
- 结果：确认。The CTA either omitted `sessionId` or reused the completed mock session ID instead of calling `createPracticePlan` and `startPracticeSession`.

### 假设 3：generated and fixture artifacts were ahead of real backend runtime wiring
- 验证方法：compare generated client/server operations, fixture files, and `backend/cmd/api/main.go` route registration.
- 结果：确认。OpenAPI/generated artifacts had `listPracticeSessions`, but backend handler/service/store and route registration were absent.

### 假设 4：picker filters and fallback UI used stale assumptions
- 验证方法：read `JDPicker.tsx`, `ResumePicker.tsx`, and `GuidedDebriefRecord.tsx` against current target job and resume API semantics.
- 结果：确认。JD picker used lifecycle `status` for parse readiness, resume picker initialized directly into version phase, and manual add lived only inside the suggestion-current branch.

## 3 根因分析

- The implementation trusted generated client and fixture coverage as runtime evidence without requiring matching backend route/service/store wiring.
- The Step 0 UI treated a question occurrence as sufficient debrief input, but the backend contract requires an answer summary for every question.
- The replay CTA treated debrief replay like route handoff, while the current practice screen requires a real started practice session.
- Existing tests mocked successful responses and did not assert real backend availability, required answer summaries, or picker recovery states.

## 4 修复方案

- Added guided-entry answer summary capture and disabled debrief submission until every recorded entry has a non-empty summary.
- Kept manual question entry available when suggestions fail or are empty.
- Changed debrief replay to create a `goal=debrief` practice plan with `sourceDebriefId`, start a fresh practice session, merge plan/session context, and navigate with the new `sessionId`.
- Changed JD picker filtering to `analysisStatus=ready`.
- Changed resume picker to always reopen on the asset list and preserve the selected asset object for version confirmation.
- Implemented real `GET /api/v1/practice/sessions` backend route, handler, service, store query, cursor envelope, and scenario fake-store support.
- Added focused frontend and backend regression tests for the reviewed failure modes.

## 6 经验教训

- Generated client, fixtures, and mocks are not sufficient evidence for runtime readiness; review must trace the handler/service/store route for every newly consumed operation.
- UI tests for contract-backed submit flows must assert required backend fields, not only successful mocked navigation.
- Replay CTAs must start a new practice session before route handoff whenever the destination route requires `sessionId`.

## 1 现象

- 从 `7e6ec756d91280b9046d34f362a3cde58bbf2d29` 之后的实现做 deep reconcile 时，仍能在场景数据、verify 脚本、fixture README、frontend 测试和 backend e2e helper 中搜到旧 `resumeVersionId`、`resumeAssetId`、`getResumeVersion`、`topbar-nav-jd_match`、`五入口` / `Five primary` 口径。
- `E2E.P0.035` 初始因本地 migration 未运行而跳过；执行 `test/scenarios/env-setup.sh --with-migrations` 后，真实失败暴露为 Postgres JSONB 重新序列化导致的字符串精确比较误判。
- `E2E.P0.016` 的 Playwright evidence 已输出 `resumeId`，但 verify gate 仍要求旧 `resumeVersionId`。
- `E2E.P0.098` full-funnel backend gate 仍使用旧 auth challenge token 格式和已删除的 `practice_plans.resume_asset_id` 字段。
- `openapi/fixtures/README.md` 仍记录旧 60 operation 计数和 `/resume-versions/{resumeVersionId}/exports` 路径。

## 2 诊断过程

### 假设 1：失败主要来自本地环境未迁移

- 验证方法：读取 `test/scenarios/README.md` 与 e2e suite README，执行 `bash test/scenarios/env-setup.sh --with-migrations`，然后重跑 P0.035 setup/trigger/verify。
- 结果：部分确认。migration 缺失会让 P0.035 跳过，但环境恢复后实际失败来自 JSONB 断言，而不是 schema 缺失。

### 假设 2：残留是运行时代码回归

- 验证方法：反查 OpenAPI/generated artifact、backend route catalog、frontend route normalization、pixel-parity、scenario trigger/verify 和负向搜索结果。
- 结果：排除主要运行时代码回归。运行时代码已基本落在新 D-20 flat resume / D-17 四入口 / D-18 workspace 嵌入口径，残留集中在场景资产、测试 helper、README 计数和少量路由别名。

### 假设 3：backend full-funnel helper 仍绑定旧 schema / auth contract

- 验证方法：运行 P0.098 trigger，读取 `backend/cmd/api/full_funnel_journey_scenario_test.go` 和 `backend/cmd/api/full_funnel_browser_server_test.go`。
- 结果：确认。测试 helper 使用非 6 位验证码，并查询已删除的 `resume_asset_id` 列；browser server state 仍暴露旧字段名。

## 3 根因分析

- 根本原因：D-17/D-18/D-20 多层 refactor 已覆盖主要 runtime，但是 deep reconcile 没有完整覆盖 scenario data、expected outcome、verify shell、fixture README、backend e2e helper SQL、JSONB 比较方式和 frontend pixel/e2e 测试命名。
- 次级根因：部分 gate 只检查历史 marker 或字符串序列化结果，没有把当前业务语义作为断言对象。JSONB 比较依赖原始字符串格式，场景 verify 依赖旧字段名，fixture README 依赖人工维护计数。
- 涉及路径：
  - `test/scenarios/e2e/*`
  - `frontend/src/app/*`
  - `frontend/tests/*`
  - `backend/cmd/api/full_funnel_journey_scenario_test.go`
  - `backend/cmd/api/full_funnel_browser_server_test.go`
  - `backend/internal/resume/store/assets_integration_test.go`
  - `openapi/fixtures/README.md`
  - `scripts/lint/*`
  - `scripts/mock_contract/*`
- 之前没有发现，是因为历史 checklist / PASS 状态覆盖了主路径验证，但没有在完成后重新对当前产品不变量执行全仓负向搜索和场景脚本级反查。

## 6 经验教训

- 大规模删除 / 重命名 refactor 的 deep reconcile 必须覆盖 scenario data、expected outcome、verify scripts、fixture README、backend e2e helper SQL 和 frontend pixel/e2e specs，不能只看 runtime 代码。
- 负向搜索需要区分 live-contract 残留和 intentional negative assertion；最终 allowlist 应只剩负向断言。
- JSONB / map 类结构断言应使用语义比较，避免被数据库重新序列化格式制造 false failure。
- 场景 PASS gate 应优先绑定当前字段语义和本轮 evidence，而不是历史 marker。

# BUG-0107: review follow-up exposed provider schema, report status, and privacy retry drift

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-26 | 关联 Commit: `fix(backend): close review correctness follow-ups (BUG-0107)`
> 关联日志: [2026-05-26](../work-journal/2026-05-26.md)

## 1 现象

- Review 指出 OpenAI-compatible adapter 在所有 `OutputSchema` 存在时都发送 `response_format={"type":"json_object"}`，会与 `jd_match.search` / `jd_match.recommendation` 的顶层 array schema 冲突。
- Review 指出报告逐题评估 fallback 聚合在 `overall_status` 缺失或非法时依赖 Go map 遍历顺序；mixed dimension 中 `needs_work` 可能被 `meets_bar` 覆盖。
- Review 指出 privacy delete runner 在 completion transaction 已提交、但 async job 尚未 finalize 前崩溃时，retry 会遇到 `privacy_requests.user_id=NULL` tombstone，无法把已完成 cleanup 幂等标记为成功。

## 2 诊断过程

### 假设 1：provider request builder 没有区分 schema 顶层类型

- 验证方法：读取 `backend/internal/ai/aiclient/providers/openai_compatible/adapter.go` 和 provider contract tests，新增 array schema request capture regression。
- 结果：确认。现状下 `TestComplete_LeavesResponseFormatUnsetForArrayOutputSchema` 失败，request 仍包含 `map[type:json_object]`。

### 假设 2：报告 fallback 聚合存在顺序依赖

- 验证方法：读取 `backend/internal/review/generate_report.go` 的 `overallStatusFromDimensionResults`，新增 mixed statuses regression 循环验证 `needs_work` dominance。
- 结果：确认。现状下测试返回过 `meets_bar`，证明 map iteration 顺序可影响结果。

### 假设 3：completed tombstone 缺少 retry-idempotent 信号

- 验证方法：读取 `backend/internal/privacy/runner/store.go` / `delete_handler.go`，为 `user_id=NULL,status=completed` 增加 SQLStore 和 handler regression。
- 结果：确认。现状下缺少 `ErrPrivacyDeleteAlreadyCompleted` sentinel，handler 会把 lookup failure 视作 retryable。

## 3 根因分析

- OpenAI-compatible adapter 只把 `OutputSchema` 是否存在作为 provider JSON mode 开关，没有读取 schema 顶层 `type`。
- `overallStatusFromDimensionResults` 在遍历 map 时遇到 `meets_bar` 立即返回，未先扫描所有 dimension 中是否存在更差状态。
- `privacy_requests.user_id` tombstone migration 让 completed request 可以保留审计行，但 lookup API 仍假设 `user_id` 永远非空，缺少“已完成且无需重跑 cleanup”的终态表达。
- 之前未发现的原因是测试只覆盖 object schema request、单一或顺序无关的 report statuses，以及首次执行 privacy delete completion，没有覆盖 crash-after-commit retry。

## 4 修复方案

- `backend/internal/ai/aiclient/providers/openai_compatible/adapter.go`：解析 output schema 顶层 `type`，仅在 `type=="object"` 时发送 `json_object`，array schema 不发送 object mode。
- `backend/internal/review/generate_report.go`：先扫描全部 dimension status，让 `needs_work` 优先于 `meets_bar`，只有全 strong 时返回 `strong`。
- `backend/internal/privacy/runner/store.go` / `delete_handler.go`：新增 `ErrPrivacyDeleteAlreadyCompleted`，当 `privacy_requests.user_id` 为 NULL 且 `status='completed'` 时让 handler 直接返回成功，避免重跑 domain cleanup 或 dead-letter 已完成任务。
- 同步更新 `backend/cmd/api/main_test.go` 的 SQL mock column contract。

## 6 经验教训

- Provider request assertions must capture the wire payload for every supported top-level output schema shape, not just object schemas.
- Status aggregation helpers should encode dominance explicitly; map iteration must never decide user-visible terminal status.
- Tombstone migrations need a paired retry-idempotence contract at the lookup/handler boundary, especially when async job finalization happens after domain cleanup.

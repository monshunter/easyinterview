# BUG-0039: frontend shell auth state and user menu parity drift

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-10 | 关联 Commit: `fix(frontend-shell): restore auth state and user menu parity`
> 关联日志: `-`

## 1 现象

- 用户在 `localhost:5173` 对比正式前端与 `ui-design` 静态原型时发现：已登录态用户区渲染为常驻的 `用户画像` / `设置与隐私` / `退出登录` 三个文本按钮，而 `ui-design/src/app.jsx` 是头像 chip + dropdown。
- `退出登录` 无法真正回到非登录态；刷新或重新读取 `/me` 后仍显示 authenticated 用户区。
- Vite dev 默认 fixture-backed mock 首屏缺少可靠的非登录态闭环，导致“已登录 / 未登录 / 登录后 / 登出后”全流程不能由当前 mock 数据覆盖。

## 2 诊断过程

### 假设 1：该能力还没进入已实施计划

- 验证方法：反查 `docs/spec/frontend-shell/plans/001-app-shell-auth-settings/plan.md`、`checklist.md`、`002-app-shell-visual-system` 与 `003-ui-design-pixel-parity-gate`。
- 结果：排除。`001` 已要求未登录显示登录/注册、已登录显示用户菜单并分流 profile/settings/logout；`002` 已要求用户菜单按钮对齐 `ui-design/src/app.jsx`；`003` 已标记 pixel parity gate completed。

### 假设 2：正式前端 TopBar 实现没有源级复刻用户菜单结构

- 验证方法：读取 `frontend/src/app/topbar/TopBar.tsx`、`topbar.css`、`TopBar.test.tsx`，并对照 `ui-design/src/app.jsx`。
- 结果：确认。正式前端旧实现把已登录用户区直接渲染成三个 inline 文本按钮，测试只断言文字存在，没有断言头像 chip、dropdown header、email、backdrop、菜单关闭行为和旧 inline 结构负向约束。

### 假设 3：logout 无效来自 generated client 或后端契约缺失

- 验证方法：读取 `frontend/src/api/devMockClient.ts`、`frontend/src/api/mockTransport.ts`、Auth fixtures 与 `AppRuntimeProvider` 刷新 `/me` 的调用路径。
- 结果：确认根因在 dev mock。`createDevMockClient()` 原先只是无状态 fixture transport，`getMe` 默认场景为 authenticated；`logout` 返回成功后没有改变后续 `/me` 响应，所以 App 会重新进入 authenticated。

## 3 根因分析

- 根本原因：D1 frontend-shell 的实现和验证把 Auth operation fixture 当成单次响应，而没有建模 session state；同时用户菜单测试只验证 route 文案存在，未验证 `ui-design` 的交互结构。
- 涉及路径：
  - `frontend/src/app/topbar/TopBar.tsx`
  - `frontend/src/app/topbar/topbar.css`
  - `frontend/src/api/devMockClient.ts`
  - `frontend/src/app/App.tsx`
  - `frontend/src/app/topbar/TopBar.test.tsx`
  - `frontend/src/api/devMockClient.test.ts`
  - `frontend/src/app/AppAuthDispatch.test.tsx`
  - `frontend/src/app/scenarios/p0-032-dev-mock-auth-state-and-user-menu.test.tsx`
- 之前没有发现的原因：
  - 已完成 checklist 的 gate 没有覆盖“默认非登录 -> 登录 -> dropdown 分流 -> logout -> 非登录”的连续状态流。
  - pixel parity 的真实浏览器覆盖重点在 TopBar 常规结构、auth shell 和 layout，未覆盖已登录用户 dropdown。
  - `screens.spec.ts` 将 profile/settings 的真实浏览器覆盖转交给 jsdom smoke，导致浏览器层无法捕捉已登录用户菜单结构漂移。

## 4 修复方案

- 选择方案：在原 owner plan `frontend-shell/001-app-shell-auth-settings` 原地修订 Phase 6，补齐源级复刻和 stateful dev mock 两条 remediation，再按 TDD/BDD 落地代码与场景。
- 主要修改：
  - `TopBar` 已登录用户区改为头像 chip；点击后显示 dropdown，包含姓名、脱敏邮箱、用户画像、设置与隐私、退出登录、backdrop 与 Escape 关闭行为。
  - `App` 把 runtime user 传入 TopBar，避免 TopBar 使用硬编码用户展示。
  - `createDevMockClient()` 维护本地 `signedIn` 状态：默认 `/me` 为 unauthenticated，`verifyAuthEmailChallenge` 成功后变 authenticated，`logout` 成功后变 unauthenticated；显式 `Prefer` 场景仍保留 fixture override。
  - 新增 `E2E.P0.032` 场景资产和 Vitest 场景，覆盖 dev mock auth state 与用户菜单分流闭环。
- 放弃方案：只在 TopBar 内隐藏/显示旧三按钮。该方案无法对齐 `ui-design` 头像 dropdown，也不能修复 logout 后 `/me` 仍 authenticated 的根因。

## 6 经验教训

- Stateful 用户流程不能只用单 operation fixture 验证；需要在 mock runtime 中显式建模 session transition。
- UI parity gate 不能只断言文本存在；对 dropdown、modal、popover 这类交互控件，应同时断言关闭状态、打开状态、header/body/动作项、backdrop/Escape、点击后分流和旧结构负向约束。
- 完整 Playwright pixel suite 还暴露了相邻测试债务：workspace hydrated pixel tests 依赖无效的 `resume-unbound` 路由上下文，且本地 Darwin screenshot baseline 未入库。该问题不属于本次 auth 修复范围，但应由 workspace/pixel owner 后续修订。

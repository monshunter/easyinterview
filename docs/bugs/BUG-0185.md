# BUG-0185: resume tailor task run succeeded before business output was accepted

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-17 | 关联 Commit: `fix(ai): close review findings (BUG-0185)`
> 关联日志: [2026-07-17](../work-journal/2026-07-17.md)

## 1 现象

- `resume_tailor` 的 provider 返回 `{"suggestions":[]}` 时，observability decorator 将唯一 `ai_task_runs` 行写成 `success/ok`。
- handler 随后由 `decodeTailorAIResponse` 拒绝该输出，并让 async job 以 `AI_OUTPUT_INVALID` 失败。
- 同一次 AI 调用因此同时具有成功 task run 和失败业务 job，影响诊断、指标与审计可信度。

## 2 诊断过程

### 假设 1：移除 handler 手写 task run 后遗漏了失败写入

- 验证方法：反查 `TailorHandlerOptions`、`buildResumeRuntime` 与 observability decorator 的 writer 注入。
- 结果：排除。decorator 是有意保留的唯一 writer；恢复 handler writer 会重新引入重复行和双 owner。

### 假设 2：decorator 与业务 decoder 的终态边界不一致

- 验证方法：对比 `resume.tailor.bullet_suggestions` output schema 与 `decodeTailorAIResponse` / `normalizeSuggestions` 的拒绝条件，并运行空数组、空白 bullet 的 decorator integration test。
- 结果：确认。schema 允许空数组和空白字符串，decorator 因此先写 success；decoder 后续才判为不可用。

## 3 根因分析

- `config/prompts/resume.tailor.bullet_suggestions/v0.1.0.schema.json` 只声明字段存在与类型，没有声明至少一条 suggestion，也没有拒绝只含空白的 `originalBullet` / `suggestedBullet`。
- `backend/internal/ai/aiclient/observability.Decorator.Complete` 在 provider/schema validation 完成后持久化 task run；`backend/internal/resume/jobs.decodeTailorAIResponse` 的业务语义校验发生在此之后。
- Phase 16 只验证了重复 writer 已移除，没有覆盖“schema 可接受但 decoder 拒绝”的终态一致性负例。

## 4 修复方案

- 保持 observability decorator 为唯一 `ai_task_runs` writer，不恢复 handler 手写行。
- 把 bullet suggestions schema 收紧为 `minItems: 1`，并用 `minLength` + non-whitespace `pattern` 对齐 decoder 的 original/suggested bullet 约束。
- 在 `TestTailorHandlerModeRoutingAndFailurePaths` 增加空数组与空白 bullet 回归，直接断言唯一 task-run 行为 failed 且业务不落 ready/outbox。
- 同批修复 `backend/internal/review/regenerate_report.go` 的 S1016，恢复全量静态分析零输出。
- 放弃“post-decode 再更新 task run”方案，因为它会让业务 handler重新拥有持久化终态，并引入第二次写入失败与竞态窗口。

## 5 验证

- RED：两个新增 case 均复现 `JobOutcome=AI_OUTPUT_INVALID`、`ai_task_runs.status=success`。
- GREEN：focused/all resume jobs tests、`make lint-prompts`、focused review tests、`go vet ./...`、`staticcheck ./...` 均通过。
- 全量：根 `make test`（Python 584/4583、全部 Go package、frontend 126 files/1027 tests）、`make build`、`make lint`、context/docs/index gates 全部通过。

## 6 经验教训

- 单写者只解决重复持久化问题，不能自动保证写入时点就是业务终态；schema validator 与下游 decoder 必须共享接受集合。
- 对每个 decorator-owned task run，至少保留一个“结构合法但业务不可用”的负例，并同时断言 task run、业务 job 与副作用终态。

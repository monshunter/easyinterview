# BUG-0112: auth mail-link login stalled on empty 202 response and backend verify link

> 模块: frontend | 严重度: high | 状态: resolved
> 发现日期: 2026-05-27 | 关联 Commit: `fix(auth): restore mail-link login flow (BUG-0112)`
> 关联日志: `-`

## 1 现象

- 用户在 `/auth/login` 或 `/auth/register` 提交邮箱后，页面没有反馈，也没有进入可输入 token 的验证入口。
- 浏览器控制台出现多条 `Unexpected end of JSON input`，堆栈指向 generated `EasyInterviewClient.request` 和 auth screen submit handler。
- 本地 Mailpit 邮件链接指向 backend `GET /api/v1/auth/email/verify?token=...`，点击后离开 frontend App，无法完成前端登录态刷新和 pendingAction 恢复。

## 2 诊断过程

### 假设 1：backend `startAuthEmailChallenge` 返回体与 generated client 不匹配

- 验证方法：检查 `backend/internal/auth/handler.go`、OpenAPI Auth 202 response、generated TS client 和 frontend auth submit 测试。
- 结果：确认。backend 对 `POST /api/v1/auth/email/start` 返回 `202 Accepted` 且无 body；generated TS client 对所有非 204 成功响应执行 `response.json()`，因此空响应触发 JSON parse error，阻断后续 UI 状态更新。

### 假设 2：本地 Mailpit 链接没有回到 frontend callback

- 验证方法：检查 `deploy/dev-stack/.env.example`、`config/config.yaml`、SMTP writer test、P0.100 hybrid runbook 与用户截图中的邮件链路。
- 结果：确认。本地默认 `EMAIL_VERIFY_BASE_URL` 指向 backend API verify URL；即使用户收到邮件，也不会回到 `/auth/verify` 页面完成 generated client verify、session refresh 和 URL token 清理。

### 假设 3：端口默认值分散硬编码导致 callback、CORS 和 API base 漂移

- 验证方法：搜索 `5173` / `8080` / CORS allowlist / frontend real API base。
- 结果：部分确认。backend dev CORS 使用固定 frontend origins，frontend real mode 暗含 backend 8080 默认。修复中将 CORS origin 改为从 `EMAIL_VERIFY_BASE_URL` 派生，并要求 dev real mode 显式设置 `VITE_EI_API_BASE_URL`。

## 3 根因分析

- 根本原因是前后端真实运行契约没有覆盖 `202 no-content` 成功响应：fixture 曾用 `{}` 代表 `startAuthEmailChallenge` 202，掩盖了真实 backend 空 body。
- 第二个根因是本地邮件回调 URL 把 backend API endpoint 暴露给用户点击，而不是进入 frontend `auth_verify` callback，由 frontend 统一兑换 session 和清理 token。
- 涉及路径：
  - `openapi/templates/ts/client.tmpl`
  - `frontend/src/api/generated/client.ts`
  - `frontend/src/app/auth/AuthVerifyScreen.tsx`
  - `frontend/src/app/App.tsx`
  - `frontend/src/app/routeUrl.ts`
  - `backend/cmd/api/main.go`
  - `backend/internal/auth/smtp_delivery_writer.go`
  - `deploy/dev-stack/.env.example`
  - `config/config.yaml`
  - `openapi/fixtures/Auth/startAuthEmailChallenge.json`
- 之前没有发现，是因为 mock fixture 返回 `{}`，而不是空 body；前端测试只覆盖了 mock 成功路径，没有覆盖真实 backend 空响应和 Mailpit callback URL。

## 4 修复方案

- generated TS client 改为先读取 response text；`204` 或空 text 直接视为成功 `undefined`，非空成功响应再 `JSON.parse`。
- `auth_verify` 允许一次性 `token` query，进入页面后自动调用 `verifyAuthEmailChallenge`，成功后使用 replace 导航恢复 pending route 或 Home，并清理 URL token；手动 token 输入保留为 fallback。
- 本地 `EMAIL_VERIFY_BASE_URL` 默认改为 frontend `/auth/verify` callback；SMTP writer 文案、dev-stack runbook、P0.100 材料和 config/spec 同步。
- backend dev CORS allowlist 从 `EMAIL_VERIFY_BASE_URL` 解析 frontend origin；frontend dev real mode 必须显式配置 `VITE_EI_API_BASE_URL`，避免端口默认分裂。
- `startAuthEmailChallenge` fixture 改为真实无 body，并刷新 fixture examples / mock-contract gate。

## 6 经验教训

- 对 `202 Accepted` / `204 No Content` / empty-body success 必须使用真实空 body fixture 或专门 client test；不能用 `{}` 代替 no-content。
- 邮件 magic-link 的用户可点击 URL 应回到 frontend callback，backend API verify endpoint 只应由 frontend generated client 调用。
- 本地 callback origin、CORS origin 和 frontend API base 必须来自同一 env/config 口径，不能在多个 runtime 入口分散写默认端口。

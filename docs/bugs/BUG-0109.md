# BUG-0109: scenario environment lifecycle was coupled to individual scenario runners

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-27 | 关联 Commit: `fix(test): decouple scenario env lifecycle (BUG-0109)`
> 关联日志: [2026-05-27](../work-journal/2026-05-27.md)

## 1 现象

- 用户需要通过 skill 独立启动、清理、重新编译和部署测试环境或本地前后端联调环境，用于人工验证或 Agent 验证特定场景。
- 当前仓库只有 `make dev-*` 和具体 `test/scenarios/e2e/p0-*` 场景脚本；`test/scenarios/env-setup.sh` / `env-verify.sh` / `env-cleanup.sh` / `env-redeploy.sh` 顶层环境入口不存在。
- `/scenario-env` 只能描述 setup / verify / cleanup / status，`/scenario-redeploy` 虽提到 `env-redeploy.sh`，但仓库没有实际入口，导致环境管理与具体场景执行耦合，无法只构建环境后交给人工或 Agent 做目标验证。

## 2 诊断过程

### 假设 1：已有 framework-owned env scripts，只是 skill 未优先调用

- 验证方法：检查 `test/scenarios/README.md`、`test/scenarios/e2e/README.md`、`find test/scenarios -maxdepth 3 -type f -name 'env-*.sh'`。
- 结果：排除。README 只把 env scripts 写成约定路径，实际不存在顶层 `env-*.sh`。

### 假设 2：根 Makefile 已经有 scenario-env lifecycle target

- 验证方法：反查根 `Makefile` 的 phony target 与 `make -n scenario-env-*`。
- 结果：排除。根 Makefile 只有 `dev-up` / `dev-down` / `dev-doctor` / `dev-reset` / `dev-logs` / `dev-pull`，没有 scenario environment 级别的 setup/status/verify/cleanup/redeploy 入口。

### 假设 3：skill 层已能表达 rebuild/redeploy 的 host-run 边界

- 验证方法：读取 `.agent-skills/scenario-env/SKILL.md` 与 `.agent-skills/scenario-redeploy/SKILL.md`。
- 结果：确认问题。`scenario-env` 未覆盖 rebuild/redeploy，也未列出 `env-status.sh` / `env-redeploy.sh`；`scenario-redeploy` 没有当前 repo 的 `deps|backend|frontend|all` host-run 目标边界。

## 3 根因分析

- 根本原因：历史设计只把 dev-stack 外部依赖生命周期落在 `make dev-*`，把 BDD 执行生命周期落在具体场景 `setup/trigger/verify/cleanup`；缺少中间层的 framework-owned shared environment lifecycle。
- 涉及路径：
  - `test/scenarios/README.md`
  - `test/scenarios/e2e/README.md`
  - `.agent-skills/scenario-env/SKILL.md`
  - `.agent-skills/scenario-redeploy/SKILL.md`
  - `Makefile`
  - `docs/spec/local-dev-stack/plans/001-bootstrap/*`
- 之前没有发现，是因为场景验收通常从具体 `p0-*` wrapper 进入，验证目标是场景 PASS，而不是“只构建共享环境并暂停给人工/Agent 使用”。

## 4 修复方案

- 选择方案：在 `local-dev-stack/001-bootstrap` 原地追加 Phase 6，把共享环境 lifecycle 固化为顶层 `test/scenarios/env-*.sh` + 根 `scenario-env-*` Make target + skill 入口。
- 新增 `test/scenarios/env-setup.sh` / `env-status.sh` / `env-verify.sh` / `env-cleanup.sh` / `env-redeploy.sh`：
  - setup/status/verify/cleanup 独立于具体场景目录。
  - cleanup 默认保留命名卷，显式 `--with-volumes` 才 reset。
  - redeploy 支持 `deps|backend|frontend|all`，当前 host-run 口径下刷新依赖和 build artifacts，不启动长驻 backend/frontend 进程。
- 更新根 `Makefile`，新增 `scenario-env-setup` / `scenario-env-status` / `scenario-env-verify` / `scenario-env-cleanup` / `scenario-env-redeploy`。
- 更新 `scenario-env` / `scenario-redeploy` skill，使其优先使用顶层 env scripts，并区分 Docker Compose 外部依赖、host-run backend/frontend 命令和 repo-tracked scenario runner。
- 更新 `test/scenarios/README.md`、`test/scenarios/e2e/README.md`、`deploy/dev-stack/README.md`，明确共享环境入口与具体场景 runner 的边界。

- `test/scenarios/env-setup.sh && test/scenarios/env-verify.sh && test/scenarios/env-cleanup.sh`：dev-stack 4 个 dependency 全 OK，cleanup 停止容器并移除 network，保留命名卷。
- `test/scenarios/env-redeploy.sh backend && test/scenarios/env-redeploy.sh frontend`：backend `go build ./cmd/...` 与 frontend production build 通过。
- `git diff --check`：通过。

## 6 经验教训

- 环境生命周期 gate 不能只嵌在具体 BDD 场景 wrapper 内；需要有 framework-owned 顶层入口，支持开发者或 Agent 只管理环境。
- Skill 文档如果声明可以 redeploy/rebuild，仓库必须有对应 repo-tracked entrypoint，否则 skill 只能给出不可执行建议。
- 本地联调环境应持续区分三层：Docker Compose 外部依赖、host-run backend/frontend 进程、具体 scenario runner。

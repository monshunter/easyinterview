# BUG-0090: frontend owner scenario wrappers widened scoped gates and ignored hash routes

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-23 | 关联 Commit: `fix(test): close frontend owner scenario full-run gates (BUG-0090)`
> 关联日志: [2026-05-23](../work-journal/2026-05-23.md)

## 1 现象

- `/scenario-run` 全量执行 frontend owner gate 场景时，P0.066 / P0.067 在 trigger 阶段失败，日志显示实际运行了 213 个 frontend Vitest 文件，而不是场景 README 声明的 scoped debrief 文件。
- P0.087 在 Playwright pixel parity 阶段失败，`resume-branch-flow-form` 和 `resume-rewrites-tab` 一直不可见，desktop / mobile 四个用例均超时。
- 影响范围覆盖 `frontend-debrief` P0.065-P0.069 wrapper 的证据可信度，以及 `frontend-resume-workshop` P0.087 的 pixel route harness。

## 2 诊断过程

### 假设 1：real backend gate 新增检查导致失败

- 验证方法：读取 `.test-output/runs/scenario-run-20260522T172049Z/e2e/p0-066-*/trigger.log`、P0.067 trigger log 和 P0.087 trigger log。
- 结果：排除。P0.066 / P0.067 的失败点是无关的 `DebriefPickerRegression.test.tsx`；P0.087 的失败点是 route harness 没有进入目标 DOM。

### 假设 2：debrief wrapper 没有真正执行 scoped runner

- 验证方法：读取 P0.065-P0.069 `scripts/trigger.sh`，对比 trigger log 中的实际 command。
- 结果：确认。脚本使用 `pnpm --filter @easyinterview/frontend test -- --run ...`，而 package script 已经是 `vitest run`，导致最终 command 形如 `vitest run -- --run ...`，扩大成全 frontend suite。

### 假设 3：P0.087 hash route 被 canonical URL 优先级吞掉

- 验证方法：读取 `frontend/src/app/routeStore.ts` 的 initial route 优先级和 `frontend/tests/pixel-parity/resume-workshop-branch-rewrites-edit.spec.ts` 的 `gotoHashRoute`。
- 结果：确认。测试 URL 为 `/?pixelRoute=N#route=resume_versions...`；`routeStore` 将非空 search 视为 meaningful canonical location，先解析 `/?pixelRoute=N`，不会再走 hash adapter。

## 3 根因分析

- debrief scenario wrapper 混用了 package script 参数透传与 Vitest 自身参数，`-- --run` 让 scoped 文件列表没有成为 runner 的真实过滤条件。
- P0.087 为了刷新 pixel route 加了 query 参数，但当前 URL 路由实现规定 canonical path/search 优先于 hash adapter；带 query 的 hash route 不再是合法 pixel harness 入口。
- 之前代表性抽样没有覆盖 P0.066 / P0.067 / P0.087 的失败组合，因此没有及时暴露 full-run 证据缺口。

## 4 修复方案

- P0.065-P0.069 trigger 统一改为 `pnpm --filter @easyinterview/frontend exec vitest run --reporter=verbose ...`，确保场景 README 声明的 scoped 测试文件是真实 runner 输入。
- P0.066 / P0.068 verify 增加关键 test file 名称断言，防止宽泛 `Tests passed` 重新变成假阳性。
- P0.087 `gotoHashRoute` 改为 `/#route=resume_versions...&pixelRoute=N`，保持 path/search 为空，让 hash adapter 按设计解析目标 route。

## 5 验证

- `bash -n` 覆盖修改过的 debrief trigger/verify 与 P0.087 trigger。
- 修复后受影响场景复跑通过：`scenario-run-fixcheck-20260522T172739Z`，P0.065-P0.069 + P0.087 全部 PASS。
- 当前工作树完整目标清单复跑通过：`scenario-run-full-20260522T172858Z`，P0.018-P0.021、P0.044-P0.047、P0.056-P0.059、P0.065-P0.069、P0.081-P0.087 共 24 个场景全部 PASS，`failures=0`。

## 6 经验教训

- Scenario wrapper 不能只看测试文件参数是否写在脚本里，必须从 trigger log 反查 runner command 和实际执行的 spec/test 文件。
- 对 hash-based pixel parity harness，任何 query string 都可能改变 routeStore bootstrap 优先级；hash route 应保持 bare `/` path，不通过 query 参数做刷新 nonce。

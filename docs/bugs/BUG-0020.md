# BUG-0020: frontend shell language switch was state-only

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-07 | 关联 Commit: `fix(frontend-shell): restore app shell i18n`
> 关联日志: [2026-05-07](../work-journal/2026-05-07.md)

## 1 现象

- 用户复查最近 `frontend-shell/001-app-shell-auth-settings` 实施后发现正式前端没有 i18n；TopBar 有语言 select，但切换后 D1 shell 文案仍是中文。
- 静态原型 `ui-design/src/app.jsx` 已持有 `lang` 并把 `lang` 传给 TopBar、auth、profile/settings 和业务 screen；正式前端只实现了 `DisplayPreferencesProvider.lang` 状态。
- OpenAPI/runtime 契约已存在 `Accept-Language`，但正式前端没有把当前 locale 传给 D1 generated client 操作。
- Review remediation 暴露后续口径问题：把 runtime `defaultUiLanguage` / `/me.uiLanguage` 接成 UI locale 来源会让登录态刷新覆盖前端语言偏好；最终产品口径修订为默认跟随浏览器 locale，未知时 fallback English，语言切换只关联前端设置。

## 2 诊断过程

### 假设 1：产品架构没有锁定 i18n

- 验证方法：读取 `docs/ui-design/ui-architecture.md`、`docs/ui-design/auth-and-entry.md`、`docs/spec/frontend-shell/spec.md` 与 OpenAPI locale 字段。
- 结果：部分确认。UI 文档、frontend-shell spec 和 OpenAPI 已锁定语言切换 / locale hint，但 runtime locale 是否能覆盖前端语言偏好没有写清，导致实现阶段选择了错误优先级。

### 假设 2：正式前端已有隐藏 i18n 层

- 验证方法：搜索 `useTranslation`、`i18n`、message catalog、`Accept-Language`、`defaultUiLanguage`、`uiLanguage` 等关键字，并读取 `TopBar.tsx` / auth / profile / settings 实现。
- 结果：排除。正式前端没有 translation catalog；`TopBar.tsx` 注释还写明 label 是中文，测试只断言 `lang` select value 变为 `en`。

### 假设 3：这只是后续 D2-D6 业务页面范围

- 验证方法：读取 `frontend-shell` spec/plan/checklist scope。
- 结果：排除。D1 scope 明确包含 TopBar 全局显示控制、auth shell、profile/settings shell 和 runtime bootstrap；即使 D2-D6 业务内容未实现，D1 可见静态文案也应支持中英切换。

## 3 根因分析

- `frontend-shell` plan/checklist 2.2 的验证过窄，只要求“主题色、暗色、语言切换在登录前后保持稳定”，没有要求语言切换改变用户可见文案。
- 正式前端把 `lang` 当成 display preference 状态保存，但没有建立 message catalog / typed helper，也没有把 D1 shell 组件接到同一个 locale source。
- 初版修复把 `defaultUiLanguage` / `/me.uiLanguage` 接成 runtime locale bootstrap，但这和用户确认的产品口径冲突：UI 语言应是前端显示偏好，初始值只跟随浏览器 locale，不依赖登录态。
- D1 auth operations 直接调用 generated client，没有合并当前 UI locale 的 `Accept-Language` header。

## 4 修复方案

- 原地修订 `frontend-shell/001-app-shell-auth-settings` spec/plan/checklist/BDD 到 i18n remediation：新增 C-7、D-7、2.4 / 2.5 / 2.6 / 2.8 与 E2E.P0.004，并明确 UI 语言默认跟随浏览器 locale、未知时 fallback English、登录态和 runtime locale 不覆盖前端设置。
- 新增 `frontend/src/app/i18n/locales/zh.ts` 与 `frontend/src/app/i18n/locales/en.ts` 独立 locale 文件；`frontend/src/app/i18n/messages.ts` 只保留导入聚合、BCP 47 locale 归一化和 `useI18n()` helper。
- TopBar、auth、profile/settings 和 placeholder shell 改为通过 i18n helper 渲染静态文案，保留 RouteName / testid / params 为稳定英文 key。
- `DisplayPreferencesProvider` 改为从浏览器 locale 初始化语言，未知或缺失时 fallback English；删除 App shell 中从 runtime config 或 `/me` 回写 UI 语言的 bootstrap 逻辑。
- `App` 在 runtime config、`/me` 和 D1 auth operations 上合并当前 UI locale 的 `Accept-Language` display hint。
- TopBar 语言切换固化为下拉框，新增结构测试阻止 `zh` / `en` 多语言 message map 回流到同一文件。
- 新增 E2E.P0.004 场景目录和 Vitest scenario，验证语言下拉框、中英切换、用户菜单、auth/profile/settings shell、route key 稳定性和 `Accept-Language` header。

## 6 经验教训

- 对显示偏好类功能，测试不能只断言 state / select value；必须断言用户可见语义效果。
- 静态原型已有 `lang` 透传时，正式前端 shell 的迁移 checklist 必须覆盖 message catalog、runtime locale 初始化和 request header 三个层面。
- UI locale 是前端显示偏好，不应由 runtime config 或登录态覆盖；浏览器 locale、English fallback 和登录态不覆盖都应在 focused tests 中固定。

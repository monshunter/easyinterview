# BUG-0130: parse launch allowed practice planning without a bound resume

> 模块: frontend | 严重度: high | 状态: resolved
> 发现日期: 2026-06-30 | 关联 Commit: `fix(frontend-home): require resume binding before practice (BUG-0130)`
> 关联日志: [2026-06-30](../work-journal/2026-06-30.md)

## 1 现象

- 首页 JD 解析完成后，`parse` preview 可以直接确认并进入后续面试规划，没有要求用户上传新简历或选择已有简历。
- 成功 handoff 仍可能使用 `resume-unbound`，导致生成的面试规划只基于 JD 做单边分析，缺失候选人简历资料。
- 影响范围覆盖首次 JD 导入后的核心模拟面试启动路径：`parse` → `workspace` → `practice`。

## 2 诊断过程

### 假设 1：简历绑定应由 Workspace 后置补齐

- 验证方法：反查当前 product / UI 真理源、`frontend-home-job-picks-and-parse` owner spec/plan、`ui-design` Parse launch 区域以及 Workspace resume picker 交互。
- 结果：排除。当前流程要求 Parse 离开前必须拥有真实 ready resume；Workspace 只保留 `autoStartPractice=1` 的技术桥接，不应作为用户可见的第二确认入口。

### 假设 2：正式前端已经在 Parse 读取并选择简历

- 验证方法：新增 `ParseResumeBinding.test.tsx`，断言 ready preview 后会调用 `listResumes` 并渲染 resume binding 区域。
- 结果：排除。红灯显示旧实现没有调用 `listResumes`，仍保留旧 confirm flow。

### 假设 3：只修 ParseScreen 不足以闭环

- 验证方法：检查 P0.016 BDD wrapper、Playwright gate、component tests、owner plan/checklist 和 scenario verify 脚本。
- 结果：确认。旧 P0.016 成功口径曾把 `workspace-missing-resume` 当作可接受目标状态，必须同步改为 ready resume binding、Save plan / Start interview 两条成功出口，以及拒绝 `resume-unbound` 成功 marker。

## 3 根因分析

- `frontend/src/app/screens/parse/ParseScreen.tsx` 的旧成功路径沿用 `interviewContextFromTargetJob` 默认 fallback，没有在 Parse preview 阶段读取 `listResumes`、过滤 ready 简历或要求用户选择真实 `resumeId`。
- `docs/spec/frontend-home-job-picks-and-parse/plans/001-home-jd-import-and-parse` 的历史 P0.016 gate 曾围绕 Confirm → Workspace 缺简历状态收口，未把“绑定 ready 简历后才能保存或启动”作为成功条件。
- `test/scenarios/e2e/p0-016-parse-confirm-to-workspace` 的旧 wrapper 没有拒绝 `resume-unbound` 成功 handoff，因此没有及时暴露 JD-only planning 的流程缺陷。

## 4 修复方案

- `ParseScreen` 在 ready preview 且已登录运行态下调用 `listResumes`，只保留 `parseStatus=ready`、未 archived、未 deleted 的简历，按更新时间默认选中最新 ready resume。
- 新增 Parse launch / resume binding UI：有 ready 简历时可选择或切换；无 ready 简历或读取失败时禁用 `仅保存规划` 与 `立即面试`，并引导进入 `resume_versions` 创建简历。
- 将旧 `Confirm` 成功出口拆成两条显式动作：`仅保存规划` 保存 `updateTargetJob` 后进入 `workspace`，`立即面试` 保存后进入 `workspace&autoStartPractice=1` 并由现有 Workspace start-practice 链路进入 `practice`。
- 两条成功出口都携带真实 `resumeId`，并在 Vitest、Playwright、P0.016 scenario verify 中反向拒绝 `resume-unbound` 与 `workspace-missing-resume` 成功 marker。
- 原地修订 owner spec/plan/checklist/BDD/context 与 scenario 文档，增加 Phase 7 resume binding gate。

## 6 经验教训

- 首次 JD 导入后的启动路径不能把 `resume-unbound` 当作可继续的成功状态；如果下游 AI 规划依赖候选人材料，上游必须先锁定真实 ready resume。
- BDD wrapper 不应只证明页面能跳转，还要拒绝产品语义上不完整的成功 marker，例如 `resume-unbound` 和 `workspace-missing-resume`。
- Completed plan 的历史 PASS 只能作为线索；当产品流程语义改变时，owner plan、scenario README、trigger/verify 和 Playwright marker 必须一起更新。

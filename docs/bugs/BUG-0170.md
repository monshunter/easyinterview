# BUG-0170: Duplicate safe reads and mixed command/detail route ownership

> 模块: frontend | 严重度: high | 状态: resolved
> 发现日期: 2026-07-14 | 关联 Commit: `fix(core-flow): separate read paths and dedupe requests (BUG-0170)`
> 关联日志: [2026-07-14](../work-journal/2026-07-14.md)

## 1 现象

- 点击简历模块、简历详情及其他读取型页面时，同一 API 在一次用户动作中可能出现两次或更多并发 GET。
- `listResumes.items` 复用了完整 `Resume` DTO，列表响应携带正文、解析快照和结构化详情等仅详情页需要的数据。
- 新建 JD 后的解析进度与点击已解析规划卡片共用 Parse 页面；只读查询也会经过导入动画，命令入口和读取入口没有清晰边界。
- 影响 Home、Parse、Workspace、Resume、Practice 与 Report 等核心路径；重复传输、过量数据和错误路由归属会增加延迟并掩盖副作用风险。

## 2 诊断过程

### 假设 1：所有重复记录都代表浏览器实际发送了重复请求

- 验证方法：同时使用浏览器网络记录器、Chrome `PerformanceResourceTiming` 和 Playwright `page.on('request')` 统计相同 method/path 的底层请求。
- 结果：部分排除。浏览器记录器会累积重复展示同一请求，不能单独作为传输次数证据；Resource Timing 与 Playwright 才能稳定确认修复后的实际 GET 数为 1。

### 假设 2：页面级 effect 与客户端缺少并发读合并共同造成真实重复 GET

- 验证方法：在 React StrictMode 下直接统计底层 `fetch`，覆盖初次挂载、详情打开、轮询 tick、失败驱逐与显式重试。
- 结果：确认。多个 mount/read loader 直接发起相同读取，生成客户端没有 per-client concurrent safe-GET single-flight；开发态重挂载和相邻 consumer 可在同一时间窗内形成重复传输。

### 假设 3：JD 路由和 Resume DTO 本身没有区分命令侧与读取侧

- 验证方法：反查 Home 卡片导航、Parse/Workspace route authority、OpenAPI `PaginatedResume.items`、fixtures、generated clients、backend store projection 与 frontend consumer。
- 结果：确认。ready 卡片仍复用了 Parse 语义，且 Resume 列表直接引用完整详情 schema；前后端没有可执行的 summary/detail 边界。

## 3 根因分析

- 生成客户端只提供逐调用传输，没有把“同一 client 内同时发生的相同安全 GET”定义为共享 in-flight promise；页面各自承担去重会产生遗漏和不一致。
- mount loader 把 AbortSignal 传入读取请求，React StrictMode cleanup 会让可共享读取失去稳定 owner；同时缺少统一的 cancelled/sequence stale-result guard。
- `/parse` 同时承接“POST 导入后的异步命令进度”和“已存在 TargetJob 的只读详情”，把不同副作用语义压进同一路由。
- OpenAPI 的 `PaginatedResume.items` 直接引用 `Resume`，因此 backend、fixture、generated client 和 frontend 都自然继承完整详情数据。
- 既有测试多统计 hook/mock 调用或 DOM 结果，没有在 StrictMode 下统计 raw transport，也没有对列表 closed projection 和旧路由做负向断言。

## 4 修复方案

- 在每个生成客户端实例内增加并发 safe-GET single-flight：key 包含 method、规范化 URL/query/headers、可接受状态及 read/auth epoch；不增加 TTL 缓存。
- 带 caller AbortSignal 的请求明确绕过 single-flight；mutation 在开始和完成时推进 read epoch，鉴权变化推进 auth epoch；成功和失败都通过双分支 settlement cleanup 驱逐 in-flight 项。
- mount/read loader 不再传 caller signal，以 cancelled/sequence guard 丢弃 stale result；显式可取消调用继续保持独立语义。
- 将 `/parse?targetJobId=...` 限定为 POST 导入后的 queued/processing 进度；ready 后 history-replace 到 `/workspace?targetJobId=...`。已解析卡片及 Report/Practice 返回动作直接进入 Workspace 只读详情。
- 新增 closed `ResumeSummary`，required 字段精确为 `id/title/displayName/language/sourceType/parseStatus/summaryHeadline/hasReadableContent/updatedAt`；`getResume` 保持完整 `Resume`，backend 用单次列表查询直接投影，frontend 不追加详情请求。
- 用 StrictMode raw-fetch、Playwright request counter、Chrome Resource Timing、OpenAPI fixture/Prism 和旧 route/字段负向搜索固化回归门禁。

## 6 经验教训

- 请求去重应由传输客户端统一拥有，页面只负责请求时机和 stale-result 防护；否则每个模块都会重复实现并留下漏网路径。
- “命令进度”和“已保存资源读取”必须拥有不同 route authority，即使两者读取同一个 resource ID，也不能复用同一交互页面。
- 列表 DTO 应是显式 closed projection，不能为了复用类型返回详情数据；后端必须直接投影，避免用 N+1 详情读取修补列表。
- 网络问题必须用浏览器底层传输计数交叉验证；会累积或重放记录的调试工具不能作为唯一证据。
- 当前实现、场景和 pixel test 更新后，还必须运行根级 contract 聚合；防漂移测试若保留旧正向 route，会在正确实现落地后反向阻断交付。

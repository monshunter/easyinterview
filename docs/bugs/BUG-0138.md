# BUG-0138: resume detail kept generated-name placeholder and repeated polling risk

> 模块: materials | 严重度: high | 状态: resolved
> 发现日期: 2026-07-07 | 关联 Commit: `fix(resume-workshop): generate names and stop detail polling (BUG-0138)`
> 关联日志: [2026-07-07](../work-journal/2026-07-07.md)

## 1 现象

- 用户上传简历并跳转到详情页后，可见简历名长期保持“名称生成中”。
- 失败态 resume 仍有 `parsedTextSnapshot` 可展示，但 `displayName` 为空，详情页无法展示有意义的业务名称。
- 用户截图显示同一详情 URL 多次请求；该路径有性能回归风险，需要固定为只在 truly pending 且无正文时轻量轮询。

## 2 诊断过程

### 假设 1：`resume.parse` prompt 未要求模型返回 UI 名称
- 验证方法：检查 `config/prompts/resume.parse/v0.1.0.schema.json`、prompt body、seed migration 和 prompt lint 合同。
- 结果：确认。schema 和 prompt contract 没有 required `displayName`，模型即使结构化解析成功也缺少明确的名称生成目标。

### 假设 2：失败态解析不会持久化 fallback `display_name`
- 验证方法：检查 `backend/internal/resume/jobs/parse.go` 与 `backend/internal/resume/store/assets.go` 的失败路径。
- 结果：确认。`CompleteParseFailure` 只写失败状态、错误码和可选正文快照；当 AI output invalid 但正文已抽取时，详情只能拿到正文，拿不到可识别名称。

### 假设 3：详情页轮询条件缺少 current-state 回归门禁
- 验证方法：补充 `ResumeDetailView` failed-with-snapshot regression 和 E2E.P0.037 场景断言，观察 `getResume` 调用次数。
- 结果：当前源码已经只在 `upload && queued|processing && no readable body` 时轮询，新增测试确认 failed-with-snapshot 只请求一次；缺口在于此前没有场景级 gate 防止回归。

## 3 根因分析

- `resume.parse` 输出合同没有把 UI 可见 `displayName` 设为 required 字段，prompt 也未禁止“uploaded resume / pasted resume / 文件名 / raw 第一行”这类无意义名称。
- 后端只在 ready 成功路径写入 LLM-derived `display_name`；AI provider / output 失败但已抽取正文时，没有从可读正文生成 fallback 名称。
- P0.037 只验证 pending upload 轮询到正文，未覆盖 failed-with-snapshot 详情必须停止轮询并展示 backend generated 名称。

## 4 修复方案

- 扩展 `resume.parse` schema、prompt body、prompt hash、seed migration、prompt lint 和 `resolved-prompts.json`，要求模型返回有意义的 required `displayName`。
- `decodeResumeParseResponse` 优先消费 AI `displayName`，并拒绝通用标题、文件名和 raw 第一行直出；无效时回退到 structured fields。
- `CompleteParseFailure` 支持在失败事务内写入可读正文派生的 fallback `display_name`，避免 failed-with-snapshot 详情长期显示“名称生成中”。
- 为 `ResumeDetailView` 和 E2E.P0.037 增加 failed-with-snapshot 单次请求回归测试；P0.035 verify 检查 store fallback 测试名。
- 同步 backend/frontend owner spec、plan、checklist、BDD、场景 README/expected outcome 和 docs/ui-design 口径。

## 6 经验教训

- UI 可见名称属于用户体验合同，不能只依赖结构化字段“顺便派生”；prompt schema、prompt body、backend validator 和失败态 fallback 要一起覆盖。
- 只要详情已有可读正文或进入 failed/ready，就不应继续请求同一详情 URL；该条件必须有组件测试和 BDD 场景双层 gate。
- prompt 改动必须同步 seed migration、prompt lint contract 和 committed resolved prompt export，避免离线 eval drift 留到后续阶段才暴露。

# BUG-0184: interview plan cards duplicated an inactive lifecycle status and empty-location placeholder

> 模块: frontend | 严重度: low | 状态: resolved
> 发现日期: 2026-07-17 | 关联 Commit: `fix(frontend): remove stale plan card metadata (BUG-0184)`
> 关联日志: -

## 1 现象

- Home 最近面试与 Workspace 面试规划复用的卡片在公司名后和右侧徽标中重复展示同一个 `TargetJob.status`。
- 新导入 TargetJob 默认保持 `draft`，而当前前端没有提供生命周期状态维护入口，因此用户持续看到两个 `Draft`，容易误解为面试规划尚未生成完成。
- `locationText` 缺失时卡片同时显示硬编码英文 `Location not set`，在中文界面中形成无决策价值的占位信息。

## 2 诊断过程

### 假设 1：两个 Draft 分别来自解析状态和规划状态

- 验证方法：反查 `MockInterviewCard` 的两个渲染位置及 generated `TargetJob` 类型。
- 结果：排除。两处都调用 `statusLabel(job.status)`，不是两个不同字段。

### 假设 2：生命周期状态是当前训练进度来源

- 验证方法：核对 Workspace owner spec、`practiceProgress` 投影和前端状态消费者。
- 结果：排除。训练进度由 backend-persisted `TargetJob.practiceProgress` 与 mini round rail 表达，lifecycle `status` 不参与轮次推断。

### 假设 3：当前 UI 能维护 lifecycle status，因此至少应保留一处

- 验证方法：搜索正式前端对 generated `updateTargetJob` 的生产调用。
- 结果：排除。当前卡片链路只读取该字段，没有用户可操作的状态维护入口。

## 3 根因分析

- 共享 `MockInterviewCard` 同时在 company eyebrow 和独立 badge 渲染 `job.status`，导致同一事实重复出现。
- 早期通用岗位卡片保留了求职生命周期映射，但当前面试规划模块明确不负责管理多岗位资产的完整生命周期。
- 地点行使用 `job.locationText || "Location not set"`，把数据缺失转换成了硬编码英文占位，而不是省略无价值信息。
- 原测试只断言占位存在，没有从用户可见语义验证状态是否可操作、是否重复，以及缺失地点是否应隐藏。

## 4 修复方案

- 原地修订 `frontend-workspace-and-practice/001-workspace-and-interview-context` 与 `docs/ui-design/module-job-workspace.md`，明确卡片不展示 lifecycle status，空地点不渲染占位行。
- 从 `MockInterviewCard` 删除 status label/tone/color 映射、company eyebrow 后的状态文本和右侧状态徽标。
- 对 `locationText` trim 后仅在非空时渲染真实地点；后端 `TargetJob.status`、OpenAPI 与状态机保持不变。
- 先添加失败行为测试，再做最小组件修改；Home 与 Workspace 继续复用同一组件。

## 5 验证

- Red：`MockInterviewCard.test.tsx` 新增两项行为断言后，状态可见与 `Location not set` 各失败一项。
- Green：`MockInterviewCard.test.tsx` 13/13，通过共享 Home/Workspace focused 回归 41/41。
- 仓库根 `make test` 通过：后端 584 tests / 4583 subtests，前端 126 files / 1027 tests。
- Frontend typecheck、owner context、Header/INDEX、docs-check、`git diff --check` 通过；共享卡片源中的 `statusLabel`、`statusTone`、`job.status` 与 `Location not set` 残留为零。

## 6 经验教训

- API 字段存在不代表必须在每个消费界面展示；用户不可操作且不影响当前决策的生命周期字段应留在数据层，而不是作为卡片占位信息。
- 对缺失可选信息，测试应验证省略行为，不应把调试型 fallback 固化为产品合同。
- 共享卡片需要用非默认 enum 值做负向测试，避免只因 fixture 长期使用 `draft` 而掩盖重复展示。

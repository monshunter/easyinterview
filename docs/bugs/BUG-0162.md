# BUG-0162: practice interviewer lost the real resume after parse output truncation

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-07-12 | 关联 Commit: `fix(interview): ground resumes and persist round progress (BUG-0162, BUG-0163)`
> 关联日志: [2026-07-12](../work-journal/2026-07-12.md)

## 1 现象

- 用户上传的真实简历没有“智能客服”项目，但 AI 面试官要求用户介绍该项目，问题无法从简历、JD 或历史对话中找到证据。
- 故障数据中的 `parsed_text_snapshot` 保存了完整真实简历且不含“智能客服”，但简历解析任务以 `AI_OUTPUT_INVALID` 失败，provider 输出恰好到达原 `resume.parse.default.max_tokens=2048` 上限，`structured_profile` 为空。
- Practice 创建与消息链路只读取 `structured_profile`；解析失败后把真实快照丢在链路外，最终以“resume context unavailable”或缺少简历证据的通用上下文调用模型。
- 影响 Practice 首问和后续追问的事实可信度；模型可能根据岗位或通用模式补造项目，因此严重度为 `high`。

## 2 诊断过程

### 假设 1：会话绑定到了其他用户或其他简历

- 验证方法：沿 session → plan → TargetJob → resume 的用户、目标岗位和简历绑定反查数据库，并检查原始文本与快照中的项目名称。
- 结果：排除。记录属于同一用户、同一 TargetJob 和目标简历，完整快照中也不存在“智能客服”。

### 假设 2：1M 输入上下文在上传或 prompt 拼装阶段被截断

- 验证方法：比较 `original_text`、`parsed_text_snapshot`、发送给解析模型的输入及长尾 marker 测试，并检查上传读取上限和任何 slice/truncate 逻辑。
- 结果：排除输入截断。真实简历完整保存在快照并完整进入 prompt；故障发生在模型输出侧，旧 prompt 要求同时回显完整 Markdown 和结构化 JSON，2048-token 输出预算不够。

### 假设 3：只需要加强面试官提示词

- 验证方法：检查 Practice start/send 的持久化读取顺序和最终 prompt message role，注入空 `structured_profile`、尾部简历事实及伪造 `</system_policy>` 文本。
- 结果：部分确认但不足以单独修复。旧提示词缺少可信策略边界，但更早的上下文加载路径已经丢失真实简历；只改文案仍无法让模型看到证据。

## 3 根因分析

- `config/prompts/resume.parse/v0.1.0.md` 让 provider 同时生成结构化字段和完整 `markdownText`，`config/ai-profiles.yaml` 的旧 2048-token 输出预算导致合法长简历的响应被截断。
- `backend/internal/resume/jobs/parse.go` 没有把 `finish_reason=length` 单独识别为输出截断，也把 provider 回显的 Markdown 当作主快照来源，解析失败时无法形成可靠结构化结果。
- 首轮确定性快照实现又用全局字符串替换识别 `Experience` / `Skills` 等章节词；这些词出现在正文句子中时也会被拆成 Markdown 标题，导致本应作为权威事实源的 `parsed_text_snapshot` 被语义改写。
- `backend/internal/store/practice/create_plan.go` 与 `backend/internal/store/practice/messages.go` 的上下文读取只依赖 `structured_profile`，没有使用已持久化的 `parsed_text_snapshot` / `original_text` 作为事实优先来源。
- `config/prompts/practice.session.chat/v0.1.0.md` 原先没有把系统策略与不可信简历/JD/历史数据隔离；首轮修复后又发现完整 history 被笼统授权为事实源，模型上一轮写出的 assistant 幻觉可能被下一轮当成已确认候选事实。
- 之前的测试只覆盖短结构化样例，没有覆盖长尾事实、输出 `length`、空结构化结果、prompt closing-tag 注入、行内章节词和用户提供的真实简历负例；P0.035 verifier 也只搜索测试名，未要求精确 `--- PASS:` 或拒绝 `--- FAIL:` / package `FAIL`。

## 4 修复方案

- Resume parse 改为只要求结构化 JSON；完整 Markdown 快照由后端从原始输入确定性生成，不再要求模型回显。`resume.parse.default.max_tokens` 提升到 8192，并在 `finish_reason=length` 时 fail closed。
- 确定性 Markdown 构建只在整行标题或行首标题前缀处识别章节，不再对正文做全局章节词替换；`Customer Experience Platform`、`Skills-based routing` 等事实保持原文。
- 上传输入保持完整读取；8 MiB 边界超限时明确失败，不做静默切片。长简历测试在输入尾部放置 marker，证明 prompt 与持久化快照均保留尾部事实。
- Practice 上下文按 `parsed_text_snapshot → original_text → structured_profile` 选择真实证据；三者都为空时在 AI 调用和消息提交前失败。start/send 都重新校验 plan、TargetJob 与当前 resume 的绑定。
- Practice prompt 拆为可信 system policy 与 JSON 编码的 `<untrusted_interview_context_json>`；候选事实只能来自 persisted resume 或 candidate-authored `user` message，`assistant` history 仅保持连续性，不能把上一轮模型臆造放大为下一轮依据。原始 business data 不能闭合 system tag 或改变 message role，persona 也不能创建事实。
- 结构化输出在 start/send 遇到 `finish_reason=length` 时只修复一次，连续截断则 fail closed，不提交不完整消息。
- 未采用“既保留模型 Markdown 回显又继续增大输出预算”的方案，因为它重复传输源文本，仍会把长简历正确性绑定到输出长度。
- P0.035 trigger 纳入行内章节词 regression；verifier 对每个命名测试要求精确 `--- PASS:`，并在任何 `--- FAIL:` 或 package `FAIL` 出现时立即失败。另用伪造失败日志的脚本契约测试锁定该行为。

## 5 验证

- Resume/Practice focused tests 覆盖完整长尾输入、确定性快照、`finish_reason=length`、空上下文 fail-closed、当前简历绑定、start/send 和 closing-tag 注入，均通过。
- `TestParseHandlerPreservesInlineHeadingWordsInSourceSnapshot` 先复现 snapshot 被改写为 `## Experience` / `## Skills`，修复后与完整 `internal/resume/jobs` 包测试通过。
- P0.035 verifier 脚本契约测试先证明失败日志被误判为 PASS，修复后通过；实际 `setup → trigger → verify → cleanup` 全部 PASS，integration store gate 与新增 snapshot regression 均出现精确 PASS evidence。
- Prompt registry、profile、seed 与 resolved prompt hash 一致；`make lint-prompts` 通过。
- 离线评测 27/27 通过，包含本次真实简历事实集和 assistant-history 放大负例：正向要求围绕 Ferry / GitOps 等实际项目提问，负向拒绝虚构或延续“智能客服”。
- `make test` exit 0：UI contract 48/48、Python 473 tests + 3919 subtests、全部 Go packages、frontend 112 files / 764 tests 通过。
- 本次没有运行真实 provider 的对抗性 UAT；离线/stub gate 能证明上下文完整传递和策略边界，但不能数学保证生成模型永不幻觉。该限制保留为后续验收项。

## 6 经验教训

- 大上下文窗口不能替代完整链路审计；输入是否完整、输出预算是否足够、下游读取哪个持久化字段是三个独立边界。
- 模型不应回显可由后端确定性保存的长源文本；结构化输出必须检查 `finish_reason`，不能只依赖 JSON decode 错误。
- 从源文本构建权威快照时，格式化器只能识别结构边界，不能用无边界关键词替换推断语义；场景 verifier 看到测试名不等于测试通过。
- AI 业务上下文必须有可信 system policy 与不可信数据的角色/编码边界，并用具体负例验证模型没有被 persona 或简历内指令改变事实来源。
- 完整对话历史可以用于语境连续性，但模型自己生成的 assistant 消息不能成为候选人事实；否则一次幻觉会在后续轮次自我强化。
- 任何用于 AI 推理的业务上下文都应来自后端持久化事实；前端仅负责提交意图和展示结果。

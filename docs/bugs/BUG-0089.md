# BUG-0089: frontend owner plans missed real-backend handoff gates

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-23 | 关联 Commit: `fix(frontend): close real backend handoff gate drift (BUG-0089)`
> 关联日志: [2026-05-23](../work-journal/2026-05-23.md)

## 1 现象

- 已完成的 frontend owner plans 仍把 backend-resume / backend-practice / backend-review / backend-debrief / practice-voice 已落地的 operation 写成 future、missing、fixture-only 或 Phase 0/Phase 5 前置口径。
- P0.018-P0.021、P0.044-P0.047、P0.056-P0.059、P0.065-P0.069、P0.081-P0.087 等 frontend scenarios 只证明 fixture-backed UI variants，没有统一证明 `VITE_EI_API_MODE=real` 下 generated client 指向真实 backend base URL。
- 代表性 P0.044 场景重跑时，旧 verify 仍把 `createPracticeVoiceTurn` 当作缺失契约，导致当前 `practice-voice-mvp/001` 已落地的 voice owner hook 被误判为 text-loop drift。

## 2 诊断过程

### 假设 1：漂移只存在于单个 frontend plan

- 验证方法：对 `frontend-workspace-and-practice`、`frontend-report-dashboard`、`frontend-resume-workshop`、`frontend-debrief` 下 active/completed spec、plan、checklist、BDD 文档执行旧口径负向搜索。
- 结果：多个 sibling owner plan 都有相同问题，说明这是 frontend-first handoff 后未回填真实 backend gate 的系统性漂移。

### 假设 2：scenario trigger/verify 已经能证明 real backend mode

- 验证方法：直接读取各 P0 scenario `trigger.sh` / `verify.sh`，并新增集中 real-mode Vitest 后接入代表性 trigger/verify。
- 结果：原 trigger 只跑 fixture-backed UI tests；verify 也没有检查 real-mode marker、base URL 或 real generated-client test 文件名。

### 假设 3：practice text-loop 的 `createPracticeVoiceTurn` 负向断言仍然有效

- 验证方法：重跑 P0.044 trigger/verify 并反查 `frontend/src/app/screens/practice/hooks/usePracticeVoiceTurn.ts` 与 `practice-voice-mvp/001`。
- 结果：`practice-voice-mvp/001` 已完成 voice operation 与 voice surface；P0.044/P0.047 应限制该 operation 只能在 voice owner hook 出现，而不是 repo-wide 0 命中。

## 3 根因分析

- completed frontend-first plans 的完成证据停留在当时的 fixture-first 状态；后续 backend owner landing 没有强制 sibling completed plans 回填 operation matrix 和 real-mode gate。
- `/plan-code-review` 的 frontend-first handoff 规则缺少“backend landed 后反扫所有 sibling completed plans + scenario real-mode trigger/verify marker”的硬门禁。
- P0.044/P0.047 的旧负向断言没有随着 `practice-voice-mvp/001` 完成而反转，导致场景 wrapper 把当前真实 owner 实现误判为旧口径泄漏。

## 4 修复方案

- 加固 `.agent-skills/plan-code-review/SKILL.md`：新增 frontend-first handoff review gate，要求检查 real API mode、base URL、credentials、Prefer header、Idempotency-Key、provenance/response 字段、scenario marker 和 sibling completed plans。
- 新增 `frontend/src/api/frontendOwners.realApiMode.test.ts`，集中覆盖 practice / report / resume / debrief owners 在 real mode 下通过 generated client 发起请求。
- 新增 `test/scenarios/_shared/scripts/frontend-real-backend-gate.sh` 与 `frontend-real-backend-verify.sh`，并接入 P0.018-P0.021、P0.044-P0.047、P0.056-P0.059、P0.065-P0.069、P0.081-P0.087 trigger/verify。
- 修订相关 frontend specs/plans/checklists/BDD 文档，清理 stale backend-missing 口径，补充 real-mode gate 证据。
- 修订 P0.044/P0.047 verify：禁止 `getFeedbackReport` 进入 practice runtime，同时允许 `createPracticeVoiceTurn` 仅出现在 `hooks/usePracticeVoiceTurn.ts`。

## 6 经验教训

- Frontend-first handoff 的完成证据必须在 backend owner landing 后主动反扫 sibling completed plans，不能只修第一个被发现的 plan。
- Scenario wrapper 需要检查 real-mode marker 与测试文件名，避免 fixture UI PASS 单独满足真实联调完成条件。
- 负向断言应表达 owner 边界，而不是永久 repo-wide 0 命中；当相邻 owner 后续落地时，必须把旧 negative gate 反转为受控 allowlist。

# BUG-0195: Markdown 简历白色纸张在夜间模式继承浅色前景

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-20 | 关联 Commit: `fix(resume): keep Markdown paper readable in dark mode (BUG-0195)`
> 关联日志: [2026-07-20](../work-journal/2026-07-20.md)

## 1 现象

- 用户在 `resume_versions?resumeId=...` 打开 Markdown 简历，并把应用切换为夜间模式。
- 详情仍显示白色 Markdown page surface，但正文、列表和强调文本接近白色，看起来像透明或消失；标题中已有局部深色规则的内容仍可读。
- PDF 页面栈不受影响；切回浅色模式可以绕过，因此属于核心阅读体验异常但存在临时绕过方式。

## 2 诊断过程

### 假设 1：正文容器被设置了透明度

- 验证方法：在真实 Chrome 页面读取 Markdown page、body、`ul`、`li`、`strong` 的 `opacity` 与计算颜色。
- 结果：排除。所有元素 `opacity=1`，但夜间模式计算颜色为 `rgb(232, 237, 246)`，白色纸张背景为 `rgb(255, 255, 255)`。

### 假设 2：白色纸张继承了应用壳的夜间 foreground

- 验证方法：反查 `global.css` 的 `body` 颜色、`themes.css` 的 dark token，以及 `screens.css` 的 Markdown page 局部规则。
- 结果：确认。`body` 使用 `var(--ei-color-fg-primary)`，夜间值为 `#e8edf6`；Markdown page 固定了 `background: #ffffff`，但只有标题声明深色，page/body 未声明局部正文颜色。

## 3 根因分析

- `frontend/src/app/theme/global.css:31` 为整个应用设置主题化前景色，这是应用壳的正确 owner。
- `frontend/src/app/screens/screens.css:2554` 将 Markdown 预览固定为白色纸张，却没有同时建立纸张自己的墨水颜色边界；普通正文、列表、强调和代码沿 DOM 继承到夜间壳层的浅色 foreground。
- 既有 CSS parity gate 只覆盖 A4 宽度、高度和 renderer 几何，没有覆盖固定浅色 surface 在 dark mode 下的局部前景色与对比度，因此回归未被代码测试阻止。

## 4 修复方案

- 在 `.ei-resume-detail-markdown-page` 上增加固定 `color: #222222`，让正文、列表、强调和 inline code 继承纸张墨水；保留既有标题、链接与分隔线的局部颜色。
- 新增 CSS parity RED/GREEN：白色纸张必须声明 `#222222`、不得引用 `--ei-color-fg-*`，并验证与白底的对比度达到 WCAG AA。
- 保持应用壳主题 token、PDF renderer、Markdown DOM、A4 几何、API 和正文事实不变。没有采用 dark paper 方案，因为当前设计合同明确要求 Markdown 维持白色文档面。

## 5 验证

- RED：`ResumeWorkshopCssParity.test.ts` 9 项中仅新增的 page-local ink 断言失败；GREEN 后 9/9 通过。
- Resume Workshop owner：20 个文件、125 项测试通过；frontend typecheck 与生产构建通过。
- 根回归：`make test` 通过，621 tests / 4628 subtests，Go packages 全部通过。
- 真实 Chrome：light/dark 两种模式下 page/body/`ul`/`li`/`strong`/`code` 均为 `rgb(34, 34, 34)`，白底为 `rgb(255, 255, 255)`；夜间壳层 foreground 仍为 `#e8edf6`；Markdown page 保持 794px、单张连续页面、无横向溢出，console warning/error 为 0。

## 6 经验教训

- 固定浅色 surface 不应只固定背景；必须同时拥有 foreground、link、rule 等局部可读颜色边界。
- 主题回归 gate 要检查实际 surface 的背景/前景组合，而不是只验证根主题 token 或标题颜色。
- 真实浏览器诊断应同时读取 `opacity` 和 computed `color`，避免把“近白字落在白底”误判为透明动画问题。

# BUG-0113: scenario redeploy refreshed artifacts without restarting host-run services

> 模块: platform | 严重度: high | 状态: resolved
> 发现日期: 2026-05-27 | 关联 Commit: `fix(test): restart host-run services on redeploy (BUG-0113)`
> 关联日志: [2026-05-27](../work-journal/2026-05-27.md)

## 1 现象

- 用户要求修复后重新部署前后端，但 `test/scenarios/env-redeploy.sh all` 只刷新 Docker 依赖与 backend/frontend build artifacts。
- 浏览器和 Mailpit 仍连接旧的宿主机 backend 进程，最新邮件中的 magic-link 继续指向旧 backend verify URL。
- 环境脚本输出没有明确列出 frontend/backend 服务地址、日志路径、PID 文件和容器日志命令，开发者无法直接接管调试。

## 2 诊断过程

### 假设 1：代码修复未生效

- 验证方法：检查 `deploy/dev-stack/.env` 和最新源码，确认 `EMAIL_VERIFY_BASE_URL` 已改为 frontend `/auth/verify` callback。
- 结果：排除。配置和源码已经修复。

### 假设 2：运行中的 backend 仍是旧进程或旧环境

- 验证方法：检查 8080/5173 listener 与 Mailpit 最新邮件；执行 build-only redeploy 后再次触发登录邮件。
- 结果：确认。build artifact 已刷新，但宿主机 backend 未被重启，Mailpit 最新可见邮件仍来自旧运行时配置。

### 假设 3：环境流程没有把调试入口作为契约输出

- 验证方法：读取 `test/scenarios/env-*.sh`、`deploy/dev-stack/README.md` 与 `scenario-env`/`scenario-redeploy` skill。
- 结果：确认。README 有分散说明，但脚本执行完成后没有固定输出可接管的地址、日志和 PID。

## 3 根因分析

- `local-dev-stack/001-bootstrap` Phase 6 对 host-run redeploy 的定义过窄：`backend` 只执行 `go build ./cmd/...`，`frontend` 只执行 `pnpm --filter @easyinterview/frontend build`。
- 在 host-run 拓扑下，构建产物刷新不等于浏览器正在访问的服务已更新；旧进程会继续使用旧 env 和旧代码。
- 环境脚本缺少统一 runtime summary，导致 Agent 启动或重建环境后没有把服务地址、日志路径和 PID 交给开发者。

## 4 修复方案

- 新增 `test/scenarios/_shared/scripts/local-dev-runtime.sh`，统一管理 host-run backend/frontend 的停止、detached 启动、端口等待、日志路径、PID 文件和调试摘要。
- 修订 `test/scenarios/env-redeploy.sh backend|frontend|all`：先 build，再从 `deploy/dev-stack/.env` 重启对应 host-run 进程。
- 修订 `env-setup.sh`、`env-status.sh`、`env-verify.sh`，让创建、巡检、验证环境后输出同一组调试入口。
- 更新 `local-dev-stack` spec/plan/checklist、dev-stack/scenario README 与 scenario env skills，把 redeploy 语义固化为 build + restart + debug handoff。

## 5 验证

- `python3 -m pytest scripts/lint/scenario_env_contract_test.py -q`
- `test/scenarios/env-redeploy.sh backend --dry-run`
- `test/scenarios/env-setup.sh --dry-run`
- `test/scenarios/env-status.sh --dry-run`
- `test/scenarios/env-redeploy.sh all`
- `lsof -nP -iTCP:8080 -iTCP:5173 -sTCP:LISTEN`
- 重新触发 `POST /api/v1/auth/email/start` 后，Mailpit 最新邮件指向 frontend `/auth/verify` callback，不再指向 backend verify API。

## 6 经验教训

- Host-run 环境中的 redeploy 必须明确区分 build、restart、readiness 和 debug handoff；只完成 build 不能宣称本地服务已重新部署。
- Agent 启动的本地环境必须把地址、日志、PID 和重启命令作为稳定输出，否则开发者无法有效接管调试。

# BUG-0021: frontend shell TopBar drifted from ui-design source structure

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-08 | 关联 Commit: `fix(frontend-shell): restore topbar ui-design source parity`
> 关联日志: `-`

## 1 现象

- 用户通过截图复查 `frontend-shell/002-app-shell-visual-system` 后指出正式前端与 `ui-design` 静态原型差异明显：正式 TopBar 使用 native theme select、独立 custom accent 按钮、带文字 dark button 和 language select；`ui-design/src/app.jsx` 则是 theme button + menu、Custom 行内 AccentPicker、icon-only dark toggle 和 language icon toggle。
- 正式前端缺少 `ui-design` 品牌副标题 `面试训练器 · v1.0` 与五个导航 icon；视觉密度、控件层级和交互节奏与原型截图不一致。
- 既有 Vitest / Playwright gate 仍通过，因为旧 gate 主要断言控件存在、高度、padding、颜色等局部 computed style，没有反向比对 `ui-design/src/app.jsx` 的实际控件结构和交互路径。
- 旧 spec / README / scenario 文档仍把语言切换写成 dropdown/select，和当前 `ui-design/` 真理源冲突，增加了实现漂移复发风险。

## 2 诊断过程

### 假设 1：正式前端只是样式值有轻微偏差

- 验证方法：读取 `frontend/src/app/topbar/TopBar.tsx`、`topbar.css`、`ui-design/src/app.jsx` 与截图对照。
- 结果：排除。差异不是单个 CSS 值，而是 DOM 构图和交互模型不同：theme/lang 都是 native select，customAccent 是独立 popover，不是 `ui-design` menu 内的 Custom row。

### 假设 2：`ui-design` 源码本身仍要求 dropdown/select

- 验证方法：读取 `docs/ui-design/ui-architecture.md`、`ui-design/src/app.jsx` 与 `frontend-shell` spec/plan。
- 结果：部分确认。当前 UI 真理源已经是 language toggle 和 theme menu，但 `frontend-shell` 旧 spec/plan/README/scenario 仍保留 dropdown/select 口径，形成了文档层误导。

### 假设 3：pixel parity gate 应该已经挡住该问题

- 验证方法：运行原有 `pnpm --filter @easyinterview/frontend test:pixel-parity --project=desktop frontend/tests/pixel-parity/topbar.spec.ts frontend/tests/pixel-parity/layout.spec.ts` 并读取断言。
- 结果：确认 gate 缺口。旧 Playwright spec 只比较 TopBar 高度、padding、border、5 个 nav testid 等，不断言 brand subtitle、nav icons、theme menu、Custom row 内嵌 AccentPicker、language toggle 等源码级结构。

## 3 根因分析

- 002 plan 的视觉实现把“源级复刻”降解成了局部样式迁移：测试固化了 className、CSS variable 和少量 computed style，但没有把 `ui-design/src/app.jsx` 的 TopBar DOM/interaction shape 转成强断言。
- 003 pixel gate 补了真实浏览器 viewport / bounding-box / screenshot regression，但 TopBar structural assertions 仍偏浅，无法发现 native select 与 menu/toggle 的交互差异。
- D1 i18n 修复期间把语言控件临时固化为 select/dropdown，后续 D2 视觉迁移没有反向修订 001 spec、frontend README 与 E2E scenario 文案，导致旧口径继续作为事实来源存在。

## 4 修复方案

- 重写 `frontend/src/app/topbar/TopBar.tsx` 的显示控制区域：按 `ui-design/src/app.jsx` 复刻 brand subtitle、五入口 icon、theme button + menu、4 个 theme option、Custom row 内嵌 AccentPicker、icon-only dark toggle 与 `topbar-lang-toggle`。
- 更新 `frontend/src/app/topbar/topbar.css`：新增 theme menu、menu backdrop、theme option、custom accent section、icon-only control、brand subtitle 与 nav icon 的源级样式。
- 收紧 Vitest 与 Playwright gate：`TopBarVisual.test.tsx` / `TopBar.test.tsx` / `topbar.spec.ts` 断言旧 native select 不存在，并验证 theme menu、custom option、language toggle、brand subtitle、nav icons 和 custom accent picker 交互。
- 更新受影响的 i18n、scenario、pixel parity 测试选择器，避免旧 `topbar-theme-select` / `topbar-lang-select` / `topbar-custom-accent-button` 继续作为正向契约。
- 原地修订 `frontend-shell` spec/history、001/002 plan/checklist/BDD、frontend README 与 E2E expected outcome，把旧 dropdown/select 口径改为当前 `ui-design` 源码一致的 menu/toggle 口径，并在 002 checklist 追加 L2 remediation gate。

## 6 经验教训

- UI parity gate 不能只比较局部 computed style；对 `ui-design` 源级复刻，必须把 DOM shape、控件类型、菜单层级、图标、aria 和主要交互路径纳入断言。
- 当 `ui-design/` 真理源更新或被重新确认时，旧 spec/plan/README/scenario 的正向口径必须同步删除或改写，否则会把历史实现细节误当成当前契约。
- Pixel parity gate 应区分“视觉几何不重叠”和“源码结构一致”两个层面；前者不能替代后者。

# BUG-0105: manual full-funnel real-provider gate missed runtime-only blockers

> 模块: test | 严重度: high | 状态: resolved
> 发现日期: 2026-05-26 | 关联 Commit: `fix(manual-uat): close real provider full funnel blockers (BUG-0105)`
> 关联日志: `-`

## 1 现象

- 执行 `E2E.P0.100` manual UAT 时，fixture/focused gates 已通过，但真实浏览器 + backend + provider 链路连续暴露阻断。
- 前端真实 preview 首次无法跨 origin 调用 backend；resume/report 的 schema-backed provider 输出在真实模型下失败；practice follow-up 在 UI 中按计时器重复追加；report 最终卡在 `REPORT FAILED / AI_OUTPUT_INVALID`。
- 影响范围覆盖登录、JD/resume 解析、practice follow-up、report.generate/report.question_assessment 和 next-round handoff，属于核心 full-funnel 验收阻断。

## 2 诊断过程

### 假设 1：本地 dev-stack 依赖不健康
- 验证方法：运行 `make dev-doctor`，并确认 Postgres migration 已执行。
- 结果：排除。Postgres、Redis、MinIO、Mailpit 均可用，migration 成功。

### 假设 2：真实 frontend preview 未正确打到 backend
- 验证方法：用浏览器和 network/console 观察 `127.0.0.1:4174` 发出的 API 请求。
- 结果：确认。dev-only CORS 缺失会阻断跨 origin 请求；未用 `VITE_EI_API_BASE_URL` 构建 preview 时，`/api/v1/*` 会被 Vite SPA fallback 当成 HTML 返回。

### 假设 3：真实 provider 输出与 schema-backed consumer 不匹配
- 验证方法：查询 `ai_task_runs` 的脱敏状态、feature_key、profile、model、latency 和 validation_status，并用 focused tests 复现 adapter/request/prompt payload 问题。
- 结果：确认。OpenAI-compatible adapter 未对 `OutputSchema` 请求 JSON object mode；profile timeout 低于真实结构化输出耗时；report assessment prompt 把 rubric 传成 `[]`，真实模型遵守“不发明维度”要求返回空 `dimension_results`。

### 假设 4：follow-up 重复来自后端重复生成
- 验证方法：对比 `ai_task_runs` 中 followup_generate 次数与 UI transcript 重复次数，并审查 `AssistantActionRenderer` effect dependencies。
- 结果：排除后端重复生成，确认前端 effect 随 `elapsed` callback identity 变化重复派发同一个 assistant action。

## 3 根因分析

- 计划的自动 gate 覆盖了 fixture/focused contract，但没有在真实 frontend preview、真实 backend、真实 provider、Mailpit session cookie 和 report runner 的组合下执行完整路径。
- 真实模式的关键运行契约没有被单点 gate 固化：
  - backend `cmd/api` 缺少 dev-only CORS allowlist。
  - `openai_compatible` schema-backed calls 未设置 provider JSON-object response mode。
  - `config/ai-profiles.yaml` 的 report/resume/target/practice timeout budgets 偏低。
  - `report.assessment.default` 仍保留 15s timeout，真实 provider 在逐题评估中接近或超过该边界。
  - `review.questionAssessmentPayload` 未把 registry rubric 传入 prompt。
  - `appendSessionEvent(answer_submitted)` 未写入 `practice_turns.answer_summary`，report assessment 只能拿到空摘要。
  - `practice.turn.lightweight_observe.answerSummary` 被错误列为 schema required，真实 provider 偶发漏字段时会把辅助 observation 变成 fail-close。
  - `AssistantActionRenderer` 把 callback identity 放入 action-dispatch effect 依赖。
- 之前没有发现，是因为 fake AI / fixture client 不会暴露真实 provider latency、provider JSON mode、prompt遵循空 rubric 的行为，也不会暴露 preview origin 与 backend origin 的组合问题。

## 4 修复方案

- 在 `backend/cmd/api` 加 dev-only CORS allowlist，并用 focused tests 固化允许 origin、credentials、preflight header 和非 dev 禁用语义。
- 在 OpenAI-compatible adapter 中为 schema-backed calls 设置 `response_format: {"type":"json_object"}`，并用 provider contract test 锁定请求体。
- 提高 full-funnel 相关 AI profile timeout budgets，并增加 catalog guard。
- 修正 `report.generate` / `report.question_assessment` prompt payload，传入 registry rubric dimensions；修正 active `report.question_assessment` prompt/schema 的 `review_status` enum 示例。
- 在 `answer_submitted` 中调用 `practice.turn.lightweight_observe` 生成并持久化 `answer_summary`，report 仍只消费 sanitized summary；`answerSummary` 保留在 prompt/parser/example 中但不作为 schema required。
- 修复 `AssistantActionRenderer`，让 side-effect dispatch 只跟随 action object 变化，不随 callback identity 变化重复触发。
- manual UAT evidence 记录 real-mode frontend 构建必须带 `VITE_EI_API_BASE_URL=http://127.0.0.1:8080/api/v1`。

## 6 经验教训

- Real-provider manual UAT 不能由 fixture/focused tests 代替；完成 gate 必须至少捕获 real frontend base URL、no `Prefer` fixture header、provider/profile/model/latency/task-run count 和 final report/next-round state。
- 对 AI prompt payload，不只检查 prompt 文件和 schema；还要在 production caller/focused test 中断言关键 business context 非空。
- 对 side-effect-only React components，effect 依赖应以事件/action identity 为边界，callback ref 更新不应重新派发业务事件。

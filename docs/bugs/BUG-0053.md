# BUG-0053: appendSessionEvent accepted stale turn submissions

> 模块: practice | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-13 | 关联 Commit: `feat(backend-practice): complete event loop and completion`
> 关联日志: [2026-05-13](../work-journal/2026-05-13.md)

## 1 现象

- 在实现 `backend-practice/002-event-loop-and-completion` 的 `appendSessionEvent` vertical slice 时，两个客户端围绕同一 current turn 竞争提交时，后到请求可能仍以旧 `turnId` 进入状态机。
- 若缺少 stale-turn 校验，第二个请求可能继续写入 event / outbox 或触发下一步 AssistantAction，破坏 `practice_session_events.seq_no` 与当前 turn 的一一对应。
- 影响范围集中在 `POST /practice/sessions/{sessionId}/events kind='answer_submitted' | 'hint_requested' | 'turn_skipped'` 等依赖当前 turn 的请求。

## 2 诊断过程

### 假设 1：仅靠 `clientEventId` 幂等可以拦住重复提交

- 验证方法：检查 replay / mismatch 路径与双客户端不同 `clientEventId` 的提交矩阵。
- 结果：排除。不同客户端使用不同 `clientEventId` 时，per-session replay 不会触发；必须校验请求 payload 中的 `turnId` 是否仍等于锁内读取的 latest turn。

### 假设 2：repository 的 `SELECT FOR UPDATE` 足以表达并发冲突

- 验证方法：检查锁内上下文读取、service state machine 与 append commit 的职责划分。
- 结果：部分确认。行锁能串行化写入，但不能自动判断第二个请求引用的是旧 turn；需要 service 在锁定后的 reservation context 上做 current-turn 语义校验。

## 3 根因分析

- 根本原因：Phase 2 初始 append flow 只验证 `clientEventId` fingerprint 与 session ownership，没有把 `answer_submitted` / `hint_requested` / `turn_skipped` 的 `payload.turnId` 绑定到锁定后的 latest turn。
- 涉及的代码路径：`backend/internal/practice/append_session_event_service.go`、`backend/internal/store/practice/append_event.go`、`backend/cmd/api/practice_http_scenario_test.go`。
- 之前没有发现的原因：原 BDD 文档强调 seq_no 序列化，但缺少“不同 clientEventId 指向同一个 stale turn”的明确负向断言。

## 4 修复方案

- 在 `AppendSessionEvent` service 中，对依赖当前 turn 的 event kind 读取 `payload.turnId` 并与 repository reservation 返回的 latest turn 做一致性校验。
- stale turn 返回 `PRACTICE_SESSION_CONFLICT`，不触发 AI、不写新 event、不泄露另一请求 payload。
- 新增 `append_session_event_service_test.go` stale-turn conflict 覆盖，并在 `TestE2EP0038PracticeEventConcurrentSeqNoStaleTurnConflict` 中固化 Go HTTP scenario 证据。
- 同步 002 checklist / BDD checklist，把 P0.038 从“仅并发 seq_no”修订为 stale-turn 并发竞争代理 + repository/DB 约束组合验证。

## 6 经验教训

- 双轨幂等不能替代业务游标校验；不同 `clientEventId` 的请求仍必须绑定当前 turn。
- BDD gate 中的并发语义应同时覆盖“写入序号不重”和“旧客户端视图被拒绝”两类风险。

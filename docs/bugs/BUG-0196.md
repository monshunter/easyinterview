# BUG-0196: scenario cleanup left repo-managed host runtimes running

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-20 | 关联 Commit: `fix(test): stop host runtimes during cleanup (BUG-0196)`
> 关联日志: [2026-07-20](../work-journal/2026-07-20.md)

## 1 现象

- 执行 `test/scenarios/env-cleanup.sh` 后，PostgreSQL、Redis、MinIO 与 Mailpit 容器均已停止，但 repo-managed frontend/backend 仍监听 `127.0.0.1:10900/10901`。
- `.test-output/local-dev/backend.pid` 与 `frontend.pid` 仍存在，开发者需要额外调用内部 stop target 才能把共享环境完整清理。
- 默认 cleanup 没有删除命名卷，数据安全边界符合预期；缺口只在 host-run app 生命周期。

## 2 诊断过程

### 假设 1：cleanup 已停止 host-run，只是端口被无关进程占用

- 验证方法：检查 listener、pidfile、进程组与当前命令。
- 结果：排除。`10900` 对应 pidfile 管理的 Vite 进程组，`10901` 对应 pidfile 管理的 `go run ./backend/cmd/api` 进程组。

### 假设 2：顶层 cleanup 只委派 Compose 依赖清理

- 验证方法：读取 `test/scenarios/env-cleanup.sh`、根 `dev-down` target 与共享 runtime helper。
- 结果：确认。标准路径只执行 `make dev-down`，而已有安全停止逻辑仅被 redeploy/full-container 切换路径消费。

## 3 根因分析

- `env-cleanup.sh` 的实现仍停留在早期“共享环境只等于 Compose 外部依赖”的合同；后来 `env-redeploy.sh` 已开始持有 host-run backend/frontend，却没有反向修订 cleanup consumer。
- Spec、plan、runbook 与 Skill 只强调“默认保留命名卷”，没有可执行断言要求 cleanup 在 `dev-down` / `dev-reset` 前停止 repo-managed host runtime。
- 既有 live gate 只验证依赖容器停止，没有同时检查 listener 与 pidfile，因此历史 PASS 未覆盖当前完整环境拓扑。

## 4 修复方案

- 在 `local-dev-runtime.sh` 中增加 `stop_host_runtimes`，组合复用现有 PID、命令和 repo cwd ownership 检查。
- `env-cleanup.sh` 在默认 `make dev-down` 和显式 `make dev-reset` 前统一调用该 helper；dry-run 明确输出 host stop 在 dependency cleanup 之前。
- 同步 local-dev-stack Spec/plan/checklist/History、dev-stack/scenario runbook 与 `/scenario-env` Skill，并增加静态顺序、dry-run 双路径、ownership safety 和文档一致性 contract。
- 未采用直接 `kill` 端口 listener 的方案，因为它无法证明进程归属，可能终止手工服务或其它仓库环境。

## 5 验证

- `python3 -m pytest scripts/lint/scenario_env_contract_test.py -q`：28 passed。
- `bash -n test/scenarios/env-cleanup.sh test/scenarios/_shared/scripts/local-dev-runtime.sh`：通过。
- 真实 `env-setup.sh` → `env-redeploy.sh all` → `env-cleanup.sh`：`10900/10901` 关闭，两个 pidfile 消失，EasyInterview 容器清空，三个命名卷保留。
- 同机 Ferry `kubectl port-forward` 继续监听 `8025`，证明 cleanup 未终止手工/无关进程。
- 根 `make test` 最终重跑通过：Python 624 tests / 4628 subtests、Go 全包、frontend 137 files / 1126 tests。

## 6 经验教训

- 环境 lifecycle 的 clean state 必须按当前拓扑同时验证容器、repo-owned host process、pidfile、listener 与数据卷，不能只检查 Compose。
- 新的 runtime producer 接入共享环境后，需要反查 setup/status/verify/cleanup/redeploy 五个 consumer，而不是只修启动或重启路径。

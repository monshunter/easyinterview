# BUG-0175: E2E evidence redaction missed URL-encoded account email

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-15 | 关联 Commit: `feat(settings): simplify account settings and harden evidence privacy (BUG-0175)`
> 关联日志: [2026-07-15](../work-journal/2026-07-15.md)

## 1 现象

- 扩展后的 `E2E.P0.101` 业务断言全部通过，但首轮 `trigger.log` 的最终 URL 保留了 percent-encoded 的完整测试账号 email。
- 原脱敏函数只替换 email 原文；浏览器把同一值编码进 URL 后绕过了替换。
- 该问题不影响 Settings 正常显示完整账号，但使场景证据不满足隐私合同，因此首轮结果不能作为验收 PASS。

## 2 诊断过程

### 假设 1：页面显示完整 email 本身违反隐私合同

- 验证方法：对照 OPENAPI-007、Settings BDD 与用户确认口径。
- 结果：排除。authenticated response 与 Settings 应显示完整 email，限制仅针对日志和 E2E 证据。

### 假设 2：URL 编码产生了脱敏函数未覆盖的等价表示

- 验证方法：检查首轮输出中的 final URL，并让增强后的 `verify.sh` 同时搜索当前 run 的原文与 `encodeURIComponent` 结果。
- 结果：确认。增强验证器先对旧输出 RED，证明 percent-encoded email 可绕过仅原文替换的 gate。

## 3 根因分析

- `test/scenarios/e2e/p0-101-auth-email-code-profile-setup/scripts/trigger.sh` 的脱敏逻辑只处理 email 原文，没有处理浏览器 URL 中的 percent-encoded 等价表示。
- `verify.sh` 原先也没有从当前 run 输入派生编码后的负向检查值，因此日志即使泄漏仍可能误报 PASS。
- 场景框架 README 只要求“脱敏证据”，此前未明确列出 URL 编码等价表示。

## 4 修复方案

- `trigger.sh` 同时替换 email 原文与 `encodeURIComponent(email)` 结果。
- `verify.sh` 从当前 setup 输入派生原文和 URL-encoded 值，任一命中都 fail closed。
- 在场景框架 README 固化“原文 + 实际日志编码等价表示”的隐私门禁，避免其他浏览器场景重复遗漏。
- 不对 Settings 的 authenticated 页面值做掩码，也不放宽真实账号字段断言。

## 5 验证

- RED：增强后的 `verify.sh` 对首轮遗留输出明确拒绝 current-run URL-encoded email。
- GREEN：清理后重新运行真实 frontend/backend/Mailpit 场景，run `e2e-p0-101-20260715114513-19516` PASS；设置齿轮、完整账号字段与 logout 均通过，证据中原文和 URL-encoded email 均为零命中。
- 回归：四个场景脚本 `bash -n` 通过，根 `make test`、文档与差异门禁通过。

## 6 经验教训

- 隐私值进入 URL、JSON、HTML 或 shell 输出后可能产生多个等价表示，证据 gate 必须从当前 run 输入派生并覆盖真实编码路径。
- 用户界面展示合同与测试证据留存合同应分开描述，避免为了日志脱敏错误地掩码 authenticated UI。

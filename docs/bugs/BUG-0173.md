# BUG-0173: evidence privacy gate rejected benign PNG metadata

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-15 | 关联 Commit: `feat(report): implement readonly conversation record (BUG-0172, BUG-0173)`
> 关联日志: [2026-07-15](../work-journal/2026-07-15.md)

## 1 现象

- P0.099 的截图证据校验器只要发现 PNG `iCCP`、`tEXt`、`zTXt` 或 `iTXt` chunk 就失败，没有判断其中是否包含项目用户数据或 secret。
- Chrome 生成的合法截图可以携带普通色彩配置等技术元数据，因此真实浏览器验收会被与隐私无关的开发过程信息阻断。
- 用户数据、认证材料、JD、简历、transcript 与运行时 secret 仍必须 fail closed；本 Bug 不放宽这些内容的保护。

## 2 诊断过程

### 假设 1：所有 PNG metadata 都可能还原用户数据

- 验证方法：读取 `validate_evidence.py` 的 chunk 扫描逻辑，并构造只包含普通 `iCCP` profile 的有效 PNG。
- 结果：排除。旧逻辑只按 chunk 类型拒绝，无法证明 payload 含用户内容。

### 假设 2：隐私规则应按内容风险而不是开发文件结构判断

- 验证方法：增加同一校验入口的对照测试：普通 `iCCP` 应通过，包含 `ei_session=` 的 `tEXt` 必须失败。
- 结果：确认。Red 测试先证明普通技术元数据被误拒绝；Green 测试证明内容级扫描可以同时允许技术 profile 并拦截会话材料。

## 3 根因分析

- `validate_evidence.py` 将 PNG metadata chunk 类型本身定义为隐私违规，把“可能承载文本的容器”误当成“已经包含用户隐私的数据”。
- 原有测试只覆盖无 metadata 的最小 PNG，没有覆盖 benign metadata 与 sensitive metadata 的正反对照。
- 设计文档此前没有明确区分项目用户数据门禁与开发过程文件完整性门禁，导致校验器范围扩大。

## 4 修复方案

- 保留 PNG chunk、CRC、尺寸、manifest SHA-256 与 current-run digest 的独立完整性校验。
- 对 `iCCP`、`tEXt`、`zTXt`、`iTXt` payload 运行现有敏感内容模式扫描，只在命中项目用户数据、认证材料或 secret 时拒绝。
- 在 e2e-scenarios-p0 spec/plan/checklist 与场景 README 中固化隐私范围，避免后续再次按开发元数据类型一刀切。
- 不引入新的 metadata allowlist，也不移除对未知/损坏 PNG 结构的 fail-closed 校验。

## 5 验证

- Red：`test_png_metadata_gate_allows_technical_profile_but_rejects_session_material` 证明旧校验器错误拒绝普通 `iCCP`。
- Green：`python3 test/scenarios/e2e/p0-099-report-generating-live-ui/scripts/test_report_conversation_evidence.py` 共 7 项通过；普通 `iCCP` 被接受，含 `ei_session=` 的 `tEXt` 被拒绝。
- 真实回归：P0.099 current run `e2e-p0-099-20260715T021319Z-57232` 完成精确六张 Chrome 截图、人工 no-OCR 审计、live API/PostgreSQL binding 并由 `verify.sh` 返回 PASS。

## 6 经验教训

- 隐私门禁应围绕项目用户数据和 secret 的可泄露性建模，不能把文件格式的可选技术字段直接等同于隐私数据。
- 内容安全、证据完整性和开发过程可复现性是三个独立 gate；应分别失败、分别给出证据，避免一个宽泛规则掩盖真实风险。

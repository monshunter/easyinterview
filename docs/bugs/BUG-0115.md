# BUG-0115: unauthenticated home and interview routes mounted protected flows before login

> 模块: frontend | 严重度: high | 状态: resolved
> 发现日期: 2026-05-28 | 关联 Commit: `fix(frontend-shell): gate protected routes before login (BUG-0115)`
> 关联日志: [2026-05-28](../work-journal/2026-05-28.md)

## 1 现象

- 未登录用户打开 Home 时，页面仍挂载 Recent mock interviews 区块并触发 `listTargetJobs`，后端返回未授权后把 raw unauthorized 错误暴露到 Home。
- 未登录用户点击 Home 上的 JD 导入、简历创建、最近面试卡片或直接访问 `workspace`、`practice`、`report`、`debrief`、`resume_versions` 等面试相关路由时，部分业务页面会先挂载，再由页面内部 auth gate 或 API 错误收口。
- 影响范围覆盖 Home、App route dispatch、报告/练习/简历等 protected route，以及后端 route-middleware 保护证据。

## 2 诊断过程

### 假设 1：后端业务 API 没有统一 session middleware

- 验证方法：补充并运行 `backend/cmd/api/main_test.go` 中的 practice/profile route middleware case，连同既有 targetjob、upload、resume、report、job route case 一起检查未登录请求。
- 结果：排除。后端关键业务 API 均返回 `AUTH_UNAUTHORIZED`，问题主要在前端未登录态仍过早挂载业务入口或发起业务请求。

### 假设 2：Home 的最近面试数据 hook 未按认证态短路

- 验证方法：读取 `frontend/src/app/screens/home/useRecentTargetJobs.ts` 与 Home 视图挂载条件，并新增未登录断言。
- 结果：确认。修复前 hook 初始化即进入加载/请求路径，Recent 区块也没有签入态显示条件，导致未登录 Home 可见受保护区块和错误文案。

### 假设 3：App 缺少统一 protected route 前置门禁

- 验证方法：读取 `frontend/src/app/App.tsx` 的 route dispatch、auth/profile setup 分支，并用 `AppAuthDispatch.test.tsx` 枚举 protected routes。
- 结果：确认。修复前 protected route 可以先进入业务屏幕，导致每个业务页面各自承担未登录收口，行为不一致。

## 3 根因分析

- 根本原因：认证门禁分散在业务页面内部，App route 层没有统一把非 Home、非 `auth_*` 路由识别为 protected route；Home 最近面试区块也没有绑定 `runtime.auth.status === "authenticated"`。
- 涉及的代码路径：
  - `frontend/src/app/App.tsx:217`：统一 protected route 渲染替换和 `auth_login` pendingAction 编码。
  - `frontend/src/app/App.tsx:245`：认证加载/错误期间的 `auth-route-gate`，避免业务屏幕提前挂载。
  - `frontend/src/app/App.tsx:343`：`home` 与 `auth_*` 之外均作为 protected route。
  - `frontend/src/app/screens/home/useRecentTargetJobs.ts:19`：未认证时清空并短路，不调用 `listTargetJobs`。
  - `frontend/src/app/screens/home/HomeScreen.tsx:37`：Recent mock interviews 仅在 authenticated 时显示。
- 之前没有发现的原因：已有测试更偏向业务页面内部 auth gate 或 fixture-backed happy path，缺少“未登录入口先跳登录、业务 API 不触发”的统一路由级负向覆盖。

## 4 修复方案

- 选择的方案：在 App 层加入统一 protected route guard，并保留 Home 与 `auth_*` 登录链路例外；Home 内的业务动作使用 `useRequestAuth` 携带 pendingAction；Recent mock interviews 仅签入后显示。
- 考虑过但放弃的替代方案：继续让各业务页面自行展示 auth gate。该方案会保留多入口行为不一致，并无法保证业务 API 在未登录路由进入前零触发。
- 修改的文件列表：
  - `frontend/src/app/App.tsx`
  - `frontend/src/app/screens/home/HomeScreen.tsx`
  - `frontend/src/app/screens/home/useRecentTargetJobs.ts`

## 6 经验教训

- 对认证边界，应优先在 App route 层建立统一 protected-route gate，业务页面内部 auth gate 只能作为兜底，不应成为首要入口策略。
- Home 这类公开入口若包含用户历史、最近记录或业务 API 数据，必须同时验证“未登录不显示、不请求、不泄漏 raw error”。
- 新增用户可见 auth 行为时，应配套一个场景级负向用例，覆盖 UI 原型、正式前端和后端 middleware 证据，而不只覆盖单页组件。

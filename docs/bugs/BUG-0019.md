# BUG-0019: frontend shell review remediation missed build and auth edge gates

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-07 | 关联 Commit: `fix(frontend-shell): harden app shell review remediation`
> 关联日志: [2026-05-07](../work-journal/2026-05-07.md)

## 1 现象

- PR review 复查 `frontend-shell/001-app-shell-auth-settings` 后发现 `pnpm --filter @easyinterview/frontend build` 会执行 `vite build`，但 `frontend/` 没有 Vite HTML entry，根 `make build` 因 `Could not resolve entry module "index.html"` 失败。
- `/me` 使用未知 fixture scenario 时，`AppRuntimeProvider` 把 mock wiring 错误吞成 `unauthenticated`。
- 未登录用户从 pendingAction 进入 login 后点击注册，`auth_register` route 丢失 pendingAction params。
- 只有 raw `returnTo=/practice?...` 的 verify 链路会丢失 query 中的 plan / target job / JD / resume / round 上下文。

## 2 诊断过程

### 假设 1：frontend build 已有隐式 Vite entry

- 验证方法：运行 `pnpm --filter @easyinterview/frontend build` 并检查 `frontend/` 文件列表。
- 结果：排除。命令因缺 `index.html` 失败，`frontend/` 仅有 package / config / src 文件，没有 Vite 默认入口。

### 假设 2：`/me` unknown scenario 属于可接受 signed-out fallback

- 验证方法：读取 `AppRuntimeProvider`、`mockTransport`、plan 1.3 和 `frontend/README.md` 的 fail-loud 约束，并补 focused test 使用 `Prefer: example=does-not-exist` 调 `/me`。
- 结果：排除。`401` 是 signed-out；unknown scenario 是 fixture wiring 错误，必须显式暴露。

### 假设 3：login -> register 不需要携带 pendingAction

- 验证方法：读取 `AuthLoginScreen` / `AuthRegisterScreen` 参数传递逻辑，并补 component test 从带 pendingAction 的 login 点击注册。
- 结果：排除。注册页提交后会继续进入 verify；若切换注册时丢掉 params，verify 无法恢复用户原本请求的动作。

### 假设 4：raw `returnTo` query 不承载业务上下文

- 验证方法：读取 `AuthVerifyScreen.buildResumeRoute` 和后端 / OpenAPI `returnTo` 约定，并补 focused test 覆盖 `/practice?planId=...`。
- 结果：排除。raw `returnTo` 是外部 email link fallback，query 中的 interview context 必须进入恢复 route params。

## 3 根因分析

- `frontend/package.json` 将 build 从占位脚本切换为 `tsc --noEmit && vite build`，但没有同步落地最小 Vite `index.html` / `src/main.tsx`。
- `AppRuntimeProvider` 的 `/me` catch-all 没有区分 HTTP 401 与 transport / fixture 错误，导致 mock-contract fail-loud gate 只覆盖 runtime config，漏掉 auth probe。
- `AuthLoginScreen` 注册跳转复用了空 params 写法，和 `AuthRegisterScreen` 已支持透传 pendingAction 的设计不一致。
- `AuthVerifyScreen` 只从 `route.params` 中抽 interview keys，解析 raw `returnTo` 时丢弃 query string。
- 之前的 focused tests 覆盖了主路径和 auth-only params 剥离，但没有覆盖 build smoke、login/register 切换、`/me` unknown scenario、raw `returnTo` fallback 这些边缘 gate。

## 4 修复方案

- 新增 `frontend/index.html` 和 `frontend/src/main.tsx`，让 Vite build 拥有真实入口并挂载 `App` + generated client。
- 扩展 `AuthState` 增加 `error` 状态；`/me` 仅在 HTTP 401 时收敛为 `unauthenticated`，其他错误保持 visible error。
- `AuthLoginScreen` 点击注册时透传当前 route params。
- `AuthVerifyScreen` 使用 `URL` / `URLSearchParams` 解析 raw `returnTo`，把允许的 interview context query keys 合并到恢复 route params。
- 补充 focused tests 覆盖四个 review findings。

## 6 经验教训

- 当 package script 从占位切换为真实 bundler gate 时，必须同轮运行根级 `make build`，而不是只跑 `typecheck`。
- fail-loud gate 要覆盖每个 runtime probe；`getRuntimeConfig` 通过不代表 `/me` 的 unknown scenario 也会暴露。
- pendingAction 主路径之外，还要覆盖 auth 页之间跳转和 external link fallback 两类恢复入口。

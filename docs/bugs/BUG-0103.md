# BUG-0103: empty focus competency arrays broke practice plan creation

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-05-24 | 关联 Commit: `feat(e2e-scenarios): close full funnel journey (BUG-0103)`
> 关联日志: `-`

## 1 现象

- `E2E.P0.099` full-stack Playwright journey 在点击 report 的 next-round CTA 后触发 `createPracticePlan`，请求体包含 `focusCompetencyCodes: []`。
- 真后端返回 generic `500`，导致下一轮 plan/session handoff 失败；同类空数组也会影响 baseline plan 创建。
- 问题只在真实 PostgreSQL 写入路径暴露，fixture/mock UI 路径无法覆盖。

## 2 诊断过程

### 假设 1：前端 next-round handoff 丢失必要上下文字段

- 验证方法：在 Playwright spec 中捕获 `POST /practice/plans` 与 `POST /practice/sessions`，同时检查 URL query 中的 `sourceReportId`、`targetJobId`、`resumeAssetId`、`planId` 和 `sessionId`。
- 结果：排除。前端已带上 next-round handoff 所需字段，失败发生在后端创建 plan 写入阶段。

### 假设 2：后端 create plan 写入空 focus code 时违反持久化约束

- 验证方法：增加 `TestE2EP0098CreatePracticePlanAcceptsEmptyFocusCodes`，用真实 handler + PostgreSQL 路径发送 `FocusCompetencyCodes: []`，并检查 `practice_plans.focus_competency_codes` 的 cardinality。
- 结果：确认。service/store 之间的空 slice 被传成 nil，`pq.Array(nil)` 写入 SQL NULL，违反 `practice_plans.focus_competency_codes NOT NULL`。

## 3 根因分析

- 根本原因：`backend/internal/store/practice/create_plan.go` 在调用 `pq.Array` 前没有把 nil `FocusCompetencyCodes` 归一化为空 slice。
- 涉及的代码路径：`backend/internal/store/practice/create_plan.go:30` 的 `CreatePlan` 插入路径；回归测试覆盖 `backend/internal/store/practice/create_plan_test.go:103` 与 `backend/cmd/api/full_funnel_journey_scenario_test.go:85`。
- 为什么之前没有发现：已有 store 测试使用非空 focus code 或宽松 `sqlmock.AnyArg()`，前端 fixture/mock flow 也不会触发 PostgreSQL `NOT NULL` 数组写入约束。

## 4 修复方案

- 选择的方案：在 store 写入前执行 `focusCompetencyCodes := append([]string{}, in.FocusCompetencyCodes...)`，保证 nil input 进入 `pq.Array` 时编码为 PostgreSQL 空数组而不是 NULL。
- 放弃的替代方案：在前端强制补默认 focus code。该方案会把后端持久化契约问题推给调用方，且无法覆盖其他 API consumer。
- 修改的文件列表：
  - `backend/internal/store/practice/create_plan.go`
  - `backend/internal/store/practice/create_plan_test.go`
  - `backend/cmd/api/full_funnel_journey_scenario_test.go`

## 5 验证

- `cd backend && go test -v ./internal/store/practice -run 'TestSQLRepositoryCreatePlanNormalizesEmptyFocusCompetencyCodes|TestSQLRepositoryCreatePlanWritesPlanAndAuditInOneTransaction|TestStartSessionAdvisoryLockKeyIsPostgresTextSafe|TestSQLRepositoryReserveSessionStart' -count=1`
- `cd backend && DATABASE_URL='postgres://user:***@localhost/easyinterview?sslmode=disable' go test -v ./cmd/api -run '^TestE2EP0098CreatePracticePlanAcceptsEmptyFocusCodes$' -count=1`
- `E2E.P0.099` scenario wrapper passed after the fix; Playwright completed next-round `createPracticePlan` + `startPracticeSession` against the real backend.

## 6 经验教训

- Store tests for PostgreSQL arrays should assert encoded empty-array behavior, not only use `sqlmock.AnyArg()`.
- Full-funnel real-backend scenarios are necessary for catching fixture-invisible persistence contract drift.

# BUG-0183: host-run SMTP port drift left the local Mailpit mailbox empty

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-16 | 关联 Commit: `fix(report): harden regeneration and local diagnostics (BUG-0182, BUG-0183)`
> 关联日志: -

## 1 现象

- 本地 Mailpit Web UI 正常监听，但发起 email-code challenge 后 Inbox 没有任何新邮件。
- 后端和前端都已启动，问题只影响 host-run 邮件投递；full-container 内部 Mailpit 路由与外部 SMTP 配置不应被改变。
- 用户无法通过真实邮件链路完成本地登录，只能误以为邮件任务或 Mailpit 本身失效。

## 2 诊断过程

### 假设 1：Mailpit 容器或 Web UI 不健康

- 验证方法：检查 scenario environment 状态、Mailpit Web endpoint 与 Compose service health。
- 结果：排除。Mailpit UI 可访问，容器健康。

### 假设 2：邮件任务没有创建或被其他 runtime 领取

- 验证方法：发起 fresh challenge，检查 host backend 邮件任务与 SMTP 连接错误。
- 结果：部分排除。任务存在，但连接到了错误的 host 端口并失败。

### 假设 3：host-run backend 和 Compose host mapping 使用不同端口

- 验证方法：比较动态 `MAILPIT_SMTP_HOST_PORT` 与 backend 实际读取的 `EMAIL_SMTP_PORT`，同时核对 full-container 的 service-internal route。
- 结果：确认。host mapping 与 backend SMTP port 不一致；full-container 使用 `mailpit-dev:1025` 的内部路由本身正确。

## 3 根因分析

- `deploy/dev-stack/.env` 的 host-run backend SMTP 端口仍是容器内部端口，而 Compose 将 Mailpit SMTP 映射到另一个 host 端口。
- 旧启动流程只检查依赖是否健康，没有在 backend 启动前验证“loopback Mailpit provider 的业务端口必须等于当前动态 host mapping”。
- Skill 不是业务配置 owner；如果把端口写入 Skill，只会产生第三份容易漂移的真理源，因此问题必须在 scenario runtime/env 层 fail fast。

## 4 修复方案

- 在 `test/scenarios/_shared/scripts/local-dev-runtime.sh` 增加 `assert_host_mailpit_smtp_route`，读取已加载的动态环境，并在 host backend 启动前校验 loopback Mailpit SMTP port 与 Compose host mapping 一致。
- 校验失败只报告字段名与端口，不输出邮箱、验证码或 SMTP secret。
- full-container 的 service-internal Mailpit route 和 external SMTP provider 不走该 guard；Skill 只继续消费 README、Makefile 和 lifecycle script 输出，不耦合任何业务端口。
- 当前本地环境同步到有效 host mapping 后重启 backend。

## 5 验证

- 25 个 scenario environment contract tests 通过，包含 mismatch fail-fast、full-container/external SMTP 旁路和 Skill 无端口字面量负向断言。
- shell syntax、runtime verify、frontend/backend 10900/10901 endpoint 与 runtime-config 检查通过。
- 在真实环境发起 fresh challenge 后，Mailpit 收到一封新邮件；Chrome 插件使用该邮件链路完成登录，证据未记录完整邮箱或验证码。
- 最终根 `make test`、`make build`、context/docs/index/diff gates 全部通过。

## 6 经验教训

- 同一容器服务的 internal port 与 host mapping 是两套地址空间；host-run 进程必须从当前 environment projection 解析 host mapping。
- “容器健康”不等于“宿主进程路由正确”，本地环境 doctor 应检查实际消费者使用的 endpoint。
- Skill 应保持业务无关；动态配置一致性由 scenario runtime 或 environment script 承担。

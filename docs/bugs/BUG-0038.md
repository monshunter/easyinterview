# BUG-0038: jd_match search parity and pixel gate drift escaped L2 review

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-10 | 关联 Commit: `fix(frontend-jd-match): restore search parity and clean pixel gate`
> 关联日志: [2026-05-10](../work-journal/2026-05-10.md)

## 1 现象

- `/plan-code-review 002-jd-match-recommendations --fix` 复查发现正式 `SearchTab` 渲染 `Data sources` heading，缺少 `NATURAL LANGUAGE SEARCH` / `自然语言搜索` label、search icon、`SOURCES` / `数据源` label，并且 source chips 只有 4 个，缺少 Company sites / 公司官网。
- `jd_match` pixel parity 常规命令在 clean checkout 形态下失败：`toHaveScreenshot` 需要的 desktop/mobile baseline PNG 位于 `frontend/tests/pixel-parity/*-snapshots/`，但该目录被 `.gitignore` 忽略。
- 原有 SearchTab Vitest、pixel parity 与 P0.028 scenario verify 只证明容器或 4-source 旧口径存在，未能捕获 ui-design 源级差异。

## 2 诊断过程

### 假设 1：正式 SearchTab 与 ui-design 真理源存在源级结构漂移

- 验证方法：直接对照 `ui-design/src/screen-jd-match.jsx::SearchTab` 与 `frontend/src/app/screens/jd_match/SearchTab.tsx`、zh/en locale。
- 结果：确认。ui-design 要求自然语言搜索标题、search icon、source label 与 5 个 source chips；正式实现只渲染 `dataSourcesHeading` 和 4 个 source chips。

### 假设 2：现有测试已经覆盖 SearchTab 源级 parity

- 验证方法：读取 `SearchTab.test.tsx`、`jdMatchLocaleNamespaces.test.ts`、`frontend/tests/pixel-parity/jd_match.spec.ts` 和 P0.028 verify script。
- 结果：排除。测试只断言 sources 容器或 4 个 filter chips，没有断言 natural-language label、search icon、source label、company source 或 i18n key。

### 假设 3：pixel parity 失败是视觉 diff

- 验证方法：运行 `pnpm --filter @easyinterview/frontend exec playwright test tests/pixel-parity/jd_match.spec.ts` 并查看错误。
- 结果：排除。失败原因是 snapshot baseline 文件不存在；Playwright 写入 actual PNG，但目标 `*-snapshots/` 目录被 `.gitignore` 忽略，无法作为 clean checkout 常规 gate。

## 3 根因分析

- 根本原因：plan 002 完成态 evidence 把历史 PASS、弱 DOM anchor 和本地 ignored screenshot baseline 当作充分证据，没有把 ui-design SearchTab 的静态 source row 和 clean-checkout pixel gate 可复现性固化成可执行断言。
- 涉及路径：`frontend/src/app/screens/jd_match/SearchTab.tsx`、`frontend/src/app/i18n/locales/{zh,en}.ts`、`frontend/tests/pixel-parity/jd_match.spec.ts`、`test/scenarios/e2e/p0-028-jd-match-search-and-saved/scripts/verify.sh`。
- 之前未发现的原因：L2 remediation 重点放在 detail fetch、auth resume、responsive geometry 和 scenario script coverage，未反向审计 SearchTab line 480-512 的非 loading 区域，也没有在干净环境校验 ignored screenshot baseline。

## 4 修复方案

- 按 ui-design 源级复刻 Search 输入区：新增 natural-language heading、search icon、`SOURCES` / `数据源` label、Company sites / 公司官网 source chip 与原型 source counts。
- 补强 i18n 和测试：`jdMatch.search.naturalLanguageHeading`、`jdMatch.search.dataSourceCompany` 加入 zh/en locale 与 namespace gate；`SearchTab.test.tsx` 覆盖 5 source chips 和中文 company source。
- 修复 pixel gate：去掉常规运行对 ignored `*-snapshots/` baseline 的依赖，改为 DOM anchor、computed style、responsive geometry 和 focused screenshot smoke（bounding box + non-empty screenshot buffer）。
- 补强 P0.028 scenario verify，反查 SearchTab 源级 parity assertions 和 pixel source-company assertion。

## 6 经验教训

- UI parity review 必须逐段覆盖 ui-design 源文件，不应只审查主路径或当前 diff 较大的区域。
- Screenshot parity gate 若依赖 baseline，baseline 必须能在 clean checkout / CI 中复现；ignored 本地 PNG 只能作为调试产物，不能作为完成证据。
- Scenario verify 应反查关键 parity assertions 是否存在，避免单元测试从“源级复刻”退化成“容器存在”。

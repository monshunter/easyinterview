# BUG-0164: P0.058 verifier counted Go subtests as root tests

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-12 | 关联 Commit: `fix(report): ground report semantics and reliability (BUG-0164, BUG-0165, BUG-0166)`
> 关联日志: [2026-07-13](../work-journal/2026-07-13.md)

## 1 现象

- E2E.P0.058 的七个前端文件、三个后端包及全部失败/恢复 marker 均通过，但 `verify.sh` 无诊断地返回非零，未生成 `backend-evidence.json`。
- 同一份 `backend.log` 中，三个包的根测试各有一条 `RUN` / `PASS`，同时包含 11 条同名前缀 subtest 日志。
- 该误报阻断报告失败路径验收，但不代表产品代码或共享环境故障，因此严重度为 `medium`。

## 2 诊断过程

### 假设 1：后端或前端 focused runner 实际失败

- 验证方法：逐项检查 `trigger.log` / `backend.log` 的目标文件、package status、六个后端 marker、八个数据库断言以及 FAIL / no-tests 负向标记。
- 结果：排除。七个前端文件 41 个测试通过，三个后端包的精确测试均通过，要求的 marker 与数据库断言完整。

### 假设 2：共享环境或迁移状态不完整

- 验证方法：运行 repo-tracked env preflight，并只读查询 `schema_migrations`。
- 结果：排除。PostgreSQL、Redis、MinIO、Mailpit 均健康；数据库为 `version=19, dirty=false`。

### 假设 3：verifier 的根测试计数包含 subtest

- 验证方法：分别使用前缀计数与精确整行/根 PASS 锚点统计同一 `backend.log`。
- 结果：确认。前缀计数得到 `14/14`，精确根测试计数得到期望的 `3/3`。

## 3 根因分析

- `verify.sh` 使用 `grep -Fc '=== RUN   TestE2EP0058ReportFailureBackendEvidence'` 和同类 PASS 前缀计数。
- Go subtest 行形如 `TestE2EP0058ReportFailureBackendEvidence/<subtest>`，也包含根测试前缀，导致三个根测试加十一个 subtest 被统计为 14。
- `set -e` 在无说明的 `test ... -eq 3` 处直接退出，因此用户只看到 verifier 非零而没有定位信息。
- 原 scenario contract 只要求精确命名测试存在，没有锁定 verifier 必须区分根测试与 subtest，故脚本漂移未被静态门禁发现。

## 4 修复方案

- RUN 计数改为 `grep -Fxc` 精确整行，只接受三个包各一条根测试启动记录。
- PASS 计数改为根行正则，要求测试名后直接跟运行时长括号，不匹配 `/subtest`。
- 在 `scenario_env_contract_test.py` 增加 source contract，禁止恢复前缀计数。
- 保留 subtest 本身及其 marker；未采用删除 subtest 或放宽期望数量，因为两者都会削弱证据精度。

## 5 验证

- RED：修复前 `run_prefix_count=14`、`pass_prefix_count=14`，P0.058 verifier exit 1；新增 contract test 失败。
- GREEN：精确根计数为 `3/3`，focused contract test 通过。
- 复用原始完整 runner 日志重跑 `verify.sh`，输出 `E2E.P0.058 PASS` 并生成 schema-valid `report-backend-evidence.v2`。
- `cleanup.sh` 成功，保留脱敏 backend artifact 与 `investigation.json`，移除 transient logs/setup metadata。

## 6 经验教训

- Go `-run '^TestName$'` 只限制根测试选择，不会阻止该测试内部执行 subtest；verifier 对 runner 日志计数时必须明确根行边界。
- 验收脚本应优先使用精确整行或结构化结果，不能用会把层级日志折叠在一起的前缀计数。
- 本问题归入现有 PATTERNS 模式 4“Completed checklist 掩盖未执行的 runner gate”；无需新增平行模式。

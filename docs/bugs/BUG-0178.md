# BUG-0178: host-run backend claimed full-container email jobs without the delivery secret

> 模块: platform | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-16 | 关联 Commit: `feat(auth): add production smtp delivery (BUG-0178)`
> 关联日志: [2026-07-16](../work-journal/2026-07-16.md)

## 1 现象

- full-container backend 发起 email-code challenge 后，`async_jobs` 可被宿主机残留 backend 抢先 lease。
- 抢占任务的宿主机进程没有发起进程内 transient delivery secret，任务因此失败或重试，容器内 SMTP writer 没有机会投递。
- 影响本地 Mailpit 与外部 SMTP 的部署验收；单独检查 Compose 健康状态无法发现该竞争。

## 2 诊断过程

### 假设 1：SMTP provider、TLS 或认证配置错误

- 验证方法：分别探测 Mailpit plain SMTP、外部 implicit TLS/auth，并检查 provider 对 `MAIL FROM`、`RCPT TO`、`DATA` 的脱敏阶段结果。
- 结果：排除；两类 SMTP endpoint 都能接受对应连接和邮件数据。

### 假设 2：full-container backend 没有创建或消费 email job

- 验证方法：检查 challenge 202 响应、`async_jobs` lease/attempt 状态和 backend 容器日志。
- 结果：部分排除；job 已创建，但 lease owner 不属于预期容器进程。

### 假设 3：宿主机 backend 与容器 backend 竞争同一队列

- 验证方法：核对 repo PID 文件、运行进程与 job lease 时序；停止仓库管理的 host-run app 后重复 provider-only Mailpit challenge。
- 结果：确认；停止 host-run backend 后任务由容器 runner 一次成功投递并可完成 verify/session/me 闭环。

## 3 根因分析

- backend internal runner 随每个 backend 进程启动并消费同一 `async_jobs` 表，而验证码明文只存在于发起 challenge 的进程内 transient delivery secret store。
- `dev-container-up` 早期没有先停止仓库 PID 文件管理的 host-run backend，造成两个 runner 同时可 lease 任务。
- Compose health、HTTP health 和 SMTP connectivity 都不会单独揭示跨进程 transient secret 所有权，因此问题只在真实发码链路出现。

## 4 修复方案

- `dev-container-up` 在启动 full-container profile 前，只停止仓库 PID 文件管理的 host-run backend/frontend，不杀无关进程、不清理命名卷。
- scenario env contract 锁定上述生命周期语义，Mailpit provider-only live gate证明任务一次成功。
- MVP 明确只支持一个 active backend 实例，不新增 Redis client、共享 secret store或加密 job payload；多副本部署前由 backend-auth owner 原地设计共享一次性 secret 与跨实例消费合同。
- 未选择当前引入共享基础设施，因为用户明确要求 MVP 尽量少依赖，且单实例已满足当前生产使用规模。

## 5 验证

- provider-only Mailpit：challenge 202、收到 code-only 邮件、verify 200、session cookie 与 `/me` 200，job `succeeded` 且 attempts=1。
- 外部 SMTP：implicit TLS/auth、授权 sender 与 provider DATA 接收均成功；应用 job `succeeded` 且 attempts=1。
- 用户确认 `EMAIL_FROM_ADDRESS` 对应真实收件箱收到 “EasyInterview sign-in code”。
- `python3 -m pytest scripts/lint/scenario_env_contract_test.py -q`、Compose config、根 `make test`、`make build`、`make lint-config` 通过；证据不包含密码、完整邮箱或验证码。

## 6 经验教训

- 只要任务依赖进程内 secret，环境生命周期就必须保证只有该进程的 runner 能消费任务；HTTP/Compose healthy 不能代替真实任务链路。
- 未来启用多副本前必须先改变 secret 所有权模型，不能把单实例 live PASS 外推为多副本安全。

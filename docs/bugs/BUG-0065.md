# BUG-0065: debrief.generate prompt baseline used non-current output schema

> 模块: debrief | 严重度: high | 状态: resolved
> 发现日期: 2026-05-16 | 关联 Commit: `feat(backend-debrief): close 001 debrief record and analysis baseline`
> 关联日志: [2026-05-16](../work-journal/2026-05-16.md)

## 1 现象

- `backend/internal/debrief.GenerateHandler` 按当前 backend-debrief plan 解析 `debrief.generate` AI 输出，要求顶层 JSON 为 `questions` + `riskItems`。
- `config/prompts/debrief.generate/v0.1.0*.md` 仍要求模型返回旧 `timeline` / `lessons` / `follow_up_actions` 结构。
- 单元测试使用 mock AI 输出，因此 handler 测试可以通过；真实 provider 若遵循旧 prompt，会导致 worker 返回 `AI_OUTPUT_INVALID`，debrief 保持 `draft`。

## 2 诊断过程

### 假设 1：worker parser 与 plan schema 是否一致
- 验证方法：读取 `backend/internal/debrief/generate_handler.go` 的 `parseGenerateResponse` 与 backend-debrief spec / plan 的 worker 描述。
- 结果：确认 parser 与当前 plan 一致，要求 `questions[*].aiAnalysis` 与 `riskItems[*]`。

### 假设 2：F3 prompt baseline 是否仍是旧 schema
- 验证方法：读取 `config/prompts/debrief.generate/v0.1.0.md` 与 `v0.1.0.en.md`。
- 结果：确认 prompt 要求旧 `timeline` / `lessons` / `follow_up_actions` 输出，与 worker parser 不兼容。

### 假设 3：seed migration 是否也需要同步
- 验证方法：运行 `python3 scripts/lint/prompt_lint.py --prompts-dir config/prompts --migrations-dir migrations`。
- 结果：更新 prompt hash 后，lint 指出 `migrations/000002_seed_baseline_prompt_rubric_versions.up.sql` seed row template_hash drift，说明 seed migration 必须同步。

## 3 根因分析

- `debrief.generate` 是 F3 既有 baseline，后续 backend-debrief plan 将 worker 输出语义改为 `questions` / `riskItems`，但没有在 Phase 4 前把 prompt/rubric baseline 与 seed migration 反向校对到 worker parser。
- Mock AI 单测验证了 parser 对目标 JSON 的处理，但无法证明真实 prompt 会产出同样 schema。
- 涉及路径：
  - `backend/internal/debrief/generate_handler.go`
  - `config/prompts/debrief.generate/v0.1.0.md`
  - `config/prompts/debrief.generate/v0.1.0.en.md`
  - `config/rubrics/debrief.generate/v0.1.0.yaml`
  - `config/rubrics/debrief.generate/v0.1.0.en.yaml`
  - `migrations/000002_seed_baseline_prompt_rubric_versions.up.sql`

## 4 修复方案

- 将 `debrief.generate` multi/en prompt 改为要求严格输出顶层 `questions` 与 `riskItems`。
- 明确 `questions` 必须保留输入顺序与原始 `questionText` / `myAnswerSummary` / `interviewerReaction` 语义，并新增 `aiAnalysis`。
- 明确 `riskItems[].severity` 只允许 `low` / `medium` / `high`。
- 同步 prompt `template_hash` 与 seed migration prompt rows。
- 更新 rubric 描述，使其评价 question analysis 与 risk item 质量，而不是旧 lessons / follow-up actions。

## 5 验证

- `python3 scripts/lint/prompt_lint.py --prompts-dir config/prompts --migrations-dir migrations` 通过。
- `python3 scripts/lint/rubric_lint.py --rubrics-dir config/rubrics` 通过。
- `./migrations/lint.sh` 通过。
- 后续全局 gate：`cd backend && go test ./... -count=1`、`python3 -m pytest scripts/lint -q`、`make codegen-check`、`make validate-fixtures`、`make docs-check`、`git diff --check` 通过。

## 6 经验教训

- 任何 worker parser 依赖 F3 prompt structured output 时，不能只测 mock AI response；plan close-out 必须反查 active prompt body 与 seed migration 是否仍输出 parser 期望的 schema。
- 后续可把“prompt 输出 schema 与 parser 期望一致”沉淀为 F3 lint 或 plan gate。

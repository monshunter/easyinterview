# BUG-0179: owner checklist compression deleted activation markers and preflight accepted incidental mentions

> 模块: test | 严重度: medium | 状态: resolved
> 发现日期: 2026-07-16 | 关联 Commit: `feat(auth): use Redis for email delivery secrets (BUG-0179)`
> 关联日志: [2026-07-16](../work-journal/2026-07-16.md)

## 1 现象

- 根级 `make test` 在 `TestV020ActivationOwnerMarkersReady` 失败，报告 `prompt-rubric-registry/004-real-model-profile-and-evals/checklist.md` 缺少 v0.2 report rubric owner marker。
- 该 owner 的实现、rubric、judge 与 28-case offline eval 资产仍存在，但 2026-07-15 的 completed-plan 压缩删除了原 Phase 8 verified marker comments。
- 修复 RED 证据首次提到缺失 marker 名称时，旧 preflight 又把这条失败说明误当作通过证据，继续失败在第二个 marker，暴露假阳性。

## 2 诊断过程

### 假设 1：report v0.2 rubric 或 eval 实现已经失效

- 验证方法：读取 F3 004 owner、backend-review handoff、当前 rubric/eval truth source，并运行 prompt/rubric/profile lint、focused eval/evalkit/registry/judge tests 与 `make eval-offline`。
- 结果：排除。single-source drift-check 为 28 cases / 9 resolved prompts，offline grading no-network，Promptfoo 28/28 PASS。

### 假设 2：owner evidence 在文档压缩时漂移

- 验证方法：反查 checklist git history和 `TestV020ActivationOwnerMarkersReady` 的消费逻辑。
- 结果：确认。历史 checklist 有两个 current marker；压缩版删除了它们，但下游 preflight 保持不变。

### 假设 3：preflight 能区分通过证据与失败说明

- 验证方法：在 verified RED comment 中保留缺失 marker 名称后重跑 focused package test。
- 结果：否定。旧实现仅检查同一行包含 `<!-- verified:` 和 marker 字符串，因此失败说明被误判为通过。

## 3 根因分析

- 跨 owner 激活证据以 checklist verified marker comment 作为机器可读 handoff，但 completed-plan 压缩没有“仍被消费者引用的 marker 必须保留”约束。
- `backend/internal/ai/registry/backend_practice_preflight_test.go` 使用宽泛 substring 匹配，没有要求显式 marker 属性，无法区分 PASS marker、失败诊断和普通历史说明。
- 根 `make test` 保留了真实下游消费者，因此最终阻止了错误交付；此前缺少针对 verifier 假阳性的负向测试。

## 4 修复方案

- 原地重开 F3 004 owner，spec/plan/checklist 明确 marker 只能在当前 gate 实际重跑通过后写入，completed-plan 压缩不得删除仍被下游消费的证据。
- 新增 `hasVerifiedOwnerMarker`，只接受 verified comment 中精确的 `marker=<name>` 字段；补充失败说明、普通 evidence、unchecked text 三类负向测试。
- 将 storage owner 的既有 marker 规范化为显式属性；F3 004 在 current 28-case eval、lint、judge/profile tests 全绿后分别恢复两个 marker。
- 不采用“直接复制历史 marker”方案，因为它无法证明当前 rubric、resolved prompt、judge contract 与 eval suite 仍一致。

## 5 验证

- RED：focused activation preflight 稳定失败在第一个缺失的 004 marker。
- GREEN：显式 marker parser 四个用例通过；无 marker 时 activation preflight 仍失败，恢复 current marker 后转绿。
- `make eval-offline`：28 cases graded、no network；Promptfoo 28 passed、0 failed、0 errors。
- F3 prompt/rubric/profile/hardcode lint、eval/evalkit/registry/profile/provider/bootstrap/judge/observability focused tests通过。
- 根 `make test`：564 个 Python tests、4481 个 subtests、全部 Go packages、frontend 126 files / 1004 tests 全部通过。
- `make build`、`make lint-config`、owner context、docs/index 与 `git diff --check` 通过。

## 6 经验教训

- 机器消费的 checklist marker 是跨 owner 合同，不是可在文档压缩中删除的流水账。
- marker verifier 必须解析显式属性，并用“失败说明仅提及 marker”负向用例防止 substring 假阳性。
- 恢复 marker 前必须重跑其 owner 的当前 gate，并在证据中记录命令、case 数与结果；历史 PASS 只能用于定位，不能用于收口。

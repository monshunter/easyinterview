# BUG-0117: auth verify recovery and skipped probe state regressed after unified login

> 模块: frontend | 严重度: medium | 状态: resolved
> 发现日期: 2026-05-28 | 关联 Commit: `fix(frontend-shell): recover auth verify refresh failures (BUG-0117)`
> 关联日志: [2026-05-28](../work-journal/2026-05-28.md)

## 1 现象

- L2 code review 发现用户从 `auth_login` / `auth_verify` 打开应用时，`skipInitialAuthProbe` 会在 provider 生命周期内持续为 true。
- `refreshAuth(user)` 直接提交 verified user 后没有消费初始跳过状态，也没有推进 `authNonce`；后续 TopBar 切换语言会改变 `requestOptions`，auth effect 重新执行并把已登录用户重置成 `unauthenticated`。
- 邮箱验证码验证成功后，如果后续 `/me` 刷新失败或超时，错误会被 `AuthVerifyScreen` 当作验证码失败展示；一次性 code 已经被消费，用户重试同一 code 会失败，可能卡在 verify 页面。

## 2 诊断过程

### 假设 1：直接提交 verified user 后没有消费 public auth 初始 skip

- 验证方法：读取 `frontend/src/app/runtime/AppRuntimeProvider.tsx` 的 `skipInitialAuthProbe` effect 和 `refreshAuth` callback，并新增 focused test 先复现“直接提交用户后变更 requestOptions”的路径。
- 结果：确认。Red 阶段 `AppRuntimeProvider` 测试显示语言/requestOptions 变化后没有重新探测 `/me`，认证状态被重置。

### 假设 2：verify code 与 post-verify profile refresh 被同一个错误边界包住

- 验证方法：读取 `frontend/src/app/App.tsx` 的 `AuthVerifyScreen` adapter，新增 focused dispatch test：`verifyAuthEmailChallenge` 成功、`/api/v1/me` 返回 503。
- 结果：确认。Red 阶段页面仍停留在 `auth_verify`，并显示 code failure 状态，而不是退出 verify 并安排可恢复的 auth refresh。

## 3 根因分析

- 根本原因一：public auth route 的初始 probe skip 被建模成静态 prop 条件，而不是 provider 内部一次性生命周期状态；`refreshAuth(user)` 的 direct commit 路径没有把该 skip 标记为已消费。
- 根本原因二：验证码验证是一次性凭证消费动作，`/me` 是后续 profile/context 刷新动作；实现把两者放在同一个 promise 失败语义里，导致刷新失败被 UI 误报为验证码失败。
- 涉及的代码路径：
  - `frontend/src/app/runtime/AppRuntimeProvider.tsx`
  - `frontend/src/app/App.tsx`
  - `frontend/src/app/runtime/AppRuntimeProvider.test.tsx`
  - `frontend/src/app/AppAuthDispatch.test.tsx`
- 之前没有发现的原因：既有测试覆盖了 auth/profile 主路径，但没有覆盖 direct user commit 之后的 `requestOptions` 依赖变化，也没有覆盖 verify 成功后 `/me` 单独失败的恢复语义。

## 4 修复方案

- 选择的方案：在 `AppRuntimeProvider` 内增加一次性 skip 消费状态；初始 auth probe 只允许跳过一次，`refreshAuth(user)` direct commit 同步标记该 skip 已消费，`refreshAuth()` 无 user 时继续通过 `authNonce` 触发真实 `/me` 刷新。
- `AuthVerifyScreen` adapter 将 `verifyAuthEmailChallenge` 与 post-verify `getMe` 分离处理；code 验证成功后，如果 `/me` 失败，则调用 `runtime.refreshAuth()` 安排可恢复探测，并返回成功结果让用户离开 verify 页面。
- 同步修订 `frontend-shell` spec/plan/checklist/context/history，新增 Phase 11 对初始 skip 消费和 post-verify refresh recovery 的回归 gate。
- 考虑过但放弃的替代方案：让 `refreshAuth(user)` 直接推进 `authNonce`。该方案会把 direct commit 与异步 `/me` 探测耦合在一起，容易造成额外请求和加载态闪烁；一次性消费 skip 更贴近该缺陷的边界。

## 6 经验教训

- Auth flow 中要区分不可逆的一次性凭证消费与可重试的 profile/context refresh；测试应覆盖两段之间失败的恢复行为。
- public route 初始探测跳过必须是一次性状态，不能只依赖 provider prop；direct auth commit 后要覆盖 language/requestOptions 变化这类 effect dependency 回归。
- 后续 review auth 登录/verify/profile 变更时，应反查 `skipInitialAuthProbe`、`authNonce`、`refreshAuth(user)`、post-verify `/me` 四个边界是否仍保持分离。

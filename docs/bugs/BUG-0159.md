# BUG-0159: real conversation funnel retained PostgreSQL and observability drift

> 模块: practice | 严重度: high | 状态: resolved
> 发现日期: 2026-07-12 | 关联 Commit: `fix(practice): harden real conversation funnel (BUG-0159)`
> 关联日志: [2026-07-12](../work-journal/2026-07-12.md)

## 1 现象

- 在共享真实环境中从简历、JD 进入连续模拟面试时，空 `focusCompetencyCodes` 被写成 SQL `NULL`，触发 `practice_plans.focus_competency_codes` 的 `NOT NULL` 约束，阻断会话创建。
- 修复计划创建后，完成会话仍向已经删除的 `practice_session_events.client_event_id` 写值；报告首次失败后的重试又把 JSON 字符串写入 PostgreSQL `text[]`，并被已处于 `generating` 的状态转换拒绝，导致核心漏斗无法到达 ready report。
- 简历解析的真实 AI 调用未经过统一可观测性装饰器，provider 失败时缺少 `ai_task_runs` 与脱敏 raw debug 证据，增加了真实链路定位成本。
- 影响简历解析诊断、练习计划创建、会话完成与报告恢复，核心全漏斗没有可用绕过方式，因此严重度为 `high`。

## 2 诊断过程

### 假设 1：真实失败只来自 provider 输出波动

- 验证方法：通过 Mailpit 登录真实账号，依次创建简历、导入 JD、创建 practice plan，并检查 backend log、数据库状态与 `ai_task_runs`。
- 结果：排除。简历解析失败时确有 provider 响应问题，但运行时没有统一 AI 可观测性记录；后续 plan 创建失败来自确定性的 PostgreSQL `NOT NULL` 约束。

### 假设 2：Practice 的当前 migration 与 store SQL 已完全同步

- 验证方法：沿 `CreatePlan`、`CompleteSession` 的实际 SQL 和当前 baseline migration 对照，使用真实数据库执行，并补充 sqlmock 精确列/参数断言。
- 结果：排除。nil slice 被 `pq.Array` 投影为 `NULL`，完成 SQL 仍引用已删除的 `client_event_id`；原 focused tests 没有覆盖空数组和生命周期事件 exact column set。

### 假设 3：报告重试失败只是异步轮询时序

- 验证方法：读取 report job/error、`feedback_reports` 状态和 retry focus 写入参数，重放失败后重试路径。
- 结果：排除。retry focus 需要 PostgreSQL `text[]`，旧实现传入 JSON；首个持久化失败后报告已在 `generating`，重试转换不具备幂等性，导致恢复路径持续失败。

## 3 根因分析

- conversation simplification 删除逐题表和事件字段后，部分 store SQL 与 privacy/OpenAPI 辅助契约仍残留旧列/旧路由描述；结构测试覆盖了主 migration，却没有约束实际 SQL 的 exact column/array encoding。
- Go 的 nil slice 与空 slice 在 PostgreSQL array 写入中语义不同，`pq.Array(nil)` 生成 SQL `NULL`；业务契约要求的是空集合 `{}`。
- report persistence 使用通用 JSON helper 处理数据库原生 `text[]`，且状态更新只允许严格 `from → to`，没有覆盖 worker 在同状态恢复的幂等重试。
- resume runtime 的 parse AI bootstrap 漏掉了已有 observability decorator，单元测试只验证 resolver/provider 可构建，没有验证真实 runtime wrapper。

## 4 修复方案

- 在 `buildResumeRuntime` 中，当数据库与 resolver 可用时统一包装 resume parse AI，并增加 runtime wiring 测试。
- `CreatePlan` 在 `pq.Array` 前把 nil competency codes 规范化为空 slice；`CompleteSession` 删除 `client_event_id` 写入，并用 sqlmock 锁定当前 lifecycle-only 列集合。
- 报告持久化对 retry focus 使用 `pq.Array`；状态更新允许当前状态等于目标状态，从而支持失败恢复的幂等重试。
- Privacy matrix 用 `practice_messages` 替代 `practice_turns`，删除 `question_assessments`；OpenAPI 的幂等说明改为 `/messages` + `clientMessageId`，并增加负向契约断言。
- 放弃只在场景脚本中绕过空数组或手动重置 report 状态的方案，因为它会掩盖真实持久化契约错误。

## 6 经验教训

- 删除结构化题目模型时，必须把 migration、实际 store SQL、privacy matrix、OpenAPI 描述和失败恢复作为同一契约面审计，不能只验证 schema 和生成类型。
- 对 PostgreSQL array 字段必须显式区分 nil 与空集合，并用真实 SQL 参数断言或 live gate覆盖。
- 异步 worker 的状态转换必须允许同目标状态重入；否则一次持久化失败会把可恢复任务永久卡在中间态。
- 真实全漏斗验收应保留 provider observability 和数据库终态证据，这两者能快速区分 AI 输出问题、应用 SQL 漂移与浏览器轮询时序。

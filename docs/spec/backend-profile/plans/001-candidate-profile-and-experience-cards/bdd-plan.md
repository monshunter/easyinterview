## Phase 3 + 4: privacy delete lifecycle

| 场景 ID | 场景 | Given | When | Then | 验证入口 |
|---------|------|-------|------|------|----------|
| E2E.P0.093 | candidate profile + experience cards privacy delete + audit tombstone + 无敏感字段泄漏 | A2 dev stack 拉起；用户 A 已登录 + 有 candidate_profile 行（headline / currentRole 非空）+ 5 experience_card 行（含 title / situation / task / action / result / metrics / skills 真实内容）；privacy_delete job 模拟入口已就位；缺少 live env、no-op、skip 时本场景必须 fail | 调 internal `DeleteCandidateProfileForUser(userId=user_A)`；触发完整删除链路 | （1）删除顺序：先 `experience_cards.DeleteForUser(user_A)`（DELETE 5 行）→ 再 `candidate_profiles.DeleteForUser(user_A)`（DELETE 1 行）；（2）audit_events 表新增 tombstone 行：`user_id=user_A` / `event_name='profile.privacy_delete'` / `metadata` 含 `experienceCardCount=5` / `deletedAt` 时间戳；**不含** title / companyName / situation / task / action / result / metrics / skills / headline / currentRole / region 任何字段值；（3）DB 状态：`candidate_profiles` 0 行（user_A）/ `experience_cards` 0 行（user_A）；其他用户不受影响；（4）后续 `getMyProfile(user_A)` 若用户仍存在（auth 层未删 user）按 D-1 seed 新 profile；（5）`CountExperienceCardsBySource(user_A)` 返回 `{manual: 0, resume_parse: 0, practice_report: 0, debrief: 0}`；（6）log / audit / outbox payload 写入路径不序列化 raw card content 或 profile 字段；允许 SQL/store 层出现必要列名，禁止把列值写入日志或事件；（7）失败回滚：模拟 `candidate_profiles.DeleteForUser` 抛错，必须事务回滚（experience_cards 不被删除）并写 audit failure 记录；（G 旧口径）grep `mistake|growth|drill|experiences|star` in privacy delete code + audit payload：0 命中 | `test/scenarios/e2e/p0-093-profile-privacy-delete-lifecycle/`；trigger 必须包含 `cd backend && go test ./internal/profile/service -run TestPrivacyDeleteOrderAndAudit -count=1 -v` + `cd backend && go test ./cmd/api -run TestProfileHTTPScenario -count=1 -v` |

# Product Scope Spec

> **版本**: 1.5
> **状态**: active
> **更新日期**: 2026-05-03

## 1 背景与目标

### 1.1 修订背景

`easyinterview-spec-v1-0.md` 是 2026-04-26 导入仓库的产品规格输入，承担过产品愿景、P0/P1 技术文档和工程拆分的初始依据。随着 `ui-design/` 静态原型和 `docs/ui-design/` 文档在 2026-05-01 到 2026-05-03 连续收敛，当前产品形态已经和旧文档出现实质漂移：

- 旧 spec 仍把热身、反问专练、单题深钻、独立错题本、多轮计划、成长看板等写入较早阶段或 MVP 主路径。
- 当前 UI 已收敛为 `首页 / 岗位推荐 / 模拟面试 / 简历 / 复盘` 五个一级入口。
- 当前 UI 明确移除了入口前练习模式卡片、独立错题队列、独立报告一级入口、独立成长中心、独立经历库和独立语音页面骨架。
- 当前报告只有 `Report Dashboard(sessionId)` 一种形态，并且必须区分 `复练当前轮` 与 `进入下一轮`。
- 当前复盘模块面向真实面试，文本添加和语音添加是同一份复盘记录的两种添加方式。
- 当前 UI 静态设计和 `docs/ui-design/` 未列为保留、重定义或明确规划例外的内容，默认视为已决策丢弃，不进入 P1/P2/P3 自动恢复列表。

因此，本 spec 取代根目录旧 spec，成为 EasyInterview 当前产品范围、阶段边界、非目标、质量红线和文档真理源关系的正式入口。旧 spec 保留为历史参考，不再作为当前开发和评审的直接依据。

### 1.2 一句话定义

**EasyInterview 是一款围绕具体目标岗位、JD、简历资产和真实面试流程设计的 AI 面试训练产品。**

它把“看到岗位 -> 导入 JD -> 确认目标面试规划 -> 完成模拟面试 -> 获得证据化报告 -> 复练当前轮或进入下一轮 -> 真实面试后复盘”的准备过程标准化、可重复化和可追踪化。

### 1.3 当前版本目标

本 spec 的目标不是重新扩张产品愿景，而是把当前已经收敛的产品事实固定下来：

1. **替代旧产品真理源**：将当前产品范围从根目录旧 spec 迁入 `docs/spec/product-scope/spec.md`，纳入 spec-centric 文档治理。
2. **对齐当前 UI 真理源**：把 `ui-design/` 和 `docs/ui-design/` 已确认的导航、路由、模块边界和删除范围纳入产品层判断。
3. **保留旧 spec 的产品水准**：继续覆盖目标用户、JTBD、产品原则、阶段路线、质量评估、隐私伦理和工程治理要求，而不是只写 UI 清单。
4. **约束后续实施**：后续 child spec、plan、OpenAPI、DB、前端实现和场景测试不得恢复已删除模块，也不得绕过当前产品边界创造平行流程。

### 1.4 真理源关系

| 层级 | 当前真理源 | 负责内容 |
|------|------------|----------|
| 产品范围 | 本文档 | 用户、JTBD、P0/P1/P2/P3 边界、非目标、质量和伦理红线 |
| UI / 交互 | `docs/ui-design/` + `ui-design/` | 当前静态 UI、一级导航、页面职责、目标路由、视觉和交互目标 |
| 工程拆分 | `docs/spec/engineering-roadmap/spec.md` | child subspec、wave、依赖 DAG、mock-first 集成策略 |
| 技术契约 | Layer B/F active spec + 已编码 truth source（`openapi/`、`shared/`、`migrations/`、`config/`） | API、DB、事件、共享枚举、AI 网关、可观测性等工程约束；`easyinterview-tech-docs/` 仅为历史技术输入 |
| 历史参考 | `easyinterview-spec-v1-0.md` | 旧版产品愿景和历史取舍，不再作为当前 active truth source |

当文档出现冲突时，产品范围以本文档为准，UI 具体交互以 `docs/ui-design/` 为准，工程落地路径以对应 child spec / plan 为准。若工程 roadmap 与本文档出现阶段漂移，必须先通过 `/plan-review` 或新的设计修订对齐，再进入 `/implement`。

## 2 范围

### 2.1 In Scope

- EasyInterview 当前产品定义、目标用户、JTBD 和产品原则。
- 当前 P0 MVP 闭环：`JD 导入 -> 目标面试规划 -> 模拟面试 -> Report Dashboard -> 复练当前轮 / 进入下一轮 -> 真实面试复盘`。
- 当前一级 UI 模块：`首页`、`岗位推荐`、`模拟面试`、`简历`、`复盘`。
- 用户菜单能力：`用户画像`、`设置与隐私`、`认证流程`。
- 模拟面试会话边界：文本面试、语音面试、语音转文字、带提示练习、严格模拟、面试官角色、问题推进、结束并生成报告。
- 报告边界：session-scoped Dashboard、准备度档位、维度状态、题目回顾、证据详情、复练计划。
- 简历边界：原始简历树、结构化主版本、岗位定制版本、创建/解析/确认、版本详情、改写建议和手动编辑。
- 复盘边界：真实面试上下文选择、文本/语音共享复盘记录、复盘分析、复盘面试。
- P1/P2/P3 的方向性阶段路线：只围绕当前 UI 已保留能力做工程化、质量化和规模化扩展，并单独标注明确规划例外。
- 评分、Prompt/Rubric/Model 版本追踪、隐私、伦理、数据来源和可观测性红线。
- 与 `docs/ui-design/`、`docs/spec/engineering-roadmap/`、旧根 spec 的职责边界。

### 2.2 Out of Scope

- 不在本 spec 中编写具体 API schema、DB schema、事件 payload、prompt 文本或 UI 组件代码。
- 不在本 spec 中承载 engineering-roadmap child subspec 的具体重排；roadmap 拆分与后续 child 创建规则以 `docs/spec/engineering-roadmap/spec.md` 为准。当前已通过 engineering-roadmap v3.0 删除 pending 占位模型，并将后续 workstream 限定为当前产品 / UI 已保留能力的 on-demand child。
- 不创建实现 plan、TDD checklist 或 BDD scenario；本文档是 docs-only 产品范围结晶。
- 不恢复旧版独立成长中心、多轮计划、经历库、追问树、单题 Drill、独立错题队列、报告时间线或刊物式报告页。
- 不定义 Team / EDU、企业端候选人评估、社区或真实面试中的隐形实时辅助。
- 不承诺没有校准证据的通过率、录用概率、击败候选人比例或强结论评分。

### 2.3 当前阶段边界

当前阶段仍以 P0 MVP 闭环为中心。P0 的判断标准不是“功能最多”，而是：

> 用户带着一份具体 JD 来到产品后，能否在最短时间内完成一轮有效练习，并明确知道下一次该复练当前轮、进入下一轮，还是回到真实面试复盘补材料。

凡不能增强这个闭环的能力，即使旧 spec 中曾经出现，也不得进入当前主流程。

### 2.4 默认丢弃规则

当前 UI 静态设计和 `docs/ui-design/` 是产品范围的正向清单：

- 已在 UI / UI 文档中作为目标模块、会话页面、横切控制或重定义能力出现的内容，才进入当前产品范围。
- 已在 UI / UI 文档中列为移除、折返、废弃或 dead code 的内容，视为已决策丢弃，不是“P0 暂不做”。
- 未在 UI / UI 文档中出现，也未在本 spec 中明确列为规划例外的内容，默认视为已丢弃的旧设想，不进入后续阶段自动恢复。
- 恢复任何已丢弃能力，必须先原地修订本 spec 和对应 UI 文档，再进入工程 plan；不得只靠旧 spec、旧 route、旧画板标签或历史组件恢复。

## 4 设计约束

### 4.1 产品原则

1. **极速进入价值**：首页必须允许用户直接输入 JD，不以长 onboarding 或登录作为开始前置。
2. **目标岗位优先**：练习、报告、简历建议、复盘都必须围绕 `TargetJob / JD / ResumeVersion / Round / Session` 建立上下文。
3. **完整面试优先**：产品默认组织一场完整模拟面试，而不是把用户分散到热身、单题、追问树和错题队列中。
4. **证据优先于伪精确**：报告前台展示准备度档位、维度状态、证据片段和下一步动作，不输出未经校准的精确通过率。
5. **复练优先于一次性报告**：报告必须提供 `复练当前轮` 和 `进入下一轮` 两条清晰路径，且二者不能混用。
6. **真实复盘独立成流**：复盘处理真实面试，不是模拟报告的错题回收箱。
7. **隐私默认保守**：简历、JD、面试回答、音频、画像、复盘内容都属于敏感数据，采集、留存、导出、删除必须可解释。
8. **工程与评估前置**：模型调用、prompt/rubric、feature flag、数据来源和语言必须可追踪；质量回归必须可复现。

### 4.2 UI 约束

- 顶部导航只能出现当前 UI 真理源确认的五个一级入口。
- `workspace` 的产品语义是 `模拟面试 / 当前面试规划`，不是旧的 `当前岗位` 一级模块。
- `report` 必须带 `sessionId` 或等价会话上下文；无上下文时不得展示假报告。
- `debrief` 的上下文选择必须在当前复盘页弹窗内完成，不跳转到模拟面试、报告或简历资产管理页。
- 语音面试只能通过 `practice?mode=voice&modality=voice` 或等价显式参数进入；不得保留或新增 `voice` 目标 route / route alias。
- 旧 route、旧画板标签、dead code 组件不得作为恢复旧模块的依据。

### 4.3 评分与反馈约束

报告可以展示：

- 准备度档位：例如 `未就绪 / 建议再练 / 基本可面 / 较为充分`。
- 维度状态：例如 `强项 / 达标 / 待加强`。
- 置信度或证据充分性说明。
- 具体题目、回答片段、证据缺口、推荐框架和下一步复练建议。

报告不得展示：

- 未经校准的精确录用概率。
- “击败多少候选人”式营销指标。
- 没有证据片段支撑的强烈通过 / 淘汰判断。
- 仅由语速、停顿、口头禅等表达信号推出的能力结论。

### 4.4 伦理与安全约束

- 不做真实面试中的隐形实时辅助或作弊能力。
- 不把音视频或情绪识别结果用于企业端候选人淘汰决策。
- 不把用户真实面试复盘内容默认公开或用于他人训练样本。
- 不在日志中输出原始简历、原始 JD、完整回答、完整复盘记录、音频转写全文或敏感个人信息。
- 任何 AI 生成的岗位分析、报告、简历建议和复盘建议都必须保留来源、版本、语言和模型调用元数据。

### 4.5 文档治理约束

- 本 spec 后续修订必须原地更新，不创建同主题 sibling spec。
- 任何改变当前 P0 用户行为流的修订，必须同步检查 `docs/ui-design/`、`docs/spec/engineering-roadmap/` 和相关 child spec。
- 涉及代码实现的计划必须通过 `/implement` -> `/tdd`，涉及用户行为的计划必须维护 BDD gate。
- 旧根 spec 只能作为历史引用；如果需要恢复旧能力，必须在本 spec 中重新设计和批准。
- 旧 spec、旧 route、旧画板标签或历史组件只能作为考古材料，不能单独作为恢复产品能力的依据。

## 5 模块边界

### 5.1 产品能力层

EasyInterview 的完整能力仍围绕五个产品层组织，但 UI 一级入口和底层能力层不是一一对应关系。

| 能力层 | 当前职责 | 当前 UI 承载 |
|--------|----------|--------------|
| M1 · Progressive Profile | 通过简历、JD、模拟面试、复盘和用户修正逐步沉淀候选人画像 | 用户画像、简历、岗位推荐、模拟面试和复盘的共享信号 |
| M2 · Target Job Workspace | 围绕 JD 管理岗位要点、简历绑定、轮次假设、公司轻情报和会话历史 | 首页 JD 导入、岗位推荐、模拟面试规划 |
| M3 · Mock Interview Orchestrator | 组织完整模拟面试、问题推进、追问、提示和 session 状态 | Interview Session |
| M4 · Evidence-based Review | 产出证据化报告、题目回顾、复练计划和真实面试复盘 | Report Dashboard、Debrief |
| M5 · Growth Signals | 准备度变化、报告后动作、画像证据和练习趋势等横切信号 | 不作为独立成长中心；仅能嵌入报告、画像、模拟面试规划等现有模块 |

### 5.2 当前一级 UI 模块

| 一级模块 | 用户任务 | P0 职责 | 不承担的职责 |
|----------|----------|---------|--------------|
| 首页 | 快速开始一次岗位准备 | JD 粘贴 / 上传 / URL 导入，最近模拟面试，创建简历入口，岗位推荐和复盘辅助入口 | 不做登录前营销页 |
| 岗位推荐 | 基于画像和简历发现值得准备的 JD | 推荐列表、匹配原因、不匹配原因、确认面试后进入 JD 解析 | 不做全球搜岗聚合平台 |
| 模拟面试 | 确认目标岗位、简历和轮次并发起 session | 当前面试规划、公司轻情报、会话历史、立即面试 | 不作为泛岗位资产管理中心 |
| 简历 | 管理可被岗位和面试消费的简历资产 | 原始简历树、结构化主版本、岗位定制版本、解析确认、版本详情 | 不做复杂排版设计器 |
| 复盘 | 复盘真实面试并生成复盘面试 | 上下文选择、文本/语音共享复盘记录、分析、复盘面试 | 不接收模拟报告错题作为复盘输入 |

### 5.3 会话级和上下文页面

| 页面 | 归属 | 进入方式 | 关键上下文 |
|------|------|----------|------------|
| `parse` | JD 解析确认 | 首页 JD 导入或岗位推荐确认 | `jdId / targetJobId` |
| `practice` | Interview Session | 模拟面试规划、报告复练、进入下一轮、复盘面试 | `sessionId / targetJobId / resumeVersionId / roundId / mode / modality / practiceMode` |
| `generating` | 报告生成过渡态 | 面试结束 | `sessionId / practiceMode / hintUsed / hintCount` |
| `report` | Report Dashboard | 面试结束、会话历史、相关入口 | `sessionId` |
| `company_intel` | 公司轻情报详情 | 模拟面试规划页卡片 | `targetJobId / jdId` |
| `profile` | 用户画像 | 用户菜单 | `userId` |
| `settings` | 设置与隐私 | 用户菜单 | `userId` |

### 5.4 移除或降级的能力

| 旧能力 | 当前处理 | 原因 |
|--------|----------|------|
| 未登录欢迎页 | 移除，默认进首页 | 减少首次价值路径阻力 |
| 当前岗位一级导航 | 合并进模拟面试规划 | 用户任务是准备面试，不是管理岗位资产 |
| 面试报告一级导航 | 移除，报告隶属于 session | 避免无上下文报告 |
| 练习模式卡片 | 移除 | 面试是完整过程，辅助程度在 session 内切换 |
| 热身 / 单题深钻 / 反问专练 | 已决策移除为独立流程 | 不切碎主流程；反问内容可作为完整面试中的问题或公司情报建议出现 |
| 追问树 | 移除 | 追问发生在面试会话内 |
| 独立错题队列 | 移除 | 错题价值放入报告题目回顾和本轮复练 |
| 成长中心 | 已决策移除为独立模块 | 只保留嵌入报告、画像或面试规划的趋势信号 |
| 多轮计划 | 移除独立系统 | 轮次节点在模拟面试规划和报告 CTA 中表达 |
| 经历库 / STAR 编辑器 | 移除独立模块 | 经历证据由简历和用户画像承载 |
| 报告时间线 / 刊物式报告 | 移除 | 报告统一为 Dashboard |

## 6 P0 MVP 详细规格

### 6.1 MVP 目标

P0 只回答一个问题：

> 当用户带着一份具体 JD 来到 EasyInterview，产品能否在最短路径内帮他完成一次针对性练习，拿到可追溯证据的报告，并知道下一步应该复练当前轮、进入下一轮，还是回到真实面试复盘补齐材料。

P0 成功标准：

- 用户不登录也能看到 JD 导入和准备路径。
- 用户可以围绕一个目标岗位、绑定简历和面试轮次进入完整模拟面试。
- 面试结束后生成一份有会话上下文、有证据、有下一步动作的报告。
- 报告可以直接发起当前轮复练或下一轮 session。
- 用户真实面试后可以在复盘模块记录真实问题，并生成复盘分析和复盘面试。

### 6.2 主流程 A：带着 JD 来的用户

```text
Home
  -> 粘贴 / 上传 / URL 导入 JD
  -> Parse & Confirm
  -> Mock Interview Plan
     ├─ 确认 TargetJob / JD
     ├─ 绑定 ResumeVersion
     ├─ 确认 InterviewRound
     └─ 立即面试
  -> Interview Session
  -> 结束并生成报告
  -> Report Dashboard(sessionId)
     ├─ 复练当前轮
     └─ 进入下一轮
```

### 6.3 主流程 B：从岗位推荐来的用户

```text
Job Picks
  -> 查看推荐 JD / 匹配原因 / 不匹配原因
  -> 确认面试
  -> Parse & Confirm
  -> Mock Interview Plan
  -> Interview Session
  -> Report Dashboard
```

岗位推荐的目标是减少用户找准备对象的成本，不是替代招聘平台。推荐结果必须能解释为什么匹配或不匹配，并在确认后回到同一套 JD 解析和模拟面试规划流程。

### 6.4 主流程 C：先补简历的用户

```text
Home 或 Resume
  -> 1 分钟创建简历 / 新建原始简历
  -> 上传 / 粘贴 / 轻量问答
  -> Agent 解析
  -> 预览确认保存 v1
  -> 从已有树分叉岗位定制版本
  -> 在模拟面试规划中绑定 ResumeVersion
```

简历模块的价值不是排版，而是形成可被 JD 解析、模拟面试、报告和复盘调用的证据资产。

### 6.5 主流程 D：刚面完一轮的用户

```text
Debrief
  -> 选择目标岗位 / JD
  -> 选择关联模拟面试
  -> 绑定简历
  -> 复盘记录
     ├─ 文本添加
     └─ 语音添加
  -> 复盘分析
  -> 复盘面试
```

复盘不假设 AI 推荐的问题就是真实问题。用户必须能跳过、改写、手动添加，并且语音提取的问题必须确认后才写入正式记录。

### 6.6 M1-lite：渐进式画像

#### 目标

用尽量少的前置成本，获得足够支撑岗位推荐、面试定制、简历建议和复盘分析的上下文。

#### 输入来源

- 原始简历、结构化主版本和岗位定制版本。
- 用户导入的 JD、岗位偏好、地区、语言和目标职级。
- 模拟面试中的回答、提示使用、追问风险和报告结果。
- 真实面试复盘中的真实问题、回答、面试官反应和用户修正。
- 用户在画像页的直接纠偏。

#### 输出

- 候选人画像摘要。
- 可用于面试和简历建议的经历证据。
- 岗位偏好、语言偏好和界面偏好边界。
- 待补充信息提示。
- 画像来源统计和用户可修正项。

#### 已移除 / 长期不恢复

- 冗长职业咨询 onboarding。
- 性格测试式大段画像。
- 独立经历库或 STAR 编辑器。
- 无来源的能力结论。

#### 验收标准

- 没有完整简历也能开始 JD 导入和模拟面试规划。
- 用户画像每个关键判断能说明来源。
- 用户可以在画像页修正系统理解。

### 6.7 M2：目标岗位工作台

#### 目标

把一份 JD 变成可练、可复盘、可关联简历和会话历史的面试规划。

#### 用户入口

- 首页粘贴 JD。
- 首页上传 JD 文件。
- 首页粘贴岗位 URL。
- 岗位推荐确认面试。
- 最近模拟面试回到当前规划。

#### 系统处理

- 解析岗位标题、公司、地区、语言、职级和来源。
- 提炼必需项、加分项、隐性关注点和风险提示。
- 关联简历版本和当前面试轮次。
- 展示公司轻情报卡片和面试轮次节点。
- 展示当前规划下的模拟面试历史。

#### 当前不进入本模块

- 全球多平台搜岗：作为规划例外保留，但不属于当前 MVP 的目标岗位工作台。
- 复杂职位收藏生态。
- 大规模外部公司情报拼图。
- 把 `当前岗位` 做成独立一级模块。

#### 验收标准

- 任意一份可读 JD 能进入解析确认流程。
- 模拟面试规划必须明确 `TargetJob / JD / ResumeVersion / InterviewRound`。
- 当前规划历史不得混入其他公司、岗位或 JD 的会话。

### 6.8 M3：模拟面试编排器

#### 目标

围绕目标岗位组织一场完整模拟面试，而不是随机抛题或让用户先选择复杂练习模式。

#### 当前面试形式

- 文本面试：用户主要通过文本回答。
- 语音面试：用户与 AI 进行实时语音对话。
- 语音转文字：文本输入框中的麦克风只负责插入转写，不改变面试形式。

#### 当前辅助程度

- `带提示练习`：展示提示、实时观察、可调用经历和现场提示。
- `严格模拟`：隐藏提示、实时观察、可调用经历和语音现场提示。

#### 关键逻辑

- 面试会话必须有 `sessionId` 和 InterviewContext。
- 面试官角色可以在同一 session 内表达 HR、综合面试官、用人经理等语气差异。
- 追问必须和用户刚才回答、目标 JD 或简历证据直接相关。
- `结束并生成报告` 固定在右侧底部，并传递 `practiceMode / hintUsed / hintCount`。

#### 已移除 / 长期不恢复

- 入口前热身、反问专练、单题深钻、追问树或 Drill builder。
- 把严格模拟做成新的面试类型。
- 把语音做成独立页面或独立产品模块。
- 真实面试中的隐形实时辅助。

#### 验收标准

- 用户从模拟面试规划点击 `立即面试` 后直接进入完整 session。
- 用户能明确当前是文本面试还是语音面试。
- 严格模拟时提示和辅助信息必须隐藏。
- 报告生成必须带上练习方式和提示使用记录。

### 6.9 M4：证据化报告

#### 目标

给出足够具体、可操作、可复练的反馈，让用户知道下一次该怎么练。

#### 报告结构

- Header：目标岗位、轮次、会话、绑定简历和报告归属说明。
- Context Strip：`sessionId`、目标岗位、轮次、简历、沟通形式、练习方式、提示记录。
- Summary Cards：准备度、维度详情、题目回顾、下一动作。
- Detail Surface：准备度详情、维度详情、题目回顾页、证据详情、复练计划。
- Next Actions：`复练当前轮` 和 `进入下一轮`。

#### 题目回顾

题目回顾承载旧 spec 中“错题本”的核心用户价值，但不形成独立模块。每题至少需要说明：

- 原始问题。
- 用户回答摘要或证据片段。
- 有效点。
- 缺口。
- 推荐框架。
- 可能追问。
- 是否加入本轮复练。

#### 已移除 / 长期不恢复

- 报告一级导航。
- 无 session 上下文的报告页。
- 时间线报告。
- 刊物式报告页。
- 独立错题队列或单题 retry。
- 精确通过率或录用概率。

#### 验收标准

- 报告的每个主要判断能回溯到题目、回答、JD、简历或复盘来源。
- 用户看完报告后能直接选择复练当前轮或进入下一轮。
- 无 `sessionId` 时必须显示缺失状态并回到当前面试规划或历史列表。

### 6.10 M5：简历资产与岗位定制

#### 目标

让简历成为可被岗位、面试、报告和复盘复用的证据资产，而不是孤立的文档上传入口。

#### 当前范围

- 原始简历树。
- 结构化主版本。
- 岗位定制版本。
- 上传 / 粘贴 / 轻量问答。
- Agent 解析。
- 预览确认保存 v1。
- 从已有树分叉岗位定制版本。
- 版本详情：预览、改写建议、手动编辑。
- 原件预览、复制、复制为新版本、导出等操作反馈。

#### 已移除 / 不规划

- 复杂排版编辑器。
- 多模板视觉设计系统。
- 深度求职信工作台。
- 独立经历库。

#### 验收标准

- 用户可以从无简历状态创建一份可确认的结构化简历版本。
- 用户可以为目标岗位分叉岗位定制版本。
- 模拟面试规划可以绑定具体简历版本。

### 6.11 M4 扩展：真实面试复盘

#### 目标

帮助用户把真实发生过的面试问题、回答、追问、面试官反应和遗漏点沉淀成下一轮练习材料。

#### 当前范围

- 选择目标岗位 / JD。
- 选择关联模拟面试。
- 绑定简历。
- 文本添加真实问题。
- 语音连续对话复盘。
- 文本和语音共享同一份记录、同一个顶部汇总条和同一组问题卡片。
- 语音实时提取待确认卡片。
- 确认 / 编辑 / 删除 / 写入记录。
- 复盘分析。
- 生成复盘面试。

#### 已移除 / 长期不恢复

- 把 AI 推荐问题默认写成真实问题。
- 把文本添加和语音添加当成两份记录。
- 要求用户手动点录音才能继续语音复盘。
- 把复盘终点定义为下一轮面试。
- 自动接收模拟报告错题作为复盘输入。

#### 验收标准

- 三个上下文选择都在复盘页弹窗内完成。
- 语音提取的问题必须经过确认才写入正式记录。
- 从文本切到语音或从语音切回文本不得丢失已记录内容。

### 6.12 规划例外与已丢弃能力

本节不是“P0 暂不做”清单，而是区分未来规划例外和已决策丢弃的旧能力。

| 能力 | 当前判断 | 原因 |
|------|----------|------|
| 全球多平台搜岗 | 规划例外；不进入当前 MVP 闭环 | 未来可作为岗位推荐的数据源扩展单独设计，但需要先解决数据接入、合规、质量和维护成本 |
| 视频情绪识别 | 已丢弃 / 长期不做 | 解释风险高，训练价值不稳定，且容易引入不当评估 |
| 社区 | 已丢弃 / 长期不做 | 稀释单人岗位准备闭环，不在当前产品定位内 |
| Team / EDU | 已丢弃 / 长期不做 | 当前产品只面向个人训练闭环，不规划团队版或组织评估产品 |
| 企业端候选人评估 | 已丢弃 / 长期不做 | 与训练产品定位冲突，伦理负担高 |
| 独立成长中心 | 已丢弃 / 长期不恢复 | 当前只保留嵌入报告、画像和面试规划的准备度 / 趋势信号 |
| 独立多轮计划 | 已丢弃 / 长期不恢复 | 面试轮次只在模拟面试规划和报告 CTA 中表达 |
| 独立经历库 / STAR 编辑器 | 已丢弃 / 长期不恢复 | 经历证据由简历和用户画像承载 |
| 独立错题本 / 单题 Drill / 追问树 | 已丢弃 / 长期不恢复 | 题目问题留在报告题目回顾和本轮复练，不形成单题流程 |
| 报告时间线 / 刊物式报告 | 已丢弃 / 长期不恢复 | 报告统一为 session-scoped Dashboard |
| 隐形实时面试辅助 | 已丢弃 / 长期不做 | 明确违反产品伦理边界 |

## 7 阶段路线

### 7.1 P0：当前 MVP

P0 聚焦一条完整闭环：

```text
JD / 推荐岗位
  -> 当前面试规划
  -> 完整模拟面试
  -> Dashboard 报告
  -> 复练当前轮 / 进入下一轮
  -> 真实面试后复盘
```

P0 退出标准：

- 用户能在无登录前置下进入 JD 导入。
- 至少一条从 JD 到报告再到复练的路径可闭合。
- 报告判断具备证据来源。
- 简历资产能被绑定到模拟面试规划。
- 真实面试复盘能生成可进入练习的复盘面试。

### 7.2 P1：加深单人价值

P1 不以恢复旧模块为目标，而是在 P0 闭环内加深当前已保留能力：

- 画像从 Lite 走向更稳定的 Progressive Profile。
- 简历工坊增强：更稳定的岗位定制建议、与报告缺口联动、版本操作反馈更完整。
- 真实复盘增强：模拟表现和真实表现对比、真实问题覆盖差异和复盘面试质量提升。
- 多语言能力增强：目标岗位语言和界面语言可以分离。
- 准备度和趋势信号只嵌入报告、画像和岗位规划，不恢复独立成长中心。

P1 退出标准：

- 用户愿意围绕同一岗位进行 2 次以上复练。
- 真实面试复盘能产生明显回访。
- 简历改写建议能反向补齐报告中暴露的证据缺口。
- 多语言输出通过人工抽检和用户反馈。

### 7.3 P2：工程化和数据源扩展

P2 只扩展当前 UI 已保留能力，外加 `全球多平台搜岗` 这一明确规划例外：

- 生产级语音能力：STT、媒体留存、表达层反馈、隐私开关和删除链路。
- 轻量岗位情报：官方信息、公开新闻、职位上下文、来源和抓取时间。
- 岗位推荐数据源扩展：在合规边界内评估全球多平台搜岗、来源标注、freshness 和维护成本。

P2 退出标准：

- 语音练习成本、延迟、隐私留存和删除链路可控。
- 公开情报模块不会成为维护负担或合规风险。
- 外部岗位数据源可证明合规、可维护，并且不会破坏当前 JD 导入和模拟面试闭环。

### 7.4 P3：可规模化的训练产品

P3 不是恢复旧的大而全求职平台，而是把已保留的训练闭环做得更稳定、更可评估、更可扩展：

- 更成熟的岗位推荐和 JD 数据来源治理。
- 更稳定的 AI 质量评估、离线集、prompt/rubric 回归和灰度策略。
- 更完整的隐私、导出、删除、审计和用户数据控制。
- 更可靠的跨语言训练、语音训练和公司轻情报。

P3 仍然不做：

- 社区、Team / EDU 或企业端训练营产品。
- 企业端候选人淘汰评估。
- 视频情绪识别。
- 独立成长中心、独立多轮计划、独立经历库、独立错题本或单题 Drill。
- 真实面试作弊辅助。
- 无来源公司情报。
- 没有校准的录用概率承诺。

## 8 验收标准

| ID | 场景 | Given | When | Then | 对应 Plan |
|----|------|-------|------|------|-----------|
| C-1 | 产品真理源迁移 | 旧根 spec 与当前 UI 发生漂移 | 读本文档和 `easyinterview-spec-v1-0.md` | 本文档明确 active，旧根 spec 明确 superseded | docs-only |
| C-2 | 首页直达价值 | 用户未登录且有 JD | 打开 App 首页 | 可以直接粘贴 / 上传 / URL 导入 JD | 后续 frontend child |

## 9 质量、安全与评估

### 9.1 质量评估

P0 质量评估必须覆盖三类指标：

| 类型 | 指标 |
|------|------|
| 体验类 | JD 导入完成率、进入模拟面试率、报告查看率、报告后复练率、复盘创建率 |
| 内容类 | JD 解析准确性、问题与岗位相关性、追问相关性、报告证据可追溯率、简历建议可用性 |
| 工程类 | 报告生成延迟、AI 调用失败率、异步任务失败率、OpenAPI/fixtures drift、prompt/rubric 回归 |

### 9.2 Prompt / Rubric / Model 治理

所有 AI 输出必须至少能追踪：

- `feature_key`
- `prompt_id`
- `prompt_version`
- `rubric_id`
- `rubric_version`
- `model_profile`
- `model_name`
- `language`
- `feature_flag`
- `data_source`
- `input_snapshot_ref`
- `generated_at`

业务域不得 hardcode prompt 文本。Prompt、Rubric 和模型 profile 的管理归对应 quality / AI child spec。

### 9.3 数据与隐私

敏感对象包括：

- 原始简历和结构化简历。
- 原始 JD、岗位链接和来源记录。
- 面试问题、回答、追问、报告和题目回顾。
- 真实面试复盘记录、语音转写和面试官反应。
- 用户画像、偏好、修正记录。
- 音频、媒体对象和 STT 中间结果。

隐私要求：

- 用户必须能理解哪些数据被用于生成建议。
- 数据删除链路必须可审计。
- 音频和媒体留存必须可关闭、可解释、可删除。
- 日志和指标不得包含原始敏感内容。

### 9.4 安全边界

- API 和前端必须按用户身份隔离 TargetJob、ResumeVersion、InterviewSession、Report 和 DebriefRecord。
- 任何导出、删除、上传、解析、报告生成和复盘生成都必须记录审计或任务来源。
- 任何外部数据源必须记录 source、retrieved_at、freshness 和使用范围。
- 任何 feature flag 改变用户可见 AI 能力时，必须能在报告或审计信息中追踪。
